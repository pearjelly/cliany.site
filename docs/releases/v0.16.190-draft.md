# v0.16.190 发布草案

**目标版本：** `0.16.190`
**提交范围：** `v0.16.189..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 blocker tail 显式化到整包摘要。

## 用户价值

`v0.16.189` 已经让 `artifact_bundle_summary` 输出 blocker preview，维护者和自动化可以只读 summary 看到 blockers 列表入口。随着阻塞项增多，工具还需要确认列表末尾是否符合预期，避免只看前缀时漏掉后追加的发布、草案或候选任务阻塞。

`v0.16.190` 在 `artifact_bundle_summary` 中新增 `blocker_tail_count`、`blocker_tail` 和 `blocker_tail_sha256`。这样维护者和自动化可以只读整包摘要，同时看到当前 blockers 的稳定前缀、末尾和完整 hash，再决定是否展开完整 `artifact-manifest.json.blockers` 或 release readiness 输出。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.190 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 blocker tail fields，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且 `blocker_tail` 等于 `artifact-manifest.json.blockers` 的末尾片段。

## 变更分组

### Blocker tail

- `artifact_bundle_summary` 新增 `blocker_tail_count`。
- `artifact_bundle_summary` 新增 `blocker_tail`。
- `artifact_bundle_summary` 新增 `blocker_tail_sha256`。
- Tail 从 `artifact-manifest.json.blockers` 的顺序派生。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 blocker tail count/list/hash。
- 现有 `blocker_count`、`blockers_sha256`、blocker preview、publication visibility 摘要、artifact files 摘要、issue metadata 摘要、review order 摘要、issue body summary 与 inventory 摘要、artifact manifest schema/key/payload 摘要、handoff 摘要和 gate 摘要保持兼容。
- 不改变 release draft validation、publication audit 或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 blocker tail fields。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的 blocker tail 字段，并断言 tail 与 `artifact-manifest.json.blockers` 同源。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 `blockers` 列表，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `blocker_preview_count`、`blocker_preview`、`blocker_preview_sha256`、`blocker_tail_count`、`blocker_tail`、`blocker_tail_sha256`、`blocker_count` 和 `blockers_sha256` 快速确认 blockers 是否漂移。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.190 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `blocker_tail_count`、`blocker_tail` 和 `blocker_tail_sha256`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示这三个字段。
4. 确认 `blocker_tail` 等于 `artifact-manifest.json.blockers` 的末尾片段。
5. 确认 `blocker_preview`、`blocker_count`、`blockers_sha256` 和完整 `blockers` 列表仍保持同源。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.190`。
8. 将 `pyproject.toml` 版本更新为 `0.16.190`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.190`。
10. 创建 tag：`git tag v0.16.190`。
