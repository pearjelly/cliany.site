# v0.16.63 发布草案

**目标版本：** `0.16.63`
**提交范围：** `v0.16.62..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts 提供 `artifact_bundle_summary`，方便工具快速判断整包状态。

## 用户价值

`v0.16.62` 已经给 candidate issue artifacts 增加 `issue_body_summary`，让工具能快速判断 body 集合是否变化。但维护者或自动化如果只想先判断这个 artifacts 包是否可展开，还需要组合 target version、candidate 数、review order、gate 状态和创建脚本 safety 字段。

`v0.16.63` 在 `artifact-manifest.json` 中新增 `artifact_bundle_summary`，汇总 `target_version`、`candidate_count`、`body_count`、`review_item_count`、`inventory_sha256`、`candidate_issue_gate_status`、`can_create_issues`、`dry_run_supported` 和 `preflight_required`。Artifacts `README.md` 也新增 `Artifact Bundle Summary` 小节，给维护者一个同源的人类可读入口。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.63 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json` 包含 `artifact_bundle_summary`；README 包含 `Artifact Bundle Summary`、`review_item_count` 和 `candidate_issue_gate_status`；summary 字段能从现有 manifest 字段重新计算得到。

## 变更分组

### Candidate issue artifact bundle summary

- `artifact-manifest.json` 新增 `artifact_bundle_summary`。
- Summary 汇总 candidate 数、body 数、review item 数、inventory hash、gate 状态和脚本安全布尔值。
- Artifacts `README.md` 新增 `Artifact Bundle Summary` 小节。
- Summary 不包含本机绝对路径，便于复现和自动化比对。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 与 README 汇总字段。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让候选 issue artifacts 的整包状态更容易被工具和维护者审阅。

## 风险与兼容性

- `artifact_bundle_summary` 是新增 manifest 字段，不改变现有字段语义。
- Summary 基于已存在的 manifest 数据计算，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.63..HEAD`。最终带 `--release-tag v0.16.63` 的 strict gate 仍以本草案顶部的 `v0.16.62..HEAD` 作为实际 release diff 范围。

```bash
python scripts/plan_next_iteration.py --target-version 0.16.63 --json
python scripts/plan_next_iteration.py --target-version 0.16.63 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.63 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.63 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json` 包含 `artifact_bundle_summary`。
3. 确认 `artifact_bundle_summary.review_item_count` 与 `review_order` 长度一致。
4. 确认 artifacts `README.md` 包含 `Artifact Bundle Summary`。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.63`。
7. 将 `pyproject.toml` 版本更新为 `0.16.63`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.63`。
9. 创建 tag：`git tag v0.16.63`。

`v0.16.63` adds artifact bundle summary metadata to candidate issue artifacts, giving maintainers and tools a compact overview before expanding the full manifest.
