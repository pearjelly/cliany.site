# v0.16.58 发布草案

**目标版本：** `0.16.58`
**提交范围：** `v0.16.57..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts README 复用完整 validation commands。

## 用户价值

`artifact-manifest.json.validation_commands` 已经包含候选 issue artifacts 的完整复核路径：重新生成 artifacts、输出计划 JSON、运行 release readiness、复核 publication audit，以及校验案例库。但是维护者如果只打开 artifacts `README.md`，当前 `Validation Commands` 只展示了其中一部分，容易漏掉 `python scripts/release_readiness.py --target-version <version> --json` 或 `python scripts/check_release_publication.py --json`。

`v0.16.58` 让 artifacts `README.md` 的 `Validation Commands` 和 `artifact-manifest.json.validation_commands` 同源生成。这样 README 面向人，manifest 面向工具，但两者给出的验证命令一致，减少候选 issue 创建前的人机交接偏差。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.58 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `README.md` 同时包含 `plan_next_iteration.py --json`、`release_readiness.py --target-version 0.16.58 --json`、`check_release_publication.py --json` 和 `validate_cases.py --strict`；`artifact-manifest.json.validation_commands` 继续保留同一组命令。

## 变更分组

### Candidate issue artifacts validation commands

- `scripts/plan_next_iteration.py --issues-dir` 的 artifacts `README.md` 现在复用同一组 validation commands。
- `artifact-manifest.json.validation_commands` 和 README `Validation Commands` 都包含 release readiness 与 publication audit 复核命令。
- 新增 helper 统一生成 candidate issue artifacts validation commands，避免 README 和 manifest 分叉。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 README 与 manifest 的 validation commands 同源。
- `tests/test_plan_next_iteration.py` 覆盖 artifacts README 的完整验证命令。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让候选 issue artifacts 的人工 README 与机器 manifest 在验证命令上保持一致。

## 风险与兼容性

- `artifact-manifest.json.validation_commands` 的字段语义不变。
- README 新增命令只用于人工复核，不自动执行。
- 不自动创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.58..HEAD`。最终带 `--release-tag v0.16.58` 的 strict gate 仍以本草案顶部的 `v0.16.57..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.58 --json
python scripts/plan_next_iteration.py --target-version 0.16.58 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.58 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.58 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 artifacts `README.md` 的 `Validation Commands` 包含 `release_readiness.py --target-version 0.16.58 --json`。
3. 确认 artifacts `README.md` 的 `Validation Commands` 包含 `check_release_publication.py --json`。
4. 确认 `artifact-manifest.json.validation_commands` 和 README `Validation Commands` 同源。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.58`。
7. 将 `pyproject.toml` 版本更新为 `0.16.58`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.58`。
9. 创建 tag：`git tag v0.16.58`。

## Release Notes 摘要

`v0.16.58` makes candidate issue artifacts README validation commands mirror `artifact-manifest.json.validation_commands`, so maintainers see the same release readiness and publication audit checks that automation reads.
