# v0.16.163 发布草案

**目标版本：** `0.16.163`
**提交范围：** `v0.16.162..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 artifact manifest 顶层字段尾部预览显式化到整包摘要。

## 用户价值

`v0.16.162` 已经让 `artifact_bundle_summary` 输出 `artifact-manifest.json` 顶层字段 preview count/list/hash，维护工具可以确认 manifest 入口字段顺序。但只看前缀仍可能漏掉 `review_checklist`、`validation_commands` 这类尾部字段漂移。

`v0.16.163` 在 `artifact_bundle_summary` 中新增 `artifact_manifest_key_tail_count`、`artifact_manifest_key_tail` 和 `artifact_manifest_key_tail_sha256`，并在 artifacts `README.md` 的 `Artifact Bundle Summary` 段落展示同源字段。这样维护工具可以只读整包摘要，同时确认 manifest 顶层字段清单前缀和后缀，再决定是否展开完整 manifest。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.163 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 manifest key tail fields，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且 tail 等于 `artifact-manifest.json` 实际顶层字段清单最后几个 key。

## 变更分组

### Artifact manifest key tail

- `artifact_bundle_summary` 新增 `artifact_manifest_key_tail_count`。
- `artifact_bundle_summary` 新增 `artifact_manifest_key_tail`。
- `artifact_bundle_summary` 新增 `artifact_manifest_key_tail_sha256`。
- Tail 从 `ARTIFACT_MANIFEST_KEYS` 的最后几个字段派生。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 manifest key tail count/list/hash。
- 现有 artifact manifest schema/key/boundary/preview/payload 摘要、summary key 摘要、handoff 摘要和 gate 摘要保持兼容。
- 不改变 release draft validation、publication audit 或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 manifest key tail fields。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的 manifest tail 字段，并断言 tail 与 manifest 顶层字段清单后缀同源。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 artifact manifest 顶层字段清单，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `artifact_manifest_key_tail` 快速确认 manifest 顶层字段清单后缀。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.163 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `artifact_manifest_key_tail_count`、`artifact_manifest_key_tail` 和 `artifact_manifest_key_tail_sha256`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示这三个字段。
4. 确认 tail 等于 `artifact-manifest.json` 实际顶层字段清单后缀。
5. 确认 `artifact_manifest_key_preview` 和 `artifact_manifest_key_boundary_sha256` 仍与 manifest key 清单同源。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.163`。
8. 将 `pyproject.toml` 版本更新为 `0.16.163`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.163`。
10. 创建 tag：`git tag v0.16.163`。
