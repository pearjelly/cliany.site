# v0.16.35 发布草案

**目标版本：** `0.16.35`
**提交范围：** `v0.16.34..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 next iteration plan 和 candidate issue handoff 直接展示 publication audit 的 release ref 上下文。

## 用户价值

`v0.16.34` 已经把 publication audit 的 worktree 状态带进 `scripts/plan_next_iteration.py` 和 `publication-handoff.json`。但维护者在周初计划或候选 issue artifacts 里仍然看不到这批发布命令对应的 repo root、latest tag、local HEAD、tag commit 和 remote check 状态；要确认这些信息，仍需要另开 `scripts/check_release_publication.py --json`。

`v0.16.35` 新增 `publication_ref_context`，把 publication audit 的 ref 上下文带到计划 JSON、文本输出、Markdown report、`publication-handoff.json` 和 artifacts `README.md`。维护者在一个 handoff 页面里就能确认发布命令对应哪个 tag/commit，再决定是否继续执行 publish script 或派发候选 issue。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.35 --json` 输出 `publication_ref_context`，其中包含 `repo_root`、`latest_tag`、`local_head`、`tag_commit` 和 `remote_checked`；`--issues-dir` 生成的 `publication-handoff.json` 包含 `ref_context`，artifacts `README.md` 的 `Publication Handoff` 展示 latest tag 和 local HEAD。

## 变更分组

### Next iteration publication ref context

- `scripts/plan_next_iteration.py` 的 `IterationPlan` 新增 `publication_ref_context`。
- JSON、文本和 Markdown plan 展示 repo root、branch、latest tag、local HEAD、tag commit、ahead/behind 和 remote check 状态。
- `--issues-dir` 生成的 `publication-handoff.json` 新增 `ref_context`，artifacts `README.md` 展示 latest tag 和 local HEAD。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明计划器会展示 publication ref context。
- `tests/test_plan_next_iteration.py` 覆盖 JSON、Markdown、publication handoff 和 artifacts README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强 next iteration plan 和候选 issue artifacts 的发布 ref 上下文交接。

## 风险与兼容性

- `publication_ref_context` 是 additive plan JSON 字段，不移除现有 key。
- `publication-handoff.json` 新增 `ref_context`，已有 publication/worktree 字段保持兼容。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- ref context 只透传 `scripts/check_release_publication.py` 的本地 audit 结果。

```bash
python scripts/plan_next_iteration.py --target-version 0.16.35 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.35 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.35 --json` 输出 `publication_ref_context`。
2. 确认 Markdown plan 包含 `Publication Ref Context`。
3. 确认 `publication-handoff.json` 包含 `ref_context`。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.35`。
6. 将 `pyproject.toml` 版本更新为 `0.16.35`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.35`。
8. 创建 tag：`git tag v0.16.35`。

## Release Notes 摘要

`v0.16.35` carries release publication ref context into next-iteration plans and candidate issue handoff artifacts, so maintainers can see the exact tag and commit associated with publish commands.
