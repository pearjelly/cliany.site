# v0.16.214 发布草案

**目标版本：** `0.16.214`
**提交范围：** `v0.16.213..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本改善 release readiness mode 的机器可读性。

## 用户价值

`v0.16.213` 已经让 tagged release preflight 通过后，`Weekly Review` 可以提示发布已验证 tag。但这个判断仍是从版本号、latest tag 和 `commits_since_latest_tag` 状态推断出来；当维护者只运行普通 `--target-version` readiness 时，报告可能和真正的 `--release-tag` 发布前检查混在一起。

`v0.16.214` 让 `release_readiness.py` 在 JSON 和 Markdown report 中显式输出 `release_mode` 与 `release_tag`。成功的 tagged preflight 会显示 `release_mode=tagged` 和对应 tag；普通目标版本检查会显示 `release_mode=target`、`release_tag=-`，并继续把下一步显示为准备打 tag。

成功标准是：`python scripts/release_readiness.py --strict --target-version 0.16.214 --report /tmp/cliany-release-readiness.md` 生成的 `Weekly Review` 仍显示 `Ready to tag`，而 `python scripts/release_readiness.py --strict --release-tag v0.16.214 --report /tmp/cliany-release-readiness-tagged.md` 在 tag 指向 HEAD 且所有 gate 通过时显示 `Ready to publish verified tag`。

## 变更分组

### Release readiness mode

- `ReadinessReport` 新增 `release_mode` 和 `release_tag` 字段，并通过 `to_dict()` 输出到 JSON。
- Markdown report 的 `Metric` 表展示 `release_mode` 与 `release_tag`，避免只看报告的人需要反推当前检查模式。
- `Weekly Review` 只有在 `release_mode == "tagged"` 且 tag 已通过 preflight 时，才显示 `Ready to publish verified tag ...`。
- 普通 `--target-version` readiness 即使版本、tag 和 HEAD 已匹配，也继续显示 `Ready to tag ... after final validation.`。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `release_mode` / `release_tag` contract。
- `tests/test_release_readiness.py` 覆盖 tagged mode 字段、Markdown 指标和 target mode 回退文案。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护循环文档中的 mode contract。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 release readiness report 的状态表达和发布前决策提示。

## 风险与兼容性

- JSON schema 新增字段，不移除既有字段；只读取 `ok`、`blockers`、`next_actions` 或 gate fields 的工具不受影响。
- Markdown report 增加两行 metric，不改变 gate 表、release links、candidate promotion 或 next actions 的结构。
- `release_mode` 由 CLI 输入决定：传入 `--release-tag` 时为 `tagged`，否则为 `target`。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.214..HEAD`。最终带 `--release-tag v0.16.214` 的 strict gate 仍以本草案顶部的 `v0.16.213..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_release_readiness.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/release_readiness.py tests/test_release_readiness.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/release_readiness.py --target-version 0.16.214 --json
python scripts/release_readiness.py --strict --target-version 0.16.214 --report /tmp/cliany-release-readiness.md
python scripts/release_readiness.py --strict --release-tag v0.16.214 --report /tmp/cliany-release-readiness-tagged.md
python scripts/plan_next_iteration.py --target-version 0.16.214 --json
```

## 发版步骤

1. 确认 `pytest tests/test_release_readiness.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov` 通过。
2. 确认 JSON report 输出 `release_mode` 和 `release_tag`。
3. 确认普通 target readiness 的 Weekly Review 显示准备打 tag。
4. 确认 tagged release mode 的 Weekly Review 显示发布已验证 tag。
5. 确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.214`。
6. 确认 `pyproject.toml` 版本更新为 `0.16.214`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.214`。
8. 创建 tag：`git tag v0.16.214`。

## Release Notes 摘要

`v0.16.214` adds explicit release readiness mode fields so target-version checks and tagged release preflights produce clear, machine-readable next-step guidance.
