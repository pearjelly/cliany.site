# v0.16.264 发布草案

**目标版本：** `0.16.264`
**提交范围：** `v0.16.263..HEAD`
**当前状态：** 2026-07-16 release slice；适配器市场支持带固定摘要的远程 HTTPS 包安装。

## 用户价值

用户可以直接从发布者提供的 HTTPS 地址安装 adapter package，不必先手工下载到本地。远程来源要求显式的 64 位十六进制 SHA-256，下载大小限制为 64 MiB，HTTPS 重定向不得降级；摘要和本地包校验全部通过后才会安装。

远程 `--dry-run` 复用本地包的 manifest、载荷哈希、冲突和备份意图检查。下载到系统临时文件的归档在成功和失败路径都会清理，不写入 adapter、session、snapshot、backup 或持久化下载缓存。

## 变更范围

- `market install <source>` 兼容本地路径，并新增带 `--sha256` 的直接 HTTPS URL 来源。
- 远程下载使用有界流式读取，校验 `Content-Length` 和实际字节数，拒绝超过 64 MiB 的归档。
- 允许 HTTPS 到 HTTPS 重定向，拒绝 HTTP、`file`、`data` 和其他协议；错误仍使用 `INSTALL_FAILED` envelope，并清洗 URL query/fragment 和临时路径。
- target-mode release readiness 识别“版本已 bump、上一版 tag 仍为 latest、目标 tag 尚未创建”的正常过渡态；真正的 release-tag readiness 继续严格要求 tag/version 对齐。
- README、README.zh、adapter lifecycle guide、官网维护者入口和 Q3 路线图同步公开契约；不宣称存在未发布的具体 adapter asset。

## 风险与兼容性

- 本地 `market install`、`--dry-run`、`--force`、备份和 rollback 语义保持不变；远程来源只在 URL 解析和下载成功后进入同一套本地校验/安装路径。
- 远程包仍然不是签名包或可信注册表；SHA-256 必须由发布者通过独立可信渠道提供，项目不实现 `latest` 解析、GitHub API 发现或自动下载缓存。
- 远程下载失败、摘要不匹配、重定向降级、响应过大、tar/manifest/载荷校验失败都必须在运行时目录零写入的条件下返回 `INSTALL_FAILED`。
- `pypi-project-search` 保持 candidate：`llm_live_preflight_not_ready` 仍阻止真实 exploration，不能伪造 adapter package、metadata validation 或 online smoke 证据。

## 案例库映射

- 案例入口继续是 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。
- 本版改进 adapter 生命周期的分发入口，不改变 active/candidate/known-gap 状态，也不发布未验证的 candidate asset。
- 当前 known gap 仍为 `search-extraction-gap`。

## 发版步骤

1. 在 `[Unreleased]` 仍包含本版条目且 compare link 为 `v0.16.263...HEAD` 时，完成 `--target-version 0.16.264 --remote` pre-tag readiness；先确认每日 release cap 未阻塞目标版本。
2. Finalize Changelog：清空 `[Unreleased]`，新增 `## [0.16.264] - 2026-07-16`，并将 compare links 更新为 `v0.16.264...HEAD` 和 `v0.16.263...v0.16.264`。
3. 提交并推送 `master`，等待 CI 成功：

   ```bash
   git add CHANGELOG.md docs/releases/v0.16.264-draft.md
   git commit -m "chore(release): finalize v0.16.264"
   git push origin master
   ```

4. 仅在 finalization commit 已位于 `HEAD` 且每日上限允许时创建本地 tag：`git tag v0.16.264`。
5. 推送 tag 前运行：

   ```bash
   uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.264 --remote --remote-name origin
   ```

6. 仅在 release-tag check 通过后运行 `git push origin v0.16.264`。tag workflow 会创建 GitHub Release 并发布 `cliany-site==0.16.264` 到 PyPI；外部动作完成前不宣称已发布。
7. 在 `site/` 运行 `vercel link --yes --project cliany.site`、`vercel --prod --yes`，再运行 `vercel inspect www.cliany.site --wait --timeout 90s` 检查官网生产 alias。
8. 运行最终审计：

   ```bash
   uv run --extra dev --frozen python scripts/check_release_publication.py --strict --remote --distribution --json
   ```

   该审计核对远端 Git、GitHub Release 和 PyPI 分发，不替代官网 inspect；官网状态以 Vercel inspect 为准。

## Release Notes 摘要

v0.16.264 adds verified remote adapter package installation. A direct HTTPS source must be paired with a pinned SHA-256; bounded downloads, HTTPS-only redirects, existing package validation, dry-run planning, and temporary-file cleanup keep the install path scriptable without weakening local safety checks.
