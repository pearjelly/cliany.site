# v0.16.157 发布草案

**目标版本：** `0.16.157`
**提交范围：** `v0.16.156..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 artifact bundle summary 自身字段清单摘要显式化。

## 用户价值

`v0.16.156` 已经让 `artifact_bundle_summary` 输出 manifest payload count/hash，维护工具可以只读整包摘要判断 manifest payload 是否漂移。但 summary 自身的字段数量和字段清单仍没有稳定摘要，工具需要展开完整 summary 并自行比对字段形状。

`v0.16.157` 在 `artifact_bundle_summary` 中新增 `artifact_bundle_summary_key_count` 和 `artifact_bundle_summary_keys_sha256`，并在 artifacts `README.md` 的 `Artifact Bundle Summary` 段落展示同源字段。这样工具只读 summary 顶部字段时，就能先判断 summary 自身字段规模和字段清单是否漂移，再决定是否继续解析其余字段。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.157 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `artifact_bundle_summary_key_count` 和 `artifact_bundle_summary_keys_sha256`，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且 key count 等于 `artifact_bundle_summary` 实际字段数量。

## 变更分组

### Artifact bundle summary key summary

- `artifact_bundle_summary` 新增 `artifact_bundle_summary_key_count`。
- `artifact_bundle_summary` 新增 `artifact_bundle_summary_keys_sha256`。
- 这两个字段从 `artifact_bundle_summary` 自身字段清单派生。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 `artifact_bundle_summary_key_count` 和 `artifact_bundle_summary_keys_sha256`。
- 现有 `artifact_manifest_payload_key_count`、`artifact_manifest_payload_sha256`、manifest key count/hash、handoff schema version、key count 与 hash 保持兼容。
- 不改变 release draft validation、publication audit 或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 summary 自身 key count/hash。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的新字段，并断言实际 summary keys 与字段清单同源。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 artifact bundle summary 字段清单，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `artifact_bundle_summary` 判断 summary key count/hash，再决定是否解析其余 summary 字段。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.157 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `artifact_bundle_summary_key_count` 和 `artifact_bundle_summary_keys_sha256`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示这两个字段。
4. 确认 key count 等于 `artifact_bundle_summary` 实际字段数量。
5. 确认 `artifact_manifest_payload_key_count` 和 `artifact_manifest_payload_sha256` 仍与 manifest payload 同源。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.157`。
8. 将 `pyproject.toml` 版本更新为 `0.16.157`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.157`。
10. 创建 tag：`git tag v0.16.157`。
