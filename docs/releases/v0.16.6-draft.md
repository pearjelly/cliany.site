# v0.16.6 发布草案

**目标版本：** `0.16.6`
**提交范围：** `v0.16.5..HEAD`
**当前阻塞：** 需要提交计划器 issue artifacts 输出、维护者循环入口和测试；最新本地 release 仍需维护者确认后 push 到远端。

## 用户价值

`v0.16.5` 已经让 `scripts/plan_next_iteration.py` 输出 candidate issue title、labels 和 body。维护者仍需要从 JSON 或 Markdown report 里复制这些字段，再手动创建 body 文件或拼 `gh issue create` 命令。

`v0.16.6` 新增 `--issues-dir`，让计划器把这些候选案例 issue 资料离线写成可审阅 artifacts：每个 candidate 一个 Markdown body 文件、一个 `issue-metadata.json` 汇总文件，以及一个 `create-issues.sh` 脚本。脚本不会自动执行，维护者可以先检查内容，再决定是否运行 `gh issue create`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.6 --issues-dir /tmp/cliany-candidate-issues` 写出 `pypi-project-search.md`、`npm-package-search.md`、`crates-io-crate-search.md`、`issue-metadata.json` 和 `create-issues.sh`；metadata 中包含 title、labels、body file 和 create command。

## 变更分组

### Candidate issue artifacts

- `plan_next_iteration.py` 新增 `--issues-dir`。
- 每个 candidate promotion 写出一个 Markdown issue body 文件。
- `issue-metadata.json` 汇总 `case_id`、`issue_title`、`issue_labels`、`issue_body_file` 和 `create_command`。
- `create-issues.sh` 输出可审阅的 `gh issue create` 命令，并使用安全 shell quoting。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `--issues-dir` 输出 issue artifacts。
- `tests/test_plan_next_iteration.py` 固定 body 文件、metadata JSON、脚本内容和脚本权限。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环里的新说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。计划器会为 `pypi-project-search`、`npm-package-search`、`crates-io-crate-search` 三个 candidate cases 写出本地 issue artifacts。

`search-extraction-gap` 继续作为抽取质量 known-gap。本版本只读取 manifest 和 readiness 里的现有证据，不访问第三方站点，不执行真实 adapter 包构建，也不会自动调用 GitHub。

## 风险与兼容性

- 这是维护脚本和报告输出 patch，不改变 CLI runtime、adapter schema、JSON envelope 或生成代码。
- `--issues-dir` 是显式 opt-in；默认运行不会写这些 artifacts。
- 输出目录由维护者指定，建议使用 `/tmp/...` 或 CI artifact 目录，避免把生成文件提交进仓库。
- 默认验证继续离线运行，不需要真实 LLM key，也不会写入 `~/.cliany-site/`。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.6 --issues-dir /tmp/cliany-candidate-issues` 写出 body、metadata 和 create script。
2. 运行计划脚本、维护者循环文档和 release draft 相关测试。
3. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.6`。
4. 将 `pyproject.toml` 版本更新为 `0.16.6`。
5. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.6`。
6. 创建 tag：`git tag v0.16.6`。

## Release Notes 摘要

`v0.16.6` adds offline candidate issue artifacts to the next-iteration planner. Maintainers can now generate issue body files, metadata JSON, and a reviewable `gh issue create` script before deciding whether to create GitHub issues.
