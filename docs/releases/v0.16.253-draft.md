# v0.16.253 发布草案

**目标版本：** `0.16.253`
**提交范围：** `v0.16.252..HEAD`
**当前状态：** 本版把官网生产部署后的 alias 复核纳入标准发版流程，准备作为 2026-07-02 的第三个可验证 patch 发布。

## 用户价值

`v0.16.252` 已经把 PyPI latest cache 滞后变成可解释的 publication audit 状态，但官网发布仍有一个真实维护空隙：Vercel CLI 的生产部署摘要可能只展示项目上的默认 alias，维护者需要额外手动 `vercel inspect <deployment>.vercel.app` 才能确认 `www.cliany.site` 和 `cliany.site` 是否已经指向本次部署。

`v0.16.253` 将稳定命令 `cd site && vercel inspect www.cliany.site --wait --timeout 90s` 加入 standard release flow，在 `website_deploy` 之后、`remote_publication_audit` 之前执行。维护者不需要从动态部署 URL 猜测检查命令，release readiness、planner 和 candidate artifact handoff 也能用同一组 `standard_release_flow_website_inspect_*` 字段确认官网 alias 检查没有被跳过。

这让每天 1-3 个版本的 release train 更接近真正的端到端发布：GitHub Release、PyPI 和官网都在同一份结构化清单里完成部署与复核。

## 变更范围

- `scripts/release_readiness.py`：standard release flow 新增 `website_inspect` 步骤，并导出 inspect 命令及 SHA-256 摘要字段。
- `scripts/plan_next_iteration.py`：planner fallback flow、文本/Markdown report、`artifact-manifest.json`、`publication-handoff.json` 和 `artifact_bundle_summary` 透传同一组 inspect 字段。
- Docs / website：weekly maintainer loop、public roadmap、官网维护者入口同步说明官网 inspect checkpoint，并把公开基线更新到 `v0.16.253`。

## 风险与兼容性

- 不改变 CLI runtime、adapter schema、action replay、AXTree selector 或 session 存储。
- `website_deploy` 和 distribution audit 字段保持兼容；新增 `website_inspect` 字段只扩展 release handoff。
- `vercel inspect www.cliany.site --wait --timeout 90s` 只读检查当前 alias target；如果 Vercel 网络或权限不可用，标准流程会在官网复核步骤明确停住，而不是伪装 release 已经完成。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的案例状态；相关说明仍落在 [cases/README.md](../../cases/README.md)。
- 当前案例库仍保持 active 4 个、candidate 3 个、known gap 1 个。
- Known gap 保持 1 个：`search-extraction-gap`。
- Candidate 晋级仍需要 adapter package、metadata validation 和 online smoke 三段真实证据。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.252..HEAD`。完成 version bump 后，target-mode readiness 也会查找占位锚点 **提交范围：** `v0.16.253..HEAD`；该锚点只表示 tag 发布后不应再有未发布差异，不改变本版实际提交范围。创建本地 tag 后，使用 local tag-mode readiness 复核最终 tag 语义。

```bash
git diff --check
uv run --extra dev --frozen pytest tests/test_release_readiness.py tests/test_plan_next_iteration.py -q --no-cov --tb=short
uv run --extra dev --frozen pytest tests/test_release_draft_docs.py tests/test_weekly_maintainer_loop_docs.py tests/test_site_content.py -q --no-cov --tb=short
uv run --extra dev --frozen python scripts/validate_cases.py --strict
uv run --extra dev --frozen python scripts/release_readiness.py --strict --target-version 0.16.253 --remote --remote-name origin --json
```

创建本地 tag 后、推送 tag 前再运行：

```bash
CLIANY_QA_OFFLINE=1 uv run --extra dev --frozen pytest tests/ -q
uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.253 --remote --remote-name origin
```

## 发版步骤

1. 确认 `v0.16.252` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 确认 2026-07-02 当天 release count 未超过每日上限。
3. 提交并推送 `master`。
4. 创建并推送 `v0.16.253` tag。
5. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
6. 按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署官网。
7. 运行 `cd site && vercel inspect www.cliany.site --wait --timeout 90s`，确认官网 alias 已指向生产部署。
8. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json`，确认 `distribution_ok=true`、GitHub Release latest tag 为 `v0.16.253`，PyPI `pypi_version` 为 `0.16.253`。

## Release Notes 摘要

v0.16.253 adds an explicit website inspect checkpoint to the standard release flow. After production Vercel deployment, maintainers can verify `www.cliany.site` with a stable `vercel inspect` command before completing the final GitHub Release and PyPI distribution audit.
