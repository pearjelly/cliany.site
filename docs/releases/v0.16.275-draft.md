# v0.16.275 发布草案

**目标版本：** `0.16.275`
**提交范围：** `v0.16.274..HEAD`
**发布主题：** 候选案例的 machine-readable handoff 先给出必经的 live-LLM preflight，避免自动化跳过 promotion gate。

完成 version bump 后，target-mode readiness 还会检查占位锚点 **提交范围：** `v0.16.275..HEAD`；该锚点不改变实际 release diff。

## 用户价值

candidate 的 `adapter_package` 并不是可直接执行 `explore` 的普通任务：它必须先通过 `cliany-site doctor --llm-live --json`。此前 runbook 的首步已经记录了这个规则，但 JSON evidence bundle 和 promotion plan 中的 primary command 仍指向后续 `explore`，会让只消费该字段的自动化跳过 gate。

本版把 executable next command、next step 和 next handoff 作为优先字段：`adapter_package` 的 primary command 与 queue command 都是 doctor preflight；真实的 explore 命令与原 task handoff 仍以 `task_command`、`task_handoff` 保留，供 preflight 成功后继续使用。`cliany-site cases` 的表格和快速命令使用同一安全顺序。已经完成 adapter package 的 candidate 直接给出 metadata validation，不会重复增加 LLM preflight。

这只修复本地 handoff 语义，不声称 live LLM、adapter package、第三方 online smoke 或任何 candidate promotion 已成功。

## 变更分组

### Candidate promotion contract

- `src/cliany_site/commands/cases.py` 为每个 promotion task 明确记录 `next_step`、`next_command`、`next_command_source`、`next_command_missing` 和 `next_handoff`。
- evidence bundle 的 `primary_next_task_command` 与 promotion plan 的 `primary_command` 现在是可立即执行的安全动作；后续任务命令以 `*_task_command` 保留。
- task queue 的 `command` 同样代表下一条可执行命令，避免 queue consumer 从 `adapter_package` 直接触发 explore。
- `metadata_validation` 与 `online_smoke` 保持各自的直接可执行命令，测试覆盖不会误加 preflight。

### Evidence visibility

- 人类可读的 evidence bundle、promotion plan、案例表格和快速命令都先呈现 doctor preflight，再说明通过后的 task command。
- `cases/README.md` 和 `cases/manifest.json` 的 candidate 状态不变；本版没有将 PyPI、npm 或 crates.io candidate 提升为 active。
- `search-extraction-gap` 继续保留为已知提取质量缺口，和这次 handoff 修复一起提醒维护者：浏览器动作完成、数据读取完成和 candidate promotion 都是不同的事实。

## 案例库映射

`pypi-project-search`、`npm-package-search` 与 `crates-io-crate-search` 仍是 candidate。`adapter_package` 仍要求 live LLM preflight、package asset、metadata validation 与 read-only online smoke 的真实证据；当 `llm_live_preflight_not_ready` 时，不得执行 explore 来生成 candidate adapter，也不得伪造 archive、在线结果或晋级状态。

本版不写入 `~/.cliany-site/`，不创建 adapter、session、snapshot 或 package artifact。active historical demo 的现有 release assets 不被改写。

## 风险与兼容性

- JSON consumers 若把 `primary_command` 或 queue `command` 当作 task command，需要改读 `primary_task_command` 或 `task_command` 才能跳到 preflight 后的动作；这是有意的安全语义修正。
- `primary_next_task` 内部仍保留原 task metadata，消费者应使用顶层 `primary_next_task_command` 或 plan queue `command` 来执行下一步。
- preflight 命令本身可能报告 `E_LLM_UNAVAILABLE` 或其他 `llm_live_preflight_not_ready` 状态；这时必须停止 candidate promotion 并附上真实 doctor evidence。
- 本版不绕过 Chrome/CDP、LLM provider 或第三方站点可用性，也不将离线测试表述为 online success。

## 发版步骤

1. 完成版本号、官网基线、CHANGELOG compare links 和本草案；在干净 HEAD 运行 target-mode readiness。
2. 提交并推送 `master`，等待 GitHub CI 与 Embodied CI 都成功；任一失败都不创建 tag。
3. 仅在两项 CI 成功后创建本地 `git tag v0.16.275`，运行 `uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.275 --remote --remote-name origin`。
4. 仅在 tag gate 通过后推送 tag，等待 GitHub Release workflow；逐版本核验 GitHub Release 与 PyPI `0.16.275`。
5. 运行 `gh release edit v0.16.275 --repo pearjelly/cliany.site --notes-file docs/releases/v0.16.275-github-release.md`，再用 `gh release view v0.16.275 --repo pearjelly/cliany.site --json body` 确认用户可读 Release Notes。
6. 在 `site/` 运行 `vercel link --yes --project cliany.site`、`vercel --prod --yes` 和 `vercel inspect www.cliany.site --wait --timeout 90s`。
7. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --strict --remote --distribution --json`；只有 audit 成功才报告发布完成。

## Release Notes 摘要

`v0.16.275` makes candidate-promotion automation run its required live-LLM preflight first and preserves the post-preflight task command separately, preventing a candidate explore command from being presented as immediately safe to execute.
