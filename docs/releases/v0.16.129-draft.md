# v0.16.129 发布草案

**目标版本：** `0.16.129`
**提交范围：** `v0.16.128..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 publication publish script path 结构化输出。

## 用户价值

`v0.16.128` 已经让维护者能在 `Candidate Issue Gate Quick Summary` 里直接看到 `publication_publish_script_command`。这解决了“如何生成可审阅脚本”的可见性问题，但工具如果只想知道脚本会写到哪里，仍需要从命令字符串中解析 `/tmp/cliany-publish-release.sh`。

`v0.16.129` 新增 `publication_publish_script_path`。计划 JSON、默认文本输出、Markdown report、candidate issue artifacts `README.md`、`artifact-manifest.json`、`publication-handoff.json` 和 `artifact_bundle_summary` 都会保留这个结构化路径，让维护者和自动化能直接定位脚本文件，而不依赖命令字符串解析。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.129 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 artifacts 中，`Candidate Issue Gate Quick Summary` 展示 `publication_publish_script_path`，`Publication Publish Script` 展示同一路径，`publication-handoff.json` 保留 `publish_script_path`，`artifact_bundle_summary` 展示 `publication_publish_script_path_sha256`。

## 变更分组

### Publication publish script path

- `IterationPlan` 新增 `publication_publish_script_path`，默认值保持 `/tmp/cliany-publish-release.sh`。
- `publication_publish_script_command` 继续使用同一路径生成 `--publish-script` 命令。
- 计划 JSON、默认文本输出和 Markdown report 展示 `publication_publish_script_path`。
- Candidate issue artifacts `README.md` 的 `Candidate Issue Gate Quick Summary` 和 `Publication Handoff` 展示发布脚本路径。
- `artifact-manifest.json` 保留 `publication_publish_script_path`，`publication-handoff.json` 保留 `publish_script_path`。
- `artifact_bundle_summary` 新增 `publication_publish_script_path_sha256`，方便工具检测路径漂移。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 quick summary、handoff 和 bundle summary 会展示 publication publish script path。
- `tests/test_plan_next_iteration.py` 覆盖 plan dict、Markdown report、artifacts README、manifest、handoff 和 bundle summary 的路径字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善候选 issue artifacts 的维护者阅读路径和机器可读发布脚本定位能力。

## 风险与兼容性

- 新增字段只拆出既有命令里的默认脚本路径。
- 现有 `publication_publish_script_command`、`Publication Publish Script` 和 publish commands 保持兼容。
- 新增 hash 只用于漂移检测，不替代脚本内容审阅。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.129 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 artifacts `README.md` 包含 `Candidate Issue Gate Quick Summary`。
3. 确认该段落展示 `publication_publish_script_path`。
4. 确认 `Publication Publish Script` 小节展示同一路径。
5. 确认 `publication-handoff.json` 包含 `publish_script_path`。
6. 确认 `artifact_bundle_summary` 包含 `publication_publish_script_path_sha256`。
7. 运行计划器、维护者循环文档和 release draft 相关测试。
8. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.129`。
9. 将 `pyproject.toml` 版本更新为 `0.16.129`。
10. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.129`。
11. 创建 tag：`git tag v0.16.129`。

## Release Notes 摘要

`v0.16.129` adds a structured publication publish script path to next-iteration plans and candidate issue artifacts, helping maintainers and automation locate `/tmp/cliany-publish-release.sh` without parsing the publish script command string.
