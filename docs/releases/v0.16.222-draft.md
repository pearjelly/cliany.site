# v0.16.222 发布草案

**目标版本：** `0.16.222`
**提交范围：** `v0.16.221..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts 直接携带 candidate 晋级证据摘要。

## 用户价值

`v0.16.221` 已经让周初计划的 JSON、默认文本输出和 Markdown report 展示 `case_promotion_evidence_summary`。维护者能从 planning 输出看到 candidate 晋级任务的 `candidate_count`、`task_count`、`pending_count`、`blocked_count`、`complete_count` 和 `primary_next_action`。

真正交接给维护者或自动化的下一步经常不是单份 report，而是 `python scripts/plan_next_iteration.py --target-version 0.16.222 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 candidate issue artifacts 包。如果 `artifact-manifest.json` 和 artifacts `README.md` 缺少同一份摘要，只读 manifest 的工具和只读 README 的人工审阅都会退回到逐个 issue body 展开。

`v0.16.222` 把 `case_promotion_evidence_summary` 接入 issue artifacts。成功标准是：`artifact-manifest.json.case_promotion_evidence_summary` 存在并与 plan summary 同源；artifacts `README.md` 包含 `Candidate Promotion Evidence Summary`；当前真实案例库仍显示 3 个 candidate、9 个 pending 子任务，并把 PyPI adapter package 作为 primary next action。

## 变更分组

### Candidate issue artifacts evidence summary

- `artifact-manifest.json` 新增顶层 `case_promotion_evidence_summary`。
- `ARTIFACT_MANIFEST_KEYS` 固定该字段顺序，放在 `candidate_cases` 之后。
- Candidate issue artifacts `README.md` 在 `Candidate Summary` 后展示 `Candidate Promotion Evidence Summary`。
- README 中的文件说明明确 manifest 包含 promotion evidence summary。
- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifacts manifest 和 README 都会携带这份摘要。

### 文档与门禁

- `tests/test_plan_next_iteration.py` 覆盖 manifest 顶层字段和 README 小节。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护循环文档中的字段说明。
- `tests/test_release_draft_docs.py` 固定本草案。
- 本版本不把该 summary 复制进 `artifact_bundle_summary`，避免扩大整包摘要 schema；工具需要完整晋级任务明细时应读取 manifest 顶层字段。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate issue artifacts 包能直接交接 candidate 晋级证据看板。

## 风险与兼容性

- `case_promotion_evidence_summary` 是 `artifact-manifest.json` 的新增顶层字段；不移除既有 `candidate_cases`、`candidate_promotions`、`issue-metadata.json` 或 issue body 内容。
- `artifact_bundle_summary` 不新增字段，避免只读整包摘要的工具被迫处理更大的 schema。
- Summary 只汇总现有 manifest 状态，不会伪造 adapter package、metadata validation 或 online smoke 已完成。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.222..HEAD`。最终带 `--release-tag v0.16.222` 的 strict gate 仍以本草案顶部的 `v0.16.221..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.222 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/plan_next_iteration.py --target-version 0.16.222 --report /tmp/cliany-next-iteration.md
python scripts/release_readiness.py --target-version 0.16.222 --json
python scripts/release_readiness.py --strict --target-version 0.16.222 --report /tmp/cliany-release-readiness.md
```

## 发版步骤

1. 确认 focused pytest 和 ruff 通过。
2. 确认 `plan_next_iteration.py --issues-dir` 生成的 `artifact-manifest.json` 包含 `case_promotion_evidence_summary`。
3. 确认 artifacts `README.md` 包含 `Candidate Promotion Evidence Summary`。
4. 确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.222`。
5. 确认 `pyproject.toml` 版本更新为 `0.16.222`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.222`。
7. 创建 tag：`git tag v0.16.222`。

## Release Notes 摘要

`v0.16.222` brings candidate promotion evidence summaries into generated issue artifacts, so maintainers and automation can review candidate next actions from both `artifact-manifest.json` and the artifacts `README.md`.
