# v0.16.15 发布草案

**目标版本：** `0.16.15`
**提交范围：** `v0.16.14..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本只让候选 issue 创建脚本在执行前复核发布可见性。

## 用户价值

`v0.16.14` 让 `scripts/plan_next_iteration.py --issues-dir` 生成的 candidate issue body 自带 `Reproduction Context`，贡献者拿到单个 issue 时能看到 target URL、candidate commands 和 offline validation commands。但 `create-issues.sh` 仍然可以在最新本地 tag 尚未公开可见时被手动执行，维护者可能先派发下一批候选任务，外部贡献者却看不到对应 release、tag 或文档状态。

`v0.16.15` 给 `create-issues.sh` 增加 release publication preflight。生成脚本会先进入当前 git worktree 根目录，运行 `python scripts/check_release_publication.py --strict --json`，并把 JSON 写入 `/tmp/cliany-issue-publication-check.json`；只有 publication audit 通过后才继续执行后续 `gh issue create` 命令。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.15 --issues-dir /tmp/cliany-candidate-issues` 写出的 `create-issues.sh` 包含 publication preflight，并且 artifacts `README.md` 说明该 preflight 不会由计划器自动执行。

## 变更分组

### Candidate issue publication preflight

- `scripts/plan_next_iteration.py` 生成的 `create-issues.sh` 在 issue 创建命令之前运行 `python scripts/check_release_publication.py --strict --json`。
- preflight JSON 固定写入 `/tmp/cliany-issue-publication-check.json`，方便维护者失败后查看具体 publication blocker。
- 脚本继续只由 `--issues-dir` 生成，不会被 `plan_next_iteration.py` 自动执行。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 candidate issue artifacts 的 `create-issues.sh` 会先做 publication preflight。
- `tests/test_plan_next_iteration.py` 固定脚本内容和 artifacts README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不提前把 candidate 标记为 active，也不创建真实 GitHub issue；它只让候选 issue 创建脚本在维护者手动执行前先确认最新 release 已经公开可见。

## 风险与兼容性

- `create-issues.sh` 新增 preflight 后，在 branch/tag 尚未发布时会提前失败，这是预期保护。
- preflight 只读取本地 git/upstream 状态；除非维护者自己运行 `--remote`，否则不会访问远端网络。
- 生成 artifacts 仍写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.15 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.15 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.15 --issues-dir /tmp/cliany-candidate-issues` 写出带 publication preflight 的 `create-issues.sh`。
2. 确认 artifacts `README.md` 说明 `create-issues.sh` 不会自动执行，并记录 `/tmp/cliany-issue-publication-check.json`。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.15`。
5. 将 `pyproject.toml` 版本更新为 `0.16.15`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.15`。
7. 创建 tag：`git tag v0.16.15`。

## Release Notes 摘要

`v0.16.15` adds a publication preflight to generated candidate issue creation scripts. Maintainers now get a strict release-publication check before `gh issue create` runs, preventing follow-up candidate work from being published before the latest local release is visible.
