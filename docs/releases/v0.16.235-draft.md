# v0.16.235 发布草案

**目标版本：** `0.16.235`
**提交范围：** `v0.16.234..HEAD`
**当前阻塞：** 需要完成 `E_LLM_UNAVAILABLE` 错误信封加固，并重新运行 release readiness、case validation、文档/官网测试、完整 pytest、官网浏览器检查和标准发布流程。

## 用户价值

v0.16.235 聚焦 `explore` 阶段的 LLM 上游不可用体验。维护者在尝试推进 `pypi-project-search` candidate 的真实 PyPI 探索时遇到上游 `502 Bad Gateway`，旧行为会把原始 HTML 网关页透传为 `E_UNKNOWN`。

本版本将这类可重试的 LLM 网关、限流和服务不可用故障统一映射为 `E_LLM_UNAVAILABLE`，并返回清洗后的 message、`retryable`、`status_code` 和 `phase`。这样自动化可以明确判断“稍后重试或切换 provider”，而不会误以为 adapter、AXTree selector map 或案例证据已经损坏。

## 变更分组

- CLI：`explore --json` 捕获 `LlmUnavailableError`，返回 `E_LLM_UNAVAILABLE` 和可重试详情。
- 引擎：LLM 重试逻辑识别 HTTP 429/500/502/503/504、Bad Gateway、Gateway Timeout、Service Unavailable 和 rate limit 等上游故障，并清洗 HTML 错误体。
- 错误码：`ErrorCode.from_exception`、`ERROR_FIX_HINTS` 和错误码覆盖测试补齐 `E_LLM_UNAVAILABLE`。
- 文档：README、README.zh 和官网文档说明 `E_LLM_UNAVAILABLE` 的含义和推荐动作。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的任何案例状态。
- 当前真实案例库仍保持 active 4 个、candidate 3 个、known gap 1 个。
- Known gap 保持 1 个：`search-extraction-gap`。
- [cases/README.md](../../cases/README.md) 的候选晋级规则仍然适用：`adapter_package`、`metadata_validation` 和 `online_smoke` 三段证据未全部 complete 前，不把 candidate 标记为 active。
- `pypi-project-search` 仍是首要 candidate；本版真实 PyPI 探索被 LLM 上游 502 阻断，因此只修复错误信封，不声称 adapter package 已生成。

## 风险与兼容性

- 下游如果只按 `E_UNKNOWN` 捕获 LLM 网关故障，需要兼容新的 `E_LLM_UNAVAILABLE`。
- `explore` 成功路径、生成 adapter schema、action runtime、AXTree 语义匹配和 codegen 逻辑不变。
- 错误消息会清洗原始 HTML，上游返回的完整 HTML 不再出现在 JSON 信封中。
- `E_LLM_UNAVAILABLE` 表示 provider 层可重试故障，不代表用户登录态、selector map 或 adapter package 损坏。
