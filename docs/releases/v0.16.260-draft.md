## 用户价值

`v0.16.259` 已经把真实 `cliany-site doctor --llm-live --json` 输出接入 candidate evidence bundle 和 issue template。下一步真正提升开源项目实用价值的方向仍是把真实案例库从“候选流程”推进到“可复用 adapter 包 + 可验证 smoke”的状态，首要目标是 `pypi-project-search/adapter_package`。

本草案先把 `v0.16.260` 的发布边界固定下来：今天不再创建第四个 release tag；维护者可以继续收集或整理 `pypi-project-search` 的 doctor preflight、adapter package、metadata validation 与 online smoke 证据，但发布窗口恢复前只准备证据和文档，不修改 package 版本、不更新官网 baseline、不把 candidate 提前标为 active。

2026-07-07 最新 preflight 刷新仍显示 blocker：CDP 可用，但 `summary.ready_for_explore=false`、`summary.capabilities.generate_adapters.ready=false`、`checks[llm_live].status=warning`、`checks[llm_live].details.error_code=E_LLM_UNAVAILABLE`，`doctor_preflight_state.status=blocked`，`reason_codes=["ready_for_explore_false","generate_adapters_not_ready","llm_live_status_warning"]`，`doctor_preflight_evidence_values_sha256=d4ab1f14b3792a778e25ebbda68aafc41f3e932de50db4735eca7e4a42e6aaad`。

如果 live LLM preflight 仍返回 `E_LLM_UNAVAILABLE` 或 `summary.capabilities.generate_adapters.ready=false`，本版应继续使用 `cliany-site cases --case-id pypi-project-search --evidence-bundle --doctor-json /tmp/cliany-doctor-preflight.json --json` 输出里的 `doctor_preflight_state` 作为 blocker evidence。只有 `doctor_preflight_state.status=ready` 时，才进入真实 `explore`、adapter package、metadata validation 和 online smoke。

当前已准备的最小可发切片是 planner 级 doctor evidence handoff：`python scripts/plan_next_iteration.py --target-version 0.16.260 --doctor-json /tmp/cliany-doctor-preflight.json --issues-dir /tmp/cliany-candidate-issues-v016260 --json` 会把同一份 doctor JSON 的 `doctor_preflight_state`、extracted values、source path 和 `--doctor-json` aware issue/evidence bundle commands 写入 `issue-metadata.json` 与 candidate issue body。这样 live LLM 仍 blocked 时，维护者可以生成可直接审阅的 blocker-ready candidate issue artifacts，而不用手工拼接 evidence bundle 和 issue body。

公开协作面也已经开始成型：`v0.16.260` milestone 追踪 `pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 三个 candidate promotion issue。生成的 `create-issues.sh` 现在会在真正运行 `gh issue create` 前检查所需 GitHub label 是否存在；如果 `case-proposal` 或 `good first issue` 缺失，脚本会列出缺失 label 并退出，避免出现部分 issue 已创建、后续 issue 因 label 缺失失败的维护事故。

维护者文档也需要跟上这条 evidence handoff：`README.md`、`README.zh.md`、`docs/good-first-issues.md`、`docs/contributor-starter.md`、`cases/README.md`、官网维护者入口、GitHub case proposal 模板和 PR 模板现在直接写明 `doctor_preflight_state_fields` / `doctor_preflight_state_statuses` 的读取契约。字段固定为 `preflight_state.status`、`preflight_state.ready_for_adapter_package`、`preflight_state.primary_reason`、`preflight_state.reason_codes` 和 `preflight_state.next_action`；状态集合固定为 `ready`、`blocked`、`missing_fields`。这样只看 README、首次贡献者入口、案例库入口、官网或 GitHub 原生模板的维护者也能判断应该继续真实 `explore`，还是先把 doctor blocker evidence 贴回 candidate issue。

自动生成的 candidate issue 正文现在也直接渲染同一份 `Doctor Preflight State Contract`：包括 `preflight_state.*` 字段、`ready` / `blocked` / `missing_fields` 三态集合，以及只有 `preflight_state.status=ready` 且 `preflight_state.ready_for_adapter_package=true` 才继续 `adapter_package` 的 gate。这样维护者从 `/tmp/cliany-candidate-issues-.../*.md` 复制 issue 正文时，不需要再回到 JSON metadata 才能看懂 doctor-backed blocker 语义。

Release readiness 现在也会检查 `scripts/plan_next_iteration.py` 是否仍保留这段 generated issue body contract。若 `Doctor Preflight State Contract`、任一 `preflight_state.*` 字段、三态集合，或 `preflight_state.status=ready` / `preflight_state.ready_for_adapter_package=true` continuation gate 从 planner 渲染代码中漂移，`project metadata validation failed` 会在发版前挡住。

## 变更范围

- 计划范围：优先推进 `pypi-project-search/adapter_package`，但不在 release cap 已满的 2026-07-07 创建 `v0.16.260` tag。
- 若 preflight ready：生成并发布 `pypi.org-<version>.cliany-adapter.tar.gz`，再补齐 metadata validation 和 online smoke 证据。
- 若 preflight blocked：发布 planner `--doctor-json` issue artifacts handoff，让 `--issues-dir` 批量候选 issue 文件直接携带当前 blocker evidence，不伪造 adapter package 证据。
- Candidate issue artifact 脚本会预检 required GitHub labels，缺失时在创建任何公开 issue 前失败并提示维护者先补 label。
- README、README.zh、Good First Issues、贡献者上手地图、案例库 README、官网维护者入口、GitHub case proposal 模板和 PR 模板补齐 doctor preflight state contract 的字段列表和三态集合，避免维护者或首次贡献者跨读 release draft 或源码才能理解 `doctor_preflight_state`。
- Candidate issue body 自动生成器补齐 `Doctor Preflight State Contract` 段落，避免复制 GitHub issue 时丢失 `preflight_state.*` 字段和三态 gate。
- Release readiness 增加 planner issue-body contract 漂移检查，避免 `scripts/plan_next_iteration.py` 后续改动移除 generated issue 正文里的 doctor preflight state gate。
- 本草案本身只解除 `release draft is missing`，不改变 `pyproject.toml`、`site/` baseline 或 `cases/manifest.json` 状态。

## 发版步骤

1. 确认当前日期已到 2026-07-08 或之后，且 `daily_release_cap_blocked=false`。
2. 确认 `v0.16.259` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
3. 运行 `cliany-site doctor --llm-live --json` 并保存到 `/tmp/cliany-doctor-preflight.json`。
4. 若 `doctor_preflight_state.status=ready`，推进 `pypi-project-search/adapter_package`；否则把 blocked evidence 写入 release notes 和 candidate issue handoff。
5. 完成 `pyproject.toml`、`CHANGELOG.md` 和官网 baseline 的 `0.16.260` 更新。
6. 运行 target-mode readiness、离线回归和 case validation。
7. 提交并推送 `master`。
8. 创建并推送 `v0.16.260` tag。
9. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
10. 按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署官网，并运行 `cd site && vercel inspect www.cliany.site --wait --timeout 90s`。
11. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json`，确认 GitHub Release、PyPI 和官网发布状态。

## Release Notes 摘要

Draft target: v0.16.260 should continue the real-case-library push after v0.16.259 made doctor-backed candidate evidence first-class. If live preflight is ready, the release should focus on producing and validating the PyPI candidate adapter package. If live preflight remains blocked, the release should ship the planner `--doctor-json` issue artifacts handoff so `--issues-dir` output carries blocker-ready doctor evidence, while leaving `pypi-project-search` as candidate and avoiding any fabricated adapter package evidence. The release also clarifies the README/README.zh, Good First Issues, contributor starter, case catalog, public website, GitHub case proposal, PR template, and generated candidate issue body doctor preflight state contract (`ready`, `blocked`, `missing_fields`) so maintainers and first-time contributors can apply blocker evidence from the primary docs and copied GitHub issue bodies. Release readiness now blocks future planner issue-body drift if that generated contract or `adapter_package` continuation gate disappears.
