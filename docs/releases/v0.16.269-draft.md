# v0.16.269 发布草案

**目标版本：** `0.16.269`
**提交范围：** `v0.16.268..HEAD`
**当前状态：** 2026-07-21 release slice；doctor 的 demo asset 与已有 adapter 运行时状态分离。

完成 version bump 后，target-mode readiness 还会检查占位锚点 **提交范围：** `v0.16.269..HEAD`；该锚点不改变实际 release diff。

## 用户价值

`cliany-site doctor --json` 现在把 `ready_for_existing_adapters` 和 `ready_for_demo_adapters` 分开。用户和自动化可以知道本机是否能运行已有 adapter，同时不会把运行时就绪误解为存在可安装的公开 demo asset；没有 asset 时会读取 `false` 并继续使用 `case_catalog_quickstart`。

## 案例库映射

本版不改变案例状态或 adapter 资产。`cases/README.md` 和 `cases/manifest.json` 仍是案例目录的唯一来源；`search-extraction-gap` 与 `llm_live_preflight_not_ready` 继续约束候选案例晋级。2026-07-21 的真实 LLM preflight 仍返回可重试的 `E_LLM_UNAVAILABLE`，因此没有生成或声明任何 candidate adapter package、online smoke 或 live explore 成功。

## 风险与兼容性

- `ready_for_existing_adapters` 是新增布尔字段，承载旧 human summary 所说的已有 adapter 运行时就绪含义。
- `ready_for_demo_adapters` 保持布尔类型，但现在严格反映 `demo_adapter_quickstart.available`；调用方应改用新字段判断已有 adapter 运行时。
- `demo_adapter_quickstart` 继续保持不可用、deprecated 和空命令列表；`case_catalog_quickstart` 仍是没有公开 asset 时的首跑路径。

## 发版步骤

1. 在 `[Unreleased]` 保留本版条目、compare link 为 `v0.16.268...HEAD` 时运行 target-mode readiness。
2. Finalize Changelog：清空 `[Unreleased]`，新增 `## [0.16.269] - 2026-07-21`，并更新 compare links。
3. 提交并推送 `master`，等待 CI 和 Embodied CI 成功。
4. 创建 `git tag v0.16.269`，然后运行 `uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.269 --remote --remote-name origin`。
5. 仅在该 gate 通过后推送 tag，等待 GitHub Release 与 PyPI 发布。
6. 在 `site/` 运行 `vercel link --yes --project cliany.site`、`vercel --prod --yes` 和 `vercel inspect www.cliany.site --wait --timeout 90s`。
7. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --strict --remote --distribution --json`。
