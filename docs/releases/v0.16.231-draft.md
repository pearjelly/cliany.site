# v0.16.231 发布草案

**目标版本：** `0.16.231`
**提交范围：** `v0.16.230..HEAD`
**当前阻塞：** 需要完成 candidate issue template 验收标准改动，并重新运行 release readiness、case validation、CLI/文档/官网测试。

## 用户价值

`v0.16.230` 已经把 candidate evidence bundle 做成可验收的证据清单。下一步要把同一套验收标准带到维护者真正创建和分发任务的入口：GitHub issue body。

`v0.16.231` 聚焦 candidate promotion issue 模板。`cliany-site cases --case-id <id> --issue-template`、`scripts/plan_next_iteration.py` 生成的 candidate issue body，以及 `scripts/validate_cases.py --report` 的 Issue Body Template 现在都会展示 `Acceptance Criteria`。维护者可以直接复制 issue body，把 adapter package、metadata validation 和 online smoke 三类任务的完成证据要求一起交给贡献者或自动化。

成功标准是：cases CLI、next iteration plan、case catalog report 的 issue body 都展示验收标准；README、README.zh、cases README 和官网 quickstart 能说明 issue template 也包含 Acceptance Criteria；`scripts/release_readiness.py --target-version 0.16.231 --json` 不再因为缺失 release draft 阻塞。

## 变更分组

### Candidate Issue Template

- `cliany-site cases --case-id <id> --issue-template` 的 Primary Evidence Task 现在展示首要任务的 Acceptance Criteria。
- Candidate issue body 新增 `## Acceptance Criteria` 小节，逐项列出 `adapter_package`、`metadata_validation` 和 `online_smoke` 的完成证据要求。
- `## Tasks` 中每个任务也附带 `Acceptance criteria` 行，让只查看 checkbox 区域的维护者不必跳转到上方小节。

### Plan 与 Report 同步

- `scripts/plan_next_iteration.py` 使用同一套 acceptance criteria 文案生成 candidate issue body，保持周计划、issue artifacts 和 CLI 模板一致。
- `scripts/validate_cases.py --report` 的 Candidate Promotion Tasks / Issue Body Template 同步输出验收标准，离线报告也能成为可复制 issue 来源。
- 相关测试覆盖 CLI JSON/Markdown issue template、next iteration plan 精确 issue body、case report Markdown 和官网内容。

### 文档与官网同步

- README / README.zh 的真实 demo 说明现在明确 `--issue-template` 会打印带 Acceptance Criteria 的 candidate 晋级 GitHub issue body。
- [cases/README.md](../../cases/README.md) 现在说明 Issue Body Template 会保留 `Acceptance Criteria`，并强调验收证据与 `promotion_evidence` 的关系。
- 官网 quickstart 现在展示 `cliany-site cases --case-id pypi-project-search --issue-template` 可生成带 `Acceptance Criteria` 的 candidate promotion issue。

## 案例库映射

- 本版本不改变 [cases/manifest.json](../../cases/manifest.json) 的案例状态，不新增 active adapter 包，不生成或提交 `~/.cliany-site/` 运行时状态。
- Active 案例保持 4 个：SuiteCRM、Apache Jira、Apache Confluence、Apache Jenkins。
- Candidate 案例保持 3 个：PyPI、npm、crates.io 搜索工作流。
- Known gap 保持 1 个：`search-extraction-gap`。
- `pypi-project-search` 仍是下一轮首要 candidate；本版本让它的 candidate promotion issue body 明确 `adapter_package` 的完成证据要求，但不声称 adapter 包已经生成。

## 风险与兼容性

- 风险：下游工具如果对 issue body 做精确快照，需要同步新增 `Acceptance Criteria` 小节；结构是新增内容，不删除旧小节。
- 风险：Acceptance Criteria 是人工和自动化交接合同，不是任务完成证明；candidate 晋级 active 仍需真实 adapter package、metadata validation 和 online smoke 三段证据全部 complete。
- 兼容性：不修改 adapter schema、生成代码、动作回放、CDP 连接或 case manifest 状态。
- 兼容性：发布节奏继续遵循每日 verified release train 和标准 release flow。
