# v0.16.204 发布草案

**目标版本：** `0.16.204`
**提交范围：** `v0.16.203..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 publication worktree status boundary 显式化到整包摘要。

## 用户价值

`v0.16.203` 已经让 `artifact_bundle_summary` 可以展示 publication publish commands 的首尾边界。维护者在审阅 candidate issue artifacts 时，还需要确认 publication worktree status 是否为空，或者在 dirty worktree 场景下能快速看见 status 列表的首尾；目前 summary 已有 status 数量和完整 hash，但只读 summary 的工具仍无法快速确认 status 列表边界是否符合预期。

`v0.16.204` 在 `artifact_bundle_summary` 中新增 `publication_worktree_status_first_item`、`publication_worktree_status_last_item` 和 `publication_worktree_status_boundary_sha256`。这样维护者和自动化可以只读整包摘要，直接确认 publication worktree status 的首尾边界、数量和完整 hash，再决定是否展开完整 `artifact-manifest.json.publication_worktree_status` 或 `publication-handoff.json.worktree_status`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.204 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 publication worktree status boundary fields，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且 first/last item 等于 `artifact-manifest.json.publication_worktree_status` 的首尾元素；无 status 行时首尾为 `null`。

## 变更分组

### Publication worktree status boundary

- `artifact_bundle_summary` 新增 `publication_worktree_status_first_item`。
- `artifact_bundle_summary` 新增 `publication_worktree_status_last_item`。
- `artifact_bundle_summary` 新增 `publication_worktree_status_boundary_sha256`。
- Boundary 从 `artifact-manifest.json.publication_worktree_status` 的顺序派生；无 status 行时首尾为 `null`。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 publication worktree status first/last item 与 boundary hash。
- 现有 `publication_worktree_status_count`、`publication_worktree_status_sha256`、publication publish command boundary、publication ref context boundary、create issues safety contract boundary、review checklist boundary、validation command boundary、release draft boundary、candidate issue gate boundary 和 publication summary 字段保持兼容。
- 不改变 publication audit、worktree status 采集、remote check、publish commands 内容、release draft validation 或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 publication worktree status boundary fields。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的 publication worktree status boundary 字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 `publication_worktree_status`，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `publication_worktree_status_first_item`、`publication_worktree_status_last_item`、`publication_worktree_status_boundary_sha256`、`publication_worktree_status_count` 和 `publication_worktree_status_sha256` 快速确认 worktree status 是否漂移。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## Release Notes 摘要

`v0.16.204` adds publication worktree status first/last item and boundary hash fields to the artifact bundle summary, helping tools inspect worktree status boundaries before expanding the full manifest.
