# v0.16.276 发布草案

**目标版本：** `0.16.276`
**提交范围：** `v0.16.275..HEAD`
**发布主题：** 将 candidate issue 的首要交接改为 preflight-first，避免公开 issue 提示维护者跳过 live-LLM gate。

完成 version bump 后，target-mode readiness 还会检查占位锚点 **提交范围：** `v0.16.276..HEAD`；该锚点不改变实际 release diff。

## 用户价值

candidate 的 `adapter_package` 需要先通过 `cliany-site doctor --llm-live --json`。v0.16.275 已让 evidence bundle、promotion plan 和案例列表把 doctor 作为下一条可执行命令，但 generated issue template 的 Primary Evidence Task 仍在 Primary Runbook 前以自然语言先写出 `explore`。复制或只阅读首段的贡献者仍可能误解这个顺序。

本版使 issue template 与其他 handoff surface 使用同一个 executable-next contract：先显示 `next_step`、`next_command` 和 next handoff；`task_command`、其 source、missing state 与 handoff 仍单独保留，说明预检成功后的动作。已经完成 adapter package 的任务直接给出 metadata validation 或 online smoke，不会被额外 preflight 阻塞。

该修复不执行 live LLM、不会生成 adapter archive，也不把 candidate 表述为 active 或在线成功。

## 变更分组

### Public issue handoff

- `cliany-site cases --case-id <id> --issue-template` 在 Primary Evidence Task 中明确写出 `Next executable step` 与 `Next executable command`。
- `adapter_package` 的第一条可执行命令是 `cliany-site doctor --llm-live --json`；其 explore 命令只出现在 `Task command after preflight` 和有序 runbook 中。
- issue-template JSON 的 primary task 暴露 `next_*` 与 `task_*` 字段，供不解析 Markdown 的 automation 使用。
- 真实的 PyPI、npm 与 crates.io public promotion issues 将在发布后按新模板同步；它们仍保持 pending，直到真实证据满足 acceptance criteria。

### Evidence boundaries

- `cases/README.md`、`cases/manifest.json`、`search-extraction-gap` 和 candidate 状态不变。
- 当 `llm_live_preflight_not_ready` 或 `E_LLM_UNAVAILABLE` 出现时，issue 仍要求停止 explore、附上 doctor evidence 并保持 `adapter_package` pending 或 blocked。

## 案例库映射

`pypi-project-search`、`npm-package-search` 与 `crates-io-crate-search` 均保持 candidate。它们的 adapter package、metadata validation 与 online smoke 仍需要真实 package path、local validation 和 read-only JSON evidence；本版只让 Issue 14、15、16 的交接顺序正确，不替代这些证据。

本版不写入 `~/.cliany-site/`，不新建 adapter、session、snapshot 或 package artifact，也不声称第三方站点返回了数据。

## 风险与兼容性

- 已经保存的外部 issue body 必须用新模板显式同步；未经同步的历史文本仍可能包含旧的自然语言顺序。
- JSON consumers 应将 `next_command` 视为可立即执行的操作，将 `task_command` 视为通过任何先决 gate 后的操作。
- metadata validation 和 online smoke 的 next/task command 相同；这一点防止把 live LLM preflight 误加到已经完成 adapter package 的流程。
- 预检失败仍是 `llm_live_preflight_not_ready`，不得伪造 LLM、adapter asset、GitHub/PyPI package 或 online smoke evidence。

## 发版步骤

1. 完成版本、CHANGELOG、官网基线、reviewed GitHub Release Notes 和本草案，在干净 HEAD 运行 target-mode readiness。
2. 提交并推送 `master`，等待 GitHub CI 和 Embodied CI 都成功；任一失败都不创建 tag。
3. 仅在 CI 成功后创建本地 `git tag v0.16.276`，运行 `uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.276 --remote --remote-name origin`。
4. 通过 tag gate 后推送 tag，等待 GitHub Release workflow，逐版本检查 GitHub Release 与 PyPI `0.16.276`。
5. 运行 `gh release edit v0.16.276 --repo pearjelly/cliany.site --notes-file docs/releases/v0.16.276-github-release.md`，随后更新 GitHub Issue 14、15、16 为新 issue template，再用 `gh release view v0.16.276 --repo pearjelly/cliany.site --json body` 检查 release notes。
6. 在 `site/` 运行 `vercel link --yes --project cliany.site`、`vercel --prod --yes` 与 `vercel inspect www.cliany.site --wait --timeout 90s`，然后浏览 www.cliany.site。
7. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --strict --remote --distribution --json`；audit 成功后才报告完成。

## Release Notes 摘要

`v0.16.276` makes candidate-promotion issue templates put the required live-LLM preflight first, while retaining the post-preflight task command explicitly for both humans and automation.
