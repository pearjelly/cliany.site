# v0.16.221 发布草案

**目标版本：** `0.16.221`
**提交范围：** `v0.16.220..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让周初计划直接展示 candidate 晋级证据汇总。

## 用户价值

`v0.16.220` 已经让 `validate_cases.py` 输出 `promotion_evidence_summary`，维护者可以从案例库报告看到 candidate 晋级任务的 pending/blocked/complete 状态和 primary next action。周初维护流程的主入口仍是 `plan_next_iteration.py`，如果这份计划看不到同样的证据汇总，维护者还需要额外打开 case catalog report 才能决定下一块切片。

`v0.16.221` 把 candidate 晋级证据汇总接入 `plan_next_iteration.py`。计划 JSON、默认文本输出和 Markdown report 都会输出 `case_promotion_evidence_summary`；Markdown report 新增 `Candidate Promotion Evidence Summary` 小节，展示 candidate/task 数量、pending/blocked/complete 数量、primary next action 和任务明细。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.221 --json` 输出 `case_promotion_evidence_summary`；`python scripts/plan_next_iteration.py --target-version 0.16.221 --report /tmp/cliany-next-iteration.md` 生成 `Candidate Promotion Evidence Summary`；当前真实案例库显示 3 个 candidate、9 个 pending 子任务，并把 PyPI adapter package 作为 primary next action。

## 变更分组

### Next iteration evidence summary

- `IterationPlan` 新增 `case_promotion_evidence_summary`。
- `plan_next_iteration.py` 会优先复用 readiness cases 中的 `promotion_evidence_summary`，测试 fixture 或旧对象缺少该字段时按 candidate cases 现算。
- Summary 会展示 `candidate_count`、`task_count`、`pending_count`、`blocked_count`、`complete_count` 和 `primary_next_action`。
- 默认文本输出新增 `case_promotion_evidence_summary`。
- Markdown report 新增 `Candidate Promotion Evidence Summary` 小节，列出 summary 指标和任务明细。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明周初计划会展示 `case_promotion_evidence_summary`。
- `tests/test_plan_next_iteration.py` 覆盖 JSON、默认文本输入路径和 Markdown report。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护循环文档中的字段说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让周初计划能直接读到 candidate 晋级证据看板。

## 风险与兼容性

- `case_promotion_evidence_summary` 是新增计划字段；不移除既有 `candidate_promotions` 或 `promotion_evidence`。
- 本版本不改变 `artifact-manifest.json` schema，也不改变 `issue-metadata.json` 内容。
- Summary 只汇总现有 manifest 状态，不会伪造 adapter package、metadata validation 或 online smoke 已完成。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.221..HEAD`。最终带 `--release-tag v0.16.221` 的 strict gate 仍以本草案顶部的 `v0.16.220..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov

python scripts/plan_next_iteration.py --target-version 0.16.221 --json
python scripts/plan_next_iteration.py --target-version 0.16.221 --report /tmp/cliany-next-iteration.md
python scripts/release_readiness.py --target-version 0.16.221 --json
python scripts/release_readiness.py --strict --target-version 0.16.221 --report /tmp/cliany-release-readiness.md
```

## 发版步骤

1. 确认 focused pytest 和 ruff 通过。
2. 确认 `plan_next_iteration.py --json` 输出 `case_promotion_evidence_summary`。
3. 确认 Markdown report 包含 `Candidate Promotion Evidence Summary`。
4. 确认默认文本输出展示 `case_promotion_evidence_summary`。
5. 确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.221`。
6. 确认 `pyproject.toml` 版本更新为 `0.16.221`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.221`。
8. 创建 tag：`git tag v0.16.221`。
