# v0.16.258 发布草案

**目标版本：** `0.16.258`
**提交范围：** `v0.16.257..HEAD`
**当前状态：** 2026-07-07 release slice 准备中；本版把 doctor preflight `preflight_state` 输出提升为 cases、validation、planner 和 issue metadata 都能读取的稳定契约。

## 用户价值

`v0.16.257` 已经让维护者可以用 `scripts/extract_doctor_preflight_evidence.py` 把 `cliany-site doctor --llm-live --json` 输出整理成 `preflight_state`，并把状态分成 `ready`、`blocked` 和 `missing_fields`。但维护工具如果只读取 `cliany-site cases --case-id <id> --evidence-bundle --json`、`scripts/validate_cases.py --json`、planner `--handoff-json` 或 `issue-metadata.json`，仍然需要知道 extractor 会产生哪些 `preflight_state.*` 字段，以及合法状态有哪些。

`v0.16.258` 把这层契约显式放进候选案例 handoff：`doctor_preflight_state_fields` 固定为 `preflight_state.status`、`preflight_state.ready_for_adapter_package`、`preflight_state.primary_reason`、`preflight_state.reason_codes`、`preflight_state.next_action`；`doctor_preflight_state_statuses` 固定为 `ready`、`blocked`、`missing_fields`。同一 handoff 还提供 fields/statuses 的 SHA-256，让自动化可以检查 state contract 是否漂移，而不需要打开 extractor 源码或解析 Markdown。

这让 `pypi-project-search/adapter_package` 继续保持谨慎：只有 doctor preflight evidence 的 `preflight_state.status=ready` 才进入真实 `explore` 和 adapter package；`blocked` 时贴 blocker evidence；`missing_fields` 时先修 selector/doctor JSON 证据。候选案例仍不提前标成 active，也不会把 runtime 状态写入仓库。

## 变更范围

- `src/cliany_site/commands/cases.py`：cases evidence bundle、promotion evidence summary 和 issue-template primary task 现在输出 `doctor_preflight_state_fields`、`doctor_preflight_state_fields_sha256`、`doctor_preflight_state_statuses` 和 `doctor_preflight_state_statuses_sha256`。
- `scripts/validate_cases.py`：case validation promotion summary 中的 primary task 和 pending task 现在暴露同一 doctor preflight state contract；非 preflight 任务输出空列表及空列表 hash。
- `scripts/plan_next_iteration.py`：candidate promotion JSON、issue metadata、compact `--handoff-json` primary candidate、task normalization 和 planner handoffs 现在透传同一 state contract。
- `README.md`、`docs/weekly-maintainer-loop.md`、`docs/candidate-promotion-runbook.md`、`site/`：维护者文档和官网说明现在明确 `doctor_preflight_state_fields`、`doctor_preflight_state_statuses` 与 `preflight_state.status` 的使用方式。
- `CHANGELOG.md`、`pyproject.toml`：准备 `0.16.258` 发布记录和版本号。
- `tests/test_cases_command.py`、`tests/test_validate_cases.py`、`tests/test_plan_next_iteration.py`：覆盖 cases、validation、planner handoff 和 issue metadata 的新增 state contract 字段及 hash。

## 风险与兼容性

- 不改变 `scripts/extract_doctor_preflight_evidence.py` 的 `preflight_state` 判定逻辑；本版只把既有输出契约暴露到上游 handoff。
- 不改变 CLI runtime、adapter schema、AXTree selector、action replay、session 存储或 adapter codegen。
- 不修改 [cases/manifest.json](../../cases/manifest.json) 的 candidate 状态，也不把 `pypi-project-search` 提前标成 active。
- 新增 JSON 字段是向后兼容扩展；依赖精确字典断言的下游测试需要接受新增字段。
- 真实 candidate promotion 仍依赖本地 Chrome、可用 LLM provider 和 `~/.cliany-site/` runtime 目录；默认验证仍走离线回归。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的案例状态；案例说明仍落在 [cases/README.md](../../cases/README.md)。
- 当前案例库保持 active 4 个、candidate 3 个、known gap 1 个。
- Known gap 保持 1 个：`search-extraction-gap`。
- 首要 candidate 仍是 `pypi-project-search`，首要任务仍是 `adapter_package`。
- 预期 adapter package 名称为 `pypi.org-<version>.cliany-adapter.tar.gz`。
- Metadata validation 命令为 `python scripts/validate_cases.py --packages-dir ~/.cliany-site/packages --include-candidate-packages --strict`。
- Online smoke 命令为 `cliany-site pypi.org search-projects --query cliany-site --limit 5 --json`。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.257..HEAD`。完成 version bump 后，target-mode readiness 也会查找占位锚点 **提交范围：** `v0.16.258..HEAD`；该锚点只表示 tag 发布后不应再有未发布差异，不改变本版实际提交范围。创建本地 tag 后，使用 local tag-mode readiness 复核最终 tag 语义。

```bash
uv run --extra dev --frozen python scripts/release_readiness.py --strict --target-version 0.16.258 --remote --remote-name origin --json
```

## 发版步骤

1. 确认 `v0.16.257` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 运行 target-mode readiness，确认 release draft、CHANGELOG、版本号和发布门禁通过。
3. 提交并推送 `master`。
4. 创建并推送 `v0.16.258` tag。
5. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
6. 按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署官网，并运行 `cd site && vercel inspect www.cliany.site --wait --timeout 90s` 复核 alias。
7. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json`，确认 GitHub Release、PyPI 和官网发布状态。

## Release Notes 摘要

v0.16.258 makes the doctor preflight state machine discoverable from candidate promotion handoffs. Cases evidence bundles, case validation summaries, planner candidate metadata, generated issue metadata, and compact planner handoffs now expose the `preflight_state.*` field contract plus the allowed `ready`, `blocked`, and `missing_fields` statuses with stable hashes. Maintainers and bots can now decide whether to run candidate `explore`, attach blocker evidence, or repair missing doctor fields without opening the extractor source or parsing Markdown.
