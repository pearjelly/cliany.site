# v0.16.236 发布草案

**目标版本：** `0.16.236`
**提交范围：** `v0.16.235..HEAD`
**当前阻塞：** 需要完成 `doctor --llm-live` 预检切片，并重新运行 release readiness、case validation、文档/官网测试、完整 pytest、官网浏览器检查和标准发布流程。

## 用户价值

v0.16.236 聚焦 `explore` 前的 LLM provider 可用性确认。维护者继续推进 `pypi-project-search` candidate 时，真实 PyPI 探索仍被上游 `502 Bad Gateway` 阻断；v0.16.235 已经让 `explore --json` 返回干净的 `E_LLM_UNAVAILABLE`，但用户仍要等到进入探索循环后才知道 provider 当前不可用。

本版本新增 opt-in 的 `cliany-site doctor --llm-live --json`。默认 `doctor` 仍只做本地 key/config/CDP/目录检查，不产生真实 LLM 调用；当用户明确追加 `--llm-live` 时，CLI 会做一次轻量 provider 预检，并把网关、限流或服务不可用故障输出为 `llm_live` warning。这样用户可以在长时间 `explore` 前先判断“稍后重试或切换 provider”，而不会误以为 adapter、AXTree selector map 或案例证据已经损坏。

## 变更分组

- CLI：`doctor` 新增 `--llm-live` 选项，只有显式指定时才实际调用一次 LLM provider。
- 健康检查：`llm_live` check 成功时返回 `ok`；上游不可用时返回 `warning`，并包含 `details.error_code=E_LLM_UNAVAILABLE`、`retryable`、`status_code` 和 `phase=llm_preflight`。
- 能力摘要：`summary.ready_for_explore` 和 `capabilities.generate_adapters` 会把失败的 live preflight 标记为 `llm_live` should-fix blocker。
- 文档：README、README.zh、官网 quickstart 和官网文档说明 live preflight 的 opt-in 边界和推荐动作。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的任何案例状态。
- 当前真实案例库仍保持 active 4 个、candidate 3 个、known gap 1 个。
- Known gap 保持 1 个：`search-extraction-gap`。
- [cases/README.md](../../cases/README.md) 的候选晋级规则仍然适用：`adapter_package`、`metadata_validation` 和 `online_smoke` 三段证据未全部 complete 前，不把 candidate 标记为 active。
- `pypi-project-search` 仍是首要 candidate；本版真实 PyPI 探索仍被 LLM 上游 502 阻断，因此只新增 provider preflight，不声称 adapter package 已生成。

## 风险与兼容性

- 默认 `cliany-site doctor --json` 不会真实调用 LLM provider，避免改变 CI、PR 门禁和离线贡献者路径。
- `doctor --llm-live` 可能消耗一次 provider 调用；因此该检查保持显式 opt-in。
- `llm_live` warning 会阻止 `ready_for_explore` 和 `generate_adapters.ready`，但不会影响 demo adapter quickstart。
- 生成 adapter schema、action runtime、AXTree 语义匹配和 codegen 逻辑不变。
- `E_LLM_UNAVAILABLE` 表示 provider 层可重试故障，不代表用户登录态、selector map 或 adapter package 损坏。
