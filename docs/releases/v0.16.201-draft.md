# v0.16.201 发布草案

**目标版本：** `0.16.201`
**提交范围：** `v0.16.200..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 create issues safety contract boundary 显式化到整包摘要。

## 用户价值

`v0.16.200` 已经让 `artifact_bundle_summary` 可以展示 review checklist 的首尾边界。维护者在审阅 candidate issue artifacts 时，还需要确认 `create-issues.sh` 的安全契约是否仍从 dry-run 支持开始，并以 preflight JSON 输出位置结束；目前 summary 已有 create issues safety contract 的 key count 和完整 hash，但只读 summary 的工具仍无法快速确认契约 key 边界是否符合预期。

`v0.16.201` 在 `artifact_bundle_summary` 中新增 `create_issues_safety_contract_first_key`、`create_issues_safety_contract_last_key` 和 `create_issues_safety_contract_key_boundary_sha256`。这样维护者和自动化可以只读整包摘要，直接确认 create issues safety contract 的 key 首尾边界、数量和完整 hash，再决定是否展开完整 `artifact-manifest.json.create_issues_safety`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.201 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 create issues safety contract boundary fields，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且 first/last key 等于 safety contract 的首尾 key。

## 变更分组

### Create issues safety contract boundary

- `artifact_bundle_summary` 新增 `create_issues_safety_contract_first_key`。
- `artifact_bundle_summary` 新增 `create_issues_safety_contract_last_key`。
- `artifact_bundle_summary` 新增 `create_issues_safety_contract_key_boundary_sha256`。
- Boundary 从 create issues safety contract 的稳定 key 顺序派生；无 contract keys 时首尾为 `null`。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 create issues safety contract first/last key 与 boundary hash。
- 现有 `create_issues_safety_contract_key_count`、`create_issues_safety_contract_sha256`、review checklist boundary、validation command boundary、release draft boundary、candidate issue gate boundary 和 publication summary 字段保持兼容。
- 不改变 `create-issues.sh`、dry-run mode、publication preflight、release draft validation 或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 create issues safety contract boundary fields。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的 create issues safety contract boundary 字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 create issues safety contract，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `create_issues_safety_contract_first_key`、`create_issues_safety_contract_last_key`、`create_issues_safety_contract_key_boundary_sha256`、`create_issues_safety_contract_key_count` 和 `create_issues_safety_contract_sha256` 快速确认 safety contract 是否漂移。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.201 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `create_issues_safety_contract_first_key`、`create_issues_safety_contract_last_key` 和 `create_issues_safety_contract_key_boundary_sha256`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示这三个字段。
4. 确认 `create_issues_safety_contract_first_key` 与 `create_issues_safety_contract_last_key` 等于 safety contract 的首尾 key。
5. 确认 `create_issues_safety_contract_key_count`、`create_issues_safety_contract_sha256` 和完整 safety contract 仍保持同源。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.201`。
8. 将 `pyproject.toml` 版本更新为 `0.16.201`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.201`。
10. 创建 tag：`git tag v0.16.201`。
