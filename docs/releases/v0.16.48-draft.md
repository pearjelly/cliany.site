# v0.16.48 发布草案

**目标版本：** `0.16.48`
**提交范围：** `v0.16.47..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts manifest 的 validation commands 覆盖 publication audit。

## 用户价值

`v0.16.47` 已经让 `artifact-manifest.json` 带有 `schema_version`。维护脚本可以先判断 manifest 格式，再解析候选案例、publication handoff、release draft handoff、复现命令和验证命令。但当前 artifacts manifest 的 `validation_commands` 仍缺少 publication audit 命令；维护者需要从 plan JSON 或 publication handoff 里再找一次如何复核发布可见性。

`v0.16.48` 让 `artifact-manifest.json` 的 `validation_commands` 包含 `python scripts/check_release_publication.py --json`。这样候选 issue artifacts 包可以直接告诉维护者：创建 issue 前先重新运行 publication audit，确认最新本地 release 是否已经可见，或者明确仍需先 push branch/tag。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.48 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `artifact-manifest.json` 的 `validation_commands` 包含 `python scripts/check_release_publication.py --json`；每周维护者循环文档说明这条 publication audit 命令。

## 变更分组

### Candidate artifacts manifest publication validation

- `artifact-manifest.json` 的 `validation_commands` 新增 `python scripts/check_release_publication.py --json`。
- Artifacts manifest 继续保留 `schema_version: 1` 和既有 publication/release draft handoff 字段。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 manifest validation commands 会包含 publication audit。
- `tests/test_plan_next_iteration.py` 覆盖 manifest validation commands。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强候选 issue artifacts manifest 的验证命令清单。

## 风险与兼容性

- `validation_commands` 是既有字段，本版本只追加 publication audit 命令。
- 不改变 plan JSON schema，不改变 `publication-handoff.json` 或 `release-draft-handoff.json`。
- 不自动执行命令，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- 这条命令只读当前 Git 状态和 release refs，用于候选 issue 创建前的人工/自动化复核。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.48..HEAD`。最终带 `--release-tag v0.16.48` 的 strict gate 仍以本草案顶部的 `v0.16.47..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.48 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.48 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.48 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `artifact-manifest.json`。
2. 确认 manifest 的 `validation_commands` 包含 `python scripts/check_release_publication.py --json`。
3. 确认每周维护者循环文档说明 manifest validation commands 包含 publication audit。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.48`。
6. 将 `pyproject.toml` 版本更新为 `0.16.48`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.48`。
8. 创建 tag：`git tag v0.16.48`。

## Release Notes 摘要

`v0.16.48` adds the publication audit command to generated `artifact-manifest.json` validation commands, helping maintainers re-check release visibility before creating candidate issues.
