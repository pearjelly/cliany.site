# v0.16.85 发布草案

**目标版本：** `0.16.85`
**提交范围：** `v0.16.84..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 `artifact_bundle_summary` 直接携带 candidate issue gate 的人工审阅布尔值。

## 用户价值

`v0.16.84` 已经让 `artifact_bundle_summary` 能检测 candidate issue gate summary 是否漂移。但维护者在展开完整 manifest 前，也需要快速知道当前 gate 是否仍然需要人工审阅。`can_create_issues` 只表达能否创建候选 issue，不能单独表达 release draft 或 publication gate 是否还需要维护者确认。

`v0.16.85` 在 `artifact_bundle_summary` 中新增 `requires_maintainer_review`，与 `candidate_issue_gate.requires_maintainer_review` 保持一致。Artifacts `README.md` 的 `Artifact Bundle Summary` 也展示该字段，让维护者和自动化能先判断是否需要人工审阅，再决定是否展开完整 `candidate_issue_gate`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.85 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `requires_maintainer_review`，并与 manifest 中的 `candidate_issue_gate.requires_maintainer_review` 一致；README 的 `Artifact Bundle Summary` 也展示这个字段。

## 变更分组

### Artifact bundle maintainer review flag

- `artifact_bundle_summary` 新增 `requires_maintainer_review`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示 maintainer review flag。
- 现有 `can_create_issues` 保持不变，两个布尔值共同表达自动化能否创建 issue 与是否仍需人工审阅。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 `requires_maintainer_review`。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate issue artifacts 的整包摘要更适合自动化判断是否需要维护者审阅。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary`，不改变 `candidate_issue_gate.requires_maintainer_review` 的既有字段语义。
- 字段来自现有 gate 结构，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.85..HEAD`。最终带 `--release-tag v0.16.85` 的 strict gate 仍以本草案顶部的 `v0.16.84..HEAD` 作为实际 release diff 范围。

```bash
python scripts/plan_next_iteration.py --target-version 0.16.85 --json

```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.85 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `requires_maintainer_review`。
3. 确认该字段与 `artifact-manifest.json.candidate_issue_gate.requires_maintainer_review` 一致。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示该字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.85`。
7. 将 `pyproject.toml` 版本更新为 `0.16.85`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.85`。
9. 创建 tag：`git tag v0.16.85`。

## Release Notes 摘要

`v0.16.85` adds the maintainer-review flag to artifact bundle summaries, helping tools detect review requirements before reading full manifests.
