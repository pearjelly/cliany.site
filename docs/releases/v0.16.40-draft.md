# v0.16.40 发布草案

**目标版本：** `0.16.40`
**提交范围：** `v0.16.39..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts 写出结构化 artifact manifest。

## 用户价值

`v0.16.39` 已经把 release draft handoff 独立成 `release-draft-handoff.json`，候选 issue artifacts 目录里现在有多个 JSON、Markdown body 和 `create-issues.sh`。维护者或自动化工具如果想知道“这个目录里哪些文件应该按什么顺序看、如何重新验证”，仍然需要扫描目录或读取 README。

`v0.16.40` 在 `scripts/plan_next_iteration.py --issues-dir` 输出目录中新增 `artifact-manifest.json`，结构化保存 target version、文件名、review order 和 validation commands。维护者可以先读这份 manifest，再按顺序审阅 publication handoff、release draft handoff、issue metadata、issue body 和创建脚本。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.40 --issues-dir /tmp/cliany-candidate-issues --json` 写出的目录包含 `artifact-manifest.json`；该 JSON 包含 `target_version`、`files`、`review_order` 和 `validation_commands`；artifacts `README.md` 的 Files 小节说明这个 manifest。

## 变更分组

### Candidate artifacts manifest

- `scripts/plan_next_iteration.py --issues-dir` 新增 `artifact-manifest.json`。
- Manifest 包含 target version、文件名、review order 和 validation commands。
- Artifacts `README.md` 的 Files 小节说明 `artifact-manifest.json` 的用途。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 candidate issue artifacts 会包含 `artifact-manifest.json`。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 内容和 README 文件列表。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强候选 issue artifacts 的结构化入口。

## 风险与兼容性

- `artifact-manifest.json` 是新增 artifact，不改变 plan JSON schema。
- `publication-handoff.json`、`release-draft-handoff.json`、`issue-metadata.json` 和 `create-issues.sh` 结构保持兼容。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- Manifest 只列出生成器已经写出的文件和验证命令，不推断外部状态。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.40 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `artifact-manifest.json`。
2. 确认 manifest 包含 `target_version`、`files`、`review_order` 和 `validation_commands`。
3. 确认 artifacts `README.md` 的 Files 小节说明 `artifact-manifest.json`。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.40`。
6. 将 `pyproject.toml` 版本更新为 `0.16.40`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.40`。
8. 创建 tag：`git tag v0.16.40`。

## Release Notes 摘要

`v0.16.40` adds an `artifact-manifest.json` index to generated candidate issue artifacts, so maintainers and tools can discover the bundle files, review order, and validation commands without scanning the directory.
