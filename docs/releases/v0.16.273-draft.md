# v0.16.273 发布草案

**目标版本：** `0.16.273`
**提交范围：** `v0.16.272..HEAD`
**当前状态：** 2026-07-23 release slice；让 `doctor` 将可安装的 active demo 直接变成首次成功路径。

## 用户价值

此前 `cases/manifest.json` 已包含四个带固定 GitHub Release HTTPS URL 和 SHA-256 的 `active` demo，但 `doctor --json` 仍把 `demo_adapter_quickstart` 标记为不可用。新用户因此只能看到案例目录，无法从环境检查直接获得一个可复制的安装、校验和只读执行路径。

本版从打包的案例目录选择第一个同时具备固定 HTTPS + SHA-256 安装命令、adapter domain 和只读命令、且无需登录的 `active` case。目前该 case 是 ASF Jira。`demo_adapter_quickstart` 返回顺序固定的安装、`verify`、只读命令，并标记 `available=true`、`deprecated=false`；human `doctor` 输出也展示同一条路径。candidate、需要登录的 SuiteCRM 和不存在的历史归档都不会进入该输出。

2026-07-23 对 Jira release archive 的隔离 `market install --sha256 --json` 已确认当前 CLI 能下载、校验并读取正确的 domain、version 与必需文件。这不等同于第三方 Jira 页面或业务命令的在线成功。当前 `cliany-site doctor --llm-live --json` 仍返回可重试的 `E_LLM_UNAVAILABLE`，即 `llm_live_preflight_not_ready`，因此没有生成 candidate adapter、没有将 candidate 提升为 active，也没有伪造 online smoke 证据。

## 变更分组

### Doctor 首次成功路径

- `src/cliany_site/commands/doctor.py` 复用打包案例清单，选择可安装、无需登录的 active case，并保留找不到合格案例时的旧兼容 fallback。
- `demo_adapter_quickstart.commands`、`install_command`、`verify_command` 和 `read_only_command` 让 JSON 客户端无需猜测固定 asset、校验命令或业务命令。
- `ready_for_demo_adapters` 仅在已有 adapter 运行环境和真实 active asset 路径都可用时为真；candidate 绝不会被选择。
- metadata v3 schema 允许既有的 `capability=api`、字符串 `api_endpoints` 和字符串 `commands`，因此 `verify` 继续执行 schema、安全扫描和 market manifest 哈希检查，而不会把已支持的 API demo 误判为 schema error。

### 文档与官网

- 10 分钟指南和官网说明：当 `ready_for_demo_adapters=true` 时，按 `demo_adapter_quickstart.commands` 的顺序运行；否则回退到案例目录。
- 路线图把 active demo 的 HTTPS/SHA-256 资产与 doctor 路径同步为已完成基础能力，同时继续把 candidate 晋级和 live LLM preflight 保留为未完成工作。

## 案例库映射

`apache-jira-issues` 是本版 doctor 引导的 active case，但其真实业务数据仍由用户执行最后一条 JSON 命令时决定。完整案例说明见 `cases/README.md`。`suitecrm-accounts`、`apache-confluence-search` 和 `apache-jenkins-jobs` 继续保持 active；`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 继续保持 candidate；`search-extraction-gap` 继续是 known-gap。

本版不声明新的 adapter asset，也不修改运行时 `~/.cliany-site/` 中的 adapters、sessions 或 snapshots。

## 风险与兼容性

- `demo_adapter_quickstart` 保持对象类型和 `commands` 列表类型。无合格 active case 或 manifest 读取失败时仍返回旧的 `available=false`、`deprecated=true` fallback，而不是猜测 URL。
- 有可用 active case 时，`reason` 与 `replacement` 不再是正向路径的必填字段；客户端应以 `available`、`deprecated` 和明确的命令字段判断当前路径。
- 静态 `verify` 成功只证明安装后的 adapter 完整性；API-capability 兼容只接受已有的字符串或对象 endpoint、字符串或对象 command 声明，不会跳过 schema、安全扫描或 manifest 哈希检查。第三方页面、网络和实际 rows 随运行时变化，最后一条只读命令不得在 release notes 中提前宣称成功。
- live LLM preflight 的 `E_LLM_UNAVAILABLE` 保持 candidate 生成的 blocker，不因 active demo 路径可用而放宽。

## 发版步骤

1. 在 `[Unreleased]` 保留本版条目、compare link 为 `v0.16.272...HEAD` 时运行 target-mode readiness。
2. Finalize Changelog：清空 `[Unreleased]`，新增 `## [0.16.273] - 2026-07-23`，并更新 compare links。
3. 提交并推送 `master`，等待 GitHub CI 与 Embodied CI 成功。
4. 创建 `git tag v0.16.273`，然后运行 `uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.273 --remote --remote-name origin`。
5. 仅在该 gate 通过后推送 tag，等待 GitHub Release 与 PyPI 发布。
6. 在 `site/` 运行 `vercel link --yes --project cliany.site`、`vercel --prod --yes` 和 `vercel inspect www.cliany.site --wait --timeout 90s`。
7. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --strict --remote --distribution --json`。

## Release Notes 摘要

`v0.16.273` makes `doctor` point new users to a real, pinned active-demo install, verify, and read-only path while keeping candidates, live LLM failures, and third-party online results honestly separated.
