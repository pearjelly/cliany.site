# v0.16.259 发布草案

**目标版本：** `0.16.259`
**提交范围：** `v0.16.258..HEAD`
**当前状态：** 2026-07-07 release slice 准备中；本版把真实 `cliany-site doctor --llm-live --json` preflight 输出接入 candidate evidence bundle 和 issue template。

## 用户价值

`v0.16.258` 已经让维护工具知道 doctor preflight state contract 长什么样，但真实推进 `pypi-project-search/adapter_package` 时仍有一个实际摩擦：当 live LLM/CDP preflight 失败，维护者需要先保存 doctor JSON，再手动运行 extractor 或复制字段，最后再把 blocker evidence 拼到 candidate issue。

本版把这条路径收进 `cliany-site cases` 本身。维护者可以先运行：

```bash
cliany-site doctor --llm-live --json > /tmp/cliany-doctor-preflight.json
```

然后运行：

```bash
cliany-site cases --case-id pypi-project-search --evidence-bundle --doctor-json /tmp/cliany-doctor-preflight.json --json
cliany-site cases --case-id pypi-project-search --issue-template --doctor-json /tmp/cliany-doctor-preflight.json
```

输出会直接包含 `doctor_preflight_evidence_values`、`doctor_preflight_evidence_ok`、`doctor_preflight_evidence_missing_count`、`doctor_preflight_evidence_values_sha256` 和 `doctor_preflight_state`，并把同一份 evidence 挂到 primary `adapter_package` task。这样 live preflight blocked 时，维护者可以把真实 blocker evidence 贴回 issue，而不是伪造 adapter package 证据。

## 变更范围

- `src/cliany_site/commands/cases.py`：新增 `--doctor-json <path>`，只在 `--issue-template` 或 `--evidence-bundle` 模式下读取本地 doctor JSON；解析现有 `DOCTOR_PREFLIGHT_EVIDENCE_SELECTORS`，生成 actual values、missing field summary、hashes 和 `doctor_preflight_state`。
- `tests/test_cases_command.py`：新增 TDD 覆盖，验证 `E_LLM_UNAVAILABLE` doctor JSON 会渲染为 blocked preflight state，并出现在 JSON evidence bundle 与 Markdown evidence bundle。
- `README.md`、`cases/README.md`、`site/`：补充 `doctor --llm-live --json > /tmp/...` 到 `cases --doctor-json ...` 的维护者路径。
- `CHANGELOG.md`、`pyproject.toml`：准备 `0.16.259` 发布记录和版本号。

## 风险与兼容性

- 不改变 `cases/manifest.json`，不把 `pypi-project-search` 提前晋级为 active。
- 不写入 runtime 状态到 repo；`--doctor-json` 只读取用户显式传入的本地 JSON 文件。
- 不改变 adapter codegen、AXTree 语义匹配、action replay 或 session 存储。
- 新增 JSON 字段是向后兼容扩展；旧的 `--evidence-bundle` 输出在未传 `--doctor-json` 时保持原有字段形状。
- 如果 doctor JSON 缺少必要字段，输出 `doctor_preflight_state.status=missing_fields`，提醒先修证据而不是继续 explore。

## 案例库映射

- 首要 candidate 仍是 `pypi-project-search`。
- 首要任务仍是 `adapter_package`。
- 本地 live preflight 证据显示 CDP 可用，但 LLM upstream 暂不可用：`checks[llm_live].status=warning`、`checks[llm_live].details.error_code=E_LLM_UNAVAILABLE`、`summary.capabilities.generate_adapters.ready=false`。
- 因此本版不生成 PyPI adapter package，不改变 candidate promotion evidence 状态；它改进的是 blocker evidence 采集路径。
- Metadata validation 命令仍为 `python scripts/validate_cases.py --packages-dir ~/.cliany-site/packages --include-candidate-packages --strict`。
- Online smoke 命令仍为 `cliany-site pypi.org search-projects --query cliany-site --limit 5 --json`。
- `search-extraction-gap` 继续保留为 known-gap，用来覆盖搜索结果抽取质量的离线 fixture，不在本版中改状态。

## 发版步骤

1. 确认 `v0.16.258` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 运行 target-mode readiness，确认 release draft、CHANGELOG、版本号和发布门禁通过。
3. 提交并推送 `master`。
4. 创建并推送 `v0.16.259` tag。
5. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
6. 按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署官网，并运行 `cd site && vercel inspect www.cliany.site --wait --timeout 90s` 复核 alias。
7. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json`，确认 GitHub Release、PyPI 和官网发布状态。

## Release Notes 摘要

v0.16.259 lets candidate promotion handoffs consume real doctor preflight output directly. `cliany-site cases --case-id <id> --evidence-bundle --doctor-json <path>` and `--issue-template --doctor-json <path>` now attach actual doctor evidence values, missing-field counts, SHA-256 drift signals, and `doctor_preflight_state` to the evidence bundle and primary adapter-package task. Maintainers can now capture a live `E_LLM_UNAVAILABLE` or CDP blocker as structured evidence without fabricating adapter package proof or manually stitching extractor output into issue comments.
