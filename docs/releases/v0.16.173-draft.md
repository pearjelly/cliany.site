# v0.16.173 发布草案

**目标版本：** `0.16.173`
**提交范围：** `v0.16.172..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 issue body summary 字段集合显式化到整包摘要。

## 用户价值

`v0.16.172` 已经让 `artifact_bundle_summary` 输出 issue body inventory first/last entry 和 boundary hash，维护工具可以只读 summary 检查 body inventory 的首尾边界。但如果工具想确认 `issue_body_summary` 的 schema 是否变化，仍只能读取完整 `artifact-manifest.json.issue_body_summary` 或只依赖整体 hash。

`v0.16.173` 在 `artifact_bundle_summary` 中新增 `issue_body_summary_key_count` 和 `issue_body_summary_keys_sha256`，并在 artifacts `README.md` 的 `Artifact Bundle Summary` 段落展示同源字段。这样维护工具可以只读整包摘要，先确认 issue body summary 字段集合，再结合既有 `issue_body_summary_sha256` 判断内容是否漂移。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.173 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 issue body summary schema fields，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且 `issue_body_summary_key_count` 等于 `artifact-manifest.json.issue_body_summary` 的字段数量。

## 变更分组

### Issue body summary schema

- `artifact_bundle_summary` 新增 `issue_body_summary_key_count`。
- `artifact_bundle_summary` 新增 `issue_body_summary_keys_sha256`。
- Key count/hash 从既有 `issue_body_summary` 的字段集合派生。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 issue body summary key count/hash。
- 现有 issue body inventory preview/tail/boundary、issue body summary hash、candidate count/preview/tail/boundary/hash、artifact manifest schema/key/payload 摘要、summary key 摘要、handoff 摘要和 gate 摘要保持兼容。
- 不改变 release draft validation、publication audit 或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 issue body summary schema fields。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的 issue body summary key count/hash，并断言 key count 与 `issue_body_summary` 字段数量同源。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 `issue_body_summary`，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `issue_body_summary_key_count`、`issue_body_summary_keys_sha256` 和 `issue_body_summary_sha256` 快速确认 issue body summary schema 与内容是否漂移。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.173 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `issue_body_summary_key_count` 和 `issue_body_summary_keys_sha256`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示这两个字段。
4. 确认 `issue_body_summary_key_count` 等于 `artifact-manifest.json.issue_body_summary` 的字段数量。
5. 确认 `issue_body_summary_sha256`、`inventory_sha256` 和完整 `issue_body_summary` 仍保持同源。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.173`。
8. 将 `pyproject.toml` 版本更新为 `0.16.173`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.173`。
10. 创建 tag：`git tag v0.16.173`。
