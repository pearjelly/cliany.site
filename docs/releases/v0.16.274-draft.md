# v0.16.274 发布草案

**目标版本：** `0.16.274`
**提交范围：** `v0.16.273..HEAD`
**当前状态：** 待下一个可发布窗口；把“动作完成”与“数据读取完成”分开，避免生成的命令在没有结果数据时报告成功。

完成 version bump 后，target-mode readiness 还会检查占位锚点 **提交范围：** `v0.16.274..HEAD`；该锚点不改变实际 release diff。

## 用户价值

`cases/cliany-site-case-review.md` 记录了 `search`、`extract_results` 和 `read_results` 的已知缺口：命令可能只完成点击、翻页或导航，就把流程当成成功，尽管并未返回可用的标题、链接、摘要或其他结果字段。这会让自动化把“浏览器动作执行了”误认为“用户要的数据已取得”。

本版让新生成的命令以提取质量兑现读取语义。名称以 `list-`、`search-`、`read-` 或 `extract-` 开头的命令，以及 action sequence 中实际包含 `extract` action 的任何命令，都会检查 `data.quality`。默认 `expects_nonempty=true` 时，零条数据、关键字段缺失和 partial 行返回 `E_EMPTY_RESULT`；`expects_nonempty=false` 仅允许业务上合法的零匹配，绝不放过 partial 或缺失字段。

探索提示也要求这类数据命令在 `action_steps` 中具备真实的 `extract` action 和稳定字段后才可标记完成。没有稳定 selector 或字段时，探索不能把命令描述成已完成的数据读取路径。

这不是对历史 adapter 的静默修改：已安装、旧版本生成的 adapters 不会自动重写。用户重新 explore 某个命令后，生成的代码才应用本版契约。当前没有宣称 live LLM 成功；`cliany-site doctor --llm-live --json` 的 `llm_live_preflight_not_ready` / `E_LLM_UNAVAILABLE` 仍阻止 candidate 生成和晋级。

## 变更分组

### 生成与探索契约

- `src/cliany_site/codegen/templates.py` 将数据质量 gate 扩展到 `read-`、`extract-`，并按实际 `extract` action 覆盖不遵循常见命名的命令。
- 质量摘要为 empty、缺少必填字段或 partial 时，生成的命令返回既有的 `E_EMPTY_RESULT`；只有显式的、合法的零匹配可以在 `expects_nonempty=false` 下成功。
- `src/cliany_site/explorer/prompts.py` 要求 named data commands 写出真实的 `extract` action、稳定 selector 和 fields，不能只通过点击或页面导航结束探索。
- `tests/test_generated_orchestration.py` 覆盖命名的 `extract-results` 空数据失败，以及非命名约定但含 `extract` action 的 `fetch-results` 也会带上质量检查。

### 文档与官网

- README、中英文官网提示和官网文档说明数据命令的覆盖范围、合法零匹配例外，以及旧 adapter 不会被静默改写。
- 这项修复针对已知 `search-extraction-gap`，不声明新的 adapter asset、第三方在线 rows 或 live LLM 成功。

## 案例库映射

`search-extraction-gap` 继续是 `cases/manifest.json` 与 `cases/README.md` 中记录的 known-gap；本版把它转化为生成代码和 prompt contract 的回归门禁。`apache-jira-issues`、`suitecrm-accounts`、`apache-confluence-search` 和 `apache-jenkins-jobs` 的 active 状态不变；candidate 的证据要求、`llm_live_preflight_not_ready` blocker 和 promotion gate 也不变。

本版没有生成或发布新的 adapter archive，也没有修改 `~/.cliany-site/` 下的 adapters、sessions 或 snapshots。任何真实站点返回的数据仍须由用户在自己的浏览器和权限上下文中执行最后一条命令后确认。

## 风险与兼容性

- 新生成的命令若此前依赖“没有提取结果也成功”的行为，现在会以 `E_EMPTY_RESULT` 失败；这是有意消除的假成功，而不是对读取命令的静默放宽。
- `expects_nonempty=false` 仍只处理合法的零匹配；empty 之外的 quality failure（尤其是 missing 或 partial）不因该声明而成功。
- 以非标准名称生成但实际包含 `extract` action 的命令同样被覆盖，避免改名绕过数据质量检查。
- 旧 adapters 不会自动变更，因此需要重新 explore 目标命令才能获得这项保护；发布说明不把这项本地代码验证表述为第三方在线成功。
- live LLM preflight 仍可能返回 `E_LLM_UNAVAILABLE`。在 `llm_live_preflight_not_ready` 时，不得生成 candidate adapter、伪造 adapter asset 或将 candidate 提升为 active。

## 发版步骤

1. 在下一个可发布窗口，完成版本号和官网基线更新；保留本版 `[Unreleased]` 条目、compare link 为 `v0.16.273...HEAD` 时运行 target-mode readiness。
2. Finalize Changelog：清空 `[Unreleased]`，新增 `## [0.16.274] - <发布日期>`，并更新 compare links。
3. 提交并推送 `master`，等待 GitHub CI 与 Embodied CI 都成功；任一失败都不创建 tag。
4. 仅在两项 CI 成功后创建本地 `git tag v0.16.274`，然后运行 `uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.274 --remote --remote-name origin`。
5. 仅在 tag gate 通过后推送 tag，等待 GitHub Release workflow 成功，并逐版本核验 GitHub Release 与 PyPI `0.16.274`。
6. 在 `site/` 运行 `vercel link --yes --project cliany.site`、`vercel --prod --yes` 和 `vercel inspect www.cliany.site --wait --timeout 90s`。
7. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --strict --remote --distribution --json`，只在 strict publication audit 通过后报告发布完成。

## Release Notes 摘要

`v0.16.274` makes newly generated data commands fail honestly when they perform browser actions without returning usable extracted data, while preserving an explicit, narrow success path for legitimate zero-match workflows.
