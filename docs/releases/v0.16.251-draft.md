# v0.16.251 发布草案

**目标版本：** `0.16.251`
**提交范围：** `v0.16.250..HEAD`
**当前阻塞：** 2026-07-01 已经发布 `v0.16.248`、`v0.16.249` 和 `v0.16.250` 三个版本。`v0.16.251` 草案用于准备下一版，但今天不应创建或推送 `v0.16.251` tag；release readiness 必须先阻断会让当天 release 数变成 `4/3` 的目标 tag。

## 用户价值

`v0.16.250` 已经完成 GitHub Release、PyPI 和官网生产部署的标准发布闭环，并让 candidate 晋级遇到任意 `llm_live` warning/error 时停止真实 `explore`。继续迭代时，维护者会自然运行 `scripts/release_readiness.py --target-version 0.16.251 --remote --remote-name origin --json` 来准备下一版。

问题在于，当前 cadence 的机器可读状态在 `3/3` 时仍然是 `daily_release_limit_ok=true`，因为它描述的是“现在是否已经超过上限”。如果 release readiness 只读取这个当前状态，就会继续给出 `git tag v0.16.251` 和 `git push origin v0.16.251`，而创建 tag 后当天会立刻变成 `4/3`。这会让“每天 1 到 3 个可验证版本”的纪律变成事后提醒，而不是发版前门禁。

`v0.16.251` 将 release readiness 改成前瞻式检查：当目标 tag 尚不存在、当天 release 数已经到达上限、并且目标 tag 会在今天新增一个 release 时，直接把目标版本发布门禁标记为 blocked，并提示维护者等到下一天再 tag。这样每日发布节奏既保持高频，也避免为了“多发一个版本”牺牲可信度。

本轮还把这个 pause action 透传到 `scripts/plan_next_iteration.py` 的顶层 `next_actions`。维护者或自动化只读取 planner JSON 时，也能看到“等到下一天再 tag”的明确动作；如果 readiness 已经给出带当前 commit-day 计数和维护手册路径的周节奏动作，planner 不再追加一条更短的重复提醒。

为避免机器读者误执行，daily-cap blocker 存在时，release readiness 和 planner 的 `publication_tag_publish_decision.target_tag_commands` 会保持空列表，并把 `target_tag_status` 标记为 `blocked_by_daily_release_cap`。这样标准发版流程仍能展示阻塞原因和下一步，但不会同时暴露 `git tag` / `git push` 命令。

同时，`next_actions` 会把 daily-cap pause action 放在首位，并过滤掉 “create a new release tag at HEAD” 这类泛化 publication 文案。维护者打开 JSON 或 Markdown report 时，第一眼看到的就是“今天停止 tag、等到下一天”，而不是一个需要再结合 blocker 才能解释的 release tag 提醒。

Candidate issue gate 也会把非草案类 release readiness blocker 纳入 `release_readiness_blockers` reason code。这样最新 release 已公开、草案已通过但目标版本仍因 daily cap blocked 时，候选 issue artifacts 会显示 `review_required`、给出 pause action，并在 evidence 中保留 `release_readiness_primary_blocker`，而不是在目标 release 不能 tag 的情况下显示无待办的 `ready`。

候选 issue artifacts 还会把 `release_readiness_blocker_count`、`release_readiness_primary_blocker` 和 `release_readiness_blockers_sha256` 提升到 `publication-handoff.json` 顶层，并写入 README 的 `Candidate Issue Gate Quick Summary`。维护者只读 handoff 或 quick summary 时，就能直接看到今天为什么不能 tag，不必展开嵌套的 gate evidence。

同时，`create-issues.sh` 会复跑 planner JSON 并检查 `candidate_issue_gate.can_create_issues`，而不是只用 strict publication visibility 作为脚本前置条件。这样在最新发布已经公开、HEAD 正常包含下一版准备提交时，候选 issue artifacts 不会因为 `latest tag != HEAD` 被误拦；如果 gate 不允许创建 issue，脚本仍会打印 preflight JSON 后退出。

发布可见性审计也会区分 `git ls-remote` 网络失败和远端 ref 真缺失。远端检查失败时，`remote_check_ok=false`、`remote_check_issues` 会保留具体失败 ref，branch/tag published 状态继续保留本地 upstream 推断，next action 只提示重跑远端审计，不会错误生成 `git push` 命令。

真实案例库的 candidate promotion plan 也开始按证据进度排序。若某个候选案例已经完成了部分 `adapter_package` / `metadata_validation` / `online_smoke` 证据，`cliany-site cases --promotion-plan` 和 release planner 的 evidence summary 会优先推荐这个更接近 `active` 的候选；如果所有候选进度相同，则继续保持 manifest 顺序。

## 变更分组

- Release readiness：新增目标 tag 的每日 release 上限前瞻判断，避免在 `3/3` 当前状态下继续建议创建第四个同日 tag。
- Next actions：当目标 tag 会让当天 release 数变成 `4/3` 时，输出明确的暂停 tag 指令。
- Next iteration planner：顶层 `next_actions` 继承 readiness 已计算出的 pause / cadence 动作，避免 planner JSON 丢失关键发版暂停信号。
- Publication decision：目标 tag release gate 继承该 blocker，并在 daily-cap blocker 存在时清空 target tag commands，避免自动化只看发布命令列表就继续执行。
- Action ordering：daily-cap pause action 在 release readiness 和 planner 顶层 `next_actions` 中排在首位，并过滤泛化 create-tag 文案。
- Candidate issue gate：非草案类 release readiness blocker 会进入 `release_readiness_blockers` reason code，并把首要 blocker / required action 带到 gate evidence。
- Candidate artifacts：`publication-handoff.json` 和 artifacts README quick summary 会直接展示 release readiness blocker 的 count、primary blocker 和 hash。
- Candidate issue script：`create-issues.sh` 以 planner 的 `candidate_issue_gate.can_create_issues` 作为执行前置条件，允许正常的下一版准备态创建候选 issue，同时保留 gate blocked 时的 JSON 诊断输出。
- Publication audit：`git ls-remote` 失败时不再等同于远端 ref 缺失，而是输出 `remote_check_ok=false` 和 rerun action，避免维护者误推已经存在的 branch/tag。
- Candidate promotion plan：候选晋级队列按 ready、pending、已完成证据数、blocked 数和命令缺失数排序，帮助维护者优先推进最接近激活的真实案例。
- Tests：新增回归覆盖，锁定“当前 cadence 合格但目标 tag 会超限”的边界，以及 release readiness / planner 不再暴露 target tag 命令、planner 透传 pause action、不重复追加简写周节奏动作、pause 优先级、candidate issue gate readiness blocker、artifact handoff 交接面与 candidate promotion 排序行为。
- Release draft：新增本草案，将 `v0.16.251` 留在准备态，等待下一可发布日期。

## 风险与兼容性

- 本版只加强 release readiness 的目标 tag 前瞻门禁，不改变 CLI runtime、adapter metadata schema、AXTree 语义匹配、action runtime 或 session 存储。
- `daily_release_limit_ok` 继续描述当前 cadence 是否已经超限；新增 blocker 描述目标 tag 创建后的 projected count。
- 已经存在的 release tag 审计不应被该前瞻 blocker 误伤，避免发布后回查 `--release-tag` 时因为同日达到上限而失败。
- 不引入脆弱 CSS selector 兜底，后续 adapter 生成仍必须依赖 AXTree 语义匹配。
- 运行时状态仍必须写入 `~/.cliany-site/`，不得把 adapters、sessions、snapshots 或 packages 写进 repo。
- 官网更新需要按 `AGENTS.md` 从 `site/` 目录部署到 `cliany.site` Vercel 项目，避免误部署到默认 `site` 项目。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.250..HEAD`。由于 2026-07-01 已经达到三次 release 上限，本草案补齐后，`v0.16.251` 的目标版本 readiness 在今天仍应因为 projected daily cap blocker 返回 blocked。下一可发布日期再运行同一组命令，确认该 blocker 消失后才能继续 tag。

```bash
git diff --check
uv run --extra dev --frozen ruff check scripts/release_readiness.py tests/test_release_readiness.py
uv run --extra dev --frozen pytest tests/test_release_readiness.py tests/test_plan_next_iteration.py -q --no-cov
uv run --extra dev --frozen pytest tests/test_cases_command.py tests/test_validate_cases.py -q --no-cov
uv run --extra dev --frozen python scripts/release_readiness.py --target-version 0.16.251 --remote --remote-name origin --json
uv run --extra dev --frozen python scripts/plan_next_iteration.py --remote --target-version 0.16.251 --json
uv run --extra dev --frozen python scripts/plan_next_iteration.py --remote --target-version 0.16.251 --issues-dir /tmp/cliany-candidate-issues-016251 --json
uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json
```

正式 tag 前再运行：

```bash
CLIANY_QA_OFFLINE=1 uv run --extra dev --frozen pytest tests/ -q
uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.251 --remote --remote-name origin
```

## 发版步骤

1. 确认 `v0.16.250` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 如果当前日期仍是 2026-07-01，停止创建 `v0.16.251` tag，因为当天已经发布 `v0.16.248`、`v0.16.249` 和 `v0.16.250`。
3. 到下一可发布日期后，重新运行 release readiness，确认 projected daily cap blocker 消失。
4. 确认 `CHANGELOG.md`、本草案、README、官网和 case evidence 状态一致。
5. 运行 release readiness、planner、publication audit、ruff 和测试。
6. 将 `CHANGELOG.md` 的 Unreleased 条目移动到 `## [0.16.251] - <date>`，并将 `pyproject.toml` 版本号更新到 `0.16.251`。
7. 提交并推送 `master`。
8. 创建并推送 `v0.16.251` tag。
9. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
10. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json`，确认 `distribution_ok=true`、GitHub Release latest tag 为 `v0.16.251`，PyPI latest version 为 `0.16.251`。
11. 官网源码如有更新，本版发布后按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署到 `cliany.site` 项目。

## Release Notes 摘要

v0.16.251 makes release readiness project the target tag against the daily release cap before suggesting tag commands. When the current day already has three release tags, planning the next target version now blocks the fourth same-day tag and tells maintainers to wait for the next release day.
