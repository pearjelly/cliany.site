# v0.16.39 发布草案

**目标版本：** `0.16.39`
**提交范围：** `v0.16.38..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts 写出结构化 release draft handoff。

## 用户价值

`v0.16.38` 已经让 artifacts `README.md` 展示 `Release Draft Handoff`，维护者打开候选 issue artifacts 首页时能看到下一版草案路径和 release draft issues。但自动化或维护脚本如果想读取这份信息，仍然需要解析 Markdown；这不如 `publication-handoff.json` 那样稳定。

`v0.16.39` 在 `scripts/plan_next_iteration.py --issues-dir` 输出目录中新增 `release-draft-handoff.json`，结构化保存 `target_version`、`release_draft_path` 和 `release_draft_issues`。维护者和脚本可以直接读取 JSON 来判断下一版草案是否准备好，再决定是否继续创建 candidate issue。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.39 --issues-dir /tmp/cliany-candidate-issues --json` 写出的目录包含 `release-draft-handoff.json`；该 JSON 包含 `target_version`、`release_draft_path` 和 `release_draft_issues`；artifacts `README.md` 的文件列表说明这个 JSON artifact。

## 变更分组

### Structured release draft handoff artifact

- `scripts/plan_next_iteration.py --issues-dir` 新增 `release-draft-handoff.json`。
- JSON 包含 target version、release draft path 和 release draft issues。
- Artifacts `README.md` 的 Files 小节说明 `release-draft-handoff.json` 的用途。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 candidate issue artifacts 会包含 `release-draft-handoff.json`。
- `tests/test_plan_next_iteration.py` 覆盖 JSON artifact 内容和 README 文件列表。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强候选 issue artifacts 的结构化 release draft 交接信息。

## 风险与兼容性

- `release-draft-handoff.json` 是新增 artifact，不改变 plan JSON schema。
- `publication-handoff.json`、`issue-metadata.json` 和 `create-issues.sh` 结构保持兼容。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- release draft issues 只复用 plan 中已有的 `release_draft_issues`。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.39 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `release-draft-handoff.json`。
2. 确认 JSON 包含 `target_version`、`release_draft_path` 和 `release_draft_issues`。
3. 确认 artifacts `README.md` 的 Files 小节说明 `release-draft-handoff.json`。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.39`。
6. 将 `pyproject.toml` 版本更新为 `0.16.39`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.39`。
8. 创建 tag：`git tag v0.16.39`。

## Release Notes 摘要

`v0.16.39` adds a structured `release-draft-handoff.json` artifact to generated candidate issue artifacts, so maintainers and tools can read the target version, draft path, and draft issues without parsing the README.
