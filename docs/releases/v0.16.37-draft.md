# v0.16.37 发布草案

**目标版本：** `0.16.37`
**提交范围：** `v0.16.36..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts README 直接展示 publication visibility summary。

## 用户价值

`v0.16.36` 已经把 `publication_visibility` 写进 next iteration plan 和 `publication-handoff.json`，并在 artifacts `README.md` 的 `Publication Handoff` 中展示 visibility status。维护者打开候选 issue artifacts 首页时，能看到当前是 `local_only`、`dirty_worktree` 或其他状态，但仍需要打开 JSON 才能看到 summary 里的具体原因，例如本地分支 ahead 多少、应该发布哪个 tag，或是否需要远端复核。

`v0.16.37` 让 `scripts/plan_next_iteration.py --issues-dir` 生成的 artifacts `README.md` 在 `Publication Handoff` 中同时展示 `visibility_summary`。维护者在派发 candidate issue 前，不用离开 README 就能看到 publication visibility 的判断原因，再决定是否先运行 publish script 或延后派发。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.37 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 artifacts `README.md` 包含 `visibility_summary`，且 `publication-handoff.json` 继续保留结构化 `visibility`。

## 变更分组

### Candidate artifacts publication visibility summary

- `scripts/plan_next_iteration.py --issues-dir` 生成的 artifacts `README.md` 在 `Publication Handoff` 下展示 `visibility_summary`。
- `publication-handoff.json` 继续保留结构化 `visibility.status` 和 `visibility.summary`。
- Markdown plan 和 JSON plan 的既有 `publication_visibility` 输出保持不变。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifacts README 会展示 visibility summary。
- `tests/test_plan_next_iteration.py` 覆盖 artifacts README 的 `visibility_summary`。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强候选 issue artifacts README 的发布可见性说明。

## 风险与兼容性

- `visibility_summary` 是 artifacts README 的新增展示行，不改变 plan JSON schema。
- `publication-handoff.json` 结构保持兼容，继续保留 `visibility`。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- summary 只复用 `publication_visibility.summary`，不重新推断远端状态。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.37 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 README 包含 `visibility_summary`。
2. 确认 `publication-handoff.json` 继续包含结构化 `visibility`。
3. 运行计划器、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.37`。
5. 将 `pyproject.toml` 版本更新为 `0.16.37`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.37`。
7. 创建 tag：`git tag v0.16.37`。

## Release Notes 摘要

`v0.16.37` surfaces the publication visibility summary in generated candidate issue artifacts README files, so maintainers can understand the release-publication blocker without opening the handoff JSON first.
