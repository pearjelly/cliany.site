# v0.16.225 发布草案

**目标版本：** `0.16.225`
**提交范围：** `v0.16.224..HEAD`
**当前阻塞：** 最新本地 release `v0.16.224` 仍需维护者确认后 push 到远端；本版本先补齐 `v0.16.225` 草案，让下一轮计划不再被 `release draft is missing` 干扰。

## 用户价值

`v0.16.224` 已经让 `artifact_bundle_summary` 能直接定位 primary candidate promotion task，但下一轮计划仍同时暴露两个不同层级的阻塞：发布可见性未完成，以及 `docs/releases/v0.16.225-draft.md` 缺失。前者需要维护者明确批准 push，后者可以在本地先解决。

`v0.16.225` 把下一版草案提前写清楚：先处理发布可见性，再继续创建 candidate issue 或推进 candidate 晋级证据。维护者读取 `plan_next_iteration.py` 输出、`publication-handoff.json` 或 `release-draft-handoff.json` 时，可以把 `candidate_issue_gate.status=blocked_by_publication` 理解为“等待发布同步”，而不是误以为还缺少下一版 release draft。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.225 --issues-dir /tmp/cliany-candidate-issues --json` 的 `release_draft_issues` 不再报告 `release draft is missing`；`candidate_issue_gate` 仍因 `publication_visibility` 为 `local_only` 而阻止创建候选 issue；`publication_next_actions` 和 `publication_publish_commands` 继续明确展示需要维护者执行的远端同步命令。

## 变更分组

### Release draft handoff cleanup

- 新增 `docs/releases/v0.16.225-draft.md`，把本轮最前置动作固定为发布可见性同步。
- 草案明确引用 `publication_visibility`、`publication_next_actions`、`publication_publish_commands`、`candidate_issue_gate`、`publication-handoff.json` 和 `release-draft-handoff.json`。
- 草案说明 `blocked_by_publication` 是当前 candidate issue gate 的预期状态：`v0.16.224` 本地 tag 尚未公开前，不创建真实 GitHub issue。

### 文档与门禁

- `tests/test_release_draft_docs.py` 固定本草案的必需章节和发布可见性字段。
- 本版本不修改案例库状态，不修改 adapter 生成逻辑，不创建真实 GitHub issue，不推送 `master` 或 tag。
- `release_draft_issues` 清空后，下一轮计划应该只剩发布同步、周提交天数和候选晋级证据这些真实待办。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

Candidate promotion 的首要任务仍是 `pypi-project-search` 的 `adapter_package` evidence。发布可见性完成前，本版本不把 candidate issue artifacts 变成真实 GitHub issue，也不把任何 candidate 标记为 active。

## 风险与兼容性

- 这是发布流程文档与测试锚点更新，不改变运行时代码。
- `publication_visibility` 仍以 `check_release_publication.py` 的真实 Git 状态为准。
- `candidate_issue_gate` 仍应在 `v0.16.224` 未发布时保持 `blocked_by_publication`。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.225..HEAD`。最终带 `--release-tag v0.16.225` 的 strict gate 仍以本草案顶部的 `v0.16.224..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_release_draft_docs.py -q --no-cov
ruff check tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.225 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/check_release_publication.py --json
python scripts/check_release_publication.py --remote --json
python scripts/release_readiness.py --target-version 0.16.225 --json
python scripts/release_readiness.py --strict --target-version 0.16.225 --report /tmp/cliany-release-readiness.md
```

## 发版步骤

1. 确认 `tests/test_release_draft_docs.py` 和 ruff 通过。
2. 确认 `plan_next_iteration.py --target-version 0.16.225 --issues-dir` 不再报告 `release draft is missing`。
3. 确认 `candidate_issue_gate.status` 在 `v0.16.224` 未发布前仍是 `blocked_by_publication`。
4. 维护者批准后执行 `git push origin master`。
5. 维护者批准后执行 `git push origin v0.16.224`。
6. 执行 `python scripts/check_release_publication.py --remote --json` 复核远端分支和 tag。
7. 准备最终发布时，确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.225`，并创建 tag：`git tag v0.16.225`。

`v0.16.225` adds the next release draft for the publication-first maintenance path, so planning output can focus on the real remote publication gate instead of a missing draft artifact.
