# v0.16.145 发布草案

**目标版本：** `0.16.145`
**提交范围：** `v0.16.144..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把第一条 publication next action 补进 artifact bundle summary。

## 用户价值

`v0.16.144` 已经让 `artifact_bundle_summary` 能同时展示第一条 release draft 阻塞原因和修复动作。维护者和自动化只读整包摘要时，仍需要展开 `publication-handoff.json` 或完整 action 列表，才能看到发布同步最先要做的一步。

`v0.16.145` 在 `artifact_bundle_summary` 中新增 `publication_primary_next_action`，并在 artifacts `README.md` 的 `Artifact Bundle Summary` 段落展示同源字段。这样工具只读 manifest summary 或 README summary，就能直接显示第一条发布同步待办，同时仍可用 `publication_next_action_count` 和 `publication_next_actions_sha256` 判断是否展开完整列表。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.145 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_primary_next_action`，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且该值与 `publication_next_actions[0]` 同源；无 publication next actions 时字段为 `null`。

## 变更分组

### Artifact bundle publication primary next action

- `artifact_bundle_summary` 新增 `publication_primary_next_action`。
- 有 publication next actions 时，该字段取第一条 action。
- 无 publication next actions 时，该字段为 `null`；README summary 用 `(none)` 展示。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示第一条发布同步待办。
- 现有 `publication_next_action_count`、`publication_next_actions_sha256`、`publication_handoff_key_count`、`publication_handoff_sha256` 和 `Publication Handoff` 保持兼容。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示第一条 publication next action。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的新字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 整包摘要的 publication primary next action 可见性。

## 风险与兼容性

- 新字段只派生自既有 `publication_next_actions`，不新增网络访问或外部状态读取。
- 无 publication next action 时字段为 `null`，方便机器读取。
- Primary action 只用于快速展示第一步，不替代完整 `publication-handoff.json` 审阅。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.145 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_primary_next_action`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示 `publication_primary_next_action`。
4. 确认有 publication next action 时，该字段等于 `publication_next_actions[0]`。
5. 确认无 publication next action 时，JSON 字段为 `null` 且 README 展示 `(none)`。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.145`。
8. 将 `pyproject.toml` 版本更新为 `0.16.145`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.145`。
10. 创建 tag：`git tag v0.16.145`。

## Release Notes 摘要

`v0.16.145` adds the primary publication next action to artifact bundle summaries, helping tools show the first release publishing step without expanding the full publication handoff.
