# v0.16.27 发布草案

**目标版本：** `0.16.27`
**提交范围：** `v0.16.26..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue body 的复现命令层级更清晰。

## 用户价值

`scripts/plan_next_iteration.py --issues-dir` 会生成 `pypi-project-search.md` 这类 candidate issue body，里面的 `Reproduction Context` 带有 target URL、candidate commands 和 offline validation commands。此前命令和字段标题都是同级 bullet，GitHub issue 里阅读时不容易分辨哪些 bullet 是字段，哪些 bullet 是具体命令。

`v0.16.27` 把 candidate commands 和 offline validation commands 渲染为 `Reproduction Context` 下的子列表。例如：

```markdown
- Candidate commands:
  - `cliany-site explore ...`
- Offline validation commands:
  - `python scripts/validate_cases.py --strict`
```

这样贡献者打开单个候选 issue 时，可以更快看到要复制哪些命令，也更容易区分复现字段和执行步骤。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.27 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `pypi-project-search.md` 中，`candidate commands` 和 `offline validation commands` 下的命令以两个空格缩进的子列表呈现。

## 变更分组

### Candidate issue body readability

- `scripts/plan_next_iteration.py` 生成的 candidate issue body 中，candidate commands 改为 `Reproduction Context` 下的子列表。
- offline validation commands 同样改为子列表。
- 命令内容、`issue-metadata.json`、`publication-handoff.json` 和 `create-issues.sh` 结构保持兼容。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 candidate issue body 的复现命令会作为子列表展示。
- `tests/test_plan_next_iteration.py` 固定 issue body 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只改善候选 issue body 的可读性。

## 风险与兼容性

- 这是 Markdown 渲染增强，不改变 JSON schema 和脚本执行行为。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.27 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `pypi-project-search.md` 包含 `Reproduction Context`。
2. 确认 candidate commands 下的命令使用两个空格缩进的子列表，例如 `cliany-site explore` 命令前有 `  - `。
3. 确认 offline validation commands 下的命令使用两个空格缩进的子列表，例如 `python scripts/validate_cases.py --strict` 命令前有 `  - `。
4. 运行计划脚本、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.27`。
6. 将 `pyproject.toml` 版本更新为 `0.16.27`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.27`。
8. 创建 tag：`git tag v0.16.27`。

## Release Notes 摘要

`v0.16.27` renders candidate issue body reproduction commands as nested lists under `Reproduction Context`, making generated GitHub issue bodies easier to scan and copy from.
