# v0.16.69 发布草案

**目标版本：** `0.16.69`
**提交范围：** `v0.16.68..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 `artifact_bundle_summary` 直接携带 publication 和 release draft 预筛字段。

## 用户价值

`v0.16.68` 已经让 `artifact_bundle_summary` 能直接判断 candidate issue gate 的 reason/action set 是否变化。但工具如果只读取整包摘要，仍然无法直接区分阻塞来自 publication 可见性，还是来自下一版 release draft。

`v0.16.69` 在 `artifact_bundle_summary` 中新增 `publication_ok`、`publication_visibility_status` 和 `release_draft_issue_count`。Artifacts `README.md` 的 `Artifact Bundle Summary` 也展示这些字段，让维护者和自动化能在展开 publication/release draft handoff 前完成更准确的预筛。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.70 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 publication/release draft 预筛字段，并与 plan 的 `publication_ok`、`publication_visibility.status` 和 `release_draft_issues` 数量一致；README 的 `Artifact Bundle Summary` 也展示这些字段。

## 变更分组

### Artifact bundle publication/draft summary

- `artifact_bundle_summary` 新增 `publication_ok`。
- `artifact_bundle_summary` 新增 `publication_visibility_status`。
- `artifact_bundle_summary` 新增 `release_draft_issue_count`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些 publication/release draft 预筛字段。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 publication/release draft 预筛字段。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate issue artifacts 的整包摘要更适合自动化区分阻塞来源。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary`，不改变 publication handoff 或 release draft handoff 的既有字段语义。
- Summary 字段复用现有 plan 数据，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.69..HEAD`。最终带 `--release-tag v0.16.69` 的 strict gate 仍以本草案顶部的 `v0.16.68..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.69 --json
python scripts/plan_next_iteration.py --target-version 0.16.69 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.69 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.69 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_ok`、`publication_visibility_status` 和 `release_draft_issue_count`。
3. 确认这些 summary 字段与 plan 的同源 publication/release draft 状态一致。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.69`。
7. 将 `pyproject.toml` 版本更新为 `0.16.69`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.69`。
9. 创建 tag：`git tag v0.16.69`。

## Release Notes 摘要

`v0.16.69` adds publication and release draft preflight fields to artifact bundle summaries, helping tools classify generated issue artifact blockers before reading the full handoff files.
