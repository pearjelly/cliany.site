# v0.16.52 发布草案

**目标版本：** `0.16.52`
**提交范围：** `v0.16.51..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue gate 直接带上机器可读判定证据。

## 用户价值

`v0.16.51` 已经让 next iteration plan 和 candidate issue artifacts 输出 `candidate_issue_gate`，能明确告诉维护者当前是 `blocked_by_publication`、`review_required` 还是 `ready`。但自动化如果要解释这个结论，仍需要继续读取 publication visibility、ref context 和 release draft handoff，再自行拼出“为什么不能创建 issue”。

`v0.16.52` 给 `candidate_issue_gate` 增加 `evidence`。这个 evidence 会记录 `publication_ok`、`publication_visibility_status`、`publication_remote_checked`、`publication_branch`、`publication_latest_tag`、`publication_ahead_count`、`release_draft_path` 和 `release_draft_issue_count`。这样工具只读取 gate 对象，就能同时知道结论、放行动作和判定依据。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.52 --issues-dir /tmp/cliany-candidate-issues --json` 写出的计划 JSON、Markdown report、`artifact-manifest.json`、`publication-handoff.json` 和 artifacts `README.md` 都能展示 candidate issue gate evidence；当最新本地 release 未公开可见时，evidence 能显示 latest tag、ahead count 和 release draft issue count。

## 变更分组

### Candidate issue gate evidence

- `candidate_issue_gate` 新增 `evidence` 对象。
- Evidence 保留 publication gate 的关键输入：publication ok、visibility status、remote checked、branch、latest tag 和 ahead count。
- Evidence 保留 release draft gate 的关键输入：release draft path 和 release draft issue count。
- Markdown report 新增 `Candidate Issue Gate Evidence` 表格。

### Artifacts handoff

- `artifact-manifest.json` 和 `publication-handoff.json` 继续写入同源 `candidate_issue_gate`，包含新增 evidence。
- Artifacts `README.md` 在 `Publication Handoff` 下展示 gate evidence latest tag、ahead count 和 release draft issue count。
- `candidate_issue_gate` 仍保留 `status`、`can_create_issues`、`requires_maintainer_review`、summary 和 required actions。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 candidate issue gate evidence 的字段和用途。
- `tests/test_plan_next_iteration.py` 覆盖 JSON、Markdown report 和 artifacts README 中的 evidence。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强创建 candidate issue 前的 gate 判定证据。

## 风险与兼容性

- `candidate_issue_gate.evidence` 是新增嵌套字段，不改变已有 gate 字段语义。
- Evidence 只复用本地 release readiness 和 publication audit 结果，不新增网络访问。
- `create-issues.sh` 的硬性 preflight 仍由 `python scripts/check_release_publication.py --strict --json` 执行。
- 不自动执行命令，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.52..HEAD`。最终带 `--release-tag v0.16.52` 的 strict gate 仍以本草案顶部的 `v0.16.51..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.52 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.52 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.52 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `artifact-manifest.json`。
2. 确认计划 JSON、Markdown report、`artifact-manifest.json` 和 `publication-handoff.json` 都包含 `candidate_issue_gate.evidence`。
3. 确认 artifacts `README.md` 展示 gate evidence latest tag、ahead count 和 release draft issue count。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.52`。
6. 将 `pyproject.toml` 版本更新为 `0.16.52`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.52`。
8. 创建 tag：`git tag v0.16.52`。

## Release Notes 摘要

`v0.16.52` adds machine-readable evidence to `candidate_issue_gate`, so maintainers and tools can audit why candidate issue creation is blocked, review-required, or ready without stitching together separate planning fields.
