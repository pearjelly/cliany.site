# v0.16.26 发布草案

**目标版本：** `0.16.26`
**提交范围：** `v0.16.25..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让候选 issue artifacts README 直接展示 publication publish script 生成命令。

## 用户价值

`v0.16.25` 已经要求维护者在运行 `create-issues.sh` 前确认 `Publication Next Actions` 已处理或明确延后。但如果维护者想按推荐路径先生成可审阅的发布脚本，仍需要打开 plan JSON 或 `publication-handoff.json` 才能找到 `publication_publish_script_command`。

`v0.16.26` 在 `scripts/plan_next_iteration.py --issues-dir` 生成的 artifacts `README.md` 里，于 `Publication Handoff` 下新增 `Publication Publish Script` 小节。维护者可以直接从 README 复制 `python scripts/check_release_publication.py --json --publish-script /tmp/cliany-publish-release.sh`，先生成脚本、审阅 branch/tag push 和 remote 复核命令，再决定是否执行。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.26 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `README.md` 包含 `Publication Handoff`、`Publication Publish Script` 和 `--publish-script /tmp/cliany-publish-release.sh`；`publication-handoff.json` 继续保留 `publication_publish_script_command`。

## 变更分组

### Candidate issue artifacts publish script entry

- `scripts/plan_next_iteration.py --issues-dir` 生成的 artifacts `README.md` 在 `Publication Handoff` 下新增 `Publication Publish Script`。
- README 会展示 `publication_publish_script_command` 对应的命令，方便维护者先生成可审阅脚本。
- `publication-handoff.json`、`create-issues.sh` 和候选 issue body 文件结构保持兼容。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifacts README 会展示 `Publication Publish Script`。
- `tests/test_plan_next_iteration.py` 固定 artifacts README 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只让候选 issue artifacts 的发布交接入口更完整。

## 风险与兼容性

- 这是 artifacts README 文案增强，不改变 JSON schema 和脚本执行行为。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

```bash
python scripts/plan_next_iteration.py --target-version 0.16.26 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.26 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.26 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 README 包含 `Publication Publish Script`。
2. 确认 README 中展示 `--publish-script /tmp/cliany-publish-release.sh`。
3. 确认 `publication-handoff.json` 继续包含 `publication_publish_script_command`。
4. 运行计划脚本、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.26`。
6. 将 `pyproject.toml` 版本更新为 `0.16.26`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.26`。
8. 创建 tag：`git tag v0.16.26`。

## Release Notes 摘要

`v0.16.26` shows the publication publish-script command directly in generated candidate issue artifacts README files, so maintainers can generate and review the release publication script from the same handoff page.
