# v0.16.241 发布草案

**目标版本：** `0.16.241`
**提交范围：** `v0.16.240..HEAD`
**当前阻塞：** 真实 LLM provider 预检仍返回 `502 Bad Gateway` / `E_LLM_UNAVAILABLE`，因此本版不生成 candidate adapter package，不运行真实 `explore`，也不把任何 candidate 标记为 active。本版把 live LLM preflight 阻断契约继续提升到 next-iteration planner 和 issue artifacts，避免维护自动化只解析 planner JSON 时跳过停机条件。

## 用户价值

`v0.16.240` 已经让 `cliany-site cases --promotion-plan` 直接展示 `llm_live_preflight_command` 和 `llm_live_preflight_blocker_note`。但维护者每天挑选下一版时通常先跑 `scripts/plan_next_iteration.py --json` 或生成 `/tmp/cliany-candidate-issues` artifacts；这些产物虽然 issue body 里有 `LLM Preflight Gate`，但机器可读 `candidate_promotions` 和 `issue-metadata.json` 仍需要解析 Markdown 才能拿到 blocker handling。

`v0.16.241` 让 planner 的 candidate promotion 记录、issue metadata、metadata summary 和 artifact README 都直接携带同一份 live LLM preflight 契约。自动化可以读取 `llm_live_preflight_command` 与 `llm_live_preflight_blocker_note` 后先运行 `cliany-site doctor --llm-live --json`；如果 `generate_adapters.ready=false` 或 `llm_live` 返回 `E_LLM_UNAVAILABLE`，就停止 candidate promotion，贴回 doctor JSON / 错误摘要，并保持 `adapter_package` pending 或 blocked。

## 变更分组

- Planner JSON：`scripts/plan_next_iteration.py --json` 的每个 `candidate_promotions[]` 项现在包含 `llm_live_preflight_command` 和 `llm_live_preflight_blocker_note`。
- Issue artifacts：`--issues-dir` 生成的 `issue-metadata.json` 和 metadata summary 同步携带这两个字段，减少 issue 创建机器人解析 Markdown 的需求。
- Human review：candidate issue artifact README 的 metadata 表和 review checklist 现在显式要求核对 LLM preflight command / blocker note。
- Tests：`tests/test_plan_next_iteration.py` 覆盖 planner JSON、issue metadata 和 artifact review checklist 中的 `E_LLM_UNAVAILABLE` 机器可读 handoff。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的案例状态；相关说明仍落在 [cases/README.md](../../cases/README.md)。
- `cases/README.md` 已记录 candidate promotion 需要真实 adapter package、metadata validation 和 online smoke 证据。
- 当前案例库仍保持 active 4 个、candidate 3 个、known gap 1 个。
- Known gap 保持 1 个：`search-extraction-gap`。
- Candidate 晋级仍需要三段真实证据：adapter package、metadata validation 和 online smoke。
- `pypi-project-search` 仍是首要 candidate，但当前 live LLM preflight 返回 `E_LLM_UNAVAILABLE` / `502 Bad Gateway`；本版只让 planner 和 issue artifacts 更清楚地记录这个 blocker，不生成或激活任何 candidate adapter。

## 风险与兼容性

- 这是 planner / issue artifact 的 additive JSON 和 Markdown 增强，不改变 adapter schema、AXTree 语义匹配、action runtime、codegen、session 存储或 marketplace package 格式。
- 新增 JSON 字段是 additive：现有读取 `candidate_promotions` 或 `issue-metadata.json` 的自动化可以忽略 `llm_live_preflight_command` 和 `llm_live_preflight_blocker_note`。
- Artifact README 多显示一列 blocker handling；交接信息更长，但能降低 issue 创建自动化直接安排 `explore` 而跳过 live preflight 的风险。

## Release note draft

v0.16.241 carries the live LLM preflight blocker contract into next-iteration planning and generated candidate issue artifacts. Candidate promotion JSON, `issue-metadata.json`, metadata summaries, and artifact review docs now expose the preflight command and `E_LLM_UNAVAILABLE` handling directly, so maintainers and issue automation can stop promotion before running real `explore` while the upstream LLM provider is unavailable.
