# v0.16.100 发布草案

**目标版本：** `0.16.100`
**提交范围：** `v0.16.99..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 publication behind count 补进 candidate issue artifacts 的整包摘要。

## 用户价值

`v0.16.99` 已经让 `artifact_bundle_summary` 能展示 `publication_remote_checked` 和 `publication_ahead_count`。维护者和自动化继续审阅候选 issue artifacts 时，还需要不展开 `publication_ref_context` 就能判断本地分支是否落后远端；如果 behind count 大于 0，发布动作就不再只是 push tag/branch，而需要先处理远端同步风险。

`v0.16.100` 在 `artifact_bundle_summary` 中新增 `publication_behind_count`。Artifacts `README.md` 的 `Artifact Bundle Summary` 也展示该字段，让工具只读整包摘要就能同时看到 ahead/behind 两个方向的发布同步规模。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.100 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_behind_count`，且该值等于 manifest 中 `publication_ref_context.behind_count`；README 的 `Artifact Bundle Summary` 也展示该字段。

## 变更分组

### Artifact bundle publication behind count

- `artifact_bundle_summary` 新增 `publication_behind_count`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示 publication behind count。
- 现有 `publication_remote_checked`、`publication_ahead_count`、`publication_ref_context` 和 `publication_visibility_status` 保持不变。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 publication behind count。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让候选 issue artifacts 的整包摘要更容易判断发布分支是否存在远端领先风险。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary`，不改变 `publication_ref_context` 或 `publication-handoff.json` 的既有结构。
- 数值来自计划器已有的 publication ref context，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.100..HEAD`。最终带 `--release-tag v0.16.100` 的 strict gate 仍以本草案顶部的 `v0.16.99..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.100 --json
python scripts/plan_next_iteration.py --target-version 0.16.100 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.100 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.100 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_behind_count`。
3. 确认该值等于 `artifact-manifest.json.publication_ref_context.behind_count`。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示该字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.100`。
7. 将 `pyproject.toml` 版本更新为 `0.16.100`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.100`。
9. 创建 tag：`git tag v0.16.100`。

## Release Notes 摘要

`v0.16.100` adds publication behind count to candidate issue artifact bundle summaries, helping tools detect remote-ahead publication risk without expanding the full publication ref context.
