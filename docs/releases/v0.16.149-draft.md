# v0.16.149 发布草案

**目标版本：** `0.16.149`
**提交范围：** `v0.16.148..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本给 release draft handoff 增加通用 primary 字段别名。

## 用户价值

`v0.16.147` 和 `v0.16.148` 已经让 `publication-handoff.json` 暴露 `primary_publish_command` 与 `primary_next_action`。维护工具读取 publication handoff 时，可以用稳定的 `primary_*` 字段直接展示首要发布命令和待办。

`release-draft-handoff.json` 目前已经有 `release_draft_primary_issue` 和 `release_draft_primary_required_action`，但字段名更长，工具如果想用同一套 handoff 读取模型，还需要为 release draft handoff 写特殊分支。`v0.16.149` 增加 `primary_issue` 和 `primary_required_action` 作为同源别名，并在 artifacts `README.md` 的 `Release Draft Handoff` 段落展示它们。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.149 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `release-draft-handoff.json` 包含 `primary_issue` 和 `primary_required_action`，artifacts README 的 `Release Draft Handoff` 展示同名字段，且它们分别与 `release_draft_primary_issue`、`release_draft_primary_required_action` 同源；无 release draft issue 时两个字段为 `null` 且 README 展示 `(none)`。

## 变更分组

### Release draft handoff primary aliases

- `release-draft-handoff.json` 新增 `primary_issue`。
- `release-draft-handoff.json` 新增 `primary_required_action`。
- Artifacts `README.md` 的 `Release Draft Handoff` 展示这两个通用别名。
- 现有 `release_draft_primary_issue`、`release_draft_primary_required_action`、`release_draft_handoff_sha256`、`release_draft_handoff_key_count` 和 `artifact_bundle_summary` 保持兼容并反映 handoff 字段变化。
- 字段值只从既有 release draft issue 列表派生，不改变 release readiness 或 publication gate 语义。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 release draft handoff 会展示通用 `primary_issue` 和 `primary_required_action`。
- `tests/test_plan_next_iteration.py` 覆盖 handoff JSON、README 展示和 handoff hash。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的 release draft handoff 可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 `release_draft_primary_issue` 和 `release_draft_primary_required_action`，不新增网络访问或外部状态读取。
- 无 release draft issue 时字段为 `null`，方便机器读取。
- 通用 primary 别名只用于快速展示首要草案问题和处理动作，不替代完整 `release_draft_issues` 与 `release_draft_required_actions` 审阅。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.149..HEAD`。最终带 `--release-tag v0.16.149` 的 strict gate 仍以本草案顶部的 `v0.16.148..HEAD` 作为实际 release diff 范围。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.149 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `release-draft-handoff.json` 包含 `primary_issue` 和 `primary_required_action`。
3. 确认 artifacts README 的 `Release Draft Handoff` 展示这两个字段。
4. 确认字段值分别等于 `release_draft_primary_issue` 和 `release_draft_primary_required_action`。
5. 确认无 release draft issue 时，JSON 字段为 `null` 且 README 展示 `(none)`。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.149`。
8. 将 `pyproject.toml` 版本更新为 `0.16.149`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.149`。
10. 创建 tag：`git tag v0.16.149`。

## Release Notes 摘要

`v0.16.149` adds generic primary issue and primary required action aliases to release draft handoff artifacts, helping tools read release draft and publication handoffs with a more consistent field model.
