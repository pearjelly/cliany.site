# v0.16.233 发布草案

**目标版本：** `0.16.233`
**提交范围：** `v0.16.232..HEAD`
**当前阻塞：** 需要完成 candidate issue artifacts acceptance visibility 改动，并重新运行 release readiness、case validation、issue artifacts 生成、文档/官网测试和完整 pytest。

## 用户价值

`v0.16.232` 已经让 cases CLI、case report 和计划器 candidate summary 都能暴露首要 candidate 任务的 acceptance criteria。下一步要补齐维护者真正创建 GitHub issue 前看到的那一层：`scripts/plan_next_iteration.py --issues-dir` 生成的 artifacts `README.md`。

`v0.16.233` 聚焦 candidate issue artifacts 的验收标准可见性。维护者生成 `/tmp/cliany-candidate-issues` 后，不必先打开单个 issue body 或 JSON metadata，就能在 artifacts README 的 `Candidate Summary` 看到首要 evidence task 的状态和验收标准，并在 `Candidate Promotion Evidence Summary` 看到 `primary_next_task_acceptance_criteria` 与每个任务的 Acceptance Criteria。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.233 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 README 同时包含 `Primary Evidence Status`、`Primary Acceptance Criteria`、`primary_next_task_acceptance_criteria` 和任务级 `Acceptance Criteria`；README、README.zh、cases README、good first issues 指南和官网 quickstart 能说明 artifacts README 的审阅入口；`scripts/release_readiness.py --target-version 0.16.233 --json` 不再因为缺失 release draft 阻塞。

## 变更分组

### Candidate Issue Artifacts

- `scripts/plan_next_iteration.py --issues-dir` 的 artifacts README `Candidate Summary` 表新增 `Primary Evidence Status` 和 `Primary Acceptance Criteria`。
- 同一 README 的 `Candidate Promotion Evidence Summary` 指标表新增 `primary_next_task_acceptance_criteria`。
- 同一 README 的 candidate promotion task 表新增 `Acceptance Criteria` 列。

### 文档与官网同步

- README / README.zh 现在说明 `python scripts/plan_next_iteration.py --issues-dir /tmp/cliany-candidate-issues` 会生成可审阅的 candidate issue artifacts，并在 README 中展示 primary evidence status / acceptance criteria。
- [cases/README.md](../../cases/README.md) 现在说明 issue artifacts README 与 case report 使用同一套 proof contract。
- [docs/good-first-issues.md](../good-first-issues.md) 现在提醒维护者创建 candidate promotion issue 前先审阅 artifacts README 的 primary evidence status / acceptance criteria。
- 官网 quickstart 现在把候选案例验证入口同步到 `plan_next_iteration.py --issues-dir` 和 `Primary Acceptance Criteria`。

## 案例库映射

- 本版本不改变 [cases/manifest.json](../../cases/manifest.json) 的案例状态，不新增 active adapter 包，不生成或提交 `~/.cliany-site/` 运行时状态。
- Active 案例保持 4 个：SuiteCRM、Apache Jira、Apache Confluence、Apache Jenkins。
- Candidate 案例保持 3 个：PyPI、npm、crates.io 搜索工作流。
- Known gap 保持 1 个：`search-extraction-gap`。
- `pypi-project-search` 仍是下一轮首要 candidate；本版本只让它的首要 `adapter_package` 状态与验收标准在 issue artifacts README 中可见，不声称 adapter 包已经生成。

## 风险与兼容性

- 风险：下游工具如果解析 artifacts README 的 `Candidate Summary` 表格列数，需要允许新增列。
- 风险：Acceptance Criteria 是交接合同，不是任务完成证明；candidate 晋级 active 仍需真实 adapter package、metadata validation 和 online smoke 三段证据全部 complete。
- 兼容性：不修改 adapter schema、生成 adapter 代码、动作回放、CDP 连接或 case manifest 状态。
- 兼容性：发布节奏继续遵循每日 verified release train 和标准 release flow。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.233..HEAD`。最终带 `--release-tag v0.16.233` 的 strict gate 仍以本草案顶部的 `v0.16.232..HEAD` 作为实际 release diff 范围。

```bash
python3 scripts/release_readiness.py --target-version 0.16.233 --json
python3 scripts/plan_next_iteration.py --target-version 0.16.233 --issues-dir /tmp/cliany-candidate-issues --json
python3 scripts/validate_cases.py --strict
uv run --extra dev pytest tests/test_plan_next_iteration.py tests/test_readme_current_features.py tests/test_site_content.py -q
```

正式 tag 前再运行：

```bash
uv run --extra dev pytest tests/ -q
python3 scripts/release_readiness.py --strict --target-version 0.16.233
```
