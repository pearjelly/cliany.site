# v0.16.254 发布草案

**目标版本：** `0.16.254`
**提交范围：** `v0.16.253..HEAD`
**当前状态：** 本版为 2026-07-03 或下一个可发版日期准备；2026-07-02 已完成 `3/3` 个 release tag，今天不再创建 `v0.16.254`。

## 用户价值

`v0.16.253` 已经把官网 alias 复核纳入标准发版流程，但当日 release train 达到上限时，readiness 和 planner 仍只会提示 “next day”。维护者和自动化看得到暂停原因，却拿不到稳定的可执行日期，容易在连续发版日里重复判断同一个阻塞条件。

`v0.16.254` 将 daily release cap 的暂停状态结构化：当目标 tag 会让当天 release count 超过上限时，readiness 和 next-iteration planner 同时输出 `daily_release_cap_blocked`、`daily_release_resume_date` 和对应 SHA-256 摘要。这样维护者可以明确知道下一次允许尝试 tag 的日期，而不是从自然语言里推断。

本版还修正 candidate issue handoff 的可执行性和安全边界：生成的 `create-issues.sh` 不再硬编码裸 `python`，而是通过 `PYTHON_BIN="${PYTHON_BIN:-python3}"` 运行 preflight 和内联 JSON gate。macOS 或精简环境缺少 `python` 时，维护者仍可直接 dry-run 或显式覆盖 `PYTHON_BIN`。当 candidate issue gate 返回 `requires_maintainer_review=true` 时，脚本会在创建公开 GitHub issue 前退出，直到维护者完成审阅并显式设置 `CLIANY_CREATE_ISSUES_ACK_REVIEW=1`。

候选案例晋级也会输出更稳定的 adapter 包名交接：cases CLI、case validation summary、next-iteration planner、issue template 和 candidate issue metadata 都会给出 `expected_adapter_package`，例如 `pypi.org-<version>.cliany-adapter.tar.gz`。维护者不再需要从 `adapter_package` 的自然语言描述里解析 release asset 名称，可以直接用该字段核对 packages-dir 与 GitHub Release asset。

当 live LLM preflight 像本轮验证一样返回 `E_LLM_UNAVAILABLE` 时，candidate promotion task object 也会直接携带 `llm_live_preflight_required`、`llm_live_preflight_command`、`llm_live_preflight_command_sha256`、`llm_live_preflight_blocker_note`、`llm_live_preflight_evidence_fields`、`doctor_preflight_evidence_fields` 和结构化 `doctor_preflight_evidence_template`。维护者或机器人只读取 primary task JSON 时，也能知道 `adapter_package` 任务必须先跑 `cliany-site doctor --llm-live --json`，失败后应贴哪些 CDP/LLM doctor 字段作为 blocker 证据。直接从 cases CLI 生成的 candidate issue body 也会展示 `Primary Runbook`、LLM/doctor blocker comments、`Doctor Preflight Evidence Fields` 和带可粘贴占位符的 `Doctor Preflight Evidence Template`，`issue_template_primary_task` 则在 JSON 输出中携带同一组 acceptance、runbook、preflight 字段、模板映射以及 `doctor_preflight_evidence_template_field_count` / `doctor_preflight_evidence_template_sha256`。`cases --promotion-plan` 也会输出 `primary_issue_template_command`、`primary_issue_template_json_command`、candidate/task 级 `issue_template_json_command`，以及 `promotion_plan.primary_doctor_preflight_evidence_template_sha256`、`promotion_plan.primary_llm_live_preflight_command_sha256`、candidate `primary_doctor_preflight_evidence_template_sha256`、`task_queue[*].doctor_preflight_evidence_template_sha256` 和 `task_queue[*].llm_live_preflight_command_sha256`，维护者可以从候选队列直接跳到可复制 issue body，并在不展开 evidence bundle 的情况下比对 doctor 模板和 preflight 命令漂移。`scripts/validate_cases.py --json` 会在 `promotion_evidence_summary.primary_next_task.doctor_preflight_evidence_template_sha256` 暴露同一漂移信号，`scripts/validate_cases.py --report` 也会在 Candidate Promotion Evidence Summary 表格输出 `primary_doctor_preflight_evidence_template_sha256`，纯文本 `scripts/validate_cases.py --strict` stdout 则输出 `promotion_evidence_primary_doctor_preflight_evidence_template_sha256` 和 `promotion_evidence_primary_llm_live_preflight_command_sha256`。下一步 planner 也会在 `candidate_promotions[].issue_template_command`、`candidate_promotions[].issue_template_json_command`、`candidate_promotions[].doctor_preflight_evidence_template`、`issue-metadata.json` 和 artifacts README 的 `Issue Template` / `Issue Template JSON` 列中输出同源 `cliany-site cases --case-id <id> --issue-template` 交接命令，并提供模板 count/hash 与 preflight command hash 别名，让只读 planner artifacts 的工具也能复现 issue body 并比对模板漂移。

面向人工交接的 `cliany-site cases --status candidate` 现在也会在 `Candidate 下一步` 中显示 `preflight_required`、`preflight_blocker` 和 `runbook_first`。不读 JSON 的贡献者同样能先看到 live LLM gate 和失败处理规则，避免直接启动真实 `explore` 或补出不可验证的 adapter 证据。

下一步计划和 issue artifacts 也会把首要任务的 preflight gate 提升为 compact alias：`case_promotion_evidence_primary_llm_live_preflight_required`、`case_promotion_evidence_primary_llm_live_preflight_command`、`case_promotion_evidence_primary_llm_live_preflight_command_sha256`、`case_promotion_evidence_primary_llm_live_preflight_blocker_note`、`case_promotion_evidence_primary_llm_live_preflight_blocker_comment`、`case_promotion_evidence_primary_doctor_preflight_blocker_comment` 和 `case_promotion_evidence_primary_doctor_preflight_evidence_template_sha256`。candidate issue metadata 与 artifacts README 也会列出 `doctor_preflight_evidence_fields`、`doctor_preflight_evidence_template_field_count`、`doctor_preflight_evidence_template_sha256` 和 `case_promotion_doctor_preflight_evidence_template_sha256`，让只读 summary 的维护工具不用展开完整 primary task，也能判断当前 candidate 是否必须先跑 doctor，并在 CDP 或 LLM gate 失败时复制标准 blocker comment。

这让每天至少一个版本、最多三个版本的节奏更可审计：今天停住，明天继续，所有 release handoff 都能用同一组字段判断是否该进入 tagging 阶段。

## 变更范围

- `scripts/check_release_cadence.py`：在 cadence report 内部保留当前评估日期，用于后续 release readiness 推导下一次可发版日期。
- `scripts/release_readiness.py`：当目标 tag 被 daily release cap 阻塞时导出 `daily_release_cap_blocked`、`daily_release_resume_date` 和摘要字段，并在文本 / Markdown report 中显式渲染这组字段。
- `scripts/plan_next_iteration.py`：planner 透传同一组 daily cap resume 字段，方便候选案例和 release handoff 自动化读取，并在文本 / Markdown report、candidate issue artifacts manifest、publication handoff 和 artifacts README 中显式渲染；candidate issue metadata 和生成的 issue body 也会携带 `expected_adapter_package` 与 task-level live preflight gate 字段；planner JSON、Markdown report 和 artifact bundle summary 同时输出 `case_promotion_evidence_primary_llm_live_preflight_required`、`case_promotion_evidence_primary_llm_live_preflight_command`、`case_promotion_evidence_primary_llm_live_preflight_command_sha256`、`case_promotion_evidence_primary_llm_live_preflight_blocker_note`、`case_promotion_evidence_primary_llm_live_preflight_blocker_comment`、`case_promotion_evidence_primary_doctor_preflight_blocker_comment`、`case_promotion_evidence_primary_doctor_preflight_evidence_template_sha256` 和 `case_promotion_doctor_preflight_evidence_template_sha256`；candidate issue metadata、issue body 和 artifacts README 同步列出 `doctor_preflight_evidence_fields`、`doctor_preflight_evidence_template`、`doctor_preflight_evidence_template_field_count` 和 `doctor_preflight_evidence_template_sha256`，issue body 还会输出 `Doctor Preflight Evidence Template`；`candidate_promotions[].issue_template_command`、`candidate_promotions[].issue_template_json_command`、`issue-metadata.json` 和 artifacts README 的 Issue Template / Issue Template JSON 列同步展示 candidate issue body 复现命令；candidate issue script 通过可覆盖的 `PYTHON_BIN` 默认使用 `python3`，并在 gate 要求维护者审阅时要求 `CLIANY_CREATE_ISSUES_ACK_REVIEW=1`。
- `src/cliany_site/commands/cases.py` 与 `scripts/validate_cases.py`：candidate evidence bundle、promotion plan、issue template 和 validation summary 输出 `expected_adapter_package`，把 adapter release asset 包名模板变成结构化字段；`adapter_package` task 同时输出 `llm_live_preflight_required=true`、`llm_live_preflight_command_sha256`、`llm_live_preflight_evidence_fields`、`doctor_preflight_evidence_fields`、`doctor_preflight_evidence_template`、`doctor_preflight_evidence_template_field_count` 和 `doctor_preflight_evidence_template_sha256`；cases CLI issue template 直接渲染 `Primary Runbook`、LLM/doctor blocker comments、`Doctor Preflight Evidence Fields` 和 `Doctor Preflight Evidence Template`，`issue_template_primary_task` 也输出同一组 acceptance、runbook、preflight 字段、模板映射和模板 digest；promotion plan 输出 `primary_issue_template_command`、`primary_issue_template_json_command`、candidate/task 级 `issue_template_json_command`、`promotion_plan.primary_doctor_preflight_evidence_template_field_count`、`promotion_plan.primary_doctor_preflight_evidence_template_sha256`、`promotion_plan.primary_llm_live_preflight_command_sha256`、candidate `primary_doctor_preflight_evidence_template_sha256`、`task_queue[*].doctor_preflight_evidence_template_sha256` 和 `task_queue[*].llm_live_preflight_command_sha256`；validation summary 的 `promotion_evidence_summary.primary_next_task.doctor_preflight_evidence_template_sha256`、Markdown report 的 `primary_doctor_preflight_evidence_template_sha256`、纯文本 stdout 的 `promotion_evidence_primary_doctor_preflight_evidence_template_sha256` 和 `promotion_evidence_primary_llm_live_preflight_command_sha256` 也会暴露同一漂移信号；human `Candidate 下一步` 输出也会展示 `preflight_required`、`preflight_blocker` 和 `runbook_first`。
- Docs：README、README.zh、官网维护者入口、case 索引和每周维护者循环文档都会展示 daily cap resume 字段、issue 创建 ACK 门禁、human candidate handoff 的 preflight 字段，以及 `Doctor Preflight Evidence Template` 交接说明。

## 风险与兼容性

- 不改变 CLI runtime、adapter schema、action replay、AXTree selector、session 存储或 adapter codegen 输出。
- 不创建 `v0.16.254` tag，不发布 PyPI，不部署官网；本草案只准备下一次可发版切片。
- 新字段为只读 release metadata；未触发 daily cap blocker 时 `daily_release_resume_date` 为 `null`。
- 如果维护者调整每日 release 上限，resume date 仍基于本次 readiness 运行日期和目标 tag 阻塞状态推导。
- 如果 candidate issue gate 仍要求维护者审阅，`create-issues.sh` 的 dry-run 不受影响；非 dry-run 创建公开 issue 前必须显式确认审阅。

## 发版步骤

1. 确认 `v0.16.253` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 等到 2026-07-03 或下一个可发版日期；不要在 2026-07-02 创建 `v0.16.254` tag。
3. 运行 target-mode readiness，确认 `daily_release_cap_blocked=false` 且草案验证通过。
4. 完成 version bump、CHANGELOG、官网基线和 release notes 摘要。
5. 提交并推送 `master`。
6. 创建并推送 `v0.16.254` tag。
7. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
8. 按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署官网，并运行 `cd site && vercel inspect www.cliany.site --wait --timeout 90s` 复核 alias。
9. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json`，确认 GitHub Release、PyPI 和官网发布状态。

## Release Notes 摘要

v0.16.254 prepares a clearer daily release cap and candidate promotion handoff. When a target tag is blocked because the day already reached the release limit, readiness and planner JSON now expose the exact next eligible resume date instead of leaving automation to infer it from prose. Candidate promotion artifacts now also expose `expected_adapter_package` and task-level live preflight gate fields such as `llm_live_preflight_required` and `llm_live_preflight_command_sha256`; direct cases evidence bundles and direct `--issue-template` handoffs include `doctor_preflight_evidence_fields`, `doctor_preflight_evidence_template`, `doctor_preflight_evidence_template_field_count`, and `doctor_preflight_evidence_template_sha256` so CDP and LLM blocker evidence survives without going through planner artifacts or Markdown parsing. The cases issue template now renders `Primary Runbook`, LLM/doctor blocker comments, `Doctor Preflight Evidence Fields`, and a `Doctor Preflight Evidence Template` with paste-ready `doctor --llm-live` placeholders, while `issue_template_primary_task` carries the same acceptance, runbook, and preflight fields in JSON. The promotion plan now exposes `primary_issue_template_command`, `primary_issue_template_json_command`, `promotion_plan.primary_llm_live_preflight_command_sha256`, and candidate/task-level `issue_template_json_command`, so maintainers can move from the ranked queue directly to a copyable issue body and compare preflight command drift. Planner JSON and candidate issue artifacts now also expose `candidate_promotions[].issue_template_command`, `candidate_promotions[].issue_template_json_command`, matching `issue-metadata.json` fields, and Issue Template / Issue Template JSON README columns, so tools that only read planner artifacts can regenerate `cliany-site cases --case-id <id> --issue-template` handoffs without deriving commands from case ids. The human `cliany-site cases --status candidate` handoff renders `preflight_required`, `preflight_blocker`, and `runbook_first`. Planner JSON, Markdown reports, and candidate issue artifact summaries also expose compact primary preflight aliases such as `case_promotion_evidence_primary_llm_live_preflight_required`, `case_promotion_evidence_primary_llm_live_preflight_command_sha256`, `case_promotion_evidence_primary_llm_live_preflight_blocker_comment`, `case_promotion_evidence_primary_doctor_preflight_blocker_comment`, `case_promotion_evidence_primary_doctor_preflight_evidence_template_sha256`, `case_promotion_doctor_preflight_evidence_template_sha256`, and candidate-level `doctor_preflight_evidence_fields`, so summary-only tools can detect the gate and copy a standard CDP/LLM blocker comment without expanding the task object. Candidate issue creation scripts default to `python3` through `PYTHON_BIN` and require `CLIANY_CREATE_ISSUES_ACK_REVIEW=1` when maintainer review is still required, keeping generated handoffs runnable and safer on systems without a bare `python`.
