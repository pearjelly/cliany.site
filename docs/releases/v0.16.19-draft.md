# v0.16.19 发布草案

**目标版本：** `0.16.19`
**提交范围：** `v0.16.18..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让候选 issue artifacts 的 README 明确要求审阅 metadata 复现字段。

## 用户价值

`v0.16.18` 让 `issue-metadata.json` 带上 `target_url`、`commands` 和 `offline_commands`，机器可以直接读取候选 issue 的复现上下文。但维护者打开 artifacts `README.md` 时，审阅清单仍只提醒看 body 文件、发布状态和非目标，没有明确要求检查 metadata 里的 target URL、candidate commands 和 offline validation commands。

`v0.16.19` 把这一步放进 artifacts `README.md` 的 Review Checklist。维护者在运行 `create-issues.sh` 前，会看到需要确认 `issue-metadata.json` 中每个 case 的 target URL、candidate commands 和 offline validation commands，减少 metadata/body 漂移后被直接创建成 GitHub issue 的风险。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.19 --issues-dir /tmp/cliany-candidate-issues` 写出的 artifacts `README.md` 包含一条审阅项，要求检查 `issue-metadata.json` 的 target URL、candidate commands 和 offline validation commands。

## 变更分组

### Candidate issue metadata review checklist

- `scripts/plan_next_iteration.py --issues-dir` 的 artifacts `README.md` 新增 metadata 复现字段审阅项。
- 审阅项要求检查 target URL、candidate commands 和 offline validation commands。
- `issue-metadata.json`、candidate body 文件和 `create-issues.sh` 的结构保持兼容。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifacts README 的审阅清单会提示核对 metadata 复现字段。
- `tests/test_plan_next_iteration.py` 固定 artifacts README 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只让维护者审阅 candidate issue artifacts 时更不容易漏掉结构化复现字段。

## 风险与兼容性

- 这是 artifacts README 文案增强，不改变 JSON schema 和脚本执行行为。
- 审阅项只引用本地生成的 `issue-metadata.json`，不访问第三方站点。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.19 --issues-dir /tmp/cliany-candidate-issues` 写出的 README 包含 metadata 复现字段审阅项。
2. 确认审阅项明确提到 target URL、candidate commands 和 offline validation commands。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.19`。
5. 将 `pyproject.toml` 版本更新为 `0.16.19`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.19`。
7. 创建 tag：`git tag v0.16.19`。

## Release Notes 摘要

`v0.16.19` adds a metadata reproduction-context review item to generated candidate issue artifact READMEs. Maintainers are now prompted to check target URLs, candidate commands, and offline validation commands before creating GitHub issues.
