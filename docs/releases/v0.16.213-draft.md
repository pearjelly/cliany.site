# v0.16.213 发布草案

**目标版本：** `0.16.213`
**提交范围：** `v0.16.212..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本改善 tagged release preflight 通过后的下一步提示。

## 用户价值

`v0.16.212` 已经让 `artifact_bundle_summary` 可以展示 candidate issue gate evidence key boundary。维护者在 tag release preflight 阶段还会阅读 `release_readiness.py --report` 的 `Weekly Review`；当 `--release-tag` 已验证 tag 指向 HEAD、版本匹配且没有 blockers 时，报告仍可能提示 “Ready to tag”，容易让维护者误以为还没有创建 tag。

`v0.16.213` 让 `Weekly Review` 在 tagged release preflight 通过后显示 “Ready to publish verified tag”。这样维护者可以更清楚地区分“下一版还没打 tag”和“当前 tag 已验证、下一步应发布”的状态。

成功标准是：`python scripts/release_readiness.py --strict --release-tag v0.16.213 --report /tmp/cliany-release-readiness.md` 在 tag 指向 HEAD 且所有 gate 通过时，`Weekly Review` 的 “What is the next smallest release slice?” 显示发布已验证 tag，而普通 `--target-version` / next-release 模式仍显示准备打 tag。

## 变更分组

### Tagged release next step

- `release_readiness.py --report` 新增 tagged release preflight 的下一步文案判断。
- 当 `current_version == target_version`、latest tag 匹配当前版本且 `commits_since_latest_tag == 0` 时，`Weekly Review` 显示 `Ready to publish verified tag ...`。
- 普通 next-release readiness 没有 blockers 时，仍显示 `Ready to tag ... after final validation.`。
- 不改变 release readiness blockers、CI gate、release workflow、case validation、package gate 或 publication audit 的判定逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 tagged release preflight 通过后，Weekly Review 会指向发布已验证 tag。
- `tests/test_release_readiness.py` 覆盖 tagged release mode 的 Weekly Review 文案。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 release readiness report 的人工决策提示。

## 风险与兼容性

- 新逻辑只改变 Markdown report 中 Weekly Review 的下一步文案，不改变 JSON schema 或 blocker 判定。
- 旧工具如果只读取 `next_actions`、`blockers` 或 gate fields，不受影响。
- 维护者在 tagged release preflight 通过后，可以从 report 直接看到下一步是发布已验证 tag。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.213..HEAD`。最终带 `--release-tag v0.16.213` 的 strict gate 仍以本草案顶部的 `v0.16.212..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_release_readiness.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/release_readiness.py tests/test_release_readiness.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/release_readiness.py --target-version 0.16.213 --json
python scripts/release_readiness.py --strict --target-version 0.16.213 --report /tmp/cliany-release-readiness.md
python scripts/plan_next_iteration.py --target-version 0.16.213 --json
```

## 发版步骤

1. 确认 `pytest tests/test_release_readiness.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov` 通过。
2. 确认普通 next-release readiness 仍显示准备打 tag。
3. 确认 tagged release mode 的 Weekly Review 显示发布已验证 tag。
4. 确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.213`。
5. 确认 `pyproject.toml` 版本更新为 `0.16.213`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.213`。
7. 创建 tag：`git tag v0.16.213`。

## Release Notes 摘要

`v0.16.213` updates release readiness Markdown reports so successful tagged release preflight points maintainers to publishing the verified tag instead of tagging again.
