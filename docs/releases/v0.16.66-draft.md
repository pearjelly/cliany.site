# v0.16.66 发布草案

**目标版本：** `0.16.66`
**提交范围：** `v0.16.65..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 `candidate_issue_gate` 提供 required actions 的数量与稳定 hash。

## 用户价值

`v0.16.65` 已经让 `candidate_issue_gate.required_actions` 同时覆盖 publication blockers 和 release draft issues。但维护工具如果只想判断 gate 待办是否变化，仍需要逐项比较 action 字符串。

`v0.16.66` 在 `candidate_issue_gate` 中新增 `required_action_count` 和 `required_actions_sha256`。Markdown report 与 candidate issue artifacts `README.md` 也展示 required actions hash，让维护者和自动化能先快速判断 gate action set 是否变化，再决定是否展开完整 actions 列表。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.67 --json` 在 publication 未发布且 release draft 有问题时，`candidate_issue_gate.required_action_count` 等于 `required_actions` 长度，`candidate_issue_gate.required_actions_sha256` 能从 `required_actions` 稳定 JSON 序列化重新计算得到；`--issues-dir` 生成的 `artifact-manifest.json`、`publication-handoff.json` 和 `README.md` 也保留同源 count/hash。

## 变更分组

### Candidate issue gate action summary

- `candidate_issue_gate` 新增 `required_action_count`。
- `candidate_issue_gate` 新增 `required_actions_sha256`。
- Markdown report 的 `Candidate Issue Gate` 展示 required action count 和 hash。
- Candidate issue artifacts `README.md` 的 `Publication Handoff` 展示 gate required action count 和 hash。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `required_action_count`、`required_actions_sha256` 和 required actions hash。
- `tests/test_plan_next_iteration.py` 覆盖 JSON、文本输出、Markdown report 和 artifacts README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让候选 issue gate 的 action set 更容易被工具审计。

## 风险与兼容性

- `required_action_count` 和 `required_actions_sha256` 是新增字段，不改变 `required_actions` 数组语义。
- Hash 基于已有 action 字符串计算，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.66..HEAD`。最终带 `--release-tag v0.16.66` 的 strict gate 仍以本草案顶部的 `v0.16.65..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.66 --json
python scripts/plan_next_iteration.py --target-version 0.16.66 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.66 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.66 --json` 输出 gate action count/hash。
2. 用缺失草案的后续目标版本确认 `required_action_count` 与 `required_actions` 长度一致。
3. 用缺失草案的后续目标版本确认 `required_actions_sha256` 能从 `required_actions` 重新计算得到。
4. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.66 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
5. 确认 `artifact-manifest.json`、`publication-handoff.json` 和 artifacts `README.md` 保留同源 count/hash。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.66`。
8. 将 `pyproject.toml` 版本更新为 `0.16.66`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.66`。
10. 创建 tag：`git tag v0.16.66`。

## Release Notes 摘要

`v0.16.66` adds count and hash metadata for candidate issue gate required actions, helping maintainers and tools detect action-set changes quickly.
