# v0.16.31 发布草案

**目标版本：** `0.16.31`
**提交范围：** `v0.16.30..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 publish script 固定生成时的仓库根目录，避免从错误目录执行时误判或误推。

## 用户价值

`v0.16.30` 已经让 `scripts/check_release_publication.py --publish-script` 生成的脚本在 push 前检查 HEAD、latest tag 和 tag commit 是否仍然匹配。但脚本通常写在 `/tmp`，维护者可能从任意目录执行它；如果当前目录不是 cliany.site 仓库，脚本里的 `git rev-parse` 和 `git push` 可能作用在错误工作区。

`v0.16.31` 让 publication report 记录 `repo_root`，并让生成的 publish script 写入 `REPO_ROOT`。脚本执行时会先 `cd "$REPO_ROOT"`，再用 `git rev-parse --show-toplevel` 确认当前仓库根目录仍然匹配；只有 repo root、HEAD、latest tag 和 tag commit 都一致后，才会继续执行 `git push origin master`、`git push origin v0.16.31` 和远端复核命令。

成功标准是：`python scripts/check_release_publication.py --json --publish-script /tmp/cliany-publish-release.sh` 写出的脚本包含 `REPO_ROOT`、`cd "$REPO_ROOT"` 和 `git rev-parse --show-toplevel`；脚本从仓库外目录启动时，也会回到生成时的 repo root 做 stale preflight。

## 变更分组

### Publication publish script repo root pinning

- `scripts/check_release_publication.py` 的 JSON、文本和 Markdown publication report 记录 `repo_root`。
- `--publish-script` 生成的脚本在 `Publication context` 中展示 `repo_root`。
- 生成脚本执行时先进入 `REPO_ROOT` 并确认 `git rev-parse --show-toplevel` 匹配，再继续 stale preflight 和 publish commands。

### 文档与门禁

- [docs/release-cadence.md](../release-cadence.md) 说明 publish script 会固定 repo root。
- `tests/test_release_publication.py` 覆盖 report 字段、脚本字段和从仓库外执行的 stale 检查。
- `tests/test_release_cadence_docs.py` 固定发布节奏文档说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强 release publication handoff 脚本的执行位置保护。

## 风险与兼容性

- `repo_root` 是 additive publication report 字段，不移除现有 JSON key。
- 脚本从任意目录运行时会切回生成时的仓库根目录；如果该目录不存在或不再是同一个 git root，会输出 `Publish script is stale` 并退出。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- `--remote` 行为保持不变；repo root 检查只读取本地文件系统和 git 状态。

## 发版前验证

```bash
pytest tests/test_release_publication.py tests/test_release_cadence_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/check_release_publication.py tests/test_release_publication.py tests/test_release_cadence_docs.py tests/test_release_draft_docs.py
python scripts/check_release_publication.py --json --publish-script /tmp/cliany-publish-release.sh
python scripts/release_readiness.py --target-version 0.16.31 --json
```

## 发版步骤

1. 确认 `python scripts/check_release_publication.py --json --publish-script /tmp/cliany-publish-release.sh` 写出的 JSON 和 Markdown report 包含 `repo_root`。
2. 确认脚本包含 `REPO_ROOT`、`cd "$REPO_ROOT"` 和 `git rev-parse --show-toplevel`。
3. 确认脚本从仓库外目录启动时仍会回到生成时的 repo root 做 stale preflight。
4. 运行 publication、release cadence 文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.31`。
6. 将 `pyproject.toml` 版本更新为 `0.16.31`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.31`。
8. 创建 tag：`git tag v0.16.31`。

## Release Notes 摘要

`v0.16.31` pins generated publish scripts to the repository root used during publication audit, so stale checks and push commands run against the intended worktree even when the script is launched from another directory.
