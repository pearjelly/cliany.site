# v0.16.55 发布草案

**目标版本：** `0.16.55`
**提交范围：** `v0.16.54..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue gate evidence 直接展示 worktree 与 release draft 的布尔门禁证据。

## 用户价值

`v0.16.54` 已经让 `candidate_issue_gate.evidence.release_draft_path` 使用仓库相对路径，交接材料不再泄漏维护者本机路径。但维护者或自动化判断 gate 原因时，仍需要从 `publication_visibility_status`、`release_draft_issue_count` 或其他字段间接推断 worktree 是否干净、release draft 是否通过。

`v0.16.55` 给 `candidate_issue_gate.evidence` 增加 `publication_worktree_clean` 和 `release_draft_ok`。这样 gate evidence 可以直接表达两个常见门禁：本地发布状态是否可继续审阅、目标 release draft 是否已经准备好。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.55 --json` 输出的 `candidate_issue_gate.evidence` 包含 `publication_worktree_clean` 和 `release_draft_ok`；Markdown report 和 candidate issue artifacts README 也展示这两个字段。

## 变更分组

### Candidate issue gate boolean evidence

- `candidate_issue_gate.evidence` 新增 `publication_worktree_clean`。
- `candidate_issue_gate.evidence` 新增 `release_draft_ok`。
- Markdown report 的 `Candidate Issue Gate Evidence` 表格展示这两个布尔字段。
- Artifacts `README.md` 的 `Publication Handoff` 展示 gate evidence worktree clean 和 release draft ok。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明新 evidence 字段。
- `tests/test_plan_next_iteration.py` 覆盖 JSON、Markdown report、文本输出和 artifacts README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强 candidate issue gate evidence 的可审计性。

## 风险与兼容性

- `candidate_issue_gate.evidence` 只新增字段，不改变已有字段语义。
- 新字段复用已有 publication audit 与 release readiness 数据，不新增网络访问。
- 不自动执行命令，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.55..HEAD`。最终带 `--release-tag v0.16.55` 的 strict gate 仍以本草案顶部的 `v0.16.54..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov

python scripts/plan_next_iteration.py --target-version 0.16.55 --json
python scripts/plan_next_iteration.py --target-version 0.16.55 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.55 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.55 --json` 输出 `publication_worktree_clean` 和 `release_draft_ok`。
2. 确认 Markdown report 的 `Candidate Issue Gate Evidence` 展示这两个字段。
3. 确认 `artifact-manifest.json`、`publication-handoff.json` 和 artifacts `README.md` 都展示同源 evidence。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.55`。
6. 将 `pyproject.toml` 版本更新为 `0.16.55`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.55`。
8. 创建 tag：`git tag v0.16.55`。

## Release Notes 摘要

`v0.16.55` adds explicit boolean gate evidence for worktree cleanliness and release draft readiness, making candidate issue gate decisions easier to audit from plans and candidate issue artifacts.
