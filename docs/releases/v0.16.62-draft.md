# v0.16.62 发布草案

**目标版本：** `0.16.62`
**提交范围：** `v0.16.61..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts 提供 `issue_body_summary`，方便快速审计整批 body 文件。

## 用户价值

`v0.16.61` 已经让 `artifact-manifest.json` 记录每个 candidate issue body 的 `byte_count` 和 `sha256`。维护者和工具可以逐个核对 body 文件，但如果只想快速判断整批 body 是否变化，还需要读取完整 `issue_body_inventory` 并自行计算摘要。

`v0.16.62` 在 `artifact-manifest.json` 中新增 `issue_body_summary`，记录 `body_count`、`total_byte_count` 和 `inventory_sha256`。Artifacts `README.md` 也新增 `Issue Body Summary` 小节，让维护者能在不解析 JSON 的情况下看到同一组汇总信息。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.62 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json` 包含 `issue_body_summary`；README 包含 `Issue Body Summary`、`body_count`、`total_byte_count` 和 `inventory_sha256`；`inventory_sha256` 可由 `issue_body_inventory` 重新计算得到。

## 变更分组

### Candidate issue body summary

- `artifact-manifest.json` 新增 `issue_body_summary`。
- `issue_body_summary` 包含 `body_count`、`total_byte_count` 和 `inventory_sha256`。
- Artifacts `README.md` 新增 `Issue Body Summary` 小节。
- Summary 基于不含本机绝对路径的 `issue_body_inventory` 计算，便于复现。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `issue_body_summary` 与 README 汇总字段。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README，并用实际 inventory 计算汇总 hash。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让候选 issue artifacts 的 body 文件集合更容易审计。

## 风险与兼容性

- `issue_body_summary` 是新增 manifest 字段，不改变 `issue_body_inventory` 或 body 文件内容。
- `inventory_sha256` 基于 inventory JSON 的稳定序列化结果，不包含本机路径。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.62..HEAD`。最终带 `--release-tag v0.16.62` 的 strict gate 仍以本草案顶部的 `v0.16.61..HEAD` 作为实际 release diff 范围。


## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.62 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json` 包含 `issue_body_summary`。
3. 确认 `issue_body_summary.inventory_sha256` 可由 `issue_body_inventory` 重新计算得到。
4. 确认 artifacts `README.md` 包含 `Issue Body Summary`。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.62`。
7. 将 `pyproject.toml` 版本更新为 `0.16.62`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.62`。
9. 创建 tag：`git tag v0.16.62`。
