# v0.16.242 发布草案

**目标版本：** `0.16.242`
**提交范围：** `v0.16.241..HEAD`
**当前阻塞：** 2026-06-17 已经发布超过 3 个版本；本草案只准备下一版，不在当天继续创建 tag、GitHub Release、PyPI 发布或官网生产部署。

## 用户价值

`v0.16.241` 之前已经把项目节奏升级为每天至少一个可验证版本，但缺少“最多 3 个版本/日”的机器可读门禁。维护者或自动化在一天内连续修复 release gate 时，可能继续 tag 多个 patch，导致外部用户看到噪声较高的 release stream。

`v0.16.242` 把每日发布上限纳入 cadence / readiness / planner 三个入口：`scripts/check_release_cadence.py --json` 直接输出今日 release tag 数量和 `daily_release_limit_ok`，`scripts/release_readiness.py` 在超过上限时阻塞 tag，`scripts/plan_next_iteration.py` 把同一组字段透传到周计划和 candidate issue artifacts。这样项目仍保持每天至少一个版本的持续动能，但达到 3 个版本后会把后续工作转为计划、验证或未发布准备，等下一天再走标准发布流程。

## 变更分组

- Cadence gate：`check_release_cadence.py` 统计当天指向当天提交的 `v*` release tags，并输出 `release_tags_today`、`release_count_today`、`max_daily_releases` 和 `daily_release_limit_ok`。
- Release readiness：超过每日上限时，strict readiness 会给出 cadence blocker 和 “pause release tagging until the next day” 的 next action。
- Planner handoff：`plan_next_iteration.py` 的 `commit_cadence` 摘要、artifact bundle summary 和 artifacts README 会显示当天发布数量、上限、是否可继续 tag，以及 release tag 列表 hash。
- Docs / website source：README、README.zh、Q3 roadmap、release cadence 文档和官网源码同步说明每天 1~3 个可验证版本。
- Tests：新增 release cadence 上限测试，并更新 README、release cadence docs 和官网内容断言。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的案例状态；相关说明仍落在 [cases/README.md](../../cases/README.md)。
- 当前案例库仍保持 active 4 个、candidate 3 个、known gap 1 个。
- Known gap 保持 1 个：`search-extraction-gap`。
- Candidate 晋级仍需要 adapter package、metadata validation 和 online smoke 三段真实证据。
- 本版不生成 candidate adapter package，也不把任何 candidate 标记为 active；每日发布上限命中时，candidate 晋级准备只能作为未发布工作保留到下一天。

## 风险与兼容性

- 这是 release tooling / docs / website source 的 additive guard，不改变 CLI adapter schema、AXTree 语义匹配、action runtime、session 存储或 marketplace package 格式。
- 今日 release tag 数量基于本地 git tag 指向提交的提交日期计算；它适合离线发版门禁，不替代 GitHub Release 远端审计。
- 当一天内已经超过上限时，readiness 会保持 blocked；维护者应停止 tag 发布，但可以继续准备下一版草案、测试和未发布代码。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.242..HEAD`。最终带 `--release-tag v0.16.242` 的 strict gate 仍以本草案顶部的 `v0.16.241..HEAD` 作为实际 release diff 范围。

```bash
uv run --extra dev pytest tests/test_release_cadence.py tests/test_release_cadence_docs.py tests/test_readme_current_features.py tests/test_site_content.py -q
uv run --extra dev pytest tests/test_release_readiness.py tests/test_plan_next_iteration.py -q
uv run ruff check scripts/check_release_cadence.py scripts/release_readiness.py scripts/plan_next_iteration.py tests/test_release_cadence.py tests/test_release_cadence_docs.py tests/test_readme_current_features.py tests/test_site_content.py
python3 scripts/check_release_cadence.py --json
python3 scripts/plan_next_iteration.py --target-version 0.16.242 --json
python3 scripts/validate_cases.py --strict
python3 scripts/release_readiness.py --target-version 0.16.242 --json
git diff --check
```

正式 tag 前再运行：

```bash
uv run --extra dev pytest tests/ -q
python3 scripts/release_readiness.py --strict --target-version 0.16.242
```

## 发版步骤

1. 等到新的自然日，确认 `release_count_today < max_daily_releases`。
2. 确认 cadence、readiness、planner、README/README.zh、官网源码、CHANGELOG 和本草案已同步。
3. 运行聚焦测试、ruff、cadence JSON、planner JSON、case validation、release readiness 和完整测试。
4. 提交并推送 `master`。
5. 创建并推送 `v0.16.242` tag。
6. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
7. 官网源码已改动；按标准发布流程在 `site/` 执行 Vercel production deploy，并记录 production deployment。

## Release note draft

v0.16.242 adds a machine-readable daily release cap to cadence, readiness, and next-iteration planning. Maintainers can now see today's release count, the 3-release cap, and whether another tag is allowed before they publish another GitHub Release, PyPI package, or website deployment.
