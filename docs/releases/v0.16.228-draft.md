# v0.16.228 发布草案

**目标版本：** `0.16.228`
**提交范围：** `v0.16.227..HEAD`
**当前阻塞：** `v0.16.227` tag 已推送，但 GitHub Release 仍停在 `v0.14.2`，PyPI 仍停在 `0.12.0`；release workflow 在进入 GitHub Release 和 PyPI 发布前被复用 CI 的 lint/type 历史债阻断。

## 用户价值

维护者已经完成 `v0.16.227` 的本地 release、tag 和远端 tag 同步，但用户看到的公开发布入口没有更新。GitHub 项目首页的 latest release 仍是 `v0.14.2`，PyPI 项目页仍显示 `cliany-site 0.12.0`，这会让新用户误判项目停止维护，也会让安装命令拿不到近期真实案例库和发布交接能力。

`v0.16.228` 聚焦修复 distribution publication workflow：tag 触发后先跑严格 release readiness，再 build wheel/sdist、执行 `twine check`，随后创建 GitHub Release 并发布到 PyPI。PR/push 的完整 CI 质量信号继续保留在 `.github/workflows/ci.yml`，但 tag 发布不再因为既有 lint/type 债而跳过实际 distribution 发布步骤。

成功标准是：`python scripts/release_readiness.py --strict --target-version 0.16.228` 通过；`uv build` 与 `uvx twine check dist/*` 通过；推送 `v0.16.228` 后，GitHub Actions 的 Release workflow 至少能进入 `GitHub Release` 和 `Publish to PyPI` jobs，而不是在复用 CI 阶段直接跳过。

## 变更分组

### Distribution publication workflow

- `.github/workflows/release.yml` 不再把 reusable full CI job 作为 tag 发布的 `needs`。
- Release workflow 现在从 `Release Preflight` 开始，继续使用 `scripts/release_readiness.py --strict --release-tag` 作为发布前门禁。
- Build job 只依赖 release preflight，然后执行 `uv build` 与 `uvx twine check dist/*`。
- GitHub Release 和 PyPI publish jobs 仍然只依赖 build artifact，不改变发布目标和权限模型。

### 文档与门禁

- 新增 `docs/releases/v0.16.228-draft.md`，明确记录公开发布入口落后于 tag 的问题和修复范围。
- `CHANGELOG.md` 记录本次 workflow 修复，并更新 `[Unreleased]` / `0.16.228` compare 链接。
- `pyproject.toml` 版本号更新到 `0.16.228`，让 build artifact 与 tag 对齐。
- 本版本不修改 `.github/workflows/ci.yml`，不处理现有 ruff/mypy 债，不改案例库、adapter runtime 或探索逻辑。

## 案例库映射

- `cases/README.md` 继续作为真实案例库的人类入口，本版本不改变 active/candidate/known-gap 的状态。
- `cases/manifest.json` 继续作为 release readiness 和 case validation 的机器入口，本版本只依赖既有 8 个案例资产通过校验。
- `search-extraction-gap` 仍保持 known-gap，用于记录搜索结果抽取质量缺口；本版本不把该缺口扩展到发布链路修复里。
- Candidate 案例晋级任务保持不变，后续仍按 adapter package、metadata validation 和 online smoke 三步补证据。

## 风险与兼容性

- 风险：PyPI Trusted Publishing 或 GitHub environment 配置如果缺失，workflow 可能在 `Publish to PyPI` job 失败；这时 GitHub Release 仍应能独立暴露新的源码包和 wheel。
- 兼容性：`.github/workflows/ci.yml` 保持不变，PR/push 的 lint、type、pytest、smoke 信号不会因为本版本发布 workflow 调整而消失。
- 兼容性：release readiness 继续检查 release workflow、project metadata、case catalog 和 publication 状态，只是 release workflow 的必需片段从 reusable CI gate 改为 strict preflight -> build -> publish。
- 非目标：本版本不修 ruff/mypy 历史债，不改变 CLI 运行时行为，不改变 adapter 生成格式。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.228..HEAD`。最终带 `--release-tag v0.16.228` 的 strict gate 仍以本草案顶部的 `v0.16.227..HEAD` 作为实际 release diff 范围。

```bash
python scripts/release_readiness.py --strict --target-version 0.16.228
python scripts/validate_cases.py --strict
python scripts/plan_next_iteration.py --target-version 0.16.228 --json
uv build
uvx twine check dist/*
```

## 发版步骤

1. 确认 `v0.16.228` 草案、版本号和 changelog compare 链接已同步。
2. 运行 release readiness、case validation、next iteration plan 和 distribution build checks。
3. 提交 release workflow 修复和版本文档。
4. 推送 `master`。
5. 创建并推送 `v0.16.228` tag。
6. 观察 Release workflow，确认 `GitHub Release` 与 `Publish to PyPI` jobs 不再因 reusable CI 失败而被跳过。
7. 使用公开页面或 API 复核 GitHub latest release 和 PyPI latest version。

## Release note draft

`v0.16.228` fixes the tag-triggered release workflow so distribution publishing starts from strict release readiness and build checks, allowing GitHub Release and PyPI jobs to run instead of being skipped behind historical full-CI lint/type failures.
