# v0.16.205 发布草案

**目标版本：** `0.16.205`
**提交范围：** `v0.16.204..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 release draft handoff boundary 显式化到整包摘要。

## 用户价值

`v0.16.204` 已经让 `artifact_bundle_summary` 可以展示 publication worktree status 的首尾边界。维护者在审阅 candidate issue artifacts 时，还需要确认 release draft handoff 的结构是否仍从 `schema_version` 开始，并以 `target_version` 结束；目前 summary 已有 handoff key count、schema version、primary issue/action 和整体 hash，但只读 summary 的工具仍无法快速确认 handoff key 边界是否符合预期。

`v0.16.205` 在 `artifact_bundle_summary` 中新增 `release_draft_handoff_first_key`、`release_draft_handoff_last_key` 和 `release_draft_handoff_key_boundary_sha256`。这样维护者和自动化可以只读整包摘要，直接确认 release draft handoff 的 key 首尾边界、字段数量和完整 hash，再决定是否展开完整 `release-draft-handoff.json`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.205 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 release draft handoff boundary fields，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且 first/last key 等于 `release-draft-handoff.json` 的首尾 key。

## 变更分组

### Release draft handoff boundary

- `artifact_bundle_summary` 新增 `release_draft_handoff_first_key`。
- `artifact_bundle_summary` 新增 `release_draft_handoff_last_key`。
- `artifact_bundle_summary` 新增 `release_draft_handoff_key_boundary_sha256`。
- Boundary 从 `release-draft-handoff.json` 的稳定 key 顺序派生；无 handoff keys 时首尾为 `null`。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 release draft handoff first/last key 与 boundary hash。
- 现有 `release_draft_handoff_key_count`、`release_draft_handoff_schema_version`、`release_draft_handoff_primary_issue`、`release_draft_handoff_primary_required_action`、`release_draft_handoff_sha256`、publication worktree status boundary、publication publish command boundary、publication ref context boundary、create issues safety contract boundary、review checklist boundary、validation command boundary、release draft boundary、candidate issue gate boundary 和 publication summary 字段保持兼容。
- 不改变 release draft handoff 内容、release draft validation、publication audit、remote check、publish commands 内容或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 release draft handoff boundary fields。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的 release draft handoff boundary 字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 `release-draft-handoff.json`，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `release_draft_handoff_first_key`、`release_draft_handoff_last_key`、`release_draft_handoff_key_boundary_sha256`、`release_draft_handoff_key_count` 和 `release_draft_handoff_sha256` 快速确认 release draft handoff 是否漂移。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.205 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `release_draft_handoff_first_key`、`release_draft_handoff_last_key` 和 `release_draft_handoff_key_boundary_sha256`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示这三个字段。
4. 确认 `release_draft_handoff_first_key` 与 `release_draft_handoff_last_key` 等于 `release-draft-handoff.json` 的首尾 key。
5. 确认 `release_draft_handoff_key_count`、`release_draft_handoff_sha256` 和完整 release draft handoff 仍保持同源。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.205`。
8. 将 `pyproject.toml` 版本更新为 `0.16.205`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.205`。
10. 创建 tag：`git tag v0.16.205`。

## Release Notes 摘要

`v0.16.205` adds release draft handoff first/last key and boundary hash fields to the artifact bundle summary, helping tools inspect handoff boundaries before expanding release-draft-handoff.json.
