# v0.16.249 发布草案

**目标版本：** `0.16.249`
**提交范围：** `v0.16.248..HEAD`
**当前阻塞：** 本草案用于准备 `v0.16.248` 之后的下一版。在 release readiness、完整验证和标准发布流程通过前，不创建或推送 `v0.16.249` tag。`pypi-project-search` candidate 晋级仍必须先通过 live LLM gate；`E_LLM_UNAVAILABLE`、`E_UNKNOWN` connection error、`ready_for_explore=false` 或 `generate_adapters.ready=false` 只能作为 blocker 证据，不能替代 adapter package、metadata validation 或 online smoke 证据。

## 用户价值

`v0.16.248` 已完成 GitHub Release、PyPI 和官网生产部署的标准发布闭环，并让 `scripts/check_release_publication.py --remote --distribution --json` 可以证明公开渠道已经暴露最新版本。`v0.16.249` 的重点是把这次真实发布状态重新校准到公开路线图、维护者路线图、官网维护者循环和下一版草案里，让外部用户看到当前基线已经是 `v0.16.248`，维护者也能继续沿着同一条真实案例晋级主线推进。

本版不把 `pypi-project-search` 提前宣传成 active，也不把 live LLM preflight 失败包装成 candidate adapter evidence。当前价值是减少下一次发布判断成本：草案、路线图、官网和 release readiness 都指向同一个下一步，即在 live LLM gate 恢复后补齐 adapter package、metadata validation 和 online smoke 三段真实证据。

## 变更分组

- Release draft：新增本草案，承接 `v0.16.248` 后的下一版准备。
- Roadmap calibration：`docs/public-roadmap.md` 和 `docs/roadmap-2026-q3.md` 的基线更新到 `v0.16.248`，并把 GitHub Release / PyPI / 官网发布闭环写入当前状态。
- Website baseline：官网 quickstart 的 Maintainer Loop 文案显示 `Current baseline: v0.16.248`，让网站读者看到当前 release train 基线。
- Candidate gate：继续把 `cliany-site doctor --llm-live --json` 作为 `pypi-project-search` 真实 `explore` 的前置门禁；preflight 失败时保持 candidate 证据 pending。
- Changelog：`CHANGELOG.md` 的 Unreleased 记录 `v0.16.249` 下一版准备；正式发布时再移动到 `0.16.249` 小节。

## Candidate preflight evidence

`pypi-project-search` 晋级命令计划：

```bash
python -m cliany_site doctor --llm-live --json
cliany-site explore "https://pypi.org" "search Python packages for cliany-site and list project names" --json
python scripts/validate_cases.py --packages-dir ~/.cliany-site/packages --include-candidate-packages --strict
cliany-site pypi.org search-projects --query cliany-site --limit 5 --json
```

执行真实 `explore` 前必须先运行 live LLM preflight。如果 `llm_live` 返回 `E_LLM_UNAVAILABLE`、`E_UNKNOWN` connection error，或 `ready_for_explore=false` / `generate_adapters.ready=false`，本轮停止真实 `explore`，不生成或伪造 `pypi.org-<version>.cliany-adapter.tar.gz`，并将结果记录为阻塞证据。

2026-07-01 初始 readiness 显示 `v0.16.249` 目标版本 publication 基线可用：`master` 与 `origin/master` 对齐，最新 tag `v0.16.248` 已发布并指向 HEAD，GitHub Release 和 PyPI 均公开暴露 `0.16.248`。当时 release gate 的主要阻塞是本草案缺失；本草案补齐后，下一步应继续根据 release readiness 输出处理真实验证与发版准备。

2026-07-01 重试 live LLM preflight。Chrome/CDP、LLM key、provider、OpenAI-compatible base URL、runtime dirs、registry 和 browser provider 可用，但 `llm_live` 返回 `E_UNKNOWN` connection error，`phase=llm_preflight`，`ready_for_explore=false`，且 `generate_adapters.ready=false`。因此本轮停止 `pypi-project-search` 的真实 `explore`，不生成或伪造 candidate adapter package，并保持三段晋级证据为 pending。

## 风险与兼容性

- 本草案只更新文档、官网展示和下一版发布准备，不改变 CLI 参数、JSON envelope、adapter metadata schema、AXTree 语义匹配、action runtime、session 存储或 marketplace package 格式。
- Candidate 晋级证据不得用 preflight 失败代替 adapter package；失败只能解释为什么保持 pending。
- 禁止引入脆弱 CSS selector 兜底，后续 adapter 生成仍必须依赖 AXTree 语义匹配。
- 运行时状态仍必须写入 `~/.cliany-site/`，不得把 adapters、sessions、snapshots 或 packages 写进 repo。
- 官网更新需要按 `AGENTS.md` 从 `site/` 目录部署到 `cliany.site` Vercel 项目，避免误部署到默认 `site` 项目。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.248..HEAD`。最终带 `--release-tag v0.16.249` 的 strict gate 仍以本草案顶部的 `v0.16.248..HEAD` 作为实际 release diff 范围。

```bash
git diff --check
python scripts/validate_cases.py --strict
pytest tests/test_site_content.py tests/test_release_draft_docs.py tests/test_release_readiness.py -q --no-cov
python scripts/plan_next_iteration.py --target-version 0.16.249 --json
python scripts/release_readiness.py --target-version 0.16.249 --json --remote --remote-name origin
python scripts/check_release_publication.py --remote --distribution --json
```

正式 tag 前再运行：

```bash
CLIANY_QA_OFFLINE=1 pytest tests/ -q
python scripts/release_readiness.py --strict --release-tag v0.16.249 --remote --remote-name origin
```

## 发版步骤

1. 确认 `v0.16.248` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 重试 live LLM preflight；只有 `generate_adapters.ready=true` 时才进入 `pypi-project-search` 真实 `explore`。
3. 完成或明确延后 `pypi-project-search` 的 adapter package、metadata validation 和 online smoke 证据。
4. 确认 `CHANGELOG.md`、本草案、公开路线图、维护者路线图、官网和 case evidence 状态一致。
5. 运行文档相关测试、case validation、完整离线 pytest 和 strict release readiness。
6. 将 `CHANGELOG.md` 的 Unreleased 条目移动到 `## [0.16.249] - <date>`，并将 `pyproject.toml` 版本号更新到 `0.16.249`。
7. 提交并推送 `master`。
8. 创建并推送 `v0.16.249` tag。
9. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
10. 运行 `python scripts/check_release_publication.py --remote --distribution --json`，确认 `distribution_ok=true`、GitHub Release latest tag 为 `v0.16.249`，PyPI latest version 为 `0.16.249`。
11. 官网源码已更新，本版发布后按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署到 `cliany.site` 项目。
