# v0.16.98 发布草案

**目标版本：** `0.16.98`
**提交范围：** `v0.16.97..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 release draft ok 布尔值补进 candidate issue artifacts 的整包摘要。

## 用户价值

`v0.16.97` 已经让 `artifact_bundle_summary` 能展示 `release_draft_path` 和 `release_draft_path_sha256`。维护者和自动化继续审阅候选 issue artifacts 时，还需要不用解释 `release_draft_issue_count == 0` 就能直接判断 release draft gate 是否通过。

`v0.16.98` 在 `artifact_bundle_summary` 中新增 `release_draft_ok`。Artifacts `README.md` 的 `Artifact Bundle Summary` 也展示该布尔值，让工具只读整包摘要就能区分 release draft 已通过、仍有 issues，或需要展开 `release-draft-handoff.json` 查看细节。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.98 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `release_draft_ok`，且该值与 `artifact-manifest.json.release_draft_issues` 是否为空一致；README 的 `Artifact Bundle Summary` 也展示该字段。

## 变更分组

### Artifact bundle release draft ok summary

- `artifact_bundle_summary` 新增 `release_draft_ok`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示 release draft ok 布尔值。
- 现有 `release_draft_issue_count`、`release_draft_issues_sha256`、`release_draft_path` 和 `release_draft_handoff_sha256` 保持不变。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 release draft ok。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让候选 issue artifacts 的整包摘要更容易判断 release draft gate 是否通过。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary`，不改变 `release_draft_issues` 或 `release-draft-handoff.json` 的既有语义。
- 布尔值从同一组 release draft issues 推导，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.98..HEAD`。最终带 `--release-tag v0.16.98` 的 strict gate 仍以本草案顶部的 `v0.16.97..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.98 --json
python scripts/plan_next_iteration.py --target-version 0.16.98 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.98 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.98 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `release_draft_ok`。
3. 确认该值与 `artifact-manifest.json.release_draft_issues` 是否为空一致。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示该字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.98`。
7. 将 `pyproject.toml` 版本更新为 `0.16.98`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.98`。
9. 创建 tag：`git tag v0.16.98`。

## Release Notes 摘要

`v0.16.98` adds an explicit release draft ok flag to candidate issue artifact bundle summaries, helping tools decide draft gate status without expanding the full release draft handoff.
