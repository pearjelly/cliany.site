# v0.16.252 发布草案

**目标版本：** `0.16.252`
**提交范围：** `v0.16.251..HEAD`
**当前状态：** 本版修复发布后公开渠道审计在 PyPI latest 缓存滞后时的 false negative，并准备作为 2026-07-02 的第二个可验证 patch 发布。

## 用户价值

`v0.16.251` 的标准发布流程已经跑通 GitHub Release、PyPI 和官网生产部署，但最终 publication audit 期间暴露了一个真实维护问题：PyPI 的版本专属页面和 JSON endpoint 已经能返回 `0.16.251`，项目级 `https://pypi.org/pypi/cliany-site/json` 的 `info.version` 却短暂仍是 `0.16.250`。旧审计只看 project latest，因此会把一个已经上传成功的 release 判成未发布。

`v0.16.252` 将 PyPI 检查改成两层证据：先读取 project latest；如果 latest 还不是目标版本，再读取 `https://pypi.org/pypi/<project>/<version>/json`。版本专属 endpoint 确认目标版本且存在发布文件时，`distribution_ok=true`，同时报告 `pypi_latest_version` 和 `pypi_release_version`，让维护者知道这是缓存滞后而不是包未发布。

这让每天 1-3 个版本的 release train 更稳：发布后不需要无意义地等待 latest cache 才能确认 PyPI 版本已公开，也不会隐藏真正缺失的版本或文件。

## 变更范围

- `scripts/check_release_publication.py`：PyPI project latest 不匹配目标版本时，查询版本专属 JSON endpoint；目标版本存在且带文件时接受发布成功。
- Distribution report：JSON、文本和 Markdown report 新增 `pypi_latest_version` / `pypi_release_version`，`pypi_version` 继续表示用于最终发布判断的版本。
- Docs / website：README、README.zh、release cadence、weekly maintainer loop、public roadmap 和官网维护者说明同步解释这些字段。

## 风险与兼容性

- 不改变 CLI runtime、adapter schema、action replay、AXTree selector 或 session 存储。
- `pypi_version` 保持兼容；新增字段只扩展 distribution report。
- 如果版本专属 endpoint 不存在、版本不匹配或没有文件，审计仍会失败并提示等待或修复 PyPI version。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.251..HEAD`。完成 version bump 后，target-mode readiness 也会查找占位锚点 **提交范围：** `v0.16.252..HEAD`；该锚点只表示 tag 发布后不应再有未发布差异，不改变本版实际提交范围。创建本地 tag 后，使用 local tag-mode readiness 复核最终 tag 语义。

```bash
git diff --check
uv run --extra dev --frozen pytest tests/test_release_publication.py -q --no-cov
uv run --extra dev --frozen pytest tests/test_release_readiness.py tests/test_plan_next_iteration.py -q --no-cov --tb=short
uv run --extra dev --frozen python scripts/validate_cases.py --strict
uv run --extra dev --frozen python scripts/release_readiness.py --strict --target-version 0.16.252 --remote --remote-name origin --json
```

创建本地 tag 后、推送 tag 前再运行：

```bash
CLIANY_QA_OFFLINE=1 uv run --extra dev --frozen pytest tests/ -q
uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.252 --remote --remote-name origin
```

## 发版步骤

1. 确认 `v0.16.251` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 确认 2026-07-02 当天 release count 未超过每日上限。
3. 提交并推送 `master`。
4. 创建并推送 `v0.16.252` tag。
5. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
6. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json`，确认 `distribution_ok=true`、GitHub Release latest tag 为 `v0.16.252`，PyPI `pypi_version` 为 `0.16.252`。
7. 按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署官网。

## Release Notes 摘要

v0.16.252 makes the post-release publication audit resilient to PyPI project latest cache lag. When the version-specific PyPI JSON endpoint already exposes the target version and files, the audit now accepts the release while still reporting the stale latest value through `pypi_latest_version`.
