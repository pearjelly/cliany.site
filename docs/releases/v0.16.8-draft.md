# v0.16.8 发布草案

**目标版本：** `0.16.8`
**提交范围：** `v0.16.7..HEAD`
**当前阻塞：** 需要提交 issue artifacts 复现命令、维护者循环入口和测试；最新本地 release 仍需维护者确认后 push 到远端。

## 用户价值

`v0.16.7` 让 `scripts/plan_next_iteration.py --issues-dir` 生成的候选 issue artifacts 自带 `README.md`，维护者可以审阅 body、metadata 和 `create-issues.sh`。

`v0.16.8` 继续补齐这个交接包的可复现性：计划 JSON 新增 `issue_artifacts_command`，artifacts `README.md` 也会写入同一条复现命令。维护者不需要从周初命令或聊天记录里猜输出目录，只要复制 `python scripts/plan_next_iteration.py --target-version 0.16.8 --issues-dir /tmp/cliany-candidate-issues` 就能重新生成候选 issue artifacts。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.8 --json` 输出 `issue_artifacts_command`，并且 `python scripts/plan_next_iteration.py --target-version 0.16.8 --issues-dir /tmp/cliany-candidate-issues` 写出的 `README.md` 包含同一条复现命令。

## 变更分组

### Issue artifacts 复现命令

- `scripts/plan_next_iteration.py` 的计划模型新增 `issue_artifacts_command`。
- JSON 输出固定 `/tmp/cliany-candidate-issues` 作为推荐复现目录。
- `--issues-dir` 生成的 `README.md` 会在验证命令区展示复现命令。
- 默认运行不写 artifacts；只有显式传入 `--issues-dir` 才写 body、`issue-metadata.json`、`README.md` 和 `create-issues.sh`。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `issue_artifacts_command` 和 artifacts README 会保留复现命令。
- `tests/test_plan_next_iteration.py` 固定 JSON 字段和 README 内容。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环里的复现说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。计划器会为 `pypi-project-search`、`npm-package-search`、`crates-io-crate-search` 三个 candidate cases 输出 issue artifacts，并把复现命令写入 README。

`search-extraction-gap` 继续作为抽取质量 known-gap。本版本只读取 manifest、readiness 和 publication audit，不访问第三方站点，不执行真实 adapter 包构建，也不会自动调用 GitHub。

## 风险与兼容性

- 这是维护脚本和 artifact 输出 patch，不改变 CLI runtime、adapter schema、JSON envelope 或生成代码。
- `issue_artifacts_command` 是计划 JSON 的新增字段；现有字段保持兼容。
- 输出目录由维护者指定，推荐使用 `/tmp/cliany-candidate-issues` 或 CI artifact 目录，避免把生成文件提交进仓库。
- 默认验证继续离线运行，不需要真实 LLM key，也不会写入 `~/.cliany-site/`。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.8 --issues-dir /tmp/cliany-candidate-issues` 写出 README、body、metadata 和 create script。
2. 确认 JSON 输出里的 `issue_artifacts_command` 与 artifacts README 中的复现命令一致。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.8`。
5. 将 `pyproject.toml` 版本更新为 `0.16.8`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.8`。
7. 创建 tag：`git tag v0.16.8`。

## Release Notes 摘要

`v0.16.8` adds an explicit regeneration command for candidate issue artifacts. Maintainers can now copy `issue_artifacts_command` from the next-iteration JSON or the generated artifacts README to recreate the same review package.
