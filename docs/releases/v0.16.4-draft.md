# v0.16.4 发布草案

**目标版本：** `0.16.4`
**提交范围：** `v0.16.3..HEAD`
**当前阻塞：** 需要提交计划器 issue body 模板、维护者循环入口和测试；最新本地 release 仍需维护者确认后 push 到远端。

## 用户价值

`v0.16.3` 已经让 `scripts/plan_next_iteration.py` 在 JSON 和 Markdown report 中展示 `candidate_promotions`，维护者可以直接看到 candidate 案例晋级所需的 `adapter_package`、`metadata_validation` 和 `online_smoke` 证据。但创建 GitHub issue 时仍需要手写 scope、tasks、validation evidence 和 non-goals。

`v0.16.4` 把可复制的 issue body 模板放进计划器输出。周初运行 `python scripts/plan_next_iteration.py --report /tmp/cliany-next-iteration.md` 后，维护者可以直接复制 `Candidate Issue Body Templates`，把 `pypi-project-search`、`npm-package-search`、`crates-io-crate-search` 等候选案例拆成独立任务。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.4 --json` 的 `candidate_promotions` 包含 `issue_body`；Markdown report 输出 `Candidate Issue Body Templates`，每个模板包含 `Scope`、`Tasks`、`Validation Evidence` 和 `Non-goals`。

## 变更分组

### Candidate issue templates

- `CandidatePromotion` 新增 `issue_body` 字段。
- JSON 输出为每个 candidate case 提供可复制 issue body。
- 文本输出在 `candidate_promotions` 下展示 issue body。
- Markdown report 新增 `Candidate Issue Body Templates` 小节，并为每个 candidate 输出 fenced Markdown 模板。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明计划器报告会生成 GitHub issue body。
- `tests/test_plan_next_iteration.py` 固定 JSON 和 Markdown report 中的 issue body 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环里的新说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。计划器会为 `pypi-project-search`、`npm-package-search`、`crates-io-crate-search` 三个 candidate cases 生成 issue body 模板。

`search-extraction-gap` 继续作为抽取质量 known-gap。本版本只读取 manifest 和 readiness 里的现有证据，不访问第三方站点，不执行真实 adapter 包构建。

## 风险与兼容性

- 这是维护脚本和报告输出 patch，不改变 CLI runtime、adapter schema、JSON envelope 或生成代码。
- `issue_body` 是新增 JSON 字段；已有 `candidate_cases`、`candidate_promotions`、`next_actions` 和 `validation_commands` 保持兼容。
- 默认验证继续离线运行，不需要真实 LLM key，也不会写入 `~/.cliany-site/`。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.4 --json` 输出 `candidate_promotions[].issue_body`。
2. 运行计划脚本、维护者循环文档和 release draft 相关测试。
3. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.4`。
4. 将 `pyproject.toml` 版本更新为 `0.16.4`。
5. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.4`。
6. 创建 tag：`git tag v0.16.4`。

## Release Notes 摘要

`v0.16.4` adds copy-ready candidate promotion issue bodies to the next-iteration planner. Maintainers can now turn candidate case promotion evidence into GitHub issues directly from the JSON or Markdown planning report.
