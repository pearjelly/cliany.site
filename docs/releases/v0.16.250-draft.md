# v0.16.250 发布草案

**目标版本：** `0.16.250`
**提交范围：** `v0.16.249..HEAD`
**当前阻塞：** 本草案用于准备 `v0.16.249` 之后的下一版。在 release readiness、完整验证和标准发布流程通过前，不创建或推送 `v0.16.250` tag。`pypi-project-search` candidate 晋级仍必须先通过 live LLM gate；`E_LLM_UNAVAILABLE`、`E_UNKNOWN` connection error、`ready_for_explore=false` 或 `generate_adapters.ready=false` 只能作为 blocker 证据，不能替代 adapter package、metadata validation 或 online smoke 证据。

## 用户价值

`v0.16.249` 已经完成 GitHub Release、PyPI 和官网生产部署的标准发布闭环，但 2026-07-01 再次运行 `cliany-site doctor --llm-live --json` 时，live LLM preflight 返回 `E_UNKNOWN` connection error，而不是旧文案里重点提到的 `E_LLM_UNAVAILABLE`。维护者如果只按旧 blocker 文案判断，仍可能误以为 `E_UNKNOWN` 需要继续跑真实 `explore`，从而污染 candidate adapter package 证据。

`v0.16.250` 把 candidate 晋级 handoff 统一扩展为“`llm_live` warning/error 均停机”，并明确列出 `E_LLM_UNAVAILABLE` 与 `E_UNKNOWN` connection error 两种常见上游阻塞。这样 issue template、evidence bundle、promotion plan、planner artifacts、case report、README、案例库文档和官网都给出同一个判断：live gate 没过时，停止真实 `explore`，贴回 doctor JSON / 错误摘要，并让 `adapter_package` 保持 pending 或 blocked。

## 变更分组

- Cases CLI：`cliany-site cases --issue-template`、`--evidence-bundle` 和 `--promotion-plan` 输出的 `llm_live_preflight_blocker_note` 覆盖任意 `llm_live` warning/error，并点名 `E_UNKNOWN` connection error。
- Case validation：`scripts/validate_cases.py --report` 的 candidate promotion checklist 使用同一 blocker note。
- Next iteration planner：`scripts/plan_next_iteration.py` 的 candidate promotion JSON、issue metadata、artifact README 和 issue body 继承同一 handoff。
- Documentation：README、README.zh、[cases/README.md](../../cases/README.md) 和官网文档同步说明 `E_UNKNOWN` connection error 只能作为 blocker 证据。
- Release draft：新增本草案，承接 `v0.16.249` 后的下一版准备。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的案例状态；相关说明落在 [cases/README.md](../../cases/README.md)。
- 当前案例库仍保持 active 4 个、candidate 3 个、known gap 1 个。
- Known gap 保持 1 个：`search-extraction-gap`。
- Candidate 晋级仍需要 adapter package、metadata validation 和 online smoke 三段真实证据。
- 首要 candidate 仍是 `pypi-project-search`；如果 live LLM gate 未通过，`adapter_package`、`metadata_validation` 和 `online_smoke` 保持 pending。

## Candidate preflight evidence

`pypi-project-search` 晋级命令计划：

```bash
python -m cliany_site doctor --llm-live --json
cliany-site explore "https://pypi.org" "search Python packages for cliany-site and list project names" --json
python scripts/validate_cases.py --packages-dir ~/.cliany-site/packages --include-candidate-packages --strict
cliany-site pypi.org search-projects --query cliany-site --limit 5 --json
```

执行真实 `explore` 前必须先运行 live LLM preflight。如果 `llm_live` 返回 warning/error（例如 `E_LLM_UNAVAILABLE` 或 `E_UNKNOWN` connection error），或 `ready_for_explore=false` / `generate_adapters.ready=false`，本轮停止真实 `explore`，不生成或伪造 `pypi.org-<version>.cliany-adapter.tar.gz`，并将结果记录为阻塞证据。

2026-07-01 初始 readiness 显示 `v0.16.250` 目标版本 publication 基线可用：`master` 与 `origin/master` 对齐，最新 tag `v0.16.249` 已发布并指向 HEAD，GitHub Release 和 PyPI 均公开暴露 `0.16.249`。当时 release gate 的主要阻塞是本草案缺失；本草案补齐后，下一步应继续根据 release readiness 输出处理真实验证与发版准备。

2026-07-01 重试 live LLM preflight。Chrome/CDP、LLM key、provider、OpenAI-compatible base URL、runtime dirs、registry 和 browser provider 可用，但 `llm_live` 返回 `E_UNKNOWN` connection error，`phase=llm_preflight`，`ready_for_explore=false`，且 `generate_adapters.ready=false`。因此本轮停止 `pypi-project-search` 的真实 `explore`，不生成或伪造 candidate adapter package，并保持三段晋级证据为 pending。

## 风险与兼容性

- 本版只扩展 candidate promotion blocker 文案和相关测试，不改变 CLI 参数、JSON envelope、adapter metadata schema、AXTree 语义匹配、action runtime、session 存储或 marketplace package 格式。
- 新增/变更字段内容仍是 additive handoff：读取 `llm_live_preflight_blocker_note` 的自动化只会得到更宽的停机条件。
- Candidate 晋级证据不得用 preflight 失败代替 adapter package；失败只能解释为什么保持 pending 或 blocked。
- 禁止引入脆弱 CSS selector 兜底，后续 adapter 生成仍必须依赖 AXTree 语义匹配。
- 运行时状态仍必须写入 `~/.cliany-site/`，不得把 adapters、sessions、snapshots 或 packages 写进 repo。
- 官网更新需要按 `AGENTS.md` 从 `site/` 目录部署到 `cliany.site` Vercel 项目，避免误部署到默认 `site` 项目。

## 发版步骤

1. 确认 `v0.16.249` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 重试 live LLM preflight；只有 `generate_adapters.ready=true` 且 `llm_live` 没有 warning/error 时才进入 `pypi-project-search` 真实 `explore`。
3. 完成或明确延后 `pypi-project-search` 的 adapter package、metadata validation 和 online smoke 证据。
4. 确认 `CHANGELOG.md`、本草案、README、官网和 case evidence 状态一致。
5. 运行文档相关测试、case validation、完整离线 pytest 和 strict release readiness。
6. 将 `CHANGELOG.md` 的 Unreleased 条目移动到 `## [0.16.250] - <date>`，并将 `pyproject.toml` 版本号更新到 `0.16.250`。
7. 提交并推送 `master`。
8. 创建并推送 `v0.16.250` tag。
9. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
10. 运行 `python scripts/check_release_publication.py --remote --distribution --json`，确认 `distribution_ok=true`、GitHub Release latest tag 为 `v0.16.250`，PyPI latest version 为 `0.16.250`。
11. 官网源码已更新，本版发布后按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署到 `cliany.site` 项目。

## Release Notes 摘要

v0.16.250 broadens the candidate promotion live LLM preflight blocker contract from `E_LLM_UNAVAILABLE` to any `llm_live` warning/error, explicitly including the `E_UNKNOWN` connection error observed on 2026-07-01. Candidate issue templates, evidence bundles, promotion plans, planner artifacts, validation reports, README docs, case docs, and website docs now tell maintainers to stop real `explore`, attach doctor JSON/error evidence, and keep adapter package evidence pending or blocked until the live gate is healthy.
