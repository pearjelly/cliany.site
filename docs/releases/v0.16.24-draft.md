# v0.16.24 发布草案

**目标版本：** `0.16.24`
**提交范围：** `v0.16.23..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让候选 issue artifacts README 直接展示 publication next actions。

## 用户价值

`v0.16.23` 已经把 publication audit 的具体待办放进 next iteration plan 和 `publication-handoff.json`。但维护者批量生成候选 issue artifacts 后，最先看到的通常是 artifacts `README.md`；如果这份 README 只显示 publish commands，不显示为什么需要先发布，就仍然要打开 JSON 才能看到 `publication_next_actions`。

`v0.16.24` 在 `scripts/plan_next_iteration.py --issues-dir` 生成的 artifacts `README.md` 里，于 `Publication Handoff` 下新增 `Publication Next Actions` 小节。它会直接展示 Push `master`、Push tag 和 remote 复核相关待办，让维护者在创建 candidate issue 前先处理发布可见性。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.24 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `README.md` 包含 `Publication Handoff`、`Publication Next Actions` 和 publication audit 的具体 action；`publication-handoff.json` 继续保留 `publication_next_actions`。

## 变更分组

### Candidate issue artifacts publication handoff

- `scripts/plan_next_iteration.py --issues-dir` 生成的 artifacts `README.md` 在 `Publication Handoff` 下新增 `Publication Next Actions`。
- 当 publication audit 有待办时，README 会列出 push branch、push tag 和 remote refs 复核相关动作。
- 当没有 publication 待办时，README 会显示 `No publication next actions are needed.`。
- `publication-handoff.json` 结构保持兼容，继续保留 `publication_next_actions` 和 `publish_commands`。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifacts README 会展示 `Publication Next Actions`。
- `tests/test_plan_next_iteration.py` 固定 artifacts README 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只把发布阻塞放到候选 issue artifacts 的首屏审阅入口里。

## 风险与兼容性

- 这是 artifacts README 文案增强，不改变 JSON schema 和脚本执行行为。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.24 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 README 包含 `Publication Next Actions`。
2. 确认 README 中能看到 Push `master` 和 Push tag 相关 action。
3. 确认 `publication-handoff.json` 继续包含 `publication_next_actions`。
4. 运行计划脚本、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.24`。
6. 将 `pyproject.toml` 版本更新为 `0.16.24`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.24`。
8. 创建 tag：`git tag v0.16.24`。

## Release Notes 摘要

`v0.16.24` shows publication next actions directly in generated candidate issue artifacts README files, so maintainers can see release publication blockers before creating candidate issues.
