# v0.16.275

Candidate promotion output now makes the required live-LLM preflight the first executable command.

## What Changed

- `cliany-site cases --evidence-bundle --json` now reports `cliany-site doctor --llm-live --json` as `primary_next_task_command` when `adapter_package` is the next incomplete promotion task.
- Candidate promotion plans and task queues use the same executable-first command, preventing an automation consumer from jumping directly to `explore` before the preflight gate.
- The later command is retained in explicit `primary_task_command` and `task_command` fields, along with its source, missing state, and handoff.
- Human-readable case tables, evidence bundles, and promotion plans show the preflight before the post-preflight task command.
- Candidates that already have a completed package proceed directly to metadata validation; this release does not add an unnecessary LLM preflight there.

## Compatibility

For candidate `adapter_package`, `primary_command` and queue `command` now mean the safe next executable action. Consumers that intentionally need the later task action should read `primary_task_command` or `task_command`.

## Trust Boundaries

This release does not create an adapter archive, prove a live LLM request, or claim a third-party online result. A failed or unavailable doctor preflight remains a blocker for candidate promotion.

## Verification

Regression coverage verifies the JSON evidence bundle, promotion queue, human output ordering, and the metadata-validation path that must not be preflight-gated. No live LLM, adapter asset, or external website success is claimed by this release.
