# v0.16.102 发布草案

**目标版本：** `0.16.102`
**提交范围：** `v0.16.101..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 publication latest tag 和 tag commit 补进 candidate issue artifacts 的整包摘要。

## 用户价值

`v0.16.101` 已经让 `artifact_bundle_summary` 能展示 publication branch、upstream 和 remote。维护者继续审阅候选 issue artifacts 时，还需要不展开 `publication_ref_context` 就能知道这批 artifacts 对应哪个最新本地 release tag，以及这个 tag 指向哪个提交；这两个值决定了后续发布脚本是否在推送正确的 release 点。

`v0.16.102` 在 `artifact_bundle_summary` 中新增 `publication_latest_tag` 和 `publication_tag_commit`。Artifacts `README.md` 的 `Artifact Bundle Summary` 也展示这些字段，让工具只读整包摘要就能定位 release tag 与 tag commit，再决定是否展开完整 `publication_ref_context`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.102 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_latest_tag` 和 `publication_tag_commit`，且这些值分别等于 manifest 中 `publication_ref_context.latest_tag` 和 `publication_ref_context.tag_commit`；README 的 `Artifact Bundle Summary` 也展示这些字段。

## 变更分组

### Artifact bundle publication tag summary

- `artifact_bundle_summary` 新增 `publication_latest_tag`。
- `artifact_bundle_summary` 新增 `publication_tag_commit`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示最新本地 tag 和 tag commit。
- 现有 `publication_ref_context`、`publication_branch`、`publication_upstream`、`publication_remote`、`publication_ahead_count`、`publication_behind_count` 和 `publication_remote_checked` 保持不变。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 publication latest tag 和 tag commit。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让候选 issue artifacts 的整包摘要更容易定位发布 tag。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary`，不改变 `publication_ref_context` 或 `publication-handoff.json` 的既有结构。
- 数值来自计划器已有的 publication ref context，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.102..HEAD`。最终带 `--release-tag v0.16.102` 的 strict gate 仍以本草案顶部的 `v0.16.101..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.102 --json
python scripts/plan_next_iteration.py --target-version 0.16.102 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.102 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.102 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_latest_tag` 和 `publication_tag_commit`。
3. 确认这些值分别等于 `artifact-manifest.json.publication_ref_context.latest_tag` 和 `artifact-manifest.json.publication_ref_context.tag_commit`。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.102`。
7. 将 `pyproject.toml` 版本更新为 `0.16.102`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.102`。
9. 创建 tag：`git tag v0.16.102`。

## Release Notes 摘要

`v0.16.102` adds publication latest tag and tag commit fields to candidate issue artifact bundle summaries, helping tools identify release tag refs without expanding the full publication ref context.
