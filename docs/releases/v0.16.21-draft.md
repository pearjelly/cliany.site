# v0.16.21 发布草案

**目标版本：** `0.16.21`
**提交范围：** `v0.16.20..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让候选 issue artifacts README 的摘要表直接指向 body 文件。

## 用户价值

`v0.16.20` 让 artifacts `README.md` 增加 `Candidate Summary` 表，维护者能看到每个 candidate 的 target URL 和命令数量。但表格仍没有直接列出对应的 issue body 文件；维护者看到某一行后，还要回到文件列表里找 `pypi-project-search.md` 这类正文。

`v0.16.21` 在 `Candidate Summary` 表里新增 `Issue Body` 列。每个 candidate 行都会列出对应的 Markdown body 文件名，让维护者从摘要表就能直接定位正文，再审阅 scope、tasks、validation evidence 和 non-goals。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.21 --issues-dir /tmp/cliany-candidate-issues` 写出的 README 中，`Candidate Summary` 表包含 `Issue Body` 列，并为 `pypi-project-search` 展示 `pypi-project-search.md`。

## 变更分组

### Candidate issue summary body link

- `scripts/plan_next_iteration.py --issues-dir` 的 artifacts `README.md` 中，`Candidate Summary` 表新增 `Issue Body` 列。
- 每个 candidate 行展示对应的 `<case_id>.md` body 文件。
- 摘要表继续保留 target URL、candidate commands 数量和 offline validation commands 数量，方便维护者从同一行完成复现信息初筛。
- `issue-metadata.json`、`publication-handoff.json` 和 `create-issues.sh` 的结构保持兼容。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `Candidate Summary` 表会列出 issue body 文件。
- `tests/test_plan_next_iteration.py` 固定 artifacts README 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只让维护者更快从 candidate 摘要跳到对应 issue body。

## 风险与兼容性

- 这是 artifacts README 文案增强，不改变 JSON schema 和脚本执行行为。
- body 文件名由已有 case id 推导，不访问第三方站点。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

## 发版前验证

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.21 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.21 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.21 --issues-dir /tmp/cliany-candidate-issues` 写出的 README `Candidate Summary` 包含 `Issue Body` 列。
2. 确认 `pypi-project-search` 行展示 `pypi-project-search.md`。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.21`。
5. 将 `pyproject.toml` 版本更新为 `0.16.21`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.21`。
7. 创建 tag：`git tag v0.16.21`。

## Release Notes 摘要

`v0.16.21` adds an Issue Body column to generated candidate issue artifact README summaries. Maintainers can now jump from each candidate summary row to the corresponding Markdown issue body file.
