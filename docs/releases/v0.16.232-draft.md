# v0.16.232 发布草案

**目标版本：** `0.16.232`
**提交范围：** `v0.16.231..HEAD`
**当前阻塞：** 需要完成 candidate promotion summary acceptance visibility 改动，并重新运行 release readiness、case validation、CLI/文档/官网测试。

## 用户价值

`v0.16.231` 已经让 candidate promotion issue body 自带 Acceptance Criteria。下一步要减少维护者和自动化读取这些验收条件的成本：全局 `cliany-site cases --json` 和 case catalog report 应该直接告诉你首要 candidate 任务需要什么完成证据，而不是必须先展开单个 evidence bundle 或复制 issue template。

`v0.16.232` 聚焦 candidate promotion summary 的验收标准可见性。`promotion_evidence_summary.primary_next_task` 现在带 `acceptance_criteria`，并新增顶层 `primary_next_task_acceptance_criteria`。`cliany-site cases` 的人类输出、`scripts/validate_cases.py` 文本输出和 Markdown report、`scripts/plan_next_iteration.py` 的 candidate promotion 摘要也同步显示 acceptance criteria。维护者可以在第一次查看 cases summary 时就判断下一步要跑哪条命令、附什么证据、是否满足 promotion gate。

成功标准是：cases CLI JSON summary、人类输出、validate_cases JSON/text/Markdown report 都展示首要任务的 acceptance criteria；README、README.zh、cases README 和官网 quickstart 能说明全局 summary 字段；`scripts/release_readiness.py --target-version 0.16.232 --json` 不再因为缺失 release draft 阻塞。

## 变更分组

### Candidate Summary

- `cliany-site cases --json` 的 `promotion_evidence_summary.primary_next_task` 新增 `acceptance_criteria`。
- `promotion_evidence_summary` 新增 `primary_next_task_acceptance_criteria`，自动化无需展开 nested task 即可读取首要 proof contract。
- `cliany-site cases` 的人类输出在 `Candidate 下一步` 下展示 `acceptance`，单案例 detail 的 Promotion Tasks 也展示每个任务的 acceptance。

### Case Report

- `scripts/validate_cases.py --json` 的 `promotion_evidence_summary.primary_next_task` 同步包含 `acceptance_criteria`。
- `scripts/validate_cases.py` 文本输出新增 `promotion_evidence_acceptance`。
- `scripts/validate_cases.py --report` 的 Candidate Promotion Evidence Summary 指标表新增 `primary_next_task_acceptance_criteria`，任务表新增 Acceptance Criteria 列。
- `scripts/plan_next_iteration.py --json` 的单个 candidate promotion primary task 现在也包含 `acceptance_criteria`，便于下一轮计划和 issue artifacts 沿用同一 proof contract。

### 文档与官网同步

- README / README.zh 现在说明 `promotion_evidence_summary.primary_next_task_acceptance_criteria` 可直接读取首要任务完成证据要求。
- [cases/README.md](../../cases/README.md) 现在说明 Candidate Promotion Evidence Summary 会同时展示 primary next action 和 primary acceptance。
- 官网 quickstart 现在把候选案例验证入口指向带 `primary_next_task_acceptance_criteria` 的 `cliany-site cases --json`。

## 案例库映射

- 本版本不改变 [cases/manifest.json](../../cases/manifest.json) 的案例状态，不新增 active adapter 包，不生成或提交 `~/.cliany-site/` 运行时状态。
- Active 案例保持 4 个：SuiteCRM、Apache Jira、Apache Confluence、Apache Jenkins。
- Candidate 案例保持 3 个：PyPI、npm、crates.io 搜索工作流。
- Known gap 保持 1 个：`search-extraction-gap`。
- `pypi-project-search` 仍是下一轮首要 candidate；本版本只让它的首要 `adapter_package` 证据要求在全局 summary 中可见，不声称 adapter 包已经生成。

## 风险与兼容性

- 风险：下游工具如果对 `promotion_evidence_summary.primary_next_task` 做严格字段白名单，需要允许新增的 `acceptance_criteria` 字段。
- 风险：Acceptance Criteria 是交接合同，不是任务完成证明；candidate 晋级 active 仍需真实 adapter package、metadata validation 和 online smoke 三段证据全部 complete。
- 兼容性：不修改 adapter schema、生成代码、动作回放、CDP 连接或 case manifest 状态。
- 兼容性：发布节奏继续遵循每日 verified release train 和标准 release flow。
