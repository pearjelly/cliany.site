# v0.16.151 发布草案

**目标版本：** `0.16.151`
**提交范围：** `v0.16.150..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 publication handoff schema version 提升到 artifact bundle summary。

## 用户价值

`v0.16.150` 已经让 `publication-handoff.json` 带上 `schema_version: 1`。维护工具如果展开 handoff，就能先判断字段语义版本；但只读 `artifact_bundle_summary` 的工具仍只能看到 `publication_handoff_key_count` 与 `publication_handoff_sha256`，无法在展开前判断 handoff schema 是否兼容。

`v0.16.151` 在 `artifact_bundle_summary` 中新增 `publication_handoff_schema_version`，并在 artifacts `README.md` 的 `Artifact Bundle Summary` 段落展示同源字段。这样工具只读整包摘要时，就能先判断 publication handoff 语义版本，再决定是否展开完整 `publication-handoff.json`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.151 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_handoff_schema_version`，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且该值与 `publication-handoff.json.schema_version` 同源。

## 变更分组

### Artifact bundle publication handoff schema version

- `artifact_bundle_summary` 新增 `publication_handoff_schema_version`。
- 该字段从 `publication-handoff.json.schema_version` 派生。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 `publication_handoff_schema_version`。
- 现有 `publication_handoff_key_count`、`publication_handoff_sha256`、`publication-handoff.json` 和 `Publication Handoff` 保持兼容。
- 不改变 publication audit、candidate issue gate 或 release readiness 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 publication handoff schema version。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的新字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 `publication-handoff.json.schema_version`，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `artifact_bundle_summary` 判断 publication handoff schema，再决定是否展开 handoff。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.151..HEAD`。最终带 `--release-tag v0.16.151` 的 strict gate 仍以本草案顶部的 `v0.16.150..HEAD` 作为实际 release diff 范围。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.151 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_handoff_schema_version`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示 `publication_handoff_schema_version`。
4. 确认该字段等于 `publication-handoff.json.schema_version`。
5. 确认 `publication_handoff_key_count` 和 `publication_handoff_sha256` 仍与当前 handoff 同源。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.151`。
8. 将 `pyproject.toml` 版本更新为 `0.16.151`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.151`。
10. 创建 tag：`git tag v0.16.151`。

## Release Notes 摘要

`v0.16.151` adds the publication handoff schema version to artifact bundle summaries, helping tools decide whether a publication handoff is compatible before expanding the full handoff payload.
