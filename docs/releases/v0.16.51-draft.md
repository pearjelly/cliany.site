# v0.16.51 发布草案

**目标版本：** `0.16.51`
**提交范围：** `v0.16.50..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 next iteration plan 和 candidate issue artifacts 直接展示候选 issue 创建门控。

## 用户价值

`v0.16.50` 已经让 `artifact-manifest.json` 包含机器可读的 `review_checklist`。维护者可以看到创建候选 issue 前要核对什么，但工具仍需要自行推断“现在是否能继续创建候选 issue”：publication 未公开时应该硬停，release draft 只有草案问题时可以创建 issue 但必须保留人工审阅。

`v0.16.51` 新增 `candidate_issue_gate`，把这层判断结构化。计划 JSON、文本输出、Markdown report、`artifact-manifest.json`、`publication-handoff.json` 和 artifacts `README.md` 都会展示 `status`、`can_create_issues`、`requires_maintainer_review`、summary 和 `required_actions`。这样维护者和自动化无需解析长文本，就能区分 `blocked_by_publication`、`review_required` 与 `ready`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.51 --issues-dir /tmp/cliany-candidate-issues --json` 写出的计划和 artifacts 同时包含 `candidate_issue_gate`；当最新本地 release 未公开可见时，gate 输出 `blocked_by_publication` 且 `can_create_issues=false`；每周维护者循环文档说明这个字段。

## 变更分组

### Candidate issue gate

- `scripts/plan_next_iteration.py` 的计划模型新增 `candidate_issue_gate`。
- 当 publication audit 未通过时，gate 使用 `blocked_by_publication`，并复用 `publication_next_actions` 作为 required actions。
- 当 publication 已通过但 release draft 仍有 issues 时，gate 使用 `review_required`，允许创建候选 issue 但要求维护者处理或明确延后 release draft issues。
- 当 publication 和 release draft gate 都干净时，gate 使用 `ready`。

### Artifacts handoff

- Markdown report 新增 `Candidate Issue Gate` 小节。
- `artifact-manifest.json` 和 `publication-handoff.json` 写入同源 `candidate_issue_gate`。
- Artifacts `README.md` 在 `Publication Handoff` 下展示 candidate issue gate、`can_create_issues` 和 gate summary。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `candidate_issue_gate` 的状态和值。
- `tests/test_plan_next_iteration.py` 覆盖 blocked publication 和 release draft review 分支。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强创建 candidate issue 前的结构化门控。

## 风险与兼容性

- `candidate_issue_gate` 是计划 JSON 和 artifacts JSON 的新增字段，不改变既有字段语义。
- `create-issues.sh` 的硬性 preflight 仍由 `python scripts/check_release_publication.py --strict --json` 执行；本版本只让该判断更容易被提前读取。
- `review_required` 不会自动放行 release tag；它只说明 candidate issue 可以创建，但 release draft issues 仍需处理或明确延后。
- 不自动执行命令，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.51..HEAD`。最终带 `--release-tag v0.16.51` 的 strict gate 仍以本草案顶部的 `v0.16.50..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.51 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.51 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.51 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `artifact-manifest.json`。
2. 确认计划 JSON、Markdown report、`artifact-manifest.json` 和 `publication-handoff.json` 都包含 `candidate_issue_gate`。
3. 确认未发布本地 release 时 gate 为 `blocked_by_publication` 且 `can_create_issues=false`。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.51`。
6. 将 `pyproject.toml` 版本更新为 `0.16.51`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.51`。
8. 创建 tag：`git tag v0.16.51`。

## Release Notes 摘要

`v0.16.51` adds a machine-readable `candidate_issue_gate` to next-iteration plans and candidate issue artifacts, making it clear when publication blockers should stop candidate issue creation and when only maintainer review remains.
