# v0.16.41 发布草案

**目标版本：** `0.16.41`
**提交范围：** `v0.16.40..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts manifest 直接展示候选案例摘要。

## 用户价值

`v0.16.40` 已经新增 `artifact-manifest.json`，把 candidate issue artifacts 的文件名、review order 和 validation commands 变成机器可读入口。但维护者或工具想快速知道这批 artifacts 覆盖哪些 candidate cases，仍然需要读取 `issue-metadata.json` 或 README 的 Candidate Summary。

`v0.16.41` 让 `artifact-manifest.json` 直接包含 `candidate_count` 和 `candidate_cases`。这样维护者打开 manifest 就能确认本批候选 issue 覆盖范围，再决定是否继续审阅 publication handoff、release draft handoff 和 issue body。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.41 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `artifact-manifest.json` 包含 `candidate_count` 和 `candidate_cases`；artifacts `README.md` 的 Files 小节说明 manifest 会包含 candidate cases。

## 变更分组

### Candidate artifacts manifest case summary

- `artifact-manifest.json` 新增 `candidate_count`。
- `artifact-manifest.json` 新增 `candidate_cases`。
- Artifacts `README.md` 的 Files 小节说明 manifest 包含 candidate cases。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 candidate issue artifacts manifest 会包含 candidate count 和 candidate cases。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 内容和 README 文件列表。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强候选 issue artifacts manifest 的候选案例摘要。

## 风险与兼容性

- `candidate_count` 和 `candidate_cases` 是 `artifact-manifest.json` 的新增字段，不改变 plan JSON schema。
- 既有 `files`、`review_order` 和 `validation_commands` 字段保持兼容。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- Candidate case 列表只复用本次生成的 candidate promotion 数据。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.41 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `artifact-manifest.json`。
2. 确认 manifest 包含 `candidate_count` 和 `candidate_cases`。
3. 确认 artifacts `README.md` 的 Files 小节说明 manifest 包含 candidate cases。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.41`。
6. 将 `pyproject.toml` 版本更新为 `0.16.41`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.41`。
8. 创建 tag：`git tag v0.16.41`。

## Release Notes 摘要

`v0.16.41` adds candidate case summary fields to generated `artifact-manifest.json` files, so maintainers and tools can identify the candidate issue bundle scope without reading issue metadata first.
