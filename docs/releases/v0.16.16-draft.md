# v0.16.16 发布草案

**目标版本：** `0.16.16`
**提交范围：** `v0.16.15..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让候选 issue artifacts 自带发布交接信息。

## 用户价值

`v0.16.15` 让 `create-issues.sh` 在运行 `gh issue create` 前先执行 publication preflight，避免最新本地 release 尚未公开可见时继续派发候选任务。但维护者打开 `/tmp/cliany-candidate-issues` 时，仍需要回到计划 JSON 或另跑 `scripts/check_release_publication.py` 才能看到具体 next actions 和 publish commands。

`v0.16.16` 把 publication handoff 写进 issue artifacts 目录。`scripts/plan_next_iteration.py --issues-dir` 会新增 `publication-handoff.json`，包含 `publication_ok`、`next_actions`、`publish_commands` 和 `publish_script_command`；artifacts `README.md` 也会展示 `Publication Handoff` 小节和可复制 publish commands。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.16 --issues-dir /tmp/cliany-candidate-issues` 写出的 artifacts 目录包含 `publication-handoff.json`，README 展示 publication 状态和 `git push origin ...` / `python scripts/check_release_publication.py --remote --json` 命令。

## 变更分组

### Candidate issue publication handoff

- `scripts/plan_next_iteration.py --issues-dir` 新增 `publication-handoff.json`。
- `publication-handoff.json` 复用计划器已有的 `publication_ok`、`next_actions`、`publication_publish_commands` 和 `publication_publish_script_command`。
- artifacts `README.md` 新增 `Publication Handoff` 小节，方便维护者在创建 candidate issues 前先完成 release publication。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 candidate issue artifacts 会包含 `publication-handoff.json`。
- `tests/test_plan_next_iteration.py` 固定 handoff JSON 和 README 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只让候选 issue artifacts 在移交给维护者时同时带上发布可见性状态和下一步命令。

## 风险与兼容性

- `publication-handoff.json` 是新增 artifact；已有 `issue-metadata.json`、candidate body 文件和 `create-issues.sh` 保持兼容。
- publish commands 只写入文件供维护者审阅，不会由 `plan_next_iteration.py` 自动执行。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

## 发版前验证

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.16 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.16 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.16 --issues-dir /tmp/cliany-candidate-issues` 写出 `publication-handoff.json`。
2. 确认 artifacts `README.md` 展示 `Publication Handoff` 和 publish commands。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.16`。
5. 将 `pyproject.toml` 版本更新为 `0.16.16`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.16`。
7. 创建 tag：`git tag v0.16.16`。

## Release Notes 摘要

`v0.16.16` adds publication handoff metadata to generated candidate issue artifacts. Maintainers now get `publication-handoff.json` and a README section with publication status, next actions, and publish commands before creating candidate issues.
