# v0.16.257 发布草案

**目标版本：** `0.16.257`
**提交范围：** `v0.16.256..HEAD`
**当前状态：** 2026-07-03 已达到 `3/3` daily release cap；本版先准备下一次可发布切片，目标是在 2026-07-04 或之后把 candidate 晋级证据 runbook 纳入维护者路径。

## 用户价值

`v0.16.256` 已经让 planner summary 能直接给出首要 candidate 的 issue template 和 evidence bundle 命令。维护者现在能从 `case_promotion_evidence_primary_issue_template_command` 与 `case_promotion_evidence_primary_evidence_bundle_json_command` 跳到具体 handoff，但真正推进 `pypi-project-search/adapter_package` 仍需要把 live preflight、adapter package、metadata validation 和 online smoke 串成可执行顺序。

`v0.16.257` 把这个顺序整理成 [Candidate Promotion Runbook](../candidate-promotion-runbook.md)：从 `cliany-site doctor --llm-live --json` 开始，只有 preflight 通过才运行 PyPI explore；adapter package 证据必须指向 `pypi.org-<version>.cliany-adapter.tar.gz` 的本地 package 路径或 GitHub Release asset；metadata validation 使用 `python scripts/validate_cases.py --packages-dir ~/.cliany-site/packages --include-candidate-packages --strict`；online smoke 使用 `cliany-site pypi.org search-projects --query cliany-site --limit 5 --json` 并要求 `ok=true`、`data.quality.ok=true` 和 `row_count>0`。

这样下一版发版前，维护者不用在 README、case manifest、planner JSON 和 issue body 之间来回拼流程。`python scripts/plan_next_iteration.py --handoff-json` 还提供 compact handoff，只输出版本、daily cap、blockers/next actions、standard release flow 首要步骤、publication summary、首要 candidate task、issue/evidence bundle 命令、validation commands，以及首要 candidate 的 `doctor_preflight_evidence_fields`、`doctor_preflight_evidence_selectors`、selector count/hash 和 extractor JSON/Markdown 命令，避免维护脚本展开完整 planner JSON。`python scripts/plan_next_iteration.py --issues-dir /tmp/cliany-candidate-issues` 也会写出同源 `planner-handoff.json`，并把它加入 artifact files、review order 和 artifacts README 的 Planner Handoff 小节；README 也会直接展示 selector count、selector SHA-256 和 LLM error selector。本轮还给 doctor preflight 字段增加 `doctor_preflight_evidence_selectors`，把 `checks[llm_live].details.error_code` 这类语义字段映射到真实 `doctor --llm-live --json` 输出里的 `data.checks[name="llm_live"].details.error_code`，让维护者不用猜 `checks` 列表如何定位。新增 `python scripts/extract_doctor_preflight_evidence.py /tmp/cliany-doctor-preflight.json` 后，维护者可以把 doctor JSON 转成只含 `values`、`missing_fields` 和 selector/hash 的 blocker evidence；加 `--markdown` 则生成可直接粘贴到 issue 的 blocker evidence 表格。Cases evidence bundle、promotion plan、candidate issue body、compact planner handoff 和 `scripts/validate_cases.py` 的 JSON/Markdown/plain text 输出现在都会直接显示这两个 extractor 命令，维护者和机器人不再需要从文档中重构脚本路径。即使当天 release cap 已满，也可以安全准备 issue/evidence，不创建新 tag，不污染仓库 runtime 状态，也不把 candidate 提前标成 active。

## 风险与兼容性

- 不改变 CLI runtime、adapter schema、AXTree selector、action replay、session 存储或 adapter codegen。
- 不修改 [cases/manifest.json](../../cases/manifest.json) 的 candidate 状态，也不把 `pypi-project-search` 提前标成 active。
- Runbook 中涉及真实 `explore`、PyPI 页面和 adapter package 的步骤需要本地 Chrome、可用 LLM provider 和用户管理的 `~/.cliany-site/` runtime 目录；默认测试仍走离线验证。
- 当 `daily_release_cap_blocked=true` 时，不创建 `v0.16.257` tag；只准备文档、issue handoff 或 blocker evidence。
- 2026-07-03 实测 `cliany-site doctor --llm-live --json`：Chrome/CDP 可用，`run_browser_workflows.ready=true`，但 live LLM 返回 `E_LLM_UNAVAILABLE` / `502 Bad Gateway`，`generate_adapters.ready=false`；本版只记录 blocker evidence 和 selector handoff，不生成 candidate adapter package。
- 如果 live LLM preflight 返回 `E_LLM_UNAVAILABLE` 或 `generate_adapters.ready=false`，把 doctor JSON 字段作为 blocker evidence，保持 `adapter_package` pending 或 blocked。

## Release Notes 摘要

v0.16.257 prepares the next candidate-promotion release slice while the 2026-07-03 daily release cap is full. It adds a maintainer-facing Candidate Promotion Runbook that turns the `pypi-project-search/adapter_package` handoff into a concrete sequence: run live LLM preflight, generate or attach the expected `pypi.org-<version>.cliany-adapter.tar.gz` adapter package, validate candidate packages from `~/.cliany-site/packages`, and run the read-only PyPI online smoke command before updating case evidence or considering promotion to active. It also adds `python scripts/plan_next_iteration.py --handoff-json`, a compact release handoff for automation that only needs the primary next action, publication summary, and primary candidate commands.
