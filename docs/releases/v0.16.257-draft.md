# v0.16.257 发布草案

**目标版本：** `0.16.257`
**提交范围：** `v0.16.256..HEAD`
**当前状态：** 2026-07-03 已达到 `3/3` daily release cap；本版先准备下一次可发布切片，目标是在 2026-07-04 或之后把 candidate 晋级证据 runbook 纳入维护者路径。

## 用户价值

`v0.16.256` 已经让 planner summary 能直接给出首要 candidate 的 issue template 和 evidence bundle 命令。维护者现在能从 `case_promotion_evidence_primary_issue_template_command` 与 `case_promotion_evidence_primary_evidence_bundle_json_command` 跳到具体 handoff，但真正推进 `pypi-project-search/adapter_package` 仍需要把 live preflight、adapter package、metadata validation 和 online smoke 串成可执行顺序。

`v0.16.257` 把这个顺序整理成 [Candidate Promotion Runbook](../candidate-promotion-runbook.md)：从 `cliany-site doctor --llm-live --json` 开始，只有 preflight 通过才运行 PyPI explore；adapter package 证据必须指向 `pypi.org-<version>.cliany-adapter.tar.gz` 的本地 package 路径或 GitHub Release asset；metadata validation 使用 `python scripts/validate_cases.py --packages-dir ~/.cliany-site/packages --include-candidate-packages --strict`；online smoke 使用 `cliany-site pypi.org search-projects --query cliany-site --limit 5 --json` 并要求 `ok=true`、`data.quality.ok=true` 和 `row_count>0`。

这样下一版发版前，维护者不用在 README、case manifest、planner JSON 和 issue body 之间来回拼流程。即使当天 release cap 已满，也可以安全准备 issue/evidence，不创建新 tag，不污染仓库 runtime 状态，也不把 candidate 提前标成 active。

- `docs/good-first-issues.md`：从 candidate promotion issue 创建说明链接到 runbook，提醒维护者先核对四步证据再创建公开 issue 或更新 `promotion_evidence`。

- 不修改 [cases/manifest.json](../../cases/manifest.json) 的 candidate 状态，也不把 `pypi-project-search` 提前标成 active。
- Runbook 中涉及真实 `explore`、PyPI 页面和 adapter package 的步骤需要本地 Chrome、可用 LLM provider 和用户管理的 `~/.cliany-site/` runtime 目录；默认测试仍走离线验证。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.256..HEAD`。完成 version bump 后，target-mode readiness 也会查找占位锚点 **提交范围：** `v0.16.257..HEAD`；该锚点只表示 tag 发布后不应再有未发布差异，不改变本版实际提交范围。创建本地 tag 后，使用 local tag-mode readiness 复核最终 tag 语义。

```bash
git diff --check
uv run --extra dev --frozen pytest tests/test_release_draft_docs.py tests/test_release_readiness.py tests/test_plan_next_iteration.py -q --no-cov --tb=short
uv run --extra dev --frozen python scripts/validate_cases.py --strict
uv run --extra dev --frozen python scripts/release_readiness.py --strict --target-version 0.16.257 --remote --remote-name origin --json
```

如果仍在 2026-07-03，最后一条 target-mode readiness 预期会因为 `daily_release_cap_blocked=true` 停在 tag gate；这是正确暂停，不是草案缺失。2026-07-04 或之后继续发布前，再运行：

```bash
CLIANY_QA_OFFLINE=1 uv run --extra dev --frozen pytest tests/ -q
uv run --extra dev --frozen python scripts/release_readiness.py --strict --target-version 0.16.257 --remote --remote-name origin --json
```

创建本地 tag 后、推送 tag 前再运行：

```bash

```

## Release Notes 摘要

v0.16.257 prepares the next candidate-promotion release slice while the 2026-07-03 daily release cap is full. It adds a maintainer-facing Candidate Promotion Runbook that turns the `pypi-project-search/adapter_package` handoff into a concrete sequence: run live LLM preflight, generate or attach the expected `pypi.org-<version>.cliany-adapter.tar.gz` adapter package, validate candidate packages from `~/.cliany-site/packages`, and run the read-only PyPI online smoke command before updating case evidence or considering promotion to active.
