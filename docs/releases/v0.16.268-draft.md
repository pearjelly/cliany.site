# v0.16.268 发布草案

**目标版本：** `0.16.268`
**提交范围：** `v0.16.267..HEAD`
**当前状态：** 2026-07-20 release slice；停止向 JSON 客户端提供不可安装的历史 demo adapter 命令。

完成 version bump 后，target-mode readiness 还会检查占位锚点 **提交范围：** `v0.16.268..HEAD`；该锚点不改变实际 release diff。

## 用户价值

`cliany-site doctor --json` 的 `demo_adapter_quickstart` 仍保留为兼容字段，但现在清楚返回 `deprecated=true`、`available=false` 和空 `commands`。自动化可以读取 `replacement=case_catalog_quickstart`，直接转向可运行的 `cliany-site cases`，而不会尝试安装不存在的历史归档文件。

## 案例库映射

本版不改变案例状态或 adapter 资产。`cases/README.md` 和 `cases/manifest.json` 仍是案例目录的唯一来源；`search-extraction-gap` 与 `llm_live_preflight_not_ready` 继续约束候选案例晋级。2026-07-20 的真实 LLM preflight 仍返回可重试的 `E_LLM_UNAVAILABLE`，所以未生成、未声明任何 PyPI candidate adapter 包或 online smoke 成功。

## 风险与兼容性

- `demo_adapter_quickstart` 继续是对象，`commands` 继续是列表，现有 JSON 解析不需要改变类型判断。
- 旧客户端若直接执行 `commands` 会看到空列表而不是失败的历史文件路径；支持 `deprecated`、`available` 和 `replacement` 的客户端可获得明确迁移提示。
- `case_catalog_quickstart` 仍只用于查看维护案例和验证路径，不暗示某个 candidate 已具备可下载 adapter asset。

## 发版步骤

1. 在 `[Unreleased]` 保留本版条目、compare link 为 `v0.16.267...HEAD` 时运行 target-mode readiness。
2. Finalize Changelog：清空 `[Unreleased]`，新增 `## [0.16.268] - 2026-07-20`，并更新 compare links。
3. 提交并推送 `master`，等待 CI 和 Embodied CI 成功。
4. 创建 `git tag v0.16.268`，然后运行 `uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.268 --remote --remote-name origin`。
5. 仅在该 gate 通过后推送 tag，等待 GitHub Release 与 PyPI 发布。
6. 在 `site/` 运行 `vercel link --yes --project cliany.site`、`vercel --prod --yes` 和 `vercel inspect www.cliany.site --wait --timeout 90s`。
7. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --strict --remote --distribution --json`。
