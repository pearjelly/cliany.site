# v0.15.1 发布草案

**目标版本：** `0.15.1`
**提交范围：** `v0.15.0..HEAD`
**当前阻塞：** 需要提交 candidate promotion 包名占位规则、运行案例库和 release readiness 验证。

## 用户价值

`v0.15.1` 聚焦真实案例库的晋级路径：candidate 案例应该告诉维护者“下一步生成哪个包”，但不能把上一轮 release 的固定版本号写死在长期文档里。这个 patch 让 PyPI、npm 和 crates.io 候选案例继续保留明确的 adapter 包名形状，同时用 `<version>` 占位表达未来 release train，避免发布后文档和校验报告马上过期。

成功标准是：维护者运行 `scripts/validate_cases.py --strict` 或 `release_readiness.py --report` 时，candidate promotion 中的 `adapter_package` 能直接展示 `market publish` 的包名格式，并且校验器会阻止新候选案例复制固定版本号或旧 `domain.cliany-adapter-vX.Y.Z.tar.gz` 格式。

## 变更分组

### Candidate promotion 包名

- `cases/manifest.json` 中 `pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 的 `promotion.adapter_package` 使用 `<domain>-<version>.cliany-adapter.tar.gz`。
- `scripts/validate_cases.py` 应继续拒绝旧格式 `domain.cliany-adapter-vX.Y.Z.tar.gz`。
- `scripts/validate_cases.py` 还应拒绝 candidate promotion 中写死的固定版本包名，例如 `demo.example.com-0.1.0.cliany-adapter.tar.gz`。
- Markdown 和文本报告中的 Candidate Promotions 继续展示完整 promotion 清单，方便维护者从 CI artifact 直接看见晋级动作。

## 案例库映射

本版本继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json) 作为真实案例入口。Active 案例保持不变；`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate；`search-extraction-gap` 继续作为抽取质量 known-gap。

这次不新增真实站点，也不把 candidate 晋级为 active；它只收紧 candidate 晋级清单的可维护性，确保后续生成 adapter 包时不会从报告里复制过期版本号。

## 风险与兼容性

- 这是案例库和校验脚本层面的 patch，不改变 CLI runtime、adapter schema 或 JSON envelope。
- 现有 active demo adapter 安装命令仍保留历史 release asset 包名，不在本次改动中改写。
- 新规则只针对 candidate promotion 的包名提示；已经发布的真实包资产仍以实际版本号存在。

## 发版前验证

```bash
python scripts/validate_cases.py --strict
python scripts/release_readiness.py --target-version 0.15.1 --json
pytest tests/test_validate_cases.py tests/test_release_readiness.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/validate_cases.py tests/test_validate_cases.py tests/test_release_readiness.py tests/test_release_draft_docs.py
```

如本地已有 release asset 包，发版前仍可额外运行：

```bash
python scripts/release_readiness.py --target-version 0.15.1 --packages-dir ~/.cliany-site/packages --require-packages --strict
```

## 发版步骤

1. 确认 `python scripts/release_readiness.py --target-version 0.15.1 --json` 只剩可解释的节奏或包资产阻塞。
2. 运行 `python scripts/validate_cases.py --strict`。
3. 运行相关 pytest 与 ruff 检查。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.15.1`。
5. 将 `pyproject.toml` 版本更新为 `0.15.1`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.15.1`。
7. 创建 tag：`git tag v0.15.1`。

## Release Notes 摘要

`v0.15.1` tightens the candidate-case promotion path. Candidate adapter package hints now use the stable `<domain>-<version>.cliany-adapter.tar.gz` shape, and `validate_cases.py` rejects fixed-version candidate hints that would drift after the next release.
