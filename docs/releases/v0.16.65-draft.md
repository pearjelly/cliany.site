# v0.16.65 发布草案

**目标版本：** `0.16.65`
**提交范围：** `v0.16.64..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 `candidate_issue_gate.required_actions` 同时覆盖 publication blockers 和 release draft issues。

## 用户价值

`v0.16.64` 已经让 candidate issue artifacts 的整包摘要能快速检测 review order 是否变化。但当 `candidate_issue_gate.reason_codes` 同时包含 `publication_not_published` 和 `release_draft_issues` 时，`required_actions` 只列 publication 待办，维护者还要再去读取 release draft handoff 才能知道草案问题该怎么处理。

`v0.16.65` 会把 release draft issues 也合并进 `candidate_issue_gate.required_actions`，并用 `Resolve release draft issue: ...` 标明动作语义。这样计划 JSON、Markdown report、candidate issue artifacts manifest 和 handoff 都能从 gate 本身给出完整待办。

成功标准是：缺失下一版草案的场景下，例如 `python scripts/plan_next_iteration.py --target-version 0.16.66 --json`，`candidate_issue_gate.required_actions` 同时包含 publish 动作和 `Resolve release draft issue: ...`；`--issues-dir` 生成的 `artifact-manifest.json` 与 `publication-handoff.json` 也保留同源动作。

## 变更分组

### Candidate issue gate required actions

- publication blocked 时，gate required actions 保留 publication next actions。
- 如果 release draft 有问题，gate required actions 追加 `Resolve release draft issue: ...`。
- publication 已发布但 release draft 仍需审阅时，gate required actions 继续只列 release draft 修复动作。
- `reason_codes`、`reason_descriptions` 和 `evidence` 语义保持不变。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `required_actions` 会同时覆盖 publication 和 release draft 待办。
- `tests/test_plan_next_iteration.py` 覆盖 JSON、Markdown report 和 artifacts handoff。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让候选 issue gate 的下一步动作更完整、更容易自动化消费。

## 风险与兼容性

- `required_actions` 的条目更完整，但字段类型仍是字符串数组。
- release draft issue 文案来自已有 readiness 检查，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.65..HEAD`。最终带 `--release-tag v0.16.65` 的 strict gate 仍以本草案顶部的 `v0.16.64..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.65 --json
python scripts/plan_next_iteration.py --target-version 0.16.65 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.65 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.65 --json` 输出 gate required actions。
2. 用缺失草案的后续目标版本确认 `candidate_issue_gate.required_actions` 同时包含 publication 动作和 `Resolve release draft issue: ...`。
3. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.65 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
4. 确认 `artifact-manifest.json` 与 `publication-handoff.json` 保留同源 gate required actions。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.65`。
7. 将 `pyproject.toml` 版本更新为 `0.16.65`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.65`。
9. 创建 tag：`git tag v0.16.65`。

## Release Notes 摘要

`v0.16.65` makes candidate issue gate required actions include release draft fixes alongside publication blockers, so maintainers and tools can act from the gate payload alone.
