# v0.16.73 发布草案

**目标版本：** `0.16.73`
**提交范围：** `v0.16.72..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 `artifact_bundle_summary` 直接携带 publication worktree status 的数量与稳定 hash。

## 用户价值

`v0.16.72` 已经让 `artifact_bundle_summary` 能检测 publication ref context 和 publish commands 是否漂移。但当发布可见性被 dirty worktree 阻塞时，工具仍然需要展开完整 manifest 或 `Publication Handoff`，才能知道 worktree status 的文件集合是否变化。

`v0.16.73` 在 `artifact_bundle_summary` 中新增 `publication_worktree_status_count` 和 `publication_worktree_status_sha256`。Artifacts `README.md` 的 `Artifact Bundle Summary` 也展示这些字段，让维护者和自动化能在展开详细 handoff 前判断 publication worktree status 是否漂移。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.73 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 publication worktree status count/hash，并能从 manifest 中的 `publication_worktree_status` 重新计算得到；README 的 `Artifact Bundle Summary` 也展示这些字段。

## 变更分组

### Artifact bundle worktree drift summary

- `artifact_bundle_summary` 新增 `publication_worktree_status_count`。
- `artifact_bundle_summary` 新增 `publication_worktree_status_sha256`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些 worktree status 漂移摘要字段。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 publication worktree status count/hash。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate issue artifacts 的整包摘要更适合自动化判断发布前工作树证据是否变化。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary`，不改变 `publication_worktree_status` 或 publication handoff 的既有字段语义。
- Hash 基于现有 plan 字段稳定 JSON 序列化计算，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.73..HEAD`。最终带 `--release-tag v0.16.73` 的 strict gate 仍以本草案顶部的 `v0.16.72..HEAD` 作为实际 release diff 范围。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.73 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_worktree_status_count` 和 `publication_worktree_status_sha256`。
3. 确认这些摘要能从 manifest 中的 `publication_worktree_status` 重新计算得到。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.73`。
7. 将 `pyproject.toml` 版本更新为 `0.16.73`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.73`。
9. 创建 tag：`git tag v0.16.73`。

## Release Notes 摘要

`v0.16.73` adds publication worktree-status count and hash fields to artifact bundle summaries, helping tools detect release-visibility drift before reading full issue artifact manifests.
