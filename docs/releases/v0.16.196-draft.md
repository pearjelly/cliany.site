# v0.16.196 发布草案

**目标版本：** `0.16.196`
**提交范围：** `v0.16.195..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 candidate issue gate required action boundary 显式化到整包摘要。

## 用户价值

`v0.16.195` 已经让 `artifact_bundle_summary` 可以展示 candidate issue gate reason code 的首尾边界。维护者和自动化在判断能否创建 candidate issue 时，还需要审阅 `candidate_issue_gate.required_actions`，确认首个维护动作和末尾动作是否仍符合预期；目前 summary 已有 required action 数量、完整 hash 和 primary action，但只读 summary 的工具仍无法快速确认 action 列表末尾是否漂移。

`v0.16.196` 在 `artifact_bundle_summary` 中新增 `candidate_issue_gate_first_required_action`、`candidate_issue_gate_last_required_action` 和 `candidate_issue_gate_required_action_boundary_sha256`。这样维护者和自动化可以只读整包摘要，直接确认 gate required actions 的首尾边界、数量、完整 hash 和 primary action，再决定是否展开完整 `artifact-manifest.json.candidate_issue_gate.required_actions`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.196 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 candidate issue gate required action boundary fields，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且 first/last required action 等于 `artifact-manifest.json.candidate_issue_gate.required_actions` 的首尾元素。

## 变更分组

### Candidate issue gate required action boundary

- `artifact_bundle_summary` 新增 `candidate_issue_gate_first_required_action`。
- `artifact_bundle_summary` 新增 `candidate_issue_gate_last_required_action`。
- `artifact_bundle_summary` 新增 `candidate_issue_gate_required_action_boundary_sha256`。
- Boundary 从 `artifact-manifest.json.candidate_issue_gate.required_actions` 的顺序派生；无 required actions 时首尾为 `null`。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 candidate issue gate required action first/last item 与 boundary hash。
- 现有 `candidate_issue_gate_required_action_count`、`candidate_issue_gate_required_actions_sha256`、`candidate_issue_gate_primary_required_action`、`candidate_issue_gate_reason_code_count`、`candidate_issue_gate_reason_codes_sha256`、`candidate_issue_gate_first_reason_code`、`candidate_issue_gate_last_reason_code` 和 `candidate_issue_gate_reason_code_boundary_sha256` 保持兼容。
- 不改变 publication audit、release draft validation 或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 candidate issue gate required action boundary fields。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的 candidate issue gate required action boundary 字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 `candidate_issue_gate.required_actions` 列表，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `candidate_issue_gate_first_required_action`、`candidate_issue_gate_last_required_action`、`candidate_issue_gate_required_action_boundary_sha256`、`candidate_issue_gate_required_action_count`、`candidate_issue_gate_required_actions_sha256` 和 `candidate_issue_gate_primary_required_action` 快速确认 gate required actions 是否漂移。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
