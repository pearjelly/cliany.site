# v0.16.215 发布草案

**目标版本：** `0.16.215`
**提交范围：** `v0.16.214..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本改善 release draft handoff 字段窗口的可审计性。

## 用户价值

`v0.16.214` 已经让 release readiness report 显式区分 `target` 和 `tagged` 模式。维护者在周初计划里还会读取 candidate issue artifacts 的 `artifact_bundle_summary`，用它判断 `release-draft-handoff.json` 是否需要展开审阅。

`v0.16.215` 给 `artifact_bundle_summary` 增加 `release_draft_handoff_key_preview_*` 和 `release_draft_handoff_key_tail_*` 字段。这样工具只读 `artifact-manifest.json` 的 summary 就能先检查 `release-draft-handoff.json` 字段清单前后窗口是否漂移，再决定是否展开完整 handoff。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.215 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json` 在 `artifact_bundle_summary` 中包含 release draft handoff key preview/tail count、列表和 SHA-256；artifacts `README.md` 的 `Artifact Bundle Summary` 也展示这些字段。

## 变更分组

### Release draft handoff window

- `artifact_bundle_summary` 新增 `release_draft_handoff_key_preview_count`、`release_draft_handoff_key_preview` 和 `release_draft_handoff_key_preview_sha256`。
- `artifact_bundle_summary` 新增 `release_draft_handoff_key_tail_count`、`release_draft_handoff_key_tail` 和 `release_draft_handoff_key_tail_sha256`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些 preview/tail 字段。
- 不改变 `release-draft-handoff.json` 本体结构，不改变 candidate issue gate 判定，也不自动创建 issue。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 release draft handoff key preview/tail contract。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 和 README 展示。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护循环文档中的字段说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的 release draft handoff 可审计性。

## 风险与兼容性

- JSON schema 只新增 `artifact_bundle_summary` 字段，不移除既有字段；只读取旧字段的工具不受影响。
- `release-draft-handoff.json` 本体不变，字段窗口只出现在 summary 和 artifacts `README.md`。
- 新字段使用稳定 JSON SHA-256，和其他 preview/tail 摘要字段保持一致。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.215..HEAD`。最终带 `--release-tag v0.16.215` 的 strict gate 仍以本草案顶部的 `v0.16.214..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.215 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.215 --json
python scripts/release_readiness.py --strict --target-version 0.16.215 --report /tmp/cliany-release-readiness.md
```

## 发版步骤

1. 确认 `pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov` 通过。
2. 确认 `artifact_bundle_summary` 输出 release draft handoff key preview/tail 字段。
3. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些字段。
4. 确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.215`。
5. 确认 `pyproject.toml` 版本更新为 `0.16.215`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.215`。
7. 创建 tag：`git tag v0.16.215`。

## Release Notes 摘要

`v0.16.215` adds release draft handoff key preview/tail summary fields so candidate issue artifact tooling can audit handoff schema windows without opening the full handoff file.
