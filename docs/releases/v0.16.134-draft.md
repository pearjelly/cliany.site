# v0.16.134 发布草案

**目标版本：** `0.16.134`
**提交范围：** `v0.16.133..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 release draft issue count 补进 release draft handoff。

## 用户价值

`v0.16.133` 已经让 `publication-handoff.json` 能直接展示发布命令数量。维护者和自动化继续审阅 candidate issue artifacts 时，`release-draft-handoff.json` 仍只提供 `release_draft_issues` 列表；只想先判断草案门禁规模的工具必须展开完整列表并自行计数。

`v0.16.134` 在 `release-draft-handoff.json` 中新增 `release_draft_issue_count`，并在 artifacts `README.md` 的 `Release Draft Handoff` 段落展示同一字段。维护者和自动化只读 handoff 文件或 README handoff 小节，就能判断下一版草案是否还有多少待处理 issue，再决定是否展开完整列表。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.134 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `release-draft-handoff.json` 包含 `release_draft_issue_count`，artifacts README 的 `Release Draft Handoff` 展示同一字段，`artifact_bundle_summary` 中的 `release_draft_handoff_key_count` 和 `release_draft_handoff_sha256` 随新增字段稳定更新。

## 变更分组

### Release draft handoff issue count

- `release-draft-handoff.json` 新增 `release_draft_issue_count`。
- Artifacts `README.md` 的 `Release Draft Handoff` 段落展示 `release_draft_issue_count`。
- 字段值来自已有 `IterationPlan.release_draft_issues` 列表长度。
- 现有 `release_draft_issues`、`release_draft_path` 和 `artifact_bundle_summary` 保持兼容。
- `release_draft_handoff_key_count` 和 `release_draft_handoff_sha256` 会随 handoff 结构变化稳定更新。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `release-draft-handoff.json` 和 README handoff 会展示 `release_draft_issue_count`。
- `tests/test_plan_next_iteration.py` 覆盖 handoff JSON、README handoff 和 bundle summary 的新字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 release draft handoff 的草案 issue 规模可见性。

## 风险与兼容性

- 新增字段只派生自既有 release draft issues 列表长度。
- 现有 `release_draft_issues`、release draft path 和 `artifact_bundle_summary` 保持兼容。
- 计数只用于快速判断草案门禁规模，不替代完整 issue 列表审阅。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.134 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `release-draft-handoff.json` 包含 `release_draft_issue_count`。
3. 确认 artifacts README 的 `Release Draft Handoff` 展示 `release_draft_issue_count`。
4. 确认 `release_draft_issue_count` 与 `release_draft_issues` 列表长度一致。
5. 确认 `artifact_bundle_summary` 的 `release_draft_handoff_key_count` 和 `release_draft_handoff_sha256` 已随新字段更新。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.134`。
8. 将 `pyproject.toml` 版本更新为 `0.16.134`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.134`。
10. 创建 tag：`git tag v0.16.134`。

`v0.16.134` adds release draft issue count to release draft handoff artifacts, helping maintainers and automation estimate draft gate review scope from the handoff alone.
