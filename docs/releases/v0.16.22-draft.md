# v0.16.22 发布草案

**目标版本：** `0.16.22`
**提交范围：** `v0.16.21..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让周初计划更清楚地指出 release draft gate 的具体缺口。

## 用户价值

`v0.16.21` 之后，`python scripts/plan_next_iteration.py --json` 能看到下一版推荐主题和 blockers。但当下一版 release draft 不存在或缺少必要小节时，计划里只显示 `release draft validation failed`，维护者还需要另跑 `scripts/release_readiness.py` 才知道具体原因。

`v0.16.22` 在 next iteration plan 里新增 `release_draft_issues`。JSON、文本输出和 Markdown report 都会列出 release draft gate 的具体问题，例如 `release draft is missing` 或缺失的 snippet；Markdown report 会在 `Release Draft Issues` 小节里保留同样信息。这样周初计划能直接告诉维护者要创建哪个草案、补哪个小节，再进入下一版 release preflight。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.22 --json` 输出 `release_draft_issues` 字段；当目标草案缺失时，该字段包含 `release draft is missing`；当目标草案存在且通过 gate 时，该字段为空列表。

## 变更分组

### Next iteration release draft diagnostics

- `scripts/plan_next_iteration.py` 的 `IterationPlan` 新增 `release_draft_issues` 字段。
- JSON 输出会保留 release draft gate 的具体 issues。
- 文本输出会在存在问题时打印 `release_draft_issues` 列表。
- Markdown report 会在 `Release Draft` 下追加 `Release Draft Issues` 小节。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明周初计划会直接暴露 `release_draft_issues`。
- `tests/test_plan_next_iteration.py` 固定 JSON 和 Markdown report 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只提升维护者在周初计划里定位 release draft blocker 的速度。

## 风险与兼容性

- `release_draft_issues` 是新增字段，不改变既有字段语义。
- 当 `release_readiness` 没有 draft 细节时，该字段会退化为空列表，兼容测试注入的轻量 report。
- 不访问第三方站点，不使用真实 LLM key，不写入 `~/.cliany-site/` 运行时状态。

## 发版步骤

1. 确认缺失草案时，`python scripts/plan_next_iteration.py --target-version 0.16.22 --json` 的 `release_draft_issues` 包含 `release draft is missing`。
2. 创建并验证本草案后，确认同一字段变为空列表。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.22`。
5. 将 `pyproject.toml` 版本更新为 `0.16.22`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.22`。
7. 创建 tag：`git tag v0.16.22`。

## Release Notes 摘要

`v0.16.22` adds `release_draft_issues` to next iteration plans. Maintainers can now see the exact release draft blocker in JSON, text, and Markdown reports without rerunning release readiness separately.
