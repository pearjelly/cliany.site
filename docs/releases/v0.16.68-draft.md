# v0.16.68 发布草案

**目标版本：** `0.16.68`
**提交范围：** `v0.16.67..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 `artifact_bundle_summary` 直接携带 candidate issue gate reason/action 摘要。

## 用户价值

`v0.16.66` 和 `v0.16.67` 已经让 `candidate_issue_gate` 分别提供 required actions 和 reason codes 的 count/hash。但自动化如果只读取 candidate issue artifacts 的 `artifact_bundle_summary`，仍然只能看到 gate status 和 `can_create_issues`，无法判断 gate reason/action set 是否变化。

`v0.16.68` 会把 `candidate_issue_gate_reason_code_count`、`candidate_issue_gate_reason_codes_sha256`、`candidate_issue_gate_required_action_count` 和 `candidate_issue_gate_required_actions_sha256` 加入 `artifact_bundle_summary`。Artifacts `README.md` 的 `Artifact Bundle Summary` 也展示这些字段，让维护者和工具能在展开完整 manifest 前完成更高信噪比的初筛。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.69 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 gate reason/action count/hash，并与 `candidate_issue_gate` 中的同名语义字段一致；README 的 `Artifact Bundle Summary` 也展示这些字段。

## 变更分组

### Artifact bundle gate summary

- `artifact_bundle_summary` 新增 `candidate_issue_gate_reason_code_count`。
- `artifact_bundle_summary` 新增 `candidate_issue_gate_reason_codes_sha256`。
- `artifact_bundle_summary` 新增 `candidate_issue_gate_required_action_count`。
- `artifact_bundle_summary` 新增 `candidate_issue_gate_required_actions_sha256`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些 gate reason/action 摘要字段。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 gate reason/action 摘要。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate issue artifacts 的整包摘要更适合自动化预筛。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary`，不改变 `candidate_issue_gate` 既有字段语义。
- Summary 字段复用 `candidate_issue_gate` 的同源 count/hash，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.68 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 gate reason/action count/hash。
3. 确认这些 summary 字段与 `candidate_issue_gate` 中的同源字段一致。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.68`。
7. 将 `pyproject.toml` 版本更新为 `0.16.68`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.68`。
9. 创建 tag：`git tag v0.16.68`。
