# v0.16.255 发布草案

**目标版本：** `0.16.255`
**提交范围：** `v0.16.254..HEAD`
**当前状态：** 本版延续 `v0.16.254` 的 candidate promotion 方向，把首要候选案例的 adapter package 交接字段提升为 planner 顶层稳定别名。

## 用户价值

`v0.16.254` 已经让 planner 输出 `case_promotion_evidence_primary_next_task`，其中包含 `pypi-project-search` 当前的 `adapter_package` 任务、预期包名、优先级和 preflight gate。但 release automation 或维护者脚本如果只读取 summary 表、artifact bundle summary 或文本报告，仍需要展开完整 nested task 才能拿到真正要执行和验收的字段。

`v0.16.255` 将这些首要任务字段提升为 compact handoff：`case_promotion_evidence_primary_expected_adapter_package`、`case_promotion_evidence_primary_acceptance_criteria`、`case_promotion_evidence_primary_acceptance_criteria_sha256`、`case_promotion_evidence_primary_priority_rank` 和 `case_promotion_evidence_primary_priority_reason`。这些字段会出现在 planner JSON、Markdown report、纯文本输出、candidate issue artifact summary 和 artifact README 中。

这样下一步推进 `pypi-project-search` 时，工具无需解析 nested JSON 或 Markdown issue body，就能知道要生成 `pypi.org-<version>.cliany-adapter.tar.gz`、要贴哪条 acceptance evidence、以及为什么它是当前第一优先级。

## 变更范围

- `scripts/plan_next_iteration.py`：在 `IterationPlan.to_dict()`、Markdown report、纯文本输出、artifact bundle summary 和 artifact README summary 中输出 primary candidate evidence compact aliases。
- `tests/test_plan_next_iteration.py`：覆盖 planner JSON 与 candidate issue artifacts README 的新增 aliases，确保它们和 `case_promotion_evidence_summary.primary_next_task` 保持一致。
- `CHANGELOG.md`：记录本轮 planner handoff 增强。

## 风险与兼容性

- 不改变 CLI runtime、adapter schema、AXTree selector、session 存储、adapter codegen 或真实 browser workflow 执行。
- 新字段只扩展 planner/release handoff；已有 nested `case_promotion_evidence_primary_next_task` 和 `case_promotion_evidence_summary.primary_next_task` 保持兼容。
- `acceptance_criteria_sha256` 基于已有 `CANDIDATE_PROMOTION_ACCEPTANCE_CRITERIA` 文本生成；如果 acceptance 文案未来改变，hash 会同步变化并暴露漂移。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的案例状态；案例说明仍落在 [cases/README.md](../../cases/README.md)。
- 当前方向仍是把 candidate 案例推向真实可用：`pypi-project-search` 的首要任务保持 `adapter_package`。
- 该任务的 expected package handoff 为 `pypi.org-<version>.cliany-adapter.tar.gz`。
- Candidate 晋级仍需要 adapter package、metadata validation 和 online smoke 三段真实证据；如果 live LLM preflight 未通过，任务保持 pending 或 blocked。
- Known gap 保持 1 个：`search-extraction-gap`。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.254..HEAD`。完成 version bump 后，target-mode readiness 也会查找占位锚点 **提交范围：** `v0.16.255..HEAD`；该锚点只表示 tag 发布后不应再有未发布差异，不改变本版实际提交范围。创建本地 tag 后，使用 local tag-mode readiness 复核最终 tag 语义。


创建本地 tag 后、推送 tag 前再运行：

## 发版步骤

1. 确认 `v0.16.254` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 运行 target-mode readiness，确认 draft、daily cap、publication visibility 和 worktree 状态都允许进入发布。
3. 完成 version bump、CHANGELOG、官网基线和 release notes 摘要。
4. 提交并推送 `master`。
5. 创建并推送 `v0.16.255` tag。
6. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
7. 按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署官网，并运行 `cd site && vercel inspect www.cliany.site --wait --timeout 90s` 复核 alias。
8. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json`，确认 GitHub Release、PyPI 和官网发布状态。

## Release Notes 摘要

v0.16.255 makes the next candidate promotion handoff more directly actionable. The next-iteration planner now exposes the primary candidate task's expected adapter package, acceptance criteria, acceptance hash, priority rank, and priority reason as compact aliases across JSON, text/Markdown reports, artifact bundle summaries, and artifact README output. Automation can now identify the `pypi-project-search` adapter package target and acceptance evidence without expanding the nested primary task object.
