# v0.14.4 发布草案

**状态：** draft  
**目标版本：** `0.14.4`  
**预计类型：** patch  
**提交范围：** `v0.14.3..HEAD`  
**当前阻塞：** 本周提交日尚未达到 3 天；满足节奏后再移动 CHANGELOG、bump version、打 tag。

## 用户价值

`v0.14.4` 聚焦 adapter 分发可靠性、真实案例库可验证性和抽取结果的任务级反馈。它不改变用户的核心探索流程，但让团队在共享、安装、维护 demo adapter 和判断数据命令是否真的有业务结果时更容易发现问题、解释错误并阻止案例索引漂移。

## 变更分组

### Adapter 生命周期

- 新增 `docs/adapter-lifecycle.md`，说明 adapter 从生成、验证、打包、安装到回滚的完整生命周期。
- `docs/adapter-lifecycle.md` 新增临时 HOME 的离线 roundtrip 验证流程，帮助维护者在不污染真实 `~/.cliany-site/` 的情况下检查 publish/install/verify/rollback。
- README 双语 marketplace 示例现在使用 `market publish` 实际生成的 `<domain>-<version>.cliany-adapter.tar.gz` 包名，避免复制旧包名失败。
- `market install` 收紧包内容校验，拒绝缺失声明文件、缺失哈希和未声明额外文件。
- `market install --json` 对安装失败返回 `INSTALL_FAILED` 与针对性 `error.fix`，覆盖重复安装、哈希不匹配、manifest 缺失字段、未声明文件和不安全路径。
- `verify --json` 在已安装 adapter 存在 `manifest.json` 时诊断 manifest v1、domain、`files` / `file_hashes` 与已安装文件哈希。

- `doctor` 的 summary 现在包含 `recommended_next_step`，human 输出会直接显示 `下一步`，帮助首次运行用户判断先跑 demo adapter、配置 LLM key 还是修复环境。

## 案例库映射

本版本的真实场景索引见 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。Active 案例的离线样例输出保存在 `cases/examples/*.json`，用于展示 JSON envelope 形状和典型字段。Release notes 中应明确说明：`suitecrm-accounts`、`apache-jira-issues`、`apache-confluence-search` 和 `apache-jenkins-jobs` 继续作为只读 demo adapter 场景；`pypi-project-search` 是 candidate 案例，用于承接新的公开只读工作流但尚不承诺 release adapter 包；`search-extraction-gap` 继续作为抽取质量 known-gap，并已通过 `tests/fixtures/search_extraction_gap.html` 固化为离线回归。

## 风险与兼容性

- `market install` 现在更严格，手工拼接或被篡改的 adapter 包会失败；使用 `cliany-site market publish <domain>` 重新打包即可修复。
- `verify --json` 会新增 `manifest` 字段；无 market manifest 的本机 `explore` adapter 显示 `manifest.status = "missing"`，但不判定失败。
- `validate_cases.py --packages-dir` 只读取本地包文件，不访问第三方站点，也不写入 `~/.cliany-site/`。
- 生成 adapter 的 JSON 输出新增 `data.quality` 字段；这是向后兼容的附加字段，但 `browser extract --strict-quality` 会把空抽取或关键字段缺失变为 `E_EMPTY_RESULT`，`list-` / `search-` 命令的空抽取或关键字段缺失也会变为 `E_EMPTY_RESULT`，下游如果曾把这些情况当成功处理，需要同步调整。
