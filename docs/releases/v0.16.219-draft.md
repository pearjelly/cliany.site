# v0.16.219 发布草案

**目标版本：** `0.16.219`
**提交范围：** `v0.16.218..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本改善 candidate 晋级证据的结构化交接。

## 用户价值

`v0.16.218` 已经让 candidate issue artifacts 可以从 `artifact_bundle_summary` 检查 publication next actions 的前后窗口。维护者接下来真正要推进的是 candidate 案例本身：adapter 包、metadata 校验和在线只读 smoke 三类证据都需要可分工、可复核、可延续。

`v0.16.219` 给 candidate 案例增加 `promotion_evidence`，为 `adapter_package`、`metadata_validation` 和 `online_smoke` 记录 `pending` / `complete` / `blocked` 状态、证据和下一步动作。`validate_cases.py` 会校验这个结构，`plan_next_iteration.py` 会把它透传到 candidate promotion JSON、issue body、`issue-metadata.json` 和 issue metadata summary。

成功标准是：`python scripts/validate_cases.py --strict` 通过并在报告中展示 candidate `promotion_evidence`；`python scripts/plan_next_iteration.py --target-version 0.16.219 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `issue-metadata.json` 和 candidate issue body 都包含当前 evidence 状态与下一步动作。

## 变更分组

### Candidate promotion evidence

- `cases/manifest.json` 的三个 candidate 案例新增 `promotion_evidence`。
- `scripts/validate_cases.py` 校验 `promotion_evidence` 必须覆盖 `adapter_package`、`metadata_validation` 和 `online_smoke`。
- `pending` / `blocked` 状态必须声明 `next_action`；`complete` 状态必须声明 `evidence`。
- 文本输出、JSON 输出和 Markdown report 都展示 candidate promotion evidence。

### Candidate issue artifacts

- `plan_next_iteration.py` 的 `candidate_promotions` JSON 新增 `promotion_evidence`。
- Candidate issue body 的 `Tasks` 小节展示每项任务的 current status、current evidence 和 next action。
- `issue-metadata.json` 与 `issue_metadata_summary` 纳入 `promotion_evidence`，让 artifacts 包的 hash 能反映证据状态变化。
- Markdown plan 的 `Candidate Promotion Tasks` 表新增 `Promotion Evidence` 列。

### 文档与门禁

- [cases/README.md](../../cases/README.md) 说明 `promotion_evidence` 的字段、允许状态和晋级规则。
- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明周初计划和 issue artifacts 会携带 evidence 状态。
- `tests/test_validate_cases.py`、`tests/test_cases_manifest.py`、`tests/test_plan_next_iteration.py` 和 `tests/test_weekly_maintainer_loop_docs.py` 覆盖新 contract。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate 晋级证据从自然语言说明升级为可校验结构。

## 风险与兼容性

- `promotion` 原有三项说明保持不变；新增 `promotion_evidence` 是 candidate 晋级状态层。
- Candidate 仍保持 `pending`，不会伪造 adapter package、metadata validation 或 online smoke 已完成。
- `issue_metadata_summary` 的 hash 会因新增 evidence 字段变化，这是预期行为。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 focused pytest 和 ruff 通过。
2. 确认 `validate_cases.py --strict` 展示 candidate `promotion_evidence`。
3. 确认 `issue-metadata.json` 包含 `promotion_evidence`。
4. 确认 candidate issue body 展示 current status、current evidence 和 next action。
5. 确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.219`。
6. 确认 `pyproject.toml` 版本更新为 `0.16.219`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.219`。
8. 创建 tag：`git tag v0.16.219`。
