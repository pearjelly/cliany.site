# v0.16.54 发布草案

**目标版本：** `0.16.54`
**提交范围：** `v0.16.53..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue gate evidence 中的 release draft path 使用仓库相对路径。

## 用户价值

`v0.16.53` 已经让 `candidate_issue_gate.evidence` 在默认文本输出中展开，维护者可以直接看到 release draft path、latest tag 和 ahead count。但 evidence 来自 release readiness 对象时，`release_draft_path` 可能是维护者本机上的绝对路径，例如 `/Users/.../docs/releases/...`。这个字段会进入计划 JSON、artifacts manifest 和 publication handoff，不适合出现在可共享交接材料里。

`v0.16.54` 把 `candidate_issue_gate.evidence.release_draft_path` 固定为 `docs/releases/v<version>-draft.md` 形式的仓库相对路径。这样 artifacts 可以安全复制到 issue、PR 或其他机器上审阅，同时仍能准确指向目标 release draft。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.54 --json` 输出的 `candidate_issue_gate.evidence.release_draft_path` 为 `docs/releases/v0.16.54-draft.md`，且不会以 `/` 开头；`--issues-dir` 生成的 `artifact-manifest.json` 和 `publication-handoff.json` 保持同源相对路径。

## 变更分组

### Candidate issue gate evidence path

- `candidate_issue_gate.evidence.release_draft_path` 改为由 target version 生成仓库相对路径。
- 当 fixture 或 readiness report 提供绝对 draft path 时，计划输出仍使用 `docs/releases/v<version>-draft.md`。
- 其他 evidence 字段保持不变。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 release draft path 使用仓库相对路径。
- `tests/test_plan_next_iteration.py` 覆盖 evidence path 不以 `/` 开头。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强 candidate issue gate evidence 的可共享性。

## 风险与兼容性

- `candidate_issue_gate.evidence` 的字段名不变，只调整 `release_draft_path` 的值格式。
- 该路径仍指向同一个仓库内 release draft。
- 不自动执行命令，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.54..HEAD`。最终带 `--release-tag v0.16.54` 的 strict gate 仍以本草案顶部的 `v0.16.53..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.54 --json
python scripts/plan_next_iteration.py --target-version 0.16.54 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.54 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.54 --json` 输出相对 `release_draft_path`。
2. 确认 `candidate_issue_gate.evidence.release_draft_path` 不以 `/` 开头。
3. 确认 `artifact-manifest.json` 和 `publication-handoff.json` 使用同源相对路径。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.54`。
6. 将 `pyproject.toml` 版本更新为 `0.16.54`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.54`。
8. 创建 tag：`git tag v0.16.54`。

## Release Notes 摘要

`v0.16.54` keeps `candidate_issue_gate.evidence.release_draft_path` repository-relative, preventing local absolute paths from leaking into next-iteration plans and candidate issue artifacts.
