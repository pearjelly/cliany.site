# v0.16.77 发布草案

**目标版本：** `0.16.77`
**提交范围：** `v0.16.76..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 `artifact_bundle_summary` 直接携带 issue metadata 的数量与稳定 hash。

## 用户价值

`v0.16.76` 已经让 `artifact_bundle_summary` 能检测 create issues safety contract 是否漂移。但 `issue-metadata.json` 里的 title、labels、target URL、candidate commands 和 body 文件映射也决定了候选任务能否安全创建。

`v0.16.77` 在 `artifact_bundle_summary` 中新增 `issue_metadata_count` 和 `issue_metadata_sha256`，并在 manifest 中新增 `issue_metadata_summary`。这个摘要覆盖 case、title、labels、target URL、commands、offline commands 和 body 文件名，不包含本机输出路径，避免不同 artifacts 目录产生无意义漂移。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.77 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 issue metadata count/hash，并能从 `issue-metadata.json` 提取稳定字段后重新计算得到；README 的 `Artifact Bundle Summary` 也展示这些字段。

## 变更分组

### Artifact bundle issue metadata summary

- `artifact_bundle_summary` 新增 `issue_metadata_count`。
- `artifact_bundle_summary` 新增 `issue_metadata_sha256`。
- `artifact-manifest.json` 新增 `issue_metadata_summary`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些 issue metadata 漂移摘要字段。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 issue metadata count/hash。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate issue artifacts 的整包摘要更适合自动化判断 issue metadata 是否变化。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary` 和 `artifact-manifest.json`，不改变 `issue-metadata.json` 的既有字段语义。
- Hash 基于稳定 issue metadata 字段计算，不包含 `issue_body_file` 和 `create_command` 里的本机输出路径。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.77..HEAD`。最终带 `--release-tag v0.16.77` 的 strict gate 仍以本草案顶部的 `v0.16.76..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.77 --json
python scripts/plan_next_iteration.py --target-version 0.16.77 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.77 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.77 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `issue_metadata_count` 和 `issue_metadata_sha256`。
3. 确认这些摘要能从 `issue-metadata.json` 提取稳定字段后重新计算得到。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示这些字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.77`。
7. 将 `pyproject.toml` 版本更新为 `0.16.77`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.77`。
9. 创建 tag：`git tag v0.16.77`。

## Release Notes 摘要

`v0.16.77` adds stable issue-metadata count and hash fields to artifact bundle summaries, helping tools detect candidate issue metadata drift before reading full manifests.
