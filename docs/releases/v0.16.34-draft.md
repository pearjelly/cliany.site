# v0.16.34 发布草案

**目标版本：** `0.16.34`
**提交范围：** `v0.16.33..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 next iteration plan 和 candidate issue handoff 直接展示 publication audit 的 worktree 状态。

## 用户价值

`v0.16.33` 已经让 `scripts/check_release_publication.py` 的 JSON、文本和 Markdown report 展示 `worktree_clean` / `worktree_status`。但维护者每周首先看的通常是 `scripts/plan_next_iteration.py --json`、Markdown plan 或 `--issues-dir` 生成的候选 issue artifacts；如果这些入口不展示 worktree 状态，维护者仍要回到 publication audit 才能确认本地目录是否干净。

`v0.16.34` 把 publication audit 的 worktree 状态接入 next iteration plan。计划 JSON 新增 `publication_worktree_clean` 和 `publication_worktree_status`；Markdown report 新增 `Publication Worktree` 小节；`publication-handoff.json` 和 artifacts `README.md` 也会展示 `worktree_clean` / `worktree_status`。这样维护者在周初选题、候选 issue 派发和发布交接时都能看到同一份本地状态证据。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.34 --json` 输出 `publication_worktree_clean` 和 `publication_worktree_status`；`--issues-dir` 生成的 `publication-handoff.json` 包含 `worktree_clean` 和 `worktree_status`，artifacts `README.md` 的 `Publication Handoff` 展示 `worktree_clean`。

## 变更分组

### Next iteration publication worktree handoff

- `scripts/plan_next_iteration.py` 的 `IterationPlan` 新增 `publication_worktree_clean` 和 `publication_worktree_status`。
- JSON、文本和 Markdown plan 展示 publication worktree 状态。
- `--issues-dir` 生成的 `publication-handoff.json` 和 artifacts `README.md` 透传同一组 worktree 字段。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明计划器会展示 publication worktree 状态。
- `tests/test_plan_next_iteration.py` 覆盖 JSON、Markdown、publication handoff 和 artifacts README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强 next iteration plan 和候选 issue artifacts 的发布状态交接。

## 风险与兼容性

- `publication_worktree_clean` 和 `publication_worktree_status` 是 additive plan JSON 字段，不移除现有 key。
- `publication-handoff.json` 新增 `worktree_clean` 和 `worktree_status`，已有 publication 字段保持兼容。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- worktree 状态只透传 `scripts/check_release_publication.py` 的本地 audit 结果。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.34 --json` 输出 `publication_worktree_clean` 和 `publication_worktree_status`。
2. 确认 Markdown plan 包含 `Publication Worktree`。
3. 确认 `publication-handoff.json` 包含 `worktree_clean` 和 `worktree_status`。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.34`。
6. 将 `pyproject.toml` 版本更新为 `0.16.34`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.34`。
8. 创建 tag：`git tag v0.16.34`。

## Release Notes 摘要

`v0.16.34` carries release publication worktree status into next-iteration plans and candidate issue handoff artifacts, so maintainers can see dirty-worktree blockers before switching tools.
