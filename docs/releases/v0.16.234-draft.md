# v0.16.234 发布草案

**目标版本：** `0.16.234`
**提交范围：** `v0.16.233..HEAD`
**当前阻塞：** 需要完成 candidate promotion queue 改动，并重新运行 release readiness、case validation、文档/官网测试、完整 pytest、官网浏览器检查和标准发布流程。

## 用户价值

v0.16.234 为 `cliany-site cases` 增加 candidate promotion queue。维护者和贡献者现在可以用一条命令查看所有 candidate 真实案例的首要证据任务、执行命令、handoff、验收标准和 evidence bundle JSON 命令，不必逐个展开单案例 bundle。

这让真实案例库的晋级工作更容易排队和交接，同时继续遵守“证据不足不提前标记 active”的规则。

## 变更分组

- CLI：新增 `cliany-site cases --promotion-plan`，输出 Markdown candidate 晋级队列。
- JSON：新增 `promotion_plan`，包含 `primary_next_item`、每个 candidate 的 primary task 摘要和未完成 `task_queue`。
- 文档：README、README.zh、[cases/README.md](../../cases/README.md) 和官网 quickstart 同步记录 `--promotion-plan`。
- 测试：补充 JSON、人类输出、互斥参数、README 和官网内容断言。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的任何案例状态。
- 当前真实案例库仍保持 active 4 个、candidate 3 个、known gap 1 个。
- Known gap 保持 1 个：`search-extraction-gap`。
- 本次不生成 adapter package，也不写入 `~/.cliany-site/` 运行时状态。
- `pypi-project-search` 仍是首要 candidate；本版本只让 candidate 队列和验收条件更容易被维护者安排，不声称 adapter 包已经生成。

## 风险与兼容性

- 下游如果严格解析 `cliany-site cases --help` 或 `cases` 参数列表，需要兼容新的 `--promotion-plan`。
- `--promotion-plan` 的人类输出比普通列表更长；默认 `cases` 输出不受影响。
- 本次不触碰 action runtime、adapter schema、AXTree selector 语义匹配或 codegen 逻辑。
- Candidate 晋级 active 仍需真实 adapter package、metadata validation 和 online smoke 三段证据全部 complete。
