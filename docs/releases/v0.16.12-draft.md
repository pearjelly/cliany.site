# v0.16.12 发布草案

**目标版本：** `0.16.12`
**提交范围：** `v0.16.11..HEAD`
**当前阻塞：** 需要提交 next iteration plan 的 publish script 生成命令、维护者循环入口和测试；最新本地 release 仍需维护者确认后 push 到远端。

## 用户价值

`v0.16.11` 让 `scripts/check_release_publication.py --publish-script /tmp/cliany-publish-release.sh` 可以生成一个可审阅的发布脚本。但维护者每周首先看的仍是 `scripts/plan_next_iteration.py --json` 或 Markdown report；如果计划器只展示 `publication_publish_commands`，维护者还需要记住另一个命令来生成脚本。

`v0.16.12` 把这个脚本生成入口也放进 next iteration plan：JSON 新增 `publication_publish_script_command`，Markdown report 新增 `Publication Publish Script` 小节。维护者可以从同一份周初计划里选择直接复制 `publication_publish_commands`，或先运行 `python scripts/check_release_publication.py --json --publish-script /tmp/cliany-publish-release.sh` 生成脚本再审阅执行。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.12 --json` 输出 `publication_publish_script_command`；`python scripts/plan_next_iteration.py --target-version 0.16.12 --report /tmp/cliany-next-iteration.md` 的报告包含 `Publication Publish Script` 命令块。

## 变更分组

### Plan publish script command

- `scripts/plan_next_iteration.py` 的计划模型新增 `publication_publish_script_command`。
- JSON 输出固定推荐脚本路径 `/tmp/cliany-publish-release.sh`。
- Markdown report 新增 `Publication Publish Script` 小节。
- 计划器仍然只读取本地 release readiness 和 publication audit，不执行 `git push` 或生成脚本。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明周初计划会带上 publish script 生成命令。
- `tests/test_plan_next_iteration.py` 固定 JSON 和 Markdown report 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本聚焦 release planning handoff，不访问第三方业务站点，不生成 adapter 包，也不会修改案例 manifest 或离线样例。

## 风险与兼容性

- `publication_publish_script_command` 是新增字段；已有计划 JSON 字段和 Markdown 小节保持兼容。
- 命令只是生成脚本的入口，不会由计划器执行。
- 推荐输出目录是 `/tmp/...`，避免把运行时 artifact 写进仓库。
- publish script 的实际内容仍由 `scripts/check_release_publication.py` 生成，避免两处命令漂移。

## 发版前验证

```bash
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.12 --json
python scripts/plan_next_iteration.py --target-version 0.16.12 --report /tmp/cliany-next-iteration.md
python scripts/release_readiness.py --target-version 0.16.12 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.12 --json` 输出 `publication_publish_script_command`。
2. 确认 Markdown report 中的 `Publication Publish Script` 包含 `--publish-script /tmp/cliany-publish-release.sh`。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.12`。
5. 将 `pyproject.toml` 版本更新为 `0.16.12`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.12`。
7. 创建 tag：`git tag v0.16.12`。

## Release Notes 摘要

`v0.16.12` adds `publication_publish_script_command` to the next-iteration planner. Maintainers can now discover the publish-script generation command directly from the planning artifact, alongside publication push commands and candidate case work.
