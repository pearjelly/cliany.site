# v0.16.154 发布草案

**目标版本：** `0.16.154`
**提交范围：** `v0.16.153..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 artifact manifest schema version 提升到 artifact bundle summary。

## 用户价值

`artifact-manifest.json` 已经包含顶层 `schema_version`，用于脚本在解析前判断字段语义。但维护工具如果先读取 `artifact_bundle_summary` 作为轻量入口，仍需要展开完整 manifest 顶层才能确认 manifest schema。

`v0.16.154` 在 `artifact_bundle_summary` 中新增 `artifact_manifest_schema_version`，并在 artifacts `README.md` 的 `Artifact Bundle Summary` 段落展示同源字段。这样工具只读整包摘要时，就能先判断 `artifact-manifest.json` 的字段语义版本，再决定是否展开完整 manifest。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.154 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `artifact_manifest_schema_version`，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且该值等于 `artifact-manifest.json.schema_version`。

## 变更分组

### Artifact bundle manifest schema version

- `artifact_bundle_summary` 新增 `artifact_manifest_schema_version`。
- 该字段与 `artifact-manifest.json.schema_version` 同源。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 `artifact_manifest_schema_version`。
- 现有 `artifact_bundle_summary` 字段、`artifact-manifest.json` 顶层 `schema_version`、`publication_handoff_schema_version`、`release_draft_handoff_schema_version`、key count 与 hash 保持兼容。
- 不改变 release draft validation、publication audit 或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 artifact manifest schema version。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的新字段，并断言它与 manifest 顶层 schema version 同源。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 artifact manifest schema version，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `artifact_bundle_summary` 判断 manifest schema，再决定是否展开完整 manifest。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.154 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `artifact_manifest_schema_version`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示 `artifact_manifest_schema_version`。
4. 确认该字段等于 `artifact-manifest.json.schema_version`。
5. 确认 handoff schema version、key count 和 hash 仍与当前 handoff 同源。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.154`。
8. 将 `pyproject.toml` 版本更新为 `0.16.154`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.154`。
10. 创建 tag：`git tag v0.16.154`。
