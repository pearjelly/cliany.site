# v0.16.17 发布草案

**目标版本：** `0.16.17`
**提交范围：** `v0.16.16..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让候选 issue 创建脚本在 preflight 失败时直接展示 blocker。

## 用户价值

`v0.16.16` 让 candidate issue artifacts 带上 `publication-handoff.json`，维护者能在创建候选 issue 前看到 publication 状态和 publish commands。但 `create-issues.sh` 的 strict preflight 仍会把 JSON 直接重定向到 `/tmp/cliany-issue-publication-check.json`；如果脚本失败，终端只看到命令退出，维护者还要再打开文件才能理解具体 blocker。

`v0.16.17` 让生成的 `create-issues.sh` 在 publication preflight 失败时打印明确提示，并把 `/tmp/cliany-issue-publication-check.json` 内容输出到 stderr 后退出。这样维护者运行脚本时能立即看到 branch/tag 未发布等 blocker，不会误以为 `gh issue create` 本身失败。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.17 --issues-dir /tmp/cliany-candidate-issues` 写出的 `create-issues.sh` 使用 `PREFLIGHT_JSON`、`if ! python scripts/check_release_publication.py --strict --json`、失败提示、`cat "$PREFLIGHT_JSON" >&2` 和 `exit 1`。

## 变更分组

### Candidate issue preflight failure output

- `create-issues.sh` 将 preflight 输出路径保存到 `PREFLIGHT_JSON="/tmp/cliany-issue-publication-check.json"`。
- publication preflight 失败时，脚本打印 `Release publication preflight failed` 提示。
- 脚本随后把 preflight JSON 输出到 stderr 并以非零状态退出。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 preflight 失败时会打印 JSON 后退出。
- `tests/test_plan_next_iteration.py` 固定脚本内容和 artifacts README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只改善维护者手动执行 `create-issues.sh` 时的失败反馈。

## 风险与兼容性

- preflight 成功路径保持兼容，后续 `gh issue create` 命令不变。
- preflight 失败路径会多打印 JSON 到 stderr，便于 CI 或维护者终端直接保留 blocker 证据。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

## 发版前验证

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.17 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.17 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.17 --issues-dir /tmp/cliany-candidate-issues` 写出带失败输出逻辑的 `create-issues.sh`。
2. 确认 artifacts `README.md` 说明 preflight 失败时会打印 JSON 后退出。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.17`。
5. 将 `pyproject.toml` 版本更新为 `0.16.17`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.17`。
7. 创建 tag：`git tag v0.16.17`。

## Release Notes 摘要

`v0.16.17` improves generated candidate issue scripts by printing the release-publication preflight JSON when the preflight fails. Maintainers can now see the publication blocker immediately before any `gh issue create` command runs.
