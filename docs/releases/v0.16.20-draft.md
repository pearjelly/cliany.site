# v0.16.20 发布草案

**目标版本：** `0.16.20`
**提交范围：** `v0.16.19..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让候选 issue artifacts README 直接展示 candidate 摘要。

## 用户价值

`v0.16.19` 让 artifacts `README.md` 的审阅清单提醒维护者检查 `issue-metadata.json` 里的 target URL、candidate commands 和 offline validation commands。但 README 的文件区仍只列出 body 文件名；维护者想快速知道这一批 candidate 分别指向哪个站点、每个候选有多少命令，还需要打开 JSON。

`v0.16.20` 在 artifacts `README.md` 中新增 `Candidate Summary` 表。表格按 case 展示 target URL、candidate commands 数量和 offline validation commands 数量，让维护者打开目录入口时就能快速确认候选任务范围，再决定是否继续审阅 metadata/body 或运行 `create-issues.sh`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.20 --issues-dir /tmp/cliany-candidate-issues` 写出的 README 包含 `Candidate Summary` 表，并列出 `pypi-project-search`、target URL、candidate command 数量和 offline validation command 数量。

## 变更分组

### Candidate issue artifacts summary

- `scripts/plan_next_iteration.py --issues-dir` 的 artifacts `README.md` 新增 `Candidate Summary` 小节。
- 摘要表展示 case id、target URL、candidate commands 数量和 offline validation commands 数量。
- `issue-metadata.json`、`publication-handoff.json` 和 `create-issues.sh` 的结构保持兼容。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifacts README 会带 `Candidate Summary` 表。
- `tests/test_plan_next_iteration.py` 固定 artifacts README 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只让维护者更快理解一批候选 issue artifacts 的内容。

## 风险与兼容性

- 这是 artifacts README 文案增强，不改变 JSON schema 和脚本执行行为。
- 摘要表只读取本地 plan/case catalog 信息，不访问第三方站点。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.20 --issues-dir /tmp/cliany-candidate-issues` 写出的 README 包含 `Candidate Summary`。
2. 确认 `Candidate Summary` 展示 case id、target URL、candidate commands 数量和 offline validation commands 数量。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.20`。
5. 将 `pyproject.toml` 版本更新为 `0.16.20`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.20`。
7. 创建 tag：`git tag v0.16.20`。

## Release Notes 摘要

`v0.16.20` adds a Candidate Summary table to generated candidate issue artifact READMEs. Maintainers can now see each candidate case, target URL, and command counts before opening JSON metadata or creating GitHub issues.
