# v0.16.220 发布草案

**目标版本：** `0.16.220`
**提交范围：** `v0.16.219..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate 晋级证据具备报告级汇总。

## 用户价值

`v0.16.219` 已经为 candidate 案例增加 `promotion_evidence`，让 `adapter_package`、`metadata_validation` 和 `online_smoke` 的状态、证据和下一步动作变成可校验结构。维护者下一步需要的是更快地判断本周该推进哪个证据，而不是逐条展开每个 candidate。

`v0.16.220` 在 `scripts/validate_cases.py` 中新增 `promotion_evidence_summary`。JSON 输出会汇总 candidate 数量、任务数量、pending/blocked/complete 计数、按任务维度的状态计数、任务列表和 primary next action。Markdown report 新增 `Candidate Promotion Evidence Summary` 小节，默认文本输出也展示 `promotion_evidence_pending`、`promotion_evidence_blocked`、`promotion_evidence_complete` 和 `promotion_evidence_next`。

成功标准是：`python scripts/validate_cases.py --strict --json` 输出 `promotion_evidence_summary`；`python scripts/validate_cases.py --strict --report /tmp/cliany-case-catalog-report.md` 生成 `Candidate Promotion Evidence Summary`；当前真实案例库显示 3 个 candidate、9 个 pending 子任务，并把 PyPI adapter package 作为 primary next action。

## 变更分组

### Candidate evidence summary

- `CasesReport.to_dict()` 新增 `promotion_evidence_summary`。
- Summary 包含 `candidate_count`、`task_count`、`status_counts`、`task_status_counts`、`pending_count`、`blocked_count`、`complete_count`、任务列表、`primary_task` 和 `primary_next_action`。
- 默认文本输出展示 promotion evidence 的 pending/blocked/complete 数量和第一条 next action。
- Markdown report 新增 `Candidate Promotion Evidence Summary` 表，列出每个 candidate 子任务的 status、evidence 和 next action。

### 文档与门禁

- [cases/README.md](../../cases/README.md) 说明 `Candidate Promotion Evidence Summary` 可作为 candidate 晋级状态看板。
- `tests/test_validate_cases.py` 覆盖 JSON、文本和 Markdown report 中的 summary contract。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate 晋级证据的整体状态更容易被维护者和自动化读取。

## 风险与兼容性

- 单个 case 的 `promotion` 和 `promotion_evidence` 字段保持不变。
- `promotion_evidence_summary` 是新增 JSON 字段；只读取旧字段的工具不受影响。
- Summary 只汇总现有 manifest 状态，不会伪造 adapter package、metadata validation 或 online smoke 已完成。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 focused pytest 和 ruff 通过。
2. 确认 `validate_cases.py --strict --json` 输出 `promotion_evidence_summary`。
3. 确认 Markdown report 包含 `Candidate Promotion Evidence Summary`。
4. 确认文本输出展示 `promotion_evidence_pending`、`promotion_evidence_blocked`、`promotion_evidence_complete` 和 `promotion_evidence_next`。
5. 确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.220`。
6. 确认 `pyproject.toml` 版本更新为 `0.16.220`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.220`。
8. 创建 tag：`git tag v0.16.220`。

## Release Notes 摘要

`v0.16.220` adds candidate promotion evidence summaries to case validation reports, giving maintainers an immediate view of pending, blocked, and complete promotion tasks plus the next recommended evidence action.
