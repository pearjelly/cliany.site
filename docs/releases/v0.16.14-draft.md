# v0.16.14 发布草案

**目标版本：** `0.16.14`
**提交范围：** `v0.16.13..HEAD`
**当前阻塞：** 需要提交 candidate issue body 的复现上下文、维护者循环入口和测试；最新本地 release 仍需维护者确认后 push 到远端。

## 用户价值

`v0.16.13` 让 `scripts/validate_cases.py --report` 输出 `Candidate Handoff Matrix`，贡献者可以从 case catalog report 看到 candidate 的 target URL、recommended commands 和 offline validation commands。但 `scripts/plan_next_iteration.py --issues-dir` 生成的 GitHub issue body 仍只包含 promotion 任务，贡献者拿到单个 issue 时还要回到报告或 manifest 查复现上下文。

`v0.16.14` 把同一份 handoff 上下文写进 candidate issue body。每个 issue body 新增 `Reproduction Context`，包含 target URL、candidate commands 和 offline validation commands。这样 `pypi-project-search.md` 这类 artifact 可以单独作为 issue 交接材料使用。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.14 --issues-dir /tmp/cliany-candidate-issues` 写出的 candidate body 文件包含 `Reproduction Context`，并列出 target URL、candidate commands 和 offline validation commands。

## 变更分组

### Candidate issue reproduction context

- `scripts/plan_next_iteration.py` 的 `candidate_promotions` JSON 增加 `target_url`、`commands` 和 `offline_commands`。
- 生成的 `issue_body` 新增 `Reproduction Context`。
- `--issues-dir` 输出的 `pypi-project-search.md`、`npm-package-search.md`、`crates-io-crate-search.md` 会带上对应复现上下文。
- issue body 仍保留原有 tasks、validation evidence 和 non-goals。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 candidate issue body 会保留复现上下文。
- `tests/test_plan_next_iteration.py` 固定 JSON、Markdown report 和 issue artifacts 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本让 candidate issue 更容易被贡献者接手，但不会提前把 candidate 标记为 active；adapter package、metadata validation 和 online smoke 仍必须按 promotion 清单逐项完成。

## 风险与兼容性

- `target_url`、`commands` 和 `offline_commands` 是 `candidate_promotions` 的新增字段；已有字段保持兼容。
- 生成 issue body 只读取 readiness/case catalog 的本地信息，不访问第三方站点。
- 默认验证继续离线运行，不需要真实 LLM key，也不会写入 `~/.cliany-site/`。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.14 --issues-dir /tmp/cliany-candidate-issues` 写出带 `Reproduction Context` 的 body 文件。
2. 确认 body 文件包含 target URL、candidate commands 和 offline validation commands。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.14`。
5. 将 `pyproject.toml` 版本更新为 `0.16.14`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.14`。
7. 创建 tag：`git tag v0.16.14`。

## Release Notes 摘要

`v0.16.14` adds reproduction context to candidate promotion issue bodies. Generated candidate issue artifacts now carry target URLs, candidate commands, and offline validation commands alongside promotion tasks.
