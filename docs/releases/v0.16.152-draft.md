# v0.16.152 发布草案

**目标版本：** `0.16.152`
**提交范围：** `v0.16.151..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 release draft handoff schema version 提升到 artifact bundle summary。

## 用户价值

`v0.16.151` 已经让 `artifact_bundle_summary` 能展示 `publication_handoff_schema_version`。维护工具只读整包摘要时，可以判断 publication handoff 的字段语义版本；但 release draft handoff 仍只有 key count 和 hash，工具需要展开 `release-draft-handoff.json` 才能确认 schema。

`v0.16.152` 在 `artifact_bundle_summary` 中新增 `release_draft_handoff_schema_version`，并在 artifacts `README.md` 的 `Artifact Bundle Summary` 段落展示同源字段。这样工具只读整包摘要时，就能同时判断 publication handoff 和 release draft handoff 的 schema version，再决定是否展开完整 handoff 文件。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.152 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `release_draft_handoff_schema_version`，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且该值与 `release-draft-handoff.json.schema_version` 同源。

## 变更分组

### Artifact bundle release draft handoff schema version

- `artifact_bundle_summary` 新增 `release_draft_handoff_schema_version`。
- 该字段从 `release-draft-handoff.json.schema_version` 派生。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示 `release_draft_handoff_schema_version`。
- 现有 `release_draft_handoff_key_count`、`release_draft_handoff_sha256`、`release-draft-handoff.json` 和 `Release Draft Handoff` 保持兼容。
- 不改变 release draft validation、publication audit 或 candidate issue gate 的判断逻辑。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示 release draft handoff schema version。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的新字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的整包摘要可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 `release-draft-handoff.json.schema_version`，不新增网络访问或外部状态读取。
- 旧工具忽略未知 summary 字段即可继续读取现有 manifest。
- 新工具可以只读 `artifact_bundle_summary` 判断 release draft handoff schema，再决定是否展开 handoff。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.152 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `release_draft_handoff_schema_version`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示 `release_draft_handoff_schema_version`。
4. 确认该字段等于 `release-draft-handoff.json.schema_version`。
5. 确认 `release_draft_handoff_key_count` 和 `release_draft_handoff_sha256` 仍与当前 handoff 同源。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.152`。
8. 将 `pyproject.toml` 版本更新为 `0.16.152`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.152`。
10. 创建 tag：`git tag v0.16.152`。

## Release Notes 摘要

`v0.16.152` adds the release draft handoff schema version to artifact bundle summaries, helping tools decide whether a release draft handoff is compatible before expanding the full handoff payload.
