# v0.16.90 发布草案

**目标版本：** `0.16.90`
**提交范围：** `v0.16.89..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 `artifact_bundle_summary` 直接携带 publication publish script command 的稳定摘要。

## 用户价值

`v0.16.89` 已经让发布可见性对象和 summary 文本可以被整包摘要快速校验。但维护者真正执行发布前，还需要审阅 `publication_publish_script_command`，确认生成 `/tmp/cliany-publish-release.sh` 的入口没有漂移。

`v0.16.90` 在 `artifact_bundle_summary` 中新增 `publication_publish_script_command_sha256`。Artifacts `README.md` 的 `Artifact Bundle Summary` 也展示这个字段，让工具只读整包摘要就能判断发布脚本生成命令是否漂移，再决定是否展开 `artifact-manifest.json` 或重新生成发布脚本。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.90 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_publish_script_command_sha256`，并能从 manifest 中的 `publication_publish_script_command` 重新计算得到；README 的 `Artifact Bundle Summary` 也展示这个字段。

## 变更分组

### Artifact bundle publication publish script command hash

- `artifact_bundle_summary` 新增 `publication_publish_script_command_sha256`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示 publication publish script command hash。
- 现有 `publication_publish_script_command` 字段保持不变；新摘要只用于快速判断发布脚本生成命令是否需要展开核对。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 publication publish script command 摘要。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate issue artifacts 的整包摘要更适合自动化判断发布脚本生成入口是否变化。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary`，不改变 `publication_publish_script_command` 的既有字段语义。
- Hash 基于 publication publish script command 的稳定 JSON 序列化计算，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.90..HEAD`。最终带 `--release-tag v0.16.90` 的 strict gate 仍以本草案顶部的 `v0.16.89..HEAD` 作为实际 release diff 范围。

```bash
python scripts/plan_next_iteration.py --target-version 0.16.90 --json
python scripts/plan_next_iteration.py --target-version 0.16.90 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.90 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.90 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_publish_script_command_sha256`。
3. 确认这个 hash 能从 `artifact-manifest.json.publication_publish_script_command` 重新计算得到。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示该字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.90`。
7. 将 `pyproject.toml` 版本更新为 `0.16.90`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.90`。
9. 创建 tag：`git tag v0.16.90`。

## Release Notes 摘要

`v0.16.90` adds a publication publish script command hash to artifact bundle summaries, helping tools detect publish-script entry drift before reading full manifests.
