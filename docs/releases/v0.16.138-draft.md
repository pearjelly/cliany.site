# v0.16.138 发布草案

**目标版本：** `0.16.138`
**提交范围：** `v0.16.137..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 release draft primary issue 补进 release draft handoff。

## 用户价值

`v0.16.137` 已经让 `release-draft-handoff.json` 能直接展示 release draft issues hash。维护者和自动化只读 handoff 时，仍需要展开 `release_draft_issues` 列表，才能拿到第一条最直接的草案阻塞原因。

`v0.16.138` 在 `release-draft-handoff.json` 中新增 `release_draft_primary_issue`，并在 artifacts `README.md` 的 `Release Draft Handoff` 段落展示同一字段。维护者和自动化只读 handoff 文件或 README handoff 小节，就能直接知道草案当前第一条阻塞，再决定是否展开完整 issues 列表。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.138 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `release-draft-handoff.json` 包含 `release_draft_primary_issue`，artifacts README 的 `Release Draft Handoff` 展示同一字段，`artifact_bundle_summary` 中的 `release_draft_handoff_key_count` 和 `release_draft_handoff_sha256` 随新增字段稳定更新。

## 变更分组

### Release draft handoff primary issue

- `release-draft-handoff.json` 新增 `release_draft_primary_issue`。
- Artifacts `README.md` 的 `Release Draft Handoff` 段落展示 `release_draft_primary_issue`。
- 字段值来自已有 `release_draft_issues` 的第一项；无 issue 时为 `null`，README 显示 `(none)`。
- 现有 `release_draft_ok`、`release_draft_issue_count`、`release_draft_path_sha256`、`release_draft_issues_sha256` 和 `artifact_bundle_summary` 保持兼容。
- `release_draft_handoff_key_count` 和 `release_draft_handoff_sha256` 会随 handoff 结构变化稳定更新。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `release-draft-handoff.json` 和 README handoff 会展示 `release_draft_primary_issue`。
- `tests/test_plan_next_iteration.py` 覆盖 handoff JSON、README handoff 和 bundle summary 的新字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 release draft handoff 的第一条草案阻塞原因可见性。

## 风险与兼容性

- 新增字段只派生自既有 release draft issues 列表。
- 现有 release draft ok、issue count、path hash、issues hash 和 `artifact_bundle_summary` 保持兼容。
- primary issue 用于快速定位第一条阻塞，不替代完整 release draft issue 审阅。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.138 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `release-draft-handoff.json` 包含 `release_draft_primary_issue`。
3. 确认 artifacts README 的 `Release Draft Handoff` 展示 `release_draft_primary_issue`。
4. 确认 `release_draft_primary_issue` 与 `release_draft_issues` 第一项一致。
5. 确认无 release draft issue 时 `release_draft_primary_issue` 为 `null`。
6. 确认 `artifact_bundle_summary` 的 `release_draft_handoff_key_count` 和 `release_draft_handoff_sha256` 已随新字段更新。
7. 运行计划器、维护者循环文档和 release draft 相关测试。
8. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.138`。
9. 将 `pyproject.toml` 版本更新为 `0.16.138`。
10. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.138`。
11. 创建 tag：`git tag v0.16.138`。

`v0.16.138` adds primary release draft issue to release draft handoff artifacts, helping maintainers and automation identify the first draft blocker from the handoff alone.
