# v0.16.143 发布草案

**目标版本：** `0.16.143`
**提交范围：** `v0.16.142..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把第一条 release draft 修复动作补进 artifact bundle summary。

## 用户价值

`v0.16.142` 已经让 `artifact_bundle_summary` 能展示 release draft required actions 的数量和 hash。维护者和自动化只读整包摘要时，可以判断动作列表是否漂移，但如果要展示第一条最该处理的草案动作，仍需要展开完整 `release-draft-handoff.json`。

`v0.16.143` 在 `artifact_bundle_summary` 中新增 `release_draft_primary_required_action`，并在 artifacts `README.md` 的 `Artifact Bundle Summary` 段落展示同源字段。这样工具只读 manifest summary 或 README summary，就能直接显示第一条 release draft 修复动作，同时仍可用 count/hash 判断是否展开完整列表。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.143 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `release_draft_primary_required_action`，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且该值与 `release-draft-handoff.json.release_draft_required_actions[0]` 同源；无 required actions 时字段为 `null`。

## 变更分组

### Artifact bundle release draft primary required action

- `artifact_bundle_summary` 新增 `release_draft_primary_required_action`。
- 有 release draft required actions 时，该字段取第一条动作。
- 无 release draft required actions 时，该字段为 `null`；README summary 用 `(none)` 语义展示。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示第一条 release draft 修复动作。
- 现有 `release_draft_required_action_count`、`release_draft_required_actions_sha256`、`release_draft_handoff_key_count`、`release_draft_handoff_sha256` 和 `Release Draft Handoff` 保持兼容。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示第一条 release draft 修复动作。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的新字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 整包摘要的 release draft primary required action 可见性。

## 风险与兼容性

- 新字段只派生自既有 `release_draft_required_actions`，不新增网络访问或外部状态读取。
- 无 release draft issue 时字段为 `null`，方便机器读取。
- Primary action 只用于快速展示第一步，不替代完整 `release-draft-handoff.json` 审阅。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.143 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `release_draft_primary_required_action`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示 `release_draft_primary_required_action`。
4. 确认有 required actions 时，该字段等于 `release-draft-handoff.json.release_draft_required_actions[0]`。
5. 确认无 required actions 时，JSON 字段为 `null` 且 README 展示 `(none)`。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.143`。
8. 将 `pyproject.toml` 版本更新为 `0.16.143`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.143`。
10. 创建 tag：`git tag v0.16.143`。
