# v0.16.46 发布草案

**目标版本：** `0.16.46`
**提交范围：** `v0.16.45..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts manifest 直接展示复现命令。

## 用户价值

`v0.16.45` 已经让 `artifact-manifest.json` 展示 release draft handoff。维护者或工具可以从一个机器可读入口判断 publication、候选案例范围、阻塞项、下一步和 release draft 缺口，但如果想重新生成同一批 candidate issue artifacts，仍然要回到 plan JSON 或 README 找 `issue_artifacts_command`。

`v0.16.46` 让 `artifact-manifest.json` 直接包含 `issue_artifacts_command`。这样 manifest 既是审阅入口，也是复现入口，维护者可以用同一个 JSON 找到当前 artifacts 的生成命令，再重新生成候选 issue body、metadata、publication handoff 和 release draft handoff。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.46 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `artifact-manifest.json` 包含 `issue_artifacts_command`；artifacts `README.md` 的 Files 小节说明 manifest 包含 reproduction command。

## 变更分组

### Candidate artifacts manifest reproduction command

- `artifact-manifest.json` 新增 `issue_artifacts_command`。
- Artifacts `README.md` 的 Files 小节说明 manifest 包含 reproduction command。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 manifest 会保留 `issue_artifacts_command`。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 内容和 README 文件列表。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强候选 issue artifacts manifest 的复现入口。

## 风险与兼容性

- `issue_artifacts_command` 是 `artifact-manifest.json` 的新增字段，不改变 plan JSON schema。
- 既有 publication、release draft、candidate、files、review order 和 validation commands 字段保持兼容。
- 不自动执行命令，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- Reproduction command 只复用当前 plan 的既有 `issue_artifacts_command`。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.46..HEAD`。最终带 `--release-tag v0.16.46` 的 strict gate 仍以本草案顶部的 `v0.16.45..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.46 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.46 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.46 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `artifact-manifest.json`。
2. 确认 manifest 包含 `issue_artifacts_command`。
3. 确认 artifacts `README.md` 的 Files 小节说明 manifest 包含 reproduction command。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.46`。
6. 将 `pyproject.toml` 版本更新为 `0.16.46`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.46`。
8. 创建 tag：`git tag v0.16.46`。

## Release Notes 摘要

`v0.16.46` adds the candidate issue artifacts reproduction command to generated `artifact-manifest.json` files, making each artifact bundle easier to regenerate from one machine-readable entrypoint.
