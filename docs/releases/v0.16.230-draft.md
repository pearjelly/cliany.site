# v0.16.230 发布草案

**目标版本：** `0.16.230`
**提交范围：** `v0.16.229..HEAD`
**当前阻塞：** 需要完成 evidence bundle 验收条件改动，并重新运行 release readiness、case validation、CLI/文档/官网测试。

## 用户价值

`v0.16.229` 已把项目维护节奏升级为“每天至少一个可验证版本”。下一步需要让每日小版本继续产生真实开源价值：不仅告诉维护者 candidate 案例下一步要跑哪条命令，还要明确“跑完以后必须贴什么证据才算完成”。

`v0.16.230` 聚焦 candidate promotion 的证据验收合同。`cliany-site cases --case-id <id> --evidence-bundle --json` 现在为 `adapter_package`、`metadata_validation` 和 `online_smoke` 三个任务输出 `acceptance_criteria`，并把首要任务的验收条件提升为 `primary_next_task_acceptance_criteria`。这样维护者、issue bot 或发布脚本可以区分“下一步命令”和“完成证据”，减少只贴命令输出、不贴可验收结果的交接空洞。

成功标准是：evidence bundle JSON 和 Markdown 都展示 acceptance criteria；README、README.zh、cases README 和官网能说明新字段；`scripts/release_readiness.py --target-version 0.16.230 --json` 不再因为缺失 release draft 阻塞。

## 变更分组

### Candidate Evidence Bundle

- `cliany-site cases --case-id <id> --evidence-bundle --json` 新增顶层 `acceptance_criteria`，按任务列出完成证据要求。
- 每个 `tasks[]` 和 `task_handoffs[]` 条目新增 `acceptance_criteria`，让自动化可以逐项读取验收条件。
- 顶层新增 `primary_next_task_acceptance_criteria`，直接暴露当前首要 pending/blocked 任务需要附上的完成证据。
- 人类 Markdown 输出新增 `Acceptance criteria` 小节，并在首要任务摘要中展示 primary acceptance。

### 文档与官网同步

- README / README.zh 的真实 demo 入口说明 evidence bundle 现在包含 `acceptance_criteria`。
- `cases/README.md` 说明顶层和任务级 acceptance criteria 的用途，以及它与 `cases/manifest.json` 中 `promotion_evidence` 的关系。
- 官网 quickstart 的 candidate case 提示同步 `acceptance_criteria`，让用户从公开页面也能理解候选案例晋级证据要求。

## 案例库映射

- 本版本不改变 [cases/manifest.json](../../cases/manifest.json) 的案例状态，不新增 active adapter 包，不生成或提交 `~/.cliany-site/` 运行时状态。
- Active 案例保持 4 个：SuiteCRM、Apache Jira、Apache Confluence、Apache Jenkins。
- Candidate 案例保持 3 个：PyPI、npm、crates.io 搜索工作流。
- Known gap 保持 1 个：`search-extraction-gap`。
- `pypi-project-search` 仍是下一轮首要 candidate；本版本为它的 `adapter_package` 任务补齐机器可读验收条件，但不声称 adapter 包已经生成。

## 风险与兼容性

- 风险：下游工具如果对 evidence bundle 做严格字段白名单，新增字段可能需要同步解析器；字段只新增不删除，既有字段语义保持不变。
- 风险：acceptance criteria 是验收合同，不是自动验证结果；真正晋级 active 仍需 adapter package、metadata validation 和 online smoke 三段证据全部 complete。
- 兼容性：不修改 adapter schema、生成代码、动作回放、CDP 连接或 case manifest 状态。
- 兼容性：发布节奏继续遵循每日 verified release train 和标准 release flow。
