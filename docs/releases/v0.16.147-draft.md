# v0.16.147 发布草案

**目标版本：** `0.16.147`
**提交范围：** `v0.16.146..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把第一条 publication publish command 补进 publication handoff。

## 用户价值

`v0.16.146` 已经让 `artifact_bundle_summary` 能直接展示 `publication_primary_publish_command`。维护工具如果只读整包摘要，可以显示第一条发布同步命令；但只读 `publication-handoff.json` 的工具仍要展开 `publish_commands` 列表，才能拿到同一个首要命令。

`v0.16.147` 在 `publication-handoff.json` 中新增 `primary_publish_command`，并在 artifacts `README.md` 的 `Publication Handoff` 段落展示同源字段。这样工具读取 publication handoff 时，可以先展示第一条发布命令，再用 `publish_command_count` 判断是否需要展开完整 `publish_commands`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.147 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `publication-handoff.json` 包含 `primary_publish_command`，artifacts README 的 `Publication Handoff` 展示同一字段，且该值与 `publish_commands[0]` 同源；无 publish command 时字段为 `null` 且 README 展示 `(none)`。

## 变更分组

### Publication handoff primary publish command

- `publication-handoff.json` 新增 `primary_publish_command`。
- 有 publish commands 时，该字段取第一条 command。
- 无 publish command 时，该字段为 `null`；README 用 `(none)` 展示。
- Artifacts `README.md` 的 `Publication Handoff` 展示第一条发布同步命令。
- 现有 `publish_command_count`、`publish_commands`、`publish_script_path_sha256`、`publish_script_command_sha256`、`publication_primary_publish_command`、`artifact_bundle_summary` 和 `publication_handoff_sha256` 保持兼容并反映 handoff 字段变化。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `publication-handoff.json` 与 artifacts README 会展示 `primary_publish_command`。
- `tests/test_plan_next_iteration.py` 覆盖 handoff JSON、README 展示和 handoff hash。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 的 publication handoff 可读性和机器可读入口。

## 风险与兼容性

- 新字段只派生自既有 `publish_commands`，不新增网络访问或外部状态读取。
- 无 publish command 时字段为 `null`，方便机器读取。
- Primary command 只用于快速展示第一条可执行命令，不替代完整 `publish_commands` 审阅。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.147 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `publication-handoff.json` 包含 `primary_publish_command`。
3. 确认 artifacts README 的 `Publication Handoff` 展示 `primary_publish_command`。
4. 确认有 publish command 时，该字段等于 `publish_commands[0]`。
5. 确认无 publish command 时，JSON 字段为 `null` 且 README 展示 `(none)`。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.147`。
8. 将 `pyproject.toml` 版本更新为 `0.16.147`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.147`。
10. 创建 tag：`git tag v0.16.147`。

`v0.16.147` adds the primary publication publish command to publication handoff artifacts, helping tools show the first executable release publishing command without expanding the full command list.
