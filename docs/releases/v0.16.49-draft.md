# v0.16.49 发布草案

**目标版本：** `0.16.49`
**提交范围：** `v0.16.48..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts manifest 的 validation commands 覆盖 release readiness。

## 用户价值

`v0.16.48` 已经让 `artifact-manifest.json` 的 `validation_commands` 包含 publication audit。维护者可以从候选 issue artifacts 包里直接看到如何复核最新本地 release 是否已经公开可见。但 artifacts 包仍缺少下一版 release gate 的复核命令，维护者还要回到 plan JSON 或周维护文档里找 `release_readiness.py`。

`v0.16.49` 让 `artifact-manifest.json` 的 `validation_commands` 包含 `python scripts/release_readiness.py --target-version <version> --json`。这样候选 issue artifacts 包可以同时告诉维护者如何复核 publication 状态、release gate、case catalog 和离线验证，减少创建候选 issue 前的判断跳转。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.49 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `artifact-manifest.json` 的 `validation_commands` 包含 `python scripts/release_readiness.py --target-version 0.16.49 --json`；每周维护者循环文档说明这条 release gate 命令。

## 变更分组

### Candidate artifacts manifest readiness validation

- `artifact-manifest.json` 的 `validation_commands` 新增 `python scripts/release_readiness.py --target-version <version> --json`。
- Artifacts manifest 继续保留 publication audit、case validation、plan reproduction 和 schema version。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 manifest validation commands 会包含 release readiness。
- `tests/test_plan_next_iteration.py` 覆盖 manifest validation commands。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强候选 issue artifacts manifest 的验证命令清单。

## 风险与兼容性

- `validation_commands` 是既有字段，本版本只追加 release readiness 命令。
- 不改变 plan JSON schema，不改变 `publication-handoff.json` 或 `release-draft-handoff.json`。
- 不自动执行命令，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- 这条命令只读项目元数据、release draft、case catalog 和本地 Git 状态，用于候选 issue 创建前的人工/自动化复核。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.49..HEAD`。最终带 `--release-tag v0.16.49` 的 strict gate 仍以本草案顶部的 `v0.16.48..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.49 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.49 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.49 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `artifact-manifest.json`。
2. 确认 manifest 的 `validation_commands` 包含 `python scripts/release_readiness.py --target-version 0.16.49 --json`。
3. 确认每周维护者循环文档说明 manifest validation commands 包含 release readiness。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.49`。
6. 将 `pyproject.toml` 版本更新为 `0.16.49`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.49`。
8. 创建 tag：`git tag v0.16.49`。

## Release Notes 摘要

`v0.16.49` adds the release readiness command to generated `artifact-manifest.json` validation commands, helping maintainers re-check release gates before creating candidate issues.
