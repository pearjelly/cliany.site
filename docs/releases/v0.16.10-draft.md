# v0.16.10 发布草案

**目标版本：** `0.16.10`
**提交范围：** `v0.16.9..HEAD`
**当前阻塞：** 需要提交 next iteration plan 的 publication handoff 命令、维护者循环入口和测试；最新本地 release 仍需维护者确认后 push 到远端。

## 用户价值

`v0.16.9` 让 `scripts/check_release_publication.py` 输出 `publish_commands`，维护者可以从 publication audit 里复制 branch/tag push 和远端复核命令。但周初选择下一版工作时，维护者首先看的通常是 `scripts/plan_next_iteration.py --json` 或 Markdown report；如果计划器只说“先发布最新本地 release”，仍然需要再跑一次 publication audit 才能拿到具体命令。

`v0.16.10` 把 publication audit 的 `publish_commands` 接入 next iteration plan。计划 JSON 新增 `publication_publish_commands`，Markdown report 新增 `Publication Publish Commands` 小节，在推荐切片旁直接显示 `git push origin master`、`git push origin v0.16.9` 和 `python scripts/check_release_publication.py --remote --json`。脚本仍然只读取本地证据，不会自动执行 push。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.10 --json` 输出 `publication_publish_commands`；`python scripts/plan_next_iteration.py --target-version 0.16.10 --report /tmp/cliany-next-iteration.md` 的报告包含 `Publication Publish Commands` 命令块。

## 变更分组

### Plan publication handoff

- `scripts/plan_next_iteration.py` 的计划模型新增 `publication_publish_commands`。
- JSON 输出会复用 publication audit 的 `publish_commands`。
- Markdown report 新增 `Publication Publish Commands` 小节。
- 默认行为仍不访问实时远端、不执行 `git push`，只把命令作为维护者交接信息输出。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明周初计划会带上 publication publish commands。
- `tests/test_plan_next_iteration.py` 固定 JSON 和 Markdown report 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本聚焦 release planning handoff，不访问第三方业务站点，不生成 adapter 包，也不会修改案例 manifest 或离线样例。

## 风险与兼容性

- `publication_publish_commands` 是新增字段；已有计划 JSON 字段和 Markdown 小节保持兼容。
- 命令来自 `scripts/check_release_publication.py` 的本地 publication audit，不新增远端网络访问。
- 脚本不会执行 `git push`；维护者仍需显式确认并手动执行命令。
- 如果 release 已公开，Markdown report 会显示无需发布命令，避免制造额外操作。

## 发版前验证

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.10 --json
python scripts/plan_next_iteration.py --target-version 0.16.10 --report /tmp/cliany-next-iteration.md
python scripts/release_readiness.py --target-version 0.16.10 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.10 --json` 输出 `publication_publish_commands`。
2. 确认 Markdown report 中的 `Publication Publish Commands` 包含 branch push、tag push 和远端复核命令。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.10`。
5. 将 `pyproject.toml` 版本更新为 `0.16.10`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.10`。
7. 创建 tag：`git tag v0.16.10`。

## Release Notes 摘要

`v0.16.10` surfaces release publication handoff commands in the next-iteration planner. Maintainers can now see the exact branch/tag push and remote verification commands from the planning artifact without running a separate publication audit first.
