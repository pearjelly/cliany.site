# v0.16.265 发布草案

**目标版本：** `0.16.265`
**提交范围：** `v0.16.264..HEAD`
**当前状态：** 2026-07-17 release slice；发布者可将已完成适配器归档的摘要交接给 release asset 安装说明。

## 用户价值

`cliany-site market publish --json` 现在为实际写入的 completed archive 返回 `package_sha256`。发布者可以将该 64 位十六进制 SHA-256 与 release asset 的 HTTPS 地址一起提供，使用户通过 `market install <source> --sha256 <package_sha256>` 进行独立、可脚本化的完整性校验。

摘要绑定的是 `market publish` 本次完成的归档，不是预计算的输入文件列表或未写入的候选路径。发行说明只交接通用 release-asset 安装步骤；本草案不声明存在任何具体的 adapter asset。

## 变更范围

- `market publish --json` 在成功完成 archive 后输出 `package_sha256`，值为小写 64 字符十六进制 SHA-256。
- 发布者可把 completed archive 的摘要携带到 release asset 的 `market install --sha256` 指令；安装端继续要求显式 HTTPS 来源和固定摘要。
- README、README.zh、adapter lifecycle guide 与官网维护者入口说明通用的摘要交接流程，不发现、命名或承诺未发布的资产。
- target-mode release readiness 继续以 `[Unreleased]`、上一版 tag 与目标 draft 的同步状态作为 pre-tag gate。

## 风险与兼容性

- `package_sha256` 仅在 `market publish` 成功完成归档时出现；失败发布不会伪造摘要或发布资产。
- 摘要不是签名或可信注册表；发布者仍须通过独立可信渠道发布 release asset URL 和 `--sha256` 值。
- 本版不改变本地安装、远程 HTTPS 下载、`--dry-run`、`--force`、备份或 `INSTALL_FAILED` 语义。
- `pypi-project-search` 仍是 candidate，并被 `E_LLM_UNAVAILABLE` 阻塞；不得声称已有 PyPI adapter asset、metadata validation 或 online smoke 证据。

## 案例库映射

- 案例入口继续是 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。
- `pypi-project-search` 保持 candidate：`llm_live_preflight_not_ready` 与 `E_LLM_UNAVAILABLE` 仍阻止真实 exploration，不能伪造 adapter package、metadata validation 或 online smoke 证据。
- 当前 known gap 仍为 `search-extraction-gap`；本版不改变案例状态或数量，也不发布未验证的 candidate asset。

## 发版前验证

本版实际 release diff 是 **提交范围：** `v0.16.264..HEAD`。target-mode readiness 是 Changelog finalization 之前的 pre-tag gate：必须在 `[Unreleased]` 仍包含本版条目、compare link 仍为 `v0.16.264...HEAD` 且本地 `v0.16.265` tag 尚不存在时运行。完成 version bump 后，该检查也会查找占位锚点 **提交范围：** `v0.16.265..HEAD`；该锚点不改变实际 release diff。

```bash
git diff --check
uv run --extra dev --frozen pytest tests/test_marketplace.py tests/test_readme_current_features.py tests/test_site_content.py tests/test_release_draft_docs.py -q --no-cov
uv run --extra dev --frozen python scripts/validate_cases.py --strict
CLIANY_QA_OFFLINE=1 uv run --extra dev --frozen pytest tests/ -q
uv build --out-dir /tmp/cliany-site-v0.16.265-dist
uvx twine check /tmp/cliany-site-v0.16.265-dist/*
uv run --extra dev --frozen python scripts/release_readiness.py --strict --target-version 0.16.265 --remote
```

target readiness 通过前不 finalization Changelog、不创建 tag、不推送、不发布、不部署。严格远端 target check 由父发布 gate 验证远端可用性。

## 发版步骤

1. 在 `[Unreleased]` 仍包含本版条目且 compare link 为 `v0.16.264...HEAD` 时，完成 `--target-version 0.16.265 --remote` pre-tag readiness；确认 `market publish --json` 的 `package_sha256` 来自 completed archive，并仅把它交接到通用 release-asset `market install --sha256` 指令。
2. Finalize Changelog：清空 `[Unreleased]`，新增 `## [0.16.265] - 2026-07-17`，并将 compare links 更新为 `v0.16.265...HEAD` 和 `v0.16.264...v0.16.265`。
3. 提交并推送 `master`，等待 CI 成功：

   ```bash
   git add CHANGELOG.md docs/releases/v0.16.265-draft.md
   git commit -m "chore(release): finalize v0.16.265"
   git push origin master
   ```

4. 仅在 finalization commit 已位于 `HEAD` 后创建本地 tag：`git tag v0.16.265`。
5. 推送 tag 前运行：

   ```bash
   uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.265 --remote --remote-name origin
   ```

6. 仅在 release-tag check 通过后运行 `git push origin v0.16.265`。tag workflow 创建 GitHub Release 并发布 `cliany-site==0.16.265` 到 PyPI；GitHub Release/PyPI audit 完成前不声称外部发布已经发生。
7. 在 `site/` 运行 `vercel link --yes --project cliany.site`、`vercel --prod --yes`，再运行 `vercel inspect www.cliany.site --wait --timeout 90s` 检查官网生产 alias。
8. 运行最终审计：

   ```bash
   uv run --extra dev --frozen python scripts/check_release_publication.py --strict --remote --distribution --json
   ```

   该审计核对远端 Git、GitHub Release 和 PyPI 分发，不替代官网 inspect；官网状态以 Vercel inspect 为准。

## Release Notes 摘要

v0.16.265 makes the exact completed adapter archive digest available through `market publish --json`. Publishers can carry `package_sha256` into generic release-asset `market install --sha256` instructions, while candidate workflows remain blocked until real exploration evidence exists.
