# v0.16.238 发布草案

**目标版本：** `0.16.238`
**提交范围：** `v0.16.237..HEAD`
**当前阻塞：** 需要完成 GitHub Actions Node 24 opt-in、release readiness 门禁、聚焦测试、完整测试、release readiness、case validation 和标准发布流程。真实 LLM provider 预检仍返回 `502 Bad Gateway`，因此本版不生成或激活任何 candidate adapter。

## 用户价值

v0.16.237 的发布流程成功更新了 GitHub Release 和 PyPI，但 GitHub Actions 在 release job 中输出了 Node 20 deprecation annotations。项目目标要求每天至少一个可验证版本；发布流水线如果继续依赖即将退役的 JavaScript action runtime，会让后续版本的稳定发布变成隐性风险。

本版本把 `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true` 加到所有 GitHub workflow 顶层环境中，提前切到 Node 24 runtime；同时让 `scripts/release_readiness.py` 校验 CI 和 release workflow 都声明了这个 opt-in。这样发布链路不只是在本次运行里少一个 warning，而是把未来 workflow runtime 迁移变成可验证的发版门禁。

## 变更分组

- CI：`ci.yml`、`release.yml`、`embodied-ci.yml`、`dep-upgrade-verify.yml` 和 `auto-reproduce.yml` 现在都声明 Node 24 action runtime opt-in。
- Release readiness：CI / release workflow 校验新增 `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: "true"` 必备片段。
- Tests：`tests/test_release_readiness.py` 新增缺少 Node 24 opt-in 的阻塞回归，并同步最小 workflow fixture。
- Dependencies hygiene：Dependabot GitHub Actions 注释补充 `actions/download-artifact@v4`，与 release workflow 中的 artifact download action 对齐。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的任何案例状态。
- [cases/README.md](../../cases/README.md) 的 candidate 晋级规则继续适用：没有 adapter package、metadata validation 和 online smoke 三段证据前，不把 candidate 标记为 active。
- 当前真实案例库仍保持 active 4 个、candidate 3 个、known gap 1 个。
- Known gap 保持 1 个：`search-extraction-gap`。
- `pypi-project-search` 仍是首要 candidate；`cliany-site doctor --llm-live --json` 仍返回 `E_LLM_UNAVAILABLE` / `502 Bad Gateway`，因此本版不运行真实 `explore` 生成 candidate adapter，也不把 candidate 标记为 active。

## 风险与兼容性

- 这是 workflow 环境变量和 release gate 加固，不改变 Python runtime、CLI 参数、adapter schema、action runtime、AXTree 语义匹配或 codegen 输出。
- 如果某个 GitHub Actions runner 或 action 对 Node 24 opt-in 有兼容问题，CI / release workflow 会直接暴露失败；当前改动保持 action 版本不变，避免同时升级多个 third-party action。
- Release readiness 会阻塞缺少 Node 24 opt-in 的 CI / release workflow fixture；自定义下游仓库如复制最小 fixture，需要同步该环境变量。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.238..HEAD`。最终带 `--release-tag v0.16.238` 的 strict gate 仍以本草案顶部的 `v0.16.237..HEAD` 作为实际 release diff 范围。

```bash
uv run --extra dev pytest tests/test_release_readiness.py -q
python3 scripts/validate_cases.py --strict
python3 scripts/release_readiness.py --target-version 0.16.238 --json
python3 scripts/plan_next_iteration.py --target-version 0.16.238 --json
uv run --extra dev pytest tests/ -q
git diff --check
```

真实上游预检烟测：

```bash
uv run cliany-site doctor --json --llm-live
```

预期：如果 LLM provider 仍返回 502 等可重试上游故障，输出 `llm_live` warning、`details.error_code=E_LLM_UNAVAILABLE`、`details.status_code` 和 `details.phase=llm_preflight`；candidate promotion 继续等待真实 adapter package、metadata validation 和 online smoke 证据。

正式 tag 前再运行：

```bash
uv run --extra dev pytest tests/ -q
python3 scripts/release_readiness.py --strict --target-version 0.16.238
```

## 发版步骤

1. 确认 workflow Node 24 opt-in、release readiness、tests、CHANGELOG 和本草案已同步。
2. 运行 release readiness、next iteration planning、case validation、聚焦测试、完整测试和真实上游预检烟测。
3. 将 `CHANGELOG.md` 的 `Unreleased` 移入 `0.16.238`，并把 `pyproject.toml` / `uv.lock` 版本号更新为 `0.16.238`。
4. 提交并推送 `master`。
5. 创建并推送 `v0.16.238` tag。
6. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
7. 官网内容未改动；按日常发布流程仍可重新执行 `site/` 的 Vercel production deploy，并记录生产 deployment。
