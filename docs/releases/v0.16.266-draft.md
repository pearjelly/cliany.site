# v0.16.266 发布草案

**目标版本：** `0.16.266`
**提交范围：** `v0.16.265..HEAD`
**当前状态：** 2026-07-20 release slice；首跑入口改为浏览维护案例及其当前验证路径，不声明不存在的 adapter release asset。

## 用户价值

新用户现在可以先安装 CLI、运行 `cliany-site doctor`，再通过 `cliany-site cases` 查看维护中的公开案例。这个路径不要求先配置 LLM key 或 Chrome/CDP，也不要求复制历史归档文件名；当用户准备为自己的站点生成命令时，再进入 `explore`。

`doctor --json` 保留既有的 `demo_adapter_quickstart` 字段以兼容已有自动化，同时新增 `case_catalog_quickstart` 作为当前推荐的首跑入口。human summary、10 分钟指南、官网首页和官网文档页使用同一条案例目录路径，避免入口彼此矛盾。

## 变更范围

- `src/cliany_site/commands/doctor.py` 将推荐下一步指向 `cliany-site cases` 和 `cliany-site cases --json`，并把它们暴露为 `case_catalog_quickstart`。
- `docs/quickstart-10min.md`、README、官网首页和官网文档页明确先浏览维护案例，再配置 LLM 生成自己的命令。
- 首跑页面的移动端命令块会换行且复制按钮不覆盖命令文本。
- `scripts/release_readiness.py` 与相关测试会阻止 `issues.apache.org.cliany-adapter-v0.14.0.tar.gz` 再次进入首跑文档或官网入口。
- `demo_adapter_quickstart` 继续保留在 JSON summary 中；本版不发布、也不暗示存在任何具体 adapter package 或 GitHub Release asset。

## 案例库映射

`cases/README.md` 和 `cases/manifest.json` 仍是公开案例目录及验证状态的唯一来源。首跑页面只引导用户通过 `cliany-site cases` 查看这些维护中的记录，不从历史发行说明推断可安装资产。

当前 `search-extraction-gap` 仍是案例库中的已知缺口；它不会被本版文案掩盖。实时 LLM 预检的 `llm_live_preflight_not_ready` 状态同样继续作为候选案例晋级的 blocker：只有拿到真实 explore、归档、元数据验证和在线 smoke 证据后，案例才可声明可发布 adapter 包。

## 风险与兼容性

- `demo_adapter_quickstart` 的字段形状和内容保持不变，调用方可继续读取它；新字段仅扩展输出。
- `case_catalog_quickstart` 只提供公开案例目录和验证路径，不保证某个案例已经具备可下载或可执行的 adapter 包。
- 2026-07-20 的 `cliany-site doctor --llm-live --json` 实测仍返回可重试的 `E_LLM_UNAVAILABLE`。因此本版不将案例晋级为 PyPI candidate、不编造在线 explore 成功或 adapter 归档。

## 发版前验证

本版实际 release diff 是 **提交范围：** `v0.16.265..HEAD`。target-mode readiness 是 Changelog finalization 之前的 pre-tag gate：必须在 `[Unreleased]` 仍包含本版条目、compare link 仍为 `v0.16.265...HEAD` 且本地 `v0.16.266` tag 尚不存在时运行。完成 version bump 后，该检查也会查找占位锚点 **提交范围：** `v0.16.266..HEAD`；该锚点不改变实际 release diff。

```bash
uv run --extra dev --frozen ruff check src/cliany_site/commands/doctor.py scripts/release_readiness.py tests/test_doctor_v3.py tests/test_quickstart_docs.py tests/test_site_content.py tests/test_release_readiness.py tests/test_release_draft_docs.py
uv run --extra dev --frozen mypy src/cliany_site/commands/doctor.py
CLIANY_QA_OFFLINE=1 uv run --extra dev --frozen pytest tests/ -q --no-cov --tb=short
bash qa/test_site_conversion.sh
uv run --extra dev --frozen python scripts/validate_cases.py --strict
uv build --out-dir /tmp/cliany-site-v0.16.266-dist
uvx twine check /tmp/cliany-site-v0.16.266-dist/*
uv run --extra dev --frozen python scripts/release_readiness.py --strict --target-version 0.16.266 --remote
```

对官网首跑区执行 Playwright 桌面和 390px 移动端检查；移动端必须完整显示 `pip install cliany-site`、`cliany-site doctor` 和 `cliany-site cases`，且复制按钮不能遮挡命令。

## 发版步骤

1. 在 `[Unreleased]` 仍包含本版条目且 compare link 为 `v0.16.265...HEAD` 时，完成 `--target-version 0.16.266 --remote` pre-tag readiness。任何 `E_LLM_UNAVAILABLE` 只作为案例晋级 blocker，不阻止本版首跑文档和 CLI 引导发布。
2. Finalize Changelog：清空 `[Unreleased]`，新增 `## [0.16.266] - 2026-07-20`，并将 compare links 更新为 `v0.16.266...HEAD` 和 `v0.16.265...v0.16.266`。
3. 提交并推送 `master`，等待 CI 成功：

   ```bash
   git add CHANGELOG.md pyproject.toml docs/releases/v0.16.266-draft.md
   git commit -m "chore(release): finalize v0.16.266"
   git push origin master
   ```

4. 仅在 finalization commit 已位于 `HEAD` 后创建本地 tag：`git tag v0.16.266`。
5. 推送 tag 前运行：

   ```bash
   uv run --extra dev --frozen python scripts/release_readiness.py --strict --release-tag v0.16.266 --remote --remote-name origin
   ```

6. 仅在 release-tag check 通过后运行 `git push origin v0.16.266`。tag workflow 创建 GitHub Release 并发布 `cliany-site==0.16.266` 到 PyPI；GitHub Release/PyPI audit 完成前不声称外部发布已经发生。
7. 在 `site/` 运行 `vercel link --yes --project cliany.site`、`vercel --prod --yes`，再运行 `vercel inspect www.cliany.site --wait --timeout 90s` 检查官网生产 alias。
8. 运行最终审计：

   ```bash
   uv run --extra dev --frozen python scripts/check_release_publication.py --strict --remote --distribution --json
   ```

## Release Notes 摘要

v0.16.266 makes the first-success path honest and usable without an LLM: install the CLI, read `doctor`, and browse maintained cases. The same path now appears in CLI guidance, docs, and the official site, while historical version-pinned archive names are blocked from returning to onboarding surfaces.
