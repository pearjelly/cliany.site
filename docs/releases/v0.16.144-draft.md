# v0.16.144 发布草案

**目标版本：** `0.16.144`
**提交范围：** `v0.16.143..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把第一条 release draft 阻塞原因补进 artifact bundle summary。

## 用户价值

`v0.16.143` 已经让 `artifact_bundle_summary` 能直接展示第一条 release draft 修复动作。维护者和自动化只读整包摘要时，能看到“下一步做什么”，但仍缺少对应的“为什么”：第一条 release draft issue 本身还需要展开 `release-draft-handoff.json` 才能看到。

`v0.16.144` 在 `artifact_bundle_summary` 中新增 `release_draft_primary_issue`，并在 artifacts `README.md` 的 `Artifact Bundle Summary` 段落展示同源字段。这样工具只读 manifest summary 或 README summary，就能同时展示第一条 release draft 阻塞原因和第一条修复动作。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.144 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `release_draft_primary_issue`，artifacts README 的 `Artifact Bundle Summary` 展示同一字段，且该值与 `release-draft-handoff.json.release_draft_primary_issue` 同源；无 release draft issue 时字段为 `null`。

## 变更分组

### Artifact bundle release draft primary issue

- `artifact_bundle_summary` 新增 `release_draft_primary_issue`。
- 有 release draft issue 时，该字段取第一条 issue。
- 无 release draft issue 时，该字段为 `null`；README summary 用 `(none)` 展示。
- Artifacts `README.md` 的 `Artifact Bundle Summary` 展示第一条 release draft 阻塞原因。
- 现有 `release_draft_primary_required_action`、`release_draft_required_action_count`、`release_draft_required_actions_sha256`、`release_draft_handoff_key_count` 和 `Release Draft Handoff` 保持兼容。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `artifact_bundle_summary` 会展示第一条 release draft 阻塞原因。
- `tests/test_plan_next_iteration.py` 覆盖 manifest summary 与 README summary 的新字段。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善 candidate issue artifacts 整包摘要的 release draft primary issue 可见性。

## 风险与兼容性

- 新字段只派生自既有 `release_draft_issues`，不新增网络访问或外部状态读取。
- 无 release draft issue 时字段为 `null`，方便机器读取。
- Primary issue 只用于快速展示首要原因，不替代完整 `release-draft-handoff.json` 审阅。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.144 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `release_draft_primary_issue`。
3. 确认 artifacts README 的 `Artifact Bundle Summary` 展示 `release_draft_primary_issue`。
4. 确认有 release draft issue 时，该字段等于 `release-draft-handoff.json.release_draft_primary_issue`。
5. 确认无 release draft issue 时，JSON 字段为 `null` 且 README 展示 `(none)`。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.144`。
8. 将 `pyproject.toml` 版本更新为 `0.16.144`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.144`。
10. 创建 tag：`git tag v0.16.144`。

## Release Notes 摘要

`v0.16.144` adds the primary release draft issue to artifact bundle summaries, helping tools show the first draft blocker and first draft fix without expanding the full handoff.
