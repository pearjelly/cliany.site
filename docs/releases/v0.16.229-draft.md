# v0.16.229 发布草案

**目标版本：** `0.16.229`
**提交范围：** `v0.16.228..HEAD`
**当前阻塞：** 需要完成本草案对应的文档/案例方向改动，并重新运行 release readiness、case validation 和针对性文档测试。

## 用户价值

`v0.16.228` 已把 tag 触发的 distribution 发布链路修到可进入 GitHub Release 和 PyPI 发布阶段。下一步需要把维护节奏从“每周至少一个版本”升级为“每天至少一个可验证版本”，让外部用户看到项目在真实案例、贡献者入口、adapter 生命周期和发布门禁上持续前进，而不是只看到零散的大批量维护提交。

`v0.16.229` 聚焦两件事：第一，固化每日 release train 的执行顺序，明确每个版本都要有用户价值、验证命令、风险和标准发布流程；第二，把下一轮产品切片继续指向真实案例库，优先推进 `pypi-project-search`、`npm-package-search` 或 `crates-io-crate-search` candidate 的 adapter package、metadata validation 与 online smoke 证据。

成功标准是：路线图、release cadence、README/README.zh 对每日发布目标表述一致；`scripts/plan_next_iteration.py --json` 能继续把 `0.16.229` 指向真实案例库或明确的发版 gate；`scripts/release_readiness.py --target-version 0.16.229 --json` 不再因为缺失 release draft 阻塞。

## 变更分组

### 每日发布节奏

- `docs/release-cadence.md` 明确每日发布循环：读计划器、选当天可验证切片、写 release draft、运行 strict gate、更新 changelog/version/docs、推 tag、等待 GitHub Release/PyPI，必要时部署官网。
- `docs/roadmap-2026-q3.md` 将北极星发布节奏更新为每天至少一个可验证版本，并说明文档、案例证据、发版门禁和官网同步也可以作为 patch 切片。
- README 和 README.zh 的路线图入口同步每日发布目标，避免新维护者继续按旧的 weekly-only 节奏理解项目。

### 真实案例库下一步

- 下一轮功能切片优先选择一个 candidate case，补齐 `adapter_package`、`metadata_validation`、`online_smoke` 三段证据中的第一段。
- 当前计划器推荐的首要任务是 `pypi-project-search` 的 `adapter_package`，后续命令来自 `cliany-site cases --case-id pypi-project-search --evidence-bundle --json`。
- 本版本不提前把 candidate 标记为 active；只有三段证据都完整后才改变案例状态。

## 案例库映射

- `cases/README.md` 继续作为真实案例库的人类入口，负责说明 active、candidate 和 known-gap 的维护状态。
- `cases/manifest.json` 继续作为 release readiness、`cliany-site cases` 和 `scripts/validate_cases.py` 的机器入口。
- Active 案例保持 4 个：SuiteCRM、Apache Jira、Apache Confluence、Apache Jenkins。
- Candidate 案例保持 3 个：PyPI、npm、crates.io 搜索工作流。
- Known gap 保持 1 个：`search-extraction-gap`，继续作为搜索抽取质量缺口的回归入口。
- 本版本改动不访问第三方站点，不生成 adapter package，不向仓库写入 `~/.cliany-site/` 运行时状态。

## 风险与兼容性

- 风险：每日发布容易把低价值变更包装成版本；本草案要求每个 patch 至少说明用户价值、验证方式和风险，避免空转。
- 风险：每日发布不能绕过 release readiness；如果 `standard_release_flow_status=blocked`，必须先关闭 blocker。
- 兼容性：不修改 CLI、JSON envelope、adapter schema、runtime、case manifest 状态或 release workflow。
- 兼容性：每周至少三天提交记录仍保留，`check_release_cadence.py` 继续作为健康度门禁。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.229..HEAD`。最终带 `--release-tag v0.16.229` 的 strict gate 仍以本草案顶部的 `v0.16.228..HEAD` 作为实际 release diff 范围。

```bash
python3 scripts/release_readiness.py --target-version 0.16.229 --json
python3 scripts/plan_next_iteration.py --target-version 0.16.229 --json
python3 scripts/validate_cases.py --strict
uv run --extra dev pytest tests/test_release_cadence_docs.py tests/test_readme_current_features.py tests/test_site_content.py -q
```

正式 tag 前再运行：

```bash
uv run --extra dev pytest tests/ -q
python3 scripts/release_readiness.py --strict --target-version 0.16.229
```

## 发版步骤

1. 确认每日发布节奏文档、README/README.zh、CHANGELOG 和本草案已同步。
2. 运行 release readiness、next iteration plan、case validation 和针对性文档测试。
3. 将 `CHANGELOG.md` 的 `Unreleased` 移入 `0.16.229`，并把 `pyproject.toml` 版本号更新为 `0.16.229`。
4. 提交并推送 `master`。
5. 创建并推送 `v0.16.229` tag。
6. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
7. 若官网文案受影响，进入 `site/` 运行 `vercel link --yes --project cliany.site` 和 `vercel --prod --yes`，并在 release notes 中记录官网同步。
