# v0.16.50 发布草案

**目标版本：** `0.16.50`
**提交范围：** `v0.16.49..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts manifest 直接展示候选任务审阅清单。

## 用户价值

`v0.16.49` 已经让 `artifact-manifest.json` 的 `validation_commands` 覆盖 release readiness。维护者可以从一个机器可读入口找到 artifacts 复现、publication audit、release gate 和 case validation 命令。但真正决定是否创建候选 issue 时，还需要人工阅读 README 的 `Review Checklist`，工具不能稳定读取这些放行动作。

`v0.16.50` 让 `artifact-manifest.json` 直接包含 `review_checklist`，并让 README 的 `Review Checklist` 与这个字段同源。这样维护者和自动化可以从同一个 JSON 判断：是否已发布最新本地 release、release draft issues 是否处理、publication next actions 是否处理、candidate metadata 和 body 是否完成审阅。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.50 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `artifact-manifest.json` 包含 `review_checklist`；artifacts `README.md` 的 `Review Checklist` 与 manifest 清单同源；每周维护者循环文档说明这个字段。

## 变更分组

### Candidate artifacts manifest review checklist

- `artifact-manifest.json` 新增 `review_checklist`。
- Artifacts `README.md` 的 `Review Checklist` 复用同一份清单。
- Artifacts manifest 继续保留 schema version、review order、validation commands、publication handoff 和 release draft handoff。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 manifest 会包含 `review_checklist`。
- `tests/test_plan_next_iteration.py` 覆盖 manifest checklist 和 README 文件列表。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强候选 issue artifacts manifest 的审阅清单。

## 风险与兼容性

- `review_checklist` 是 `artifact-manifest.json` 的新增字段，不改变 plan JSON schema。
- 既有 `README.md` 审阅清单语义保持一致，只改为与 JSON 字段同源渲染。
- 不自动执行命令，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- 清单只描述创建 candidate issue 前的人工/自动化审阅项，不改变 candidate promotion 状态。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.50..HEAD`。最终带 `--release-tag v0.16.50` 的 strict gate 仍以本草案顶部的 `v0.16.49..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.50 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.50 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.50 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `artifact-manifest.json`。
2. 确认 manifest 包含 `review_checklist`。
3. 确认 artifacts `README.md` 的 `Review Checklist` 与 manifest 清单同源。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.50`。
6. 将 `pyproject.toml` 版本更新为 `0.16.50`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.50`。
8. 创建 tag：`git tag v0.16.50`。

## Release Notes 摘要

`v0.16.50` adds a review checklist to generated `artifact-manifest.json` files, making candidate issue artifact review steps machine-readable and aligned with the README checklist.
