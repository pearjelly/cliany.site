# v0.16.28 发布草案

**目标版本：** `0.16.28`
**提交范围：** `v0.16.27..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue metadata 的 body 文件定位更稳定。

## 用户价值

`scripts/plan_next_iteration.py --issues-dir` 会生成 `issue-metadata.json`，供维护者或后续脚本批量审阅 candidate issue。此前 metadata 只有 `issue_body_file`，通常是带临时目录的完整路径；如果下游只需要在 artifacts 目录里定位 `pypi-project-search.md`，就必须解析路径或重新拼接 case id。

`v0.16.28` 为每个 candidate metadata 增加 `issue_body_name`，保留稳定的 body file name，同时继续保留 `issue_body_file` 作为 body file path。这样人工审阅、README 摘要和脚本消费可以直接使用文件名，不需要依赖绝对路径格式。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.28 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `issue-metadata.json` 中，每个 candidate 同时包含 `issue_body_name` 和 `issue_body_file`，其中 `pypi-project-search` 的 `issue_body_name` 为 `pypi-project-search.md`。

## 变更分组

### Candidate issue metadata

- `scripts/plan_next_iteration.py --issues-dir` 写出的 `issue-metadata.json` 为每个 candidate 增加 `issue_body_name`。
- `issue_body_file` 保持兼容，继续指向完整 body file path。
- artifacts `README.md` 的文件说明同步标注 body file name 和 body file path。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 `issue-metadata.json` 同时保留 body file name 和 body file path。
- `tests/test_plan_next_iteration.py` 固定 metadata 字段和 README 描述。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只增强 candidate issue artifacts 的机器可读 metadata。

## 风险与兼容性

- 这是 additive JSON schema 增强，不移除或重命名现有字段。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.28 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `issue-metadata.json` 包含 `issue_body_name`。
2. 确认 `pypi-project-search` 的 `issue_body_name` 为 `pypi-project-search.md`。
3. 确认 `issue_body_file` 仍保留完整 body file path，旧脚本不用迁移即可继续运行。
4. 运行计划脚本、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.28`。
6. 将 `pyproject.toml` 版本更新为 `0.16.28`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.28`。
8. 创建 tag：`git tag v0.16.28`。

## Release Notes 摘要

`v0.16.28` adds `issue_body_name` to generated candidate issue metadata while keeping `issue_body_file`, making issue artifacts easier to consume without parsing absolute paths.
