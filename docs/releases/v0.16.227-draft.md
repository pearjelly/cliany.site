# v0.16.227 发布草案

**目标版本：** `0.16.227`
**提交范围：** `v0.16.226..HEAD`
**当前阻塞：** 最新本地 release `v0.16.226` 仍需维护者确认后 push 到远端；本版本先补齐 `v0.16.227` 草案，让下一轮计划不再被 `release draft is missing` 干扰。

## 用户价值

`v0.16.226` 已经把本地 release publication handoff 草案补齐，并完成本地 tag。当前计划里真正排在最前面的待办仍是发布可见性：`publication_visibility.status` 仍是 `local_only`，`candidate_issue_gate.status` 仍是 `blocked_by_publication`，因此不应该创建 candidate issue 或扩大下一版范围。

`v0.16.227` 继续把这个状态写入 release draft：本地已完成 `v0.16.226`，但远端还看不到分支和 tag。维护者读取 `plan_next_iteration.py`、`publication-handoff.json` 或 `release-draft-handoff.json` 时，可以直接判断当前不是缺少草案，而是需要审阅并执行发布同步。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.227 --issues-dir /tmp/cliany-candidate-issues --json` 的 `release_draft_issues` 不再报告 `release draft is missing`；`next_actions` 继续把 `latest local release is not published` 对应的发布同步动作放在首位；当 `tag_points_at_head=false` 时，顶层动作必须明确提示先推 `master`，再选择回到当前 tag commit 或在 HEAD 创建新 release tag，而不是误写成直接推 tag；`publication_next_actions` 和 `publication_publish_commands` 继续明确展示 `git push origin master` 和 `python scripts/check_release_publication.py --remote --json`。

## 变更分组

### Release draft handoff cleanup

- 新增 `docs/releases/v0.16.227-draft.md`，把当前 `v0.16.226` 本地 release 未公开的状态固化到下一版草案。
- 草案明确引用 `publication_visibility`、`publication_next_actions`、`publication_publish_commands`、`candidate_issue_gate`、`publication-handoff.json` 和 `release-draft-handoff.json`。
- 草案说明 `blocked_by_publication` 是当前 candidate issue gate 的预期状态：`v0.16.226` 本地 tag 尚未公开前，不创建真实 GitHub issue。

### Publication action precision

- `plan_next_iteration.py` 现在复用 `check_release_publication.py` 的具体 `publication_next_actions` 作为顶层发布动作。
- 当最新 tag 不指向 HEAD 时，`publication_visibility.summary` 明确写出 `tag_points_at_head=false` 对应的人工选择：回到 tag commit 发布，或在 HEAD 创建新的 release tag。
- 本版本不再把这种状态压缩成 “push branch and tag” 的泛化提示，避免维护者误推过期 tag。
- `check_release_publication.py --publish-script` 生成的脚本在这种状态下写入 `Publication review notes`，明确 `This script will not push that tag automatically`。
- `check_release_publication.py --json` / `--report` 输出 `tag_publish_decision`，其中 `status=manual_decision_required` 和 `can_push_tag=false` 让工具不用解析自然语言也能识别 tag mismatch。
- `plan_next_iteration.py` 把同一判断透传为 `publication_tag_publish_decision`，并写入 Markdown report、`artifact-manifest.json`、`publication-handoff.json` 和 artifacts `README.md` 的 `Publication Handoff`。
- `artifact_bundle_summary` 同步输出 `publication_tag_publish_decision_status`、`publication_tag_can_push`、`publication_tag_required_action_sha256` 和 key/hash 摘要窗口，让只读整包摘要的工具也能识别 tag 发布决策。
- `candidate_issue_gate.evidence` 同步输出 `publication_tag_decision_status`、`publication_tag_can_push` 和 `publication_tag_required_action`，artifacts `README.md` 也展示 `gate_evidence_tag_decision`，让只读取 gate 的工具也能解释 publication block。
- `candidate_issue_gate` 本体同步输出 `primary_reason_code`、`primary_reason_description` 和 `primary_required_action`，artifacts `README.md` 的 `Publication Handoff` 也展示 `gate_primary_reason_code`、`gate_primary_reason_description` 和 `gate_primary_required_action`。
- `publication-handoff.json` 额外输出 `candidate_issue_gate_primary_reason_code`、`candidate_issue_gate_primary_reason_description` 和 `candidate_issue_gate_primary_required_action` 顶层别名，方便只读取 handoff 的工具不用展开 nested gate。
- `artifact_bundle_summary` 同步输出 `publication_handoff_candidate_issue_gate_primary_reason_code`、`publication_handoff_candidate_issue_gate_primary_reason_description` 和 `publication_handoff_candidate_issue_gate_primary_required_action`，让只读整包摘要的工具也能审计 handoff 顶层别名。

### 文档与门禁

- `tests/test_release_draft_docs.py` 固定本草案的必需章节和发布可见性字段。
- `plan_next_iteration.py` 现在输出结构化 `commit_cadence`，并在 Markdown report、`artifact-manifest.json` 和 `artifact_bundle_summary` 中展示 `commit_cadence_status`、`commit_cadence_missing_commit_days`、`commit_cadence_commit_days_sha256` 和 `commit_cadence_next_actions_sha256`，让维护者和自动化能直接跟踪每周提交天数缺口。
- `publication-handoff.json` 和 artifacts `README.md` 的 `Publication Handoff` 也展示 `commit_cadence_status`、`commit_cadence_missing_commit_days` 和 `commit_cadence_primary_next_action`，让维护者先审阅发布同步时也能看到本周提交天数阻塞。
- 本版本不修改案例库状态，不修改 adapter 生成逻辑，不创建真实 GitHub issue，不推送 `master` 或 tag。
- `release_draft_issues` 清空后，下一轮计划应该只剩发布同步、周提交天数和候选晋级证据这些真实待办。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

Candidate promotion 的首要任务仍是 `pypi-project-search` 的 `adapter_package` evidence。发布可见性完成前，本版本不把 candidate issue artifacts 变成真实 GitHub issue，也不把任何 candidate 标记为 active。

## 风险与兼容性

- 这是发布流程计划输出、文档与测试锚点更新，不改变 CLI adapter 运行时和案例库状态。
- `publication_visibility` 仍以 `check_release_publication.py` 的真实 Git 状态为准。
- `candidate_issue_gate` 仍应在 `v0.16.226` 未发布时保持 `blocked_by_publication`。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

完成 `v0.16.227` 后，下一轮目标版本自检会检查这段锚点：**提交范围：** `v0.16.227..HEAD`。当前带 `--release-tag v0.16.227` 的 strict gate 仍以本草案顶部的 `v0.16.226..HEAD` 作为实际 release diff 范围。

## 发版步骤

1. 确认 `tests/test_release_draft_docs.py` 和 ruff 通过。
2. 确认 `plan_next_iteration.py --target-version 0.16.227 --issues-dir` 不再报告 `release draft is missing`，且在 `tag_points_at_head=false` 时不会提示直接推旧 tag。
3. 确认 `publication_tag_publish_decision.status`、`artifact_bundle_summary.publication_tag_publish_decision_status`、`candidate_issue_gate.evidence.publication_tag_decision_status` 与 `publication-handoff.json` 中的 `tag_publish_decision.status` 一致，且 `candidate_issue_gate.status` 在 `v0.16.226` 未发布前仍是 `blocked_by_publication`。
4. 确认 `candidate_issue_gate.primary_reason_code`、`candidate_issue_gate.primary_reason_description`、`candidate_issue_gate.primary_required_action` 与 `publication-handoff.json` 的 `candidate_issue_gate_primary_*` 顶层别名、`artifact_bundle_summary.publication_handoff_candidate_issue_gate_primary_*`、artifacts `README.md` 的 `gate_primary_*` 展示一致。
5. 确认 `commit_cadence.status`、`commit_cadence.missing_commit_days` 与 `publication-handoff.json` 的 `commit_cadence_*`、`artifact_bundle_summary.commit_cadence_*` 一致，且当前缺口仍要求本周继续提交到至少 3 个独立日期。
6. 维护者批准后执行 `git push origin master`。
7. 若维护者确认继续发布 `v0.16.226` tag，则回到该 tag commit 后执行 `git push origin v0.16.226`；若选择发布当前 HEAD，则先创建新的 release tag。
8. 执行 `python scripts/check_release_publication.py --remote --json` 复核远端分支和 tag。
9. 准备最终发布时，确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.227`，并创建 tag：`git tag v0.16.227`。
