# v0.16.45 发布草案

**目标版本：** `0.16.45`
**提交范围：** `v0.16.44..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 candidate issue artifacts manifest 直接展示 release draft handoff。

## 用户价值

`v0.16.44` 已经让 `artifact-manifest.json` 展示 publication ref、worktree 和 publish-script 上下文。维护者或工具可以先读 manifest 判断 release publication 是否阻塞候选 issue 创建，但如果要判断下一版 release draft 是否已准备好，仍然需要继续读取 `release-draft-handoff.json`。

`v0.16.45` 让 `artifact-manifest.json` 直接包含 `release_draft_path` 和 `release_draft_issues`。这样候选 issue artifacts 的机器可读入口可以同时表达 publication 状态、候选案例范围、阻塞项、下一步和 release draft 缺口。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.45 --issues-dir /tmp/cliany-candidate-issues --json` 写出的 `artifact-manifest.json` 包含 release draft handoff 字段；artifacts `README.md` 的 Files 小节说明 manifest 会包含 release draft handoff。

## 变更分组

### Candidate artifacts manifest release draft handoff

- `artifact-manifest.json` 新增 `release_draft_path`。
- `artifact-manifest.json` 新增 `release_draft_issues`。
- Artifacts `README.md` 的 Files 小节说明 manifest 包含 release draft handoff。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 manifest 会复用 `release-draft-handoff.json` 的 release draft path 和 release draft issues。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 内容和 README 文件列表。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只增强候选 issue artifacts manifest 的 release draft handoff 摘要。

## 风险与兼容性

- `release_draft_path` 和 `release_draft_issues` 是 `artifact-manifest.json` 的新增字段，不改变 plan JSON schema。
- 既有 publication、candidate、files、review order 和 validation commands 字段保持兼容。
- 不自动执行 push，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。
- Release draft 字段只复用当前 plan 的既有字段和 release readiness 结果。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.45..HEAD`。最终带 `--release-tag v0.16.45` 的 strict gate 仍以本草案顶部的 `v0.16.44..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.45 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.45 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.45 --issues-dir /tmp/cliany-candidate-issues --json` 写出 `artifact-manifest.json`。
2. 确认 manifest 包含 `release_draft_path` 和 `release_draft_issues`。
3. 确认 artifacts `README.md` 的 Files 小节说明 manifest 包含 release draft handoff。
4. 运行计划器、维护者循环文档和 release draft 相关测试。
5. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.45`。
6. 将 `pyproject.toml` 版本更新为 `0.16.45`。
7. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.45`。
8. 创建 tag：`git tag v0.16.45`。

## Release Notes 摘要

`v0.16.45` adds release draft handoff fields to generated `artifact-manifest.json` files, making candidate issue artifact bundles easier to inspect from one machine-readable entrypoint.
