# v0.16.95 发布草案

**目标版本：** `0.16.95`
**提交范围：** `v0.16.94..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 publication visibility key count 补进 candidate issue artifacts 的整包摘要。

## 用户价值

`v0.16.94` 已经让 `artifact_bundle_summary` 能展示 publication ref context 的 key count。维护者和自动化继续审阅发布可见性时，还需要知道 `publication_visibility` 的结构是否漂移；目前摘要已有 `publication_visibility_sha256` 和 `publication_visibility_summary_sha256`，但没有 key count。

`v0.16.95` 在 `artifact_bundle_summary` 中新增 `publication_visibility_key_count`。Artifacts `README.md` 的 `Artifact Bundle Summary` 也展示该字段，让工具只读整包摘要就能判断 publication visibility 的结构规模，再决定是否展开完整 visibility 对象。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.95 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_visibility_key_count`，且该值等于 manifest 中 `publication_visibility` 的 key 数；README 的 `Artifact Bundle Summary` 也展示该字段。

## 变更分组

### Artifact bundle publication visibility key count

- `artifact_bundle_summary` 新增 `publication_visibility_key_count`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 展示 publication visibility key count。
- 现有 `publication_visibility`、`publication_visibility_sha256` 和 `publication_visibility_summary_sha256` 保持不变。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 artifact bundle summary 会携带 publication visibility key count。
- `tests/test_plan_next_iteration.py` 覆盖 manifest 和 README。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让候选 issue artifacts 的整包摘要更容易判断 publication visibility 结构是否变化。

## 风险与兼容性

- 新字段只出现在 `artifact_bundle_summary`，不改变 `publication_visibility` 的既有字段语义。
- 数量从同一组 publication visibility key 计算，不新增网络访问。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.95..HEAD`。最终带 `--release-tag v0.16.95` 的 strict gate 仍以本草案顶部的 `v0.16.94..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov

python scripts/plan_next_iteration.py --target-version 0.16.95 --json
python scripts/plan_next_iteration.py --target-version 0.16.95 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.95 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.95 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 `artifact-manifest.json.artifact_bundle_summary` 包含 `publication_visibility_key_count`。
3. 确认该值等于 `artifact-manifest.json.publication_visibility` 的 key 数。
4. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 展示该字段。
5. 运行计划器、维护者循环文档和 release draft 相关测试。
6. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.95`。
7. 将 `pyproject.toml` 版本更新为 `0.16.95`。
8. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.95`。
9. 创建 tag：`git tag v0.16.95`。

## Release Notes 摘要

`v0.16.95` adds publication visibility key counts to candidate issue artifact bundle summaries, helping tools detect release visibility structure drift before expanding the full visibility object.
