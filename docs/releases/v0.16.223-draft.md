# v0.16.223 发布草案

**目标版本：** `0.16.223`
**提交范围：** `v0.16.222..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让 `artifact_bundle_summary` 也能审计 candidate 晋级证据摘要。

## 用户价值

`v0.16.222` 已经让 candidate issue artifacts 的 `artifact-manifest.json` 和 artifacts `README.md` 直接展示 `case_promotion_evidence_summary`。这解决了“完整 artifacts 包可读”的问题，但只读 `artifact_bundle_summary` 的维护工具仍然只能知道 candidate cases 是否变化，不能快速判断 candidate 晋级证据摘要本身是否漂移。

`v0.16.223` 把 candidate 晋级证据摘要接入 bundle-level summary。工具只读取 `artifact_bundle_summary` 时，也可以看到 `case_promotion_evidence_summary_key_count`、`case_promotion_evidence_summary_keys_sha256`、key preview/tail/boundary、`case_promotion_evidence_summary_sha256`、candidate/task/pending/blocked/complete 计数和 `case_promotion_evidence_primary_next_action`。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.223 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 `artifact-manifest.json.artifact_bundle_summary` 包含这些 `case_promotion_evidence_*` 字段；artifacts `README.md` 的 `Artifact Bundle Summary` 展示同源字段；当前真实案例库仍显示 3 个 candidate、9 个 pending 子任务，并把 PyPI adapter package 作为 primary next action。

## 变更分组

### Bundle-level promotion evidence summary

- `ARTIFACT_BUNDLE_SUMMARY_KEYS` 新增 `case_promotion_evidence_summary_key_count`、`case_promotion_evidence_summary_keys_sha256`、`case_promotion_evidence_summary_key_boundary_sha256`、`case_promotion_evidence_summary_key_preview`、`case_promotion_evidence_summary_key_tail` 和 `case_promotion_evidence_summary_sha256`。
- `artifact_bundle_summary` 新增 `case_promotion_evidence_candidate_count`、`case_promotion_evidence_task_count`、`case_promotion_evidence_pending_count`、`case_promotion_evidence_blocked_count`、`case_promotion_evidence_complete_count` 和 `case_promotion_evidence_primary_next_action`。
- Candidate issue artifacts `README.md` 的 `Artifact Bundle Summary` 输出这些字段，供人工审阅和机器读取保持一致。
- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明只读整包摘要的维护工具如何用这些字段判断 candidate 晋级证据规模、hash 和首要动作。

### 文档与门禁

- `tests/test_plan_next_iteration.py` 固定 manifest summary 字段、hash 和 README 渲染。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护循环文档中的字段说明。
- `tests/test_release_draft_docs.py` 固定本草案。
- 本版本不改变 `artifact-manifest.json.case_promotion_evidence_summary` 的完整结构，也不改变 candidate issue body 内容。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只让 candidate issue artifacts 的整包摘要能更快暴露 candidate 晋级证据状态。

## 风险与兼容性

- 这是 `artifact_bundle_summary` 的新增字段，不移除既有字段。
- 完整任务明细仍只在 `case_promotion_evidence_summary` 顶层字段中输出；bundle summary 只保留 hash、字段窗口、计数和首要动作。
- Summary 只汇总现有 manifest 状态，不会伪造 adapter package、metadata validation 或 online smoke 已完成。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.223..HEAD`。最终带 `--release-tag v0.16.223` 的 strict gate 仍以本草案顶部的 `v0.16.222..HEAD` 作为实际 release diff 范围。

```bash
pytest tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py -q --no-cov
ruff check scripts/plan_next_iteration.py tests/test_plan_next_iteration.py tests/test_weekly_maintainer_loop_docs.py tests/test_release_draft_docs.py
python scripts/plan_next_iteration.py --target-version 0.16.223 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/plan_next_iteration.py --target-version 0.16.223 --report /tmp/cliany-next-iteration.md
python scripts/release_readiness.py --target-version 0.16.223 --json
python scripts/release_readiness.py --strict --target-version 0.16.223 --report /tmp/cliany-release-readiness.md
```

## 发版步骤

1. 确认 focused pytest 和 ruff 通过。
2. 确认 `plan_next_iteration.py --issues-dir` 生成的 `artifact_bundle_summary` 包含 `case_promotion_evidence_summary_sha256`。
3. 确认 artifacts `README.md` 的 `Artifact Bundle Summary` 包含 `case_promotion_evidence_primary_next_action`。
4. 确认 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.223`。
5. 确认 `pyproject.toml` 版本更新为 `0.16.223`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.223`。
7. 创建 tag：`git tag v0.16.223`。

## Release Notes 摘要

`v0.16.223` adds bundle-level fingerprints and counts for candidate promotion evidence summaries, so maintenance tools can detect evidence drift and show the primary next action without expanding the full artifacts manifest.
