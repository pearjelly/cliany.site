# v0.16.18 发布草案

**目标版本：** `0.16.18`
**提交范围：** `v0.16.17..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本让候选 issue metadata 自带机器可读复现上下文。

## 用户价值

`v0.16.17` 让 `create-issues.sh` 在 publication preflight 失败时直接打印 JSON blocker，维护者能更快理解为什么候选 issue 创建被拦住。但 `issue-metadata.json` 仍主要描述 issue title、labels、body file 和 create command；如果维护者想批量审阅 target URL、candidate commands 或 offline validation commands，还需要打开每个 Markdown body 或重新解析计划 JSON。

`v0.16.18` 把复现上下文写进 `issue-metadata.json`。每个 candidate 条目新增 `target_url`、`commands` 和 `offline_commands`，让维护者、脚本或后续 GitHub issue 创建工具能直接从 metadata 获取复现入口，而不需要解析 Markdown。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.18 --issues-dir /tmp/cliany-candidate-issues` 写出的 `issue-metadata.json` 中，每个 candidate 条目都包含 target URL、candidate commands 和 offline validation commands；artifacts `README.md` 也说明 metadata 包含 reproduction context。

## 变更分组

### Candidate issue metadata reproduction context

- `scripts/plan_next_iteration.py --issues-dir` 的 `issue-metadata.json` 新增 `target_url`。
- `issue-metadata.json` 新增 `commands`，保留 candidate 的推荐探索/回放命令。
- `issue-metadata.json` 新增 `offline_commands`，保留本地验证命令。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 candidate issue metadata 会保留 target URL、candidate commands 和 offline validation commands。
- `tests/test_plan_next_iteration.py` 固定 metadata JSON 和 artifacts README 输出。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环入口。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例数量和状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，也不把 candidate 标记为 active；它只让候选 issue artifacts 的 metadata 更适合脚本化审阅和后续自动化。

## 风险与兼容性

- `target_url`、`commands` 和 `offline_commands` 是 `issue-metadata.json` 的新增字段；已有字段保持兼容。
- 新字段来自 readiness/case catalog 的本地信息，不访问第三方站点。
- artifacts 继续写入维护者指定的 `--issues-dir`，不会写入 `~/.cliany-site/` 或仓库运行时状态。

```bash
python scripts/plan_next_iteration.py --target-version 0.16.18 --issues-dir /tmp/cliany-candidate-issues --json
python scripts/release_readiness.py --target-version 0.16.18 --json
```

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.18 --issues-dir /tmp/cliany-candidate-issues` 写出带结构化复现上下文的 `issue-metadata.json`。
2. 确认 artifacts `README.md` 说明 `issue-metadata.json` 包含 reproduction context。
3. 运行计划脚本、维护者循环文档和 release draft 相关测试。
4. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.18`。
5. 将 `pyproject.toml` 版本更新为 `0.16.18`。
6. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.18`。
7. 创建 tag：`git tag v0.16.18`。

## Release Notes 摘要

`v0.16.18` adds structured reproduction context to generated candidate issue metadata. Maintainers can now read target URLs, candidate commands, and offline validation commands directly from `issue-metadata.json`.
