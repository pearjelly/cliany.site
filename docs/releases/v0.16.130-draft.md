# v0.16.130 发布草案

**目标版本：** `0.16.130`
**提交范围：** `v0.16.129..HEAD`
**当前阻塞：** 最新本地 release 仍需维护者确认后 push 到远端；本版本把 publication publish script path hash 提升为 plan 级字段。

## 用户价值

`v0.16.129` 已经把 `publication_publish_script_path` 结构化输出，维护者和自动化不再需要从命令字符串中解析 `/tmp/cliany-publish-release.sh`。但如果工具只读取周初 plan JSON 或 Markdown report，而不生成 candidate issue artifacts，就还拿不到该路径的稳定摘要。

`v0.16.130` 新增 `publication_publish_script_path_sha256`。计划 JSON、默认文本输出、Markdown report、candidate issue artifacts `README.md` 的 `Candidate Issue Gate Quick Summary` 和 `artifact-manifest.json` 都会展示这个 hash，让自动化能在最轻量的计划阶段判断发布脚本路径是否漂移。

成功标准是：`python scripts/plan_next_iteration.py --target-version 0.16.130 --issues-dir /tmp/cliany-candidate-issues --json` 生成的 plan JSON 包含 `publication_publish_script_path_sha256`，artifacts README 的 `Candidate Issue Gate Quick Summary` 展示同一字段，`artifact-manifest.json` 同时保留 `publication_publish_script_path` 和 `publication_publish_script_path_sha256`。

## 变更分组

### Publication publish script path hash

- `IterationPlan` 新增 `publication_publish_script_path_sha256`。
- 该 hash 使用现有稳定 JSON SHA-256 规则计算 `/tmp/cliany-publish-release.sh`。
- 计划 JSON、默认文本输出和 Markdown report 展示 `publication_publish_script_path_sha256`。
- Candidate issue artifacts `README.md` 的 `Candidate Issue Gate Quick Summary` 展示 `publication_publish_script_path_sha256`。
- `artifact-manifest.json` 顶层保留 `publication_publish_script_path_sha256`。
- 现有 `publication_publish_script_path`、`publication_publish_script_command` 和 `artifact_bundle_summary.publication_publish_script_path_sha256` 保持兼容。

### 文档与门禁

- [docs/weekly-maintainer-loop.md](../weekly-maintainer-loop.md) 说明 plan 级别会展示 publication publish script path hash。
- `tests/test_plan_next_iteration.py` 覆盖 plan dict、Markdown report、artifacts README 和 manifest 的路径 hash。
- `tests/test_weekly_maintainer_loop_docs.py` 固定维护者循环说明。
- `tests/test_release_draft_docs.py` 固定本草案。

## 案例库映射

本版本不改变案例库状态，继续依赖 [cases/README.md](../../cases/README.md) 和 [cases/manifest.json](../../cases/manifest.json)。`pypi-project-search`、`npm-package-search` 和 `crates-io-crate-search` 仍是 candidate cases；`search-extraction-gap` 继续作为抽取质量 known-gap。

本版本不创建真实 GitHub issue，不创建真实 GitHub Release，不推送 tag，也不把 candidate 标记为 active；它只改善周初计划和候选 issue artifacts 的发布脚本路径漂移检测能力。

## 风险与兼容性

- 新增字段只派生自既有 `publication_publish_script_path`。
- 现有 `publication_publish_script_path`、`publication_publish_script_command`、`Publication Publish Script` 和 publish commands 保持兼容。
- 新增 hash 只用于漂移检测，不替代脚本内容审阅。
- 不访问 GitHub，不创建远端 tag，不访问第三方业务站点，不使用真实 LLM key。

## 发版前验证

发版前目标版本自检会检查这段锚点：**提交范围：** `v0.16.130..HEAD`。最终带 `--release-tag v0.16.130` 的 strict gate 仍以本草案顶部的 `v0.16.129..HEAD` 作为实际 release diff 范围。

## 发版步骤

1. 确认 `python scripts/plan_next_iteration.py --target-version 0.16.130 --issues-dir /tmp/cliany-candidate-issues --json` 生成 artifacts。
2. 确认 plan JSON 包含 `publication_publish_script_path_sha256`。
3. 确认 artifacts `README.md` 包含 `Candidate Issue Gate Quick Summary`。
4. 确认该段落展示 `publication_publish_script_path_sha256`。
5. 确认 `artifact-manifest.json` 顶层包含 `publication_publish_script_path_sha256`。
6. 运行计划器、维护者循环文档和 release draft 相关测试。
7. 将 `CHANGELOG.md` 的 Unreleased 移动到 `0.16.130`。
8. 将 `pyproject.toml` 版本更新为 `0.16.130`。
9. 运行 `python scripts/release_readiness.py --strict --target-version 0.16.130`。
10. 创建 tag：`git tag v0.16.130`。

## Release Notes 摘要

`v0.16.130` adds a plan-level publication publish script path hash, helping maintainers and automation detect `/tmp/cliany-publish-release.sh` path drift before generating full candidate issue artifacts.
