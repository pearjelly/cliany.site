# v0.16.256 发布草案

**目标版本：** `0.16.256`
**提交范围：** `v0.16.255..HEAD`
**当前状态：** 本版继续收窄 `pypi-project-search` candidate 晋级交接，把首要任务对应的 issue template 与 evidence bundle 命令提升为 planner 顶层别名。

## 用户价值

`v0.16.255` 已经把首要 candidate 的 adapter 包名、验收标准和优先级变成 planner 顶层字段。维护者和自动化现在能知道当前应该处理 `pypi-project-search/adapter_package`，但如果要生成 GitHub issue body 或 evidence bundle，仍需要从 `candidate_promotions[]` 里找到同一个 case，再读取 `issue_template_command` 或 `evidence_bundle_command`。

`v0.16.256` 让这一步更直接：planner JSON、纯文本输出、Markdown report、candidate issue artifact summary 和 artifact README 会直接输出 `case_promotion_evidence_primary_issue_template_command`、`case_promotion_evidence_primary_issue_template_json_command`、`case_promotion_evidence_primary_evidence_bundle_command` 和 `case_promotion_evidence_primary_evidence_bundle_json_command`。

这样只读 summary 的维护工具可以从首要 adapter_package 任务直接跳到可复制 issue body 或 evidence bundle，不必重新实现 primary case 查找逻辑，也不必从 Markdown 表格里推导命令。

## 变更范围

- `scripts/plan_next_iteration.py`：新增 primary candidate promotion lookup，并把首要 issue template / evidence bundle 命令输出到 JSON、文本、Markdown、artifact bundle summary 和 artifact README。
- `tests/test_plan_next_iteration.py`：用 red/green 测试锁定 planner JSON 与 issue artifact README 中的新增 primary command aliases。
- `CHANGELOG.md`：记录 planner handoff 增强。

## 风险与兼容性

- 不改变 CLI runtime、adapter schema、AXTree selector、session 存储、adapter codegen 或真实 browser workflow 执行。
- 新字段只读追加；已有 `candidate_promotions[].issue_template_command`、`candidate_promotions[].evidence_bundle_command` 和 primary task 字段保持兼容。
- 新 helper 只根据 `case_promotion_evidence_summary.primary_next_task.case_id` 查找已经排序好的 candidate promotion；如果没有 primary task 或 case id，别名会输出 `null` / `(none)`，不会影响现有 planner 输出。

## 案例库映射

- 本次不修改 [cases/manifest.json](../../cases/manifest.json) 的案例状态；案例说明仍落在 [cases/README.md](../../cases/README.md)。
- 当前方向仍是把 candidate 案例推向真实可用：`pypi-project-search` 的首要任务保持 `adapter_package`。
- Primary issue template handoff 为 `cliany-site cases --case-id pypi-project-search --issue-template`。
- Primary evidence bundle handoff 为 `cliany-site cases --case-id pypi-project-search --evidence-bundle --json`。
- Known gap 保持 1 个：`search-extraction-gap`。

## 发版步骤

1. 确认 `v0.16.255` 的 GitHub Release、PyPI 和官网发布状态仍可审计。
2. 运行 target-mode readiness，确认 draft、daily cap、publication visibility 和 worktree 状态都允许进入发布。
3. 完成 version bump、CHANGELOG、官网基线和 release notes 摘要。
4. 提交并推送 `master`。
5. 创建并推送 `v0.16.256` tag。
6. 等待 `.github/workflows/release.yml` 更新 GitHub Release 和 PyPI。
7. 按 `AGENTS.md` 的 Vercel 步骤从 `site/` 部署官网，并运行 `cd site && vercel inspect www.cliany.site --wait --timeout 90s` 复核 alias。
8. 运行 `uv run --extra dev --frozen python scripts/check_release_publication.py --remote --distribution --json`，确认 GitHub Release、PyPI 和官网发布状态。

## Release Notes 摘要

v0.16.256 makes primary candidate handoffs easier to execute from planner summaries. The next-iteration planner now exposes the selected candidate's issue-template and evidence-bundle commands as top-level primary aliases across JSON, text/Markdown reports, artifact bundle summaries, and artifact README output. Automation can now jump from `pypi-project-search/adapter_package` directly to reproducible issue and evidence commands without searching `candidate_promotions[]`.
