### 市场定位差异化

cliany-site 在行业中的独特位置：**唯一将「网站交互」持久化为「可复用 CLI 适配器」的工具**。

竞品对比：
- **Browser Use / Stagehand / agent-browser**：侧重一次性 AI 代理交互，无持久化命令生成
- **Playwright AI**：增强 Playwright 的 AI 辅助，但输出是测试脚本而非 CLI 命令
- **Firecrawl**：侧重抓取，/interact 功能接近但无 CLI 命令层

**核心护城河：** AXTree 语义理解 + 模糊匹配自愈 + 按域名组织的 CLI 适配器体系

## 二、六大迭代方向

### 方向 A：多模态感知 — 「看懂」而不仅仅是「读懂」

**问题：** 当前仅依赖 AXTree（无障碍树）理解页面。但很多网站的 AXTree 质量差（缺 label、role 错误），导致 LLM 理解不准确。

**方案：**
1. **截图 + Vision LLM 双通道感知**：explore 每步同时截取页面截图，用 GPT-4o/Claude Vision 进行视觉理解，与 AXTree 信息交叉验证
2. **视觉标注叠加**：在截图上标注可交互元素的序号（类似 SoM — Set-of-Mark），LLM 可直接说"点击标注 [7] 的按钮"
3. **OCR 兜底**：AXTree 无 label 时，用 OCR 从截图中提取按钮/链接文字
4. **页面布局理解**：将截图区域映射到 DOM 坐标，支持"点击页面右上角的头像"这种空间描述

**预期收益：** 探索成功率从 ~70% 提升到 ~90%+，大幅减少"找不到元素"的失败

**复杂度：** 高（需引入 Vision API、截图管道、坐标映射）  
**依赖：** 无前置依赖，可独立推进

---

### 方向 B：会话式探索 — 从「一句话定义」到「对话式协作」

**问题：** 当前 explore 是"一锤子买卖"——用户给一句话描述，LLM 全自动执行。遇到歧义（如搜索结果有多个）或需要决策（如选哪个选项），LLM 只能猜。

**方案：**
1. **交互式 explore 模式** (`--interactive`)：
   - LLM 每规划一步后暂停，展示当前页面状态 + 计划动作
   - 用户可确认 / 修改 / 跳过 / 插入自定义动作
   - 支持"回退到上一步"
2. **多轮对话探索**：
   - 首次 explore 生成基础适配器
   - 后续可对话式追加："再加一个导出 CSV 的功能"、"把搜索改成支持分页"
   - 增量更新适配器而非重新生成
3. **探索录像回放**：
   - 每步截图 + AXTree 快照存档
   - 用户可事后回看探索过程，标记需要修改的步骤
   - `cliany-site replay <domain> <command>` 回放录像

**预期收益：** 适配器质量和用户满意度大幅提升；降低"探索完发现不对要重来"的挫败感

**复杂度：** 中高  
**依赖：** 方向 A 的截图能力可增强体验，但非必须

---

### 方向 C：适配器生态进化 — 从「本地工具」到「社区平台」

**问题：** 适配器市场当前仅支持本地打包/安装。用户无法发现和共享其他人创建的高质量适配器。

**方案：**
1. **远程 Registry**：
   - 搭建 PyPI 风格的适配器索引（可基于 GitHub Releases 或独立服务）
   - `cliany-site market search github.com` — 搜索可用适配器
   - `cliany-site market install github.com/search@2.0` — 从远程安装
   - 支持 `cliany-site market publish --registry https://registry.cliany.site`
2. **适配器模板 & 脚手架**：
   - 预制常见网站（GitHub、Google、Twitter、淘宝等）的高质量适配器模板
   - `cliany-site init github.com` — 基于模板快速生成适配器骨架
3. **适配器组合**：
   - 允许一个适配器依赖另一个适配器的命令
   - 示例：`github-ci` 适配器依赖 `github.com` 的 login 和 navigate 命令
4. **社区评分与排行**：
   - 适配器健康度自动评分（自愈成功率、最后验证时间）
   - 用户星标和使用统计

**预期收益：** 构建网络效应；每新增一个高质量适配器都增加平台价值

**复杂度：** 高（需 Registry 后端、Auth、CI 自动验证）  
**依赖：** 当前市场基础设施可复用

---

### 方向 D：智能执行引擎 — 从「录制回放」到「自适应执行」

**问题：** 当前执行引擎本质上是"录制回放"——按固定步骤序列执行。网站 UI 一变就可能卡住。虽有自愈机制，但能力有限。

**方案：**
1. **目标驱动执行**：
   - 不再按固定步骤执行，而是记录每步的"意图"（如"找到搜索框并输入"）
   - 执行时 LLM 实时理解当前页面，动态规划达成意图的最佳路径
   - 退化为经典模式作为 fallback（当页面未变时直接用录制路径，更快）
2. **执行策略分层**：
   - **L0 精确匹配**：selector 完全命中 → 直接执行（最快）
   - **L1 模糊匹配**：selector 未命中但 AXTree 中有相似元素 → 自动修复（当前能力）
   - **L2 视觉定位**：AXTree 也找不到 → 截图 + Vision LLM 定位元素（新增）
   - **L3 意图重规划**：元素确实不存在 → LLM 重新规划实现同一意图的路径（新增）
3. **执行学习**：
   - 记录每次执行的修复历史
   - 高频修复模式自动固化为新 selector 候选
   - 跨用户聚合修复知识（需 Registry）

**预期收益：** 从"页面一改就废"进化到"真正可靠的自动化"；这是 cliany-site 的终极价值

**复杂度：** 极高  
**依赖：** 方向 A（多模态感知）是 L2 层的前置条件

---

### 方向 E：开发者平台化 — 从「CLI 工具」到「自动化基础设施」

**问题：** 当前虽有 SDK 和 HTTP API，但缺乏将 cliany-site 嵌入更大系统的深度集成能力。

**方案：**
1. **Webhook & 事件总线**：
   - 适配器执行完成后触发 Webhook 通知
   - 支持 `on_success` / `on_failure` 回调配置
   - 可集成到 Slack、飞书、钉钉等通知渠道
2. **MCP (Model Context Protocol) Server**：
   - 将 cliany-site 暴露为 MCP Server
   - 任何支持 MCP 的 AI 代理（Claude Desktop、Cursor、OpenCode 等）可直接调用
   - `explore`、`execute`、`list` 等作为 MCP Tools
   - 适配器命令动态注册为 MCP Resources
3. **定时任务 & 监控**：
   - `cliany-site schedule add "github.com check-stars --repo X" --cron "0 9 * * *"`
   - 定时执行 + 结果变更检测 + 告警
   - 典型场景：监控竞品页面变化、定时抓取数据
4. **Pipe & Compose 原语**：
   - `cliany-site github.com search --query X | cliany-site github.com star`
   - Unix 管道风格组合命令
   - JSON 输出自然支持 `jq` 管道

**预期收益：** 从"手动触发的工具"变成"可编排的自动化节点"；打开 B2B 和平台化的想象空间

**复杂度：** 中（MCP Server 可快速实现）到高（定时任务需持久化调度）  
**依赖：** 当前 SDK/API 基础设施可复用

---

### 方向 F：性能与规模 — 从「单任务」到「高吞吐」

**问题：** 当前所有操作串行执行，LLM 调用是瓶颈，不适合大规模自动化场景。

**方案：**
1. **本地小模型加速**：
   - 简单动作（已知按钮点击、表单填写）使用本地 LLM（如 Ollama + Qwen/Llama）
   - 复杂决策（歧义消解、多路径选择）才调用云端大模型
   - 预计降低 60%+ API 成本，提速 3-5x
2. **AXTree 智能裁剪**：
   - 只采集视口内 + 交互元素的子树，减少 token 消耗
   - 增量 diff：页面变化时只传递变化部分
   - 预计减少 50%+ token 消耗
3. **多 Tab 并行**：
   - 独立任务在不同 Tab 中并行执行
   - 共享 Session 但独立 DOM 上下文
   - 批量执行从串行变并行，吞吐量提升 N 倍
4. **流式执行**：
   - LLM 响应边生成边执行，不等完整回复
   - explore 和 execute 流水线化

**预期收益：** 适合企业级大规模自动化；降低 LLM 成本

**复杂度：** 中  
**依赖：** 无强依赖

## 三、优先级推荐

基于 **用户价值 × 差异化壁垒 × 实现可行性** 三维评估：

```
                高用户价值
                    │
         ┌──────────┼──────────┐
         │    D     │    B     │
         │  智能执行 │  会话探索  │
         │          │          │
  高壁垒 ├──────────┼──────────┤ 低壁垒
         │    A     │    E     │
         │  多模态   │  平台化   │
         │          │          │
         └──────────┼──────────┘
                    │
                低用户价值
```

### 推荐路线图

| 阶段 | 方向 | 版本 | 周期 | 核心交付 |
|------|------|------|------|----------|
| **v0.7 — 看得见** | A 多模态感知 | v0.7.0 | 2-3 周 | 截图 + Vision 双通道；SoM 标注；探索成功率指标 |
| **v0.8 — 聊得通 ✅** | B 会话式探索 | v0.8.0 | 2-3 周 | `--interactive` 模式；多轮增量更新；探索录像 |
| **v0.9 — 连得上** | E.2 MCP Server | v0.9.0 | 1-2 周 | MCP Tools/Resources 暴露；Claude Desktop 集成验证 |
| **v1.0 — 可靠执行** | D 智能执行引擎 | v1.0.0 | 3-4 周 | 意图驱动执行；L0-L3 分层策略；执行学习 |

### 关键里程碑

- **v0.7：** 可以对外 demo "AI 看着页面操作"的能力，有视觉冲击力
- **v0.8：** 用户真正用起来不再觉得"黑盒"，可控性质变
- **v0.9：** 接入 AI 代理生态，被 Claude/Cursor 用户发现
- **v1.0：** "可靠性"成为核心卖点，可以认真推广

## 五、风险与约束

| 风险 | 影响 | 缓解 |
|------|------|------|
| Vision API 成本高 | A 方向增加 API 费用 | 仅在 AXTree 不足时触发 Vision；本地 OCR 兜底 |
| 交互式模式增加延迟 | B 方向用户等待 | 非交互模式保留为默认；交互模式是 opt-in |
| MCP 协议还在演进 | E.2 需跟随上游 | 做薄适配层，核心逻辑不依赖 MCP 细节 |
| 本地小模型质量不够 | F.1 简单任务也可能出错 | 大模型兜底；逐步放宽小模型处理的任务范围 |
| 竞品快速跟进 | 差异化窗口收窄 | 加速 v0.7-v0.9，抢先建立用户心智和适配器生态壁垒 |
