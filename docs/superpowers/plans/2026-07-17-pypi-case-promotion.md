# PyPI Case Promotion Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Promote the verified public PyPI search workflow into an installable, digest-pinned active case in v0.16.265.

**Architecture:** The explorer creates a runtime adapter only after the live doctor gate succeeds. The marketplace packages and validates it, then the case catalog and public docs reference the same GitHub Release asset after the normal tag workflow publishes the release. Runtime artifacts never enter the repository.

**Tech Stack:** Python 3.11+, Click, Chrome CDP, OpenAI-compatible LLM, existing case validator, GitHub CLI/Actions, PyPI trusted publishing, Vercel.

## Global Constraints

- Keep adapters, sessions, snapshots, and packages under `~/.cliany-site/`; do not add them to Git.
- Use AXTree semantic matching only; do not introduce CSS selector fallbacks.
- Promote PyPI only after live preflight, archive validation, and online smoke all pass.
- Attach the exact archive generated by `market publish` to GitHub Release `v0.16.265`; users install it by HTTPS URL plus a 64-character SHA-256.
- Keep npm and crates.io as candidates, and keep real LLM credentials out of tests and PR workflows.
- Publish no more than one v0.16.265 tag and no more than three tags total on 2026-07-17.
