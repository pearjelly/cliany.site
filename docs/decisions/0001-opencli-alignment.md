# ADR 0001: OpenCLI 心智模型对齐与 0.9.0 架构演进

**状态**: 已接受

**日期**: 2026-04-26

## 1. browser 原子命令组

**背景**: 目前的适配器命令往往是复合型的，难以在不同站点间复用底层浏览器控制逻辑。为了提升 AI Agent 的可操作性，需要一套标准化的低层级指令。

**决策**: 引入 9 个原子命令：`state`, `navigate`, `find`, `click`, `type`, `extract`, `wait`, `screenshot`, `eval`。这些命令直接映射到 CDP 操作，但不包含业务逻辑。

**备选方案**: 继续使用高度封装的 `Workflow` 模式。放弃原因：封装过深导致 Agent 无法在遇到未知 UI 变化时进行微操。

**后果**: 
- 正面：Agent 可以通过组合这些原子命令实现任意复杂的交互，显著提升了系统的灵活性。
- 负面/权衡：增加了 CLI 的表面积，用户直接使用这些原子命令的门槛较高，主要面向机器/Agent。

## 2. 运行时 registry 动态汇聚

**背景**: 系统中存在内置命令、原子命令以及用户生成的适配器命令。目前缺乏统一的发现机制，导致 Agent 难以获知当前环境可用的所有功能。

**决策**: 实现运行时 registry，按照 `builtin > atom > adapter` 的优先级进行动态汇聚。如果发生命名冲突，使用 `qualified_name`（如 `github.com:search`）进行区分。

**备选方案**: 静态硬编码命令列表。放弃原因：无法支持动态加载的第三方适配器。

**后果**: 
- 正面：提供了一致的命令查询接口，`cliany-site --explain` 可以输出完整的系统契约。
- 负面/权衡：动态加载增加了启动时的 I/O 开销，需要精细化缓存。

## 3. verify 双模式

**背景**: 适配器生成后需要验证其可用性。现有的验证方式不够系统，且难以兼顾效率与真实性。

**决策**: 提供 `verify` 双模式。默认使用静态验证（jsonschema 校验 metadata + AST 扫描 `commands.py`）；使用 `--smoke` 标志开启冒烟验证，连接真实 CDP 实例执行动作。

**备选方案**: 仅保留冒烟测试。放弃原因：冒烟测试耗时长且依赖环境，不适合集成到所有开发环节。

**后果**: 
- 正面：实现了「快慢结合」的质量保障体系，既能快速拦截低级错误，又能确保端到端闭环。
- 负面/权衡：维护两套验证逻辑增加了代码复杂度。

## 4. AGENT.md 自动重写

**背景**: AI Agent 需要上下文才能更好地工作。目前的上下文分散在各个文档中，Agent 很难一次性获取全貌。

**决策**: 引入项目级 `AGENT.md`。每次 `explore` 完成后自动触发重写。使用 `sentinel` 标记和 `hash` 校验实现双重保护，防止用户手动修改的内容被意外覆盖。

**备选方案**: 手动维护 `AGENT.md`。放弃原因：适配器更新频繁，手动维护极易滞后。

**后果**: 
- 正面：Agent 始终拥有最新的项目快照，包括已有的适配器能力和环境状态。
- 负面/权衡：频繁的文件重写可能在某些文件监控工具中触发不必要的通知。

## 5. 零 LLM 执行原则

**背景**: 为了降低使用成本和提升响应速度，必须区分「探索」和「执行」两个阶段。

**决策**: 运行时（执行适配器命令时）默认遵循「零 LLM」原则，完全依赖生成的代码和 fuzzy 匹配逻辑。仅在显式开启 `--heal` 时才允许调用 LLM。

**备选方案**: 每次执行都让 LLM 参与规划。放弃原因：Token 消耗过高，且由于 LLM 的不稳定性可能导致执行结果不可预测。

**后果**: 
- 正面：显著降低了运行成本，执行延迟从秒级降至毫秒级。
- 负面/权衡：对生成的 selector 质量和 fuzzy 匹配算法提出了更高要求。

## 6. --heal 旁路写入 + accept-heal 显式合并

**背景**: 当自愈（Self-healing）发生时，如何安全地更新适配器 metadata 是一个难题。直接覆盖可能引入错误。

**决策**: 自愈过程采取旁路写入策略，生成 `metadata.healed.json`。用户必须通过 `accept-heal` 命令显式确认并合并这些更改。

**备选方案**: 自动合并自愈结果。放弃原因：风险较大，一旦自愈逻辑出错，会损坏原始适配器。

**后果**: 
- 正面：确保了适配器的稳定性，所有的自动修复都是可审计、可回滚的。
- 负面/权衡：增加了一个人工介入的步骤，降低了自动化闭环的流畅度。

## 7. metadata schema_version=2 硬切换

**背景**: 0.9.0 引入了大量新字段（如 smoke/heal_history），旧版本的 schema 已经无法承载。

**决策**: 进行硬切换，将 `schema_version` 提升至 2。系统拒绝加载所有 v1 适配器，并引导用户重新执行 `explore`。

**备选方案**: 实现 v1 到 v2 的复杂迁移逻辑。放弃原因：0.9.0 的心智模型发生了根本变化，旧的探索数据已不足以支撑新的原子化要求，迁移成本远高于重新探索。

**后果**: 
- 正面：代码库无需背负沉重的历史包袱，可以轻装阵前。
- 负面/权衡：破坏了向后兼容性，用户需要重新生成适配器。

## 8. 统一 JSON envelope

**背景**: 目前的 JSON 输出结构不统一，导致上层工具（如 workflow 引擎）解析成本高。

**决策**: 统一所有的 JSON 输出信封格式：`{ok, version, command, data, error, meta}`。其中 `ok` 替代 `success`，`version` 固定为 "1"，`meta` 包含 `source` 和 `duration_ms`。

**备选方案**: 保留现有的 `{success, data, error}` 结构。放弃原因：缺乏版本控制和足够的元数据支持。

**后果**: 
- 正面：增强了机器可读性，为后续的版本演进打下了基础。
- 负面/权衡：现有的脚本和下游依赖需要同步更新字段名。

## 9. QA 双路径

**背景**: 在线测试依赖网络和真实的第三方站点（如 GitHub），容易因外部因素失效；离线测试则缺乏真实性。

**决策**: 建立 QA 双路径。在线路径定期抓取真实站点；离线路径使用本地 fixture（`127.0.0.1:18080`）。通过环境变量 `CLIANY_QA_OFFLINE=1` 强制切换。

**备选方案**: 仅依赖 Mock。放弃原因：Mock 无法真实模拟复杂的 CDP 交互和 AXTree 变化。

**后果**: 
- 正面：既保证了本地开发的反馈速度，又能在 CI 环节进行真实的端到端校验。
- 负面/权衡：需要维护两套测试数据（真实 URL vs 本地 fixture）。
