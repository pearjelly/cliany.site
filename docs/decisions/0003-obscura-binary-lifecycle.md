# ADR 0003: Obscura 二进制生命周期管理策略

**状态**: 已接受

**日期**: 2026-05-11

## 背景

cliany-site 计划引入 Obscura 作为可选的浏览器 provider（Task 系列）。Obscura 是一个独立的可执行二进制文件，需要明确其版本缓存目录布局、active version 指针维护、版本回滚机制、离线行为以及自动升级策略。

## 决策

### 1. 版本缓存目录布局

所有 Obscura 二进制版本缓存在 `~/.cliany-site/bin/obscura/` 下，按版本号隔离存放：

```
~/.cliany-site/bin/obscura/
├── active           # 文本文件，内容为当前激活的版本号，如 "0.1.2"
├── 0.1.2/
│   ├── obscura      # macOS/Linux 可执行文件
│   ├── obscura.exe  # Windows 可执行文件
│   └── .ready       # 安装完成标记文件
└── 0.1.3/
    ├── obscura
    └── .ready
```

**关键约定**：
- `.ready` 文件标志该版本安装完整；缺失时视为损坏，不允许使用
- 原子下载：先写入 `.tmp` 临时文件，rename 完成后再创建 `.ready`
- 安装操作幂等：重复安装同一版本覆盖现有文件，不改变 active 指针

### 2. Active Version 指针

`active` 是纯文本文件，内容为当前激活版本字符串（如 `0.1.2`）。

- 读取：调用 `get_active_version(cache_dir)` 函数，文件不存在或内容为空时返回 `None`
- 写入：仅通过 `use_version(state, version)` 修改，确保目标版本已安装且 `.ready` 存在
- 启动时若 `active` 指向的版本目录已损坏（`.ready` 缺失），报结构化错误，不静默回退

### 3. 版本回滚机制

`rollback_version(state)` 按安装时间升序排序已安装版本，切换至当前 active 版本的前一个版本。

**约束**：
- 至少需要 2 个已安装版本才可回滚
- 已是最旧版本时拒绝回滚并抛出明确错误
- 回滚操作等同于调用 `use_version`，同样写入 `active` 文件

典型序列：`install 0.1.2` → `install 0.1.3` → `use 0.1.3` → `rollback` → active 回到 `0.1.2`

### 4. 自动升级策略（默认 opt-out）

`BinaryConfig.auto_upgrade` 默认为 `False`。用户必须显式设置才能开启自动升级。

**理由**：
- 自动升级可能破坏已验证的工作流
- 与 `BinaryConfig.pinned_version` 语义冲突：若同时指定固定版本又开启自动升级，`validate_pinned_version` 抛出错误
- 离线场景下自动升级会导致不可预知的失败

**不追踪 `latest`**：默认安装策略要求显式指定版本号，不允许将 `latest` 作为版本标识符。

### 5. 离线行为

当网络不可达时：
- 若 `active` 版本存在且 `.ready` 完整，直接使用本地缓存，不抛出网络错误
- 若无可用本地版本，结构化报错提示用户先在有网络环境下安装
- 不允许自动降级至低版本作为离线 fallback

### 6. 下载完整性校验边界

由于上游无官方 `.sha256` 文件，采用以下策略：
- 强制使用 HTTPS 下载（不接受 HTTP 重定向至 HTTPS 降级为 HTTP）
- 支持用户通过 `BinaryConfig` 提供可选的预期哈希值（实现在 Task 6-8）
- 校验失败时删除 `.tmp` 文件，不创建 `.ready`

## 备选方案

**备选 A：将版本号写入 metadata.json 而非 active 文件**
放弃原因：文本文件更简单，不依赖 JSON 解析，且与 POSIX 工具（`cat`）天然兼容。

**备选 B：默认追踪 latest 并在锁定文件中记录**
放弃原因：`latest` 在无网络时无法解析，破坏离线可用性；且与项目「零 LLM 执行」原则类似，执行时不应引入外部依赖。

**备选 C：自动升级默认开启，用户可 opt-out**
放弃原因：意外升级破坏已验证工作流的风险高于手动升级的麻烦。

## 后果

**正面**：
- 目录布局直观，可用 `ls` 手工检查状态
- 版本隔离彻底，多版本共存不干扰
- 回滚操作可预测，无需网络

**负面/权衡**：
- 需要手动管理版本清理（旧版本不自动删除，防止影响回滚候选集）
- 离线安装需要用户预先下载并放置二进制文件
